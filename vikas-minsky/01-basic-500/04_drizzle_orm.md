## 4. Drizzle ORM (121–140)

121. What is Drizzle ORM?
     Drizzle ORM is a lightweight, type-safe SQL ORM for TypeScript that embraces SQL syntax rather than hiding it. It provides a query builder API that maps closely to SQL, with full type inference from database schemas and zero runtime overhead for unused features.

122. Why choose Drizzle over Prisma?
     Drizzle offers a SQL-like query syntax, smaller bundle size, faster runtime performance, and better control over generated queries. Unlike Prisma's proprietary schema language, Drizzle uses plain TypeScript for schema definition and avoids vendor lock-in.

123. Explain schema-first design.
     Drizzle follows a schema-first approach where database schemas are defined in TypeScript using Drizzle's schema syntax. These schemas serve as the single source of truth for type inference, migrations, and query building, ensuring type safety across the stack.

124. How are migrations handled in Drizzle?
     Migrations are generated by comparing the current schema file with the database state using `drizzle-kit`. It produces SQL migration files that can be reviewed and executed, supporting both push (auto-apply) and generate (manual review) workflows.

125. Explain Drizzle relations.
     Relations in Drizzle define associations between tables using the `relations()` helper. They support one-to-one, one-to-many, and many-to-many relationships with type-safe joins and automatic eager/lazy loading capabilities.

126. How does type safety work?
     Drizzle infers TypeScript types directly from the schema definition. Every query returns fully typed results, and the query builder validates column names, operators, and join conditions at compile time, eliminating runtime type errors.

127. Explain query builders in Drizzle.
     Drizzle's query builder provides methods like `select()`, `where()`, `orderBy()`, `groupBy()`, and `having()` that mirror SQL syntax in a type-safe way. Builders compose sequentially and translate to optimized SQL queries for the target dialect.

128. How do transactions work?
     Transactions use the `db.transaction()` callback API, which receives a transaction client for all operations within the transaction. The API supports savepoints, rollback on error, and nested transactions for complex workflows.

129. Explain Drizzle with PostgreSQL.
     Drizzle has database-specific packages like `drizzle-orm/pg-core` with PostgreSQL-specific features: JSONB operators, full-text search vectors, array operations, range types, and custom PostgreSQL extensions like PostGIS.

130. How do you optimize queries in Drizzle?
     Optimize by using `prepare()` for prepared statements, enabling query logging during development with the `logger` option, selecting only needed columns instead of `select()`, adding proper indexes, and using `explain()` to analyze query plans.

131. Explain Drizzle Studio.
     Drizzle Studio is a browser-based GUI for exploring and editing database data. It connects directly to the database, displays table schemas and relations, and provides an inline editor for running queries and viewing results in real time.

132. Difference between SQL-first and ORM-first approaches?
     SQL-first approaches like Drizzle give you direct control over generated SQL with a thin abstraction layer. ORM-first approaches like Prisma abstract SQL entirely behind a high-level API, which simplifies development but can limit optimization and debugging.

133. How does Drizzle handle joins?
     Joins are handled using the `leftJoin()`, `innerJoin()`, `rightJoin()`, and `fullJoin()` methods with type-safe join conditions. Drizzle infers the joined table types and returns nested result objects that match the join structure.

134. Explain raw SQL support.
     Drizzle provides `sql` tagged template literals and `db.execute()` for executing raw SQL queries when needed. Raw queries are still type-safe when combined with Drizzle's type system, and result columns are typed based on the schema.

135. What are Drizzle adapters?
     Adapters bridge Drizzle with database drivers like `node-postgres`, `mysql2`, `better-sqlite3`, or Neon serverless. Each adapter provides a consistent API while handling driver-specific connection pooling, SSL, and configuration.

136. Explain schema validation integration.
     Drizzle integrates with Zod for runtime validation through `drizzle-zod`, which generates Zod schemas from Drizzle tables. This enables automatic validation of incoming data against database constraints before queries execute.

137. How do you seed databases?
     Database seeding uses Drizzle's `db.insert()` with arrays of seed data objects. Seed scripts run independently before tests or development, creating realistic initial data using libraries like `@faker-js/faker` for randomized content.

138. Explain migration conflicts.
     Migration conflicts occur when multiple developers create overlapping or conflicting schema changes. Drizzle's SQL-based migrations help by making conflicts visible during code review, and running migrations sequentially prevents ordering issues.

139. How do you structure large Drizzle projects?
     Large projects organize schemas by domain modules (users, posts, etc.) in separate files, centralize database connection in a `db` module, use migrations folders for versioning, and define reusable query fragments in service files.

140. Explain type inference benefits.
     Type inference eliminates manual type definitions by deriving TypeScript types from the database schema. This reduces boilerplate, prevents mismatch errors between schema and queries, and automatically updates types when the schema changes.
