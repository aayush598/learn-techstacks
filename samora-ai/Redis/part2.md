# Redis Interview Questions and Answers - Part 2

## Q1: How does Redis Cluster handle network partitions and split-brain scenarios?
**A:** Redis Cluster uses a gossip-based failure detection with a node timeout configuration (`cluster-node-timeout`). When a partition occurs: (1) If the partition isolates a minority of nodes (including at least one master), those nodes become unreachable but continue operating. (2) If a master is partitioned from the majority, after `cluster-node-timeout` milliseconds, the majority side marks it as `FAIL` and promotes its replica. (3) The partitioned master, when it reconnects, discovers it is no longer the authoritative master and demotes itself to a replica. (4) Redis Cluster uses a quorum-based approach (majority of masters must agree for failover) to prevent split-brain. (5) The `cluster-require-full-coverage` setting controls whether the cluster accepts writes when some slots are unavailable.

## Q2: How does PSYNC2 improve Redis replication over the original PSYNC?
**A:** PSYNC2 (introduced in Redis 4.0) significantly improved replication resynchronization: (1) It adds a replication ID and offset to the master's state, replacing the old run ID approach. (2) Masters can now have multiple replication IDs -- the current ID and a previous ID during failover. This allows replicas to continue partial resynchronization even after a master restart or failover. (3) PSYNC2 stores the replication state in the RDB file, so a restarting master preserves its replication ID and offset. (4) A replica can perform partial resync with a promoted replica (new master) if it was once connected to the old master. (5) PSYNC2 reduced full resyncs dramatically, making replication much more efficient during failover scenarios.

## Q3: How do you tune Redis AOF rewrite for high-write-throughput workloads?
**A:** AOF rewrite tuning for high-throughput: (1) Set `auto-aof-rewrite-percentage 100` and `auto-aof-rewrite-min-size 64mb` -- rewrite when AOF doubles in size from the last rewrite and exceeds 64MB. (2) Use `no-appendfsync-on-rewrite yes` to prevent fsync during rewrite (reduces disk contention). (3) Schedule BGREWRITEAOF during low-traffic periods. (4) Use SSD/NVMe disks for AOF writes -- HDDs cannot sustain high-throughput AOF logging. (5) Consider `appendfsync everysec` (best balance of durability and performance). (6) For extreme write throughput, evaluate switching AOF off entirely and using only RDB + replication. (7) Monitor `aof_pending_bio_fsync` and `aof_delayed_fsync` in `INFO persistence` to detect fsync bottlenecks.

## Q4: What is the Redis Jemalloc memory allocator and how does it affect memory fragmentation?
**A:** Jemalloc is Redis's default memory allocator, chosen for its excellent performance in multi-threaded workloads and low fragmentation characteristics. Key aspects: (1) Jemalloc uses size classes (arenas) that reduce external fragmentation by grouping allocations of similar sizes. (2) It maintains per-thread caches to reduce lock contention. (3) Redis reports `mem_fragmentation_ratio` = `used_memory_rss / used_memory`. A ratio of 1.0-1.5 is normal. Above 1.5 indicates fragmentation. (4) Causes of fragmentation: creating and freeing keys of varying sizes, using many small keys, high churn. (5) Solutions: restart Redis (last resort), set `maxmemory` to trigger eviction which may consolidate, upgrade to Jemalloc 5+ which has better fragmentation handling. (6) Use `INFO MEMORY` to examine `allocator_active`, `allocator_allocated`, and `allocator_resident` for detailed allocator metrics.

## Q5: How does Redis Active Defragmentation (jemalloc) work?
**A:** Redis 4.0+ includes active defragmentation that runs in the background to reduce memory fragmentation. How it works: (1) Enabled via `ACTIVE DEFRAG THRESHOLD` (default `activedefrag yes` when `jemalloc` is detected). (2) Triggered when `mem_fragmentation_ratio` exceeds `active-defrag-threshold-lower` (default 1.1). (3) Redis allocates new memory, copies data from fragmented areas, and frees the old fragmented memory. (4) Controlled by `active-defrag-cycle-min` and `active-defrag-cycle-max` (CPU usage limits as a percentage). (5) `active-defrag-max-scan-fields` limits the number of fields scanned per hash/set/zset to avoid latency spikes. (6) Works at the Jemalloc page level -- relocates allocations from sparsely populated pages to densely populate them, allowing pages to be returned to the OS. (7) Monitor effectiveness via `INFO MEMORY` fields prefixed with `allocator_`.

## Q6: How do you configure Redis Cluster for multi-region active-passive replication?
**A:** Multi-region Redis Cluster setup: (1) Deploy master nodes in the primary region and replica nodes across secondary regions. (2) Each replica is assigned to follow a master from the primary region. (3) Configure `cluster-replica-no-failover yes` on replicas in secondary regions to prevent automatic promotion (you want failover only in the primary region). (4) Use `replica-priority 0` on cross-region replicas to ensure they are never promoted by Sentinel/Cluster. (5) Monitor cross-region replication lag via `INFO replication` `master_repl_offset` vs `slave_repl_offset`. (6) For disaster recovery, manually promote a secondary region replica by issuing `CLUSTER FAILOVER` (or `REPLICAOF NO ONE`). (7) Application uses DNS-based routing to point to the active region's cluster endpoints.

## Q7: How do you implement exponential backoff for Redis reconnection in high-availability setups?
**A:** Exponential backoff for Redis reconnection: (1) Start with a base delay (100ms). (2) Multiply by a factor (2x) on each retry. (3) Add jitter (+/- 50% random) to prevent thundering herd when multiple clients reconnect simultaneously. (4) Cap the maximum delay (e.g., 30 seconds). (5) Reset to base delay on successful connection. (6) Implementation pseudocode: `delay = min(base * 2^attempts + random(0, jitter), max_delay)`. (7) Most Redis client libraries (ioredis, redis-py, Jedis) support this natively via `retryStrategy`. (8) Important: during the backoff period, queue operations or fail fast rather than blocking the application thread.

## Q8: How does Redis Cluster's resharding work internally during slot migration?
**A:** Resharding moves hash slots between nodes without cluster downtime: (1) The cluster controller (redis-cli) sends `CLUSTER SETSLOT <slot> MIGRATING <source_id>` to the source node. (2) Sends `CLUSTER SETSLOT <slot> IMPORTING <target_id>` to the target node. (3) Keys are migrated using `MIGRATE` commands: source dumps the key, target restores it, source deletes the key. (4) During migration, if a client requests a key on the source node, the source checks if the key is already migrated. If so, it returns an `-ASK` redirect to the client pointing to the target. (5) The client follows the `-ASK` redirect and sends the command to the target with `ASKING` flag. (6) After all keys in the slot are migrated, both nodes are updated: `CLUSTER SETSLOT <slot> NODE <target_id>` to all nodes. (7) Multiple slots can be migrated in parallel. Progress can be monitored with `CLUSTER GETKEYSINSLOT`.

## Q9: How do you design a Redis caching strategy to handle cache stampede without distributed locks?
**A:** Cache stampede prevention without locks: (1) **Probabilistic early expiration** -- before the TTL expires, randomly refresh the cache early. Use a jitter factor: if the key was written at `created_at` and TTL is `ttl`, compute a random early refresh window in the last `ttl * 0.1` seconds. (2) **Xfetch algorithm** -- each cache request computes a random early refresh probability: `p = (now - created_at) / (ttl * 2)`. When `p > random(0,1)`, the cache is refreshed. (3) **Stale-while-revalidate** -- serve stale data while asynchronously refreshing the cache. The first request to detect expiry triggers a background refresh. (4) **Hedged requests** -- send multiple refresh requests and use the first one that returns, canceling the others. (5) These approaches avoid the synchronization overhead of distributed locks while still preventing stampedes.

## Q10: How do you implement a sliding window rate limiter using Redis sorted sets?
**A:** Sliding window rate limiter with sorted sets: (1) Key: `rate_limit:{user_id}:{action}`. (2) Each request adds a member (unique ID or timestamp) with score = current timestamp in milliseconds: `ZADD key timestamp member`. (3) Remove old entries outside the window: `ZREMRANGEBYSCORE key -inf (now - window_ms`. (4) Count entries in the window: `ZCARD key`. (5) If count > limit, reject the request. (6) Set a TTL on the key equal to the window duration for automatic cleanup. (7) For efficiency, only keep 1-2% sample of entries for high-rate limits. (8) Alternative: use a fixed window counter with `INCR` and `EXPIRE` for simpler but less precise rate limiting. The sorted set approach provides exact sliding window counting at the cost of more memory.

## Q11: How does the Redis `XCLAIM` command work for consumer failure handling in streams?
**A:** `XCLAIM` transfers ownership of pending stream messages from one consumer to another: (1) When consumer A fails, its pending messages remain unacknowledged in the stream. (2) Consumer B claims those messages: `XCLAIM mystream groupA consumerB 3600000 1617000000000-0` (3rd arg is min-idle-time in ms -- only claim messages idle longer than this). (4) The claimed messages are delivered to consumer B. (5) `XCLAIM` also updates the message's delivery counter (tracked in the Pel list). (6) `XAUTOCLAIM` (Redis 6.2+) automates this -- it scans pending entries and claims them without specifying individual IDs. (7) The idle time threshold prevents claiming messages that are currently being processed. (8) Use `XPENDING` to inspect pending messages and their idle times.

## Q12: How do you implement a distributed rate limiter using the Generic Cell Rate Algorithm (GCRA) in Redis?
**A:** GCRA rate limiting with Redis: (1) Algorithm parameters: emission interval `T = 1 / rate` (e.g., 1 second for 1 req/s) and burst size `B`. (2) The TAT (Theoretical Arrival Time) is stored in Redis: `TAT = max(now, current_TAT) + T`. (3) Redis Lua script: `local tat = redis.call("GET", key); if not tat then tat = now end; local new_tat = math.max(tonumber(tat), now) + T; if new_tat - now <= B then redis.call("SET", key, new_tat, "EX", 2*B); return 1 else return 0 end`. (4) GCRA provides very accurate rate limiting with minimal memory (one key per limiter, not per request). (5) Unlike sliding window (which stores all timestamps), GCRA uses O(1) memory per rate limiter. (6) Used by Kong and other API gateways for production rate limiting. (7) Supports variable rate limits and burst control with minimal overhead.

## Q13: How do you implement Redis as a distributed session store with consistent hashing for cache affinity?
**A:** Distributed session store with consistent hashing: (1) Use a Redis cluster or a consistent hashing ring (e.g., twemproxy, redis-hashring library). (2) The session ID is hashed and mapped to a specific Redis node: `node = hash(session_id) % N`. (3) This provides cache affinity -- a user's session always goes to the same node, maximizing cache hits. (4) For failover, use replica nodes: if the primary node for a session fails, the consistent hashing can fall back to the replica. (5) Implement virtual nodes in the hash ring to balance load distribution. (6) Set session TTL (`EXPIRE session_id 86400`) and use sliding expiration (`touch session_id` on each request). (7) For rehashing during scaling, use a consistent hash ring with minimal key redistribution (only N/N+M keys move when adding nodes).

## Q14: How do you implement a priority queue in Redis using sorted sets?
**A:** Priority queue with Redis sorted sets: (1) Producer: `ZADD queue:priority timestamp score member` where `score` encodes priority and timestamp: `score = priority * 10^15 + (max_timestamp - now). (2) This ensures higher priority items (lower priority number) are dequeued first, and items with the same priority are ordered by time. (3) Consumer: `ZPOPMIN queue:priority` (Redis 5.0+) or `ZRANGE queue:priority 0 0 WITHSCORES` + `ZREM`. (4) For blocking: `BRPOP` doesn't work with sorted sets. Instead, use a combination: a list for signaling + sorted set for ordering. (5) `BLPOP priority_signal signal_timeout` then `ZPOPMIN queue:priority`. (6) For delayed/retry queues, use a separate sorted set with execution time as the score. A periodic task moves ready items to the main queue.

## Q15: How do you implement a delay queue in Redis?
**A:** Delay queue implementation: (1) Use a sorted set with the execution timestamp as the score and task ID as the member. (2) Producer: `ZADD delay_queue <future_timestamp> task:123`. (3) A dedicated consumer (or scheduled task) polls: `ZRANGEBYSCORE delay_queue -inf <current_timestamp> WITHSCORES LIMIT 0 100`. (4) For each returned task: `ZREM delay_queue task:123` (atomically using Lua) and push to the processing queue. (5) Alternative: use Redis Streams with a delayed consumer group -- the producer publishes to the stream, and the consumer uses `XREAD` with blocking. (6) For very long delays (hours/days), use a leaderboard-like approach with periodic scanning. (7) Redis 7.4+ introduced server-side timers for more efficient delay queue implementation.

## Q16: How does Redis handle large value compression and what are the trade-offs?
**A:** Redis does not natively compress values -- compression must be handled at the application level. Strategies: (1) Compress values before storing: `SET key compressed_data` where `compressed_data = compress(original_data)`. (2) Use `lz4`, `zstd`, `snappy`, or `gzip` compression algorithms. (3) Trade-offs: compression reduces memory usage (50-90% for text/JSON) but adds CPU overhead for compression/decompression. (4) Only compress values larger than 1KB -- small values see minimal gains. (5) Consider `RedisJSON` module for JSON data (doesn't compress but provides field-level operations). (6) For Redis on Flash (Enterprise), compression is handled at the storage layer. (7) For high-throughput read workloads, decompression CPU can become a bottleneck -- load test with realistic data. (8) Monitor `used_memory_overhead` vs `used_memory_dataset` to understand compression savings.

## Q17: How do you implement Redis Streams consumer groups with manual acknowledgment and dead-letter queues?
**A:** Stream consumer groups with DLQ: (1) Create a consumer group: `XGROUP CREATE mystream mygroup $ MKSTREAM`. (2) Consumer reads: `XREADGROUP GROUP mygroup consumer1 COUNT 10 BLOCK 5000 STREAMS mystream >`. (3) After processing successfully: `XACK mystream mygroup message_id`. (4) On failure: don't acknowledge -- the message remains pending. (5) After N failures (track via message delivery count), move to DLQ: `XADD mystream:dlq * FIELDS ...` and then `XACK mystream mygroup message_id`. (6) Monitor pending messages: `XPENDING mystream mygroup`. (7) Claim stale pending messages: `XAUTOCLAIM mystream mygroup consumer2 3600000` (claims messages pending > 1 hour). (8) DLQ monitoring alerts on DLQ size and provides a mechanism for manual reprocessing or skip.

## Q18: How do you use Redis to implement distributed unique ID generation?
**A:** Distributed unique ID with Redis: (1) **INCR-based**: `INCR id:users` -- simple atomic increment, but sequential IDs are predictable. (2) **INCRBY with sharding**: `INCRBY id:users:shard_{user_id % N} 1` -- each shard gets a range, combine shard ID with counter for unique IDs. (3) **Timestamp-prefixed**: `timestamp_ms * 1000000 + INCR id:counter` -- provides time-ordered unique IDs. (4) **Lua script** for atomic ID generation with prefix: `local ts = redis.call("TIME")[1] * 1000 + redis.call("TIME")[2] / 1000; local seq = redis.call("INCR", "id:seq"); return ts * 1000000 + (seq % 1000000)`. (5) For Snowflake-like IDs, combine Redis counter with worker ID and timestamp. (6) Redis's single-threaded nature guarantees no duplicates for `INCR` commands. (7) Set a TTL on the counter key for cleanup (unless it needs to persist across restarts).

## Q19: How do you implement Redis as a distributed semaphore?
**A:** Redis distributed semaphore: (1) Initialize N permits: `SADD semaphore:resource permit:1 permit:2 ... permit:N`. (2) Acquire: `SPOP semaphore:resource` -- returns a permit or nil if none available. (3) Release: `SADD semaphore:resource permit_id`. (4) For timed acquisition, use `BLPOP` doesn't work with sets. Use blocking with polling or Redis Streams. (5) For fairness (FIFO order), use a list instead: `RPUSH semaphore:queue` for request, `LPOP` for acquisition after processing. (6) Timeout-semaphore: each acquired permit has a TTL: `SET semaphore:acquired:{user} {permit_id} EX 30`. Use a Lua script for atomic acquire+TTL set. (7) The Redlock algorithm can also be adapted for semaphore semantics across multiple Redis masters. (8) Monitor semaphore usage with `SCARD semaphore:resource` to track available permits.

## Q20: How do you implement Redis-backed distributed locks with the Redlock algorithm correctly?
**A:** Redlock correct implementation: (1) Acquire locks on N independent Redis masters (typically 5). (2) Set lock key with NX and PX (expiry in ms): `SET key random_value NX PX 10000`. (3) Use a cryptographically random value as the lock value for safe release. (4) The effective acquisition time is: start time - current time. If the client took more than the lock TTL to acquire a majority, release all and retry. (5) A lock is acquired if a majority (N/2 + 1) of instances respond with OK and the total elapsed time is less than the lock validity. (6) To release: evaluate Lua script that deletes the key only if the value matches. (7) Retry on failure with a random delay to prevent thundering herd. (8) Controversial: Martin Kleppmann argues Redlock is not safe under certain conditions (clock jumps, long GC pauses). Alternatives: ZooKeeper, etcd, or a single Redis instance with replica promotion.

## Q21: How do you implement a cache-aside pattern with Redis that handles concurrent cache misses?
**A:** Cache-aside with concurrent miss handling: (1) On cache miss, only one request should regenerate the cache, others should wait. (2) Use SET NX (lock): `SET cache_lock:{key} <thread_id> NX EX 5`. (3) If lock acquired: query the database, `SET cache_key data EX 300`, then `DEL cache_lock`. (4) If lock not acquired: wait briefly (spin with `GET lock` + sleep) and retry the cache read. (5) Avoid busy-waiting -- use exponential backoff with `{ retries: 5, base_delay: 10ms }`. (6) For extremely high contention, use probabilistic early expiration instead of locks. (7) Alternative: use a two-phase approach -- set a "rebuilding" flag on the cache key, serve stale data while rebuilding. (8) Always set TTL on cached data and locks to prevent deadlocks from crashed processes.

## Q22: How do you configure Redis for optimal use with TLS/mTLS in production?
**A:** Redis TLS configuration: (1) Set `tls-port 6380` (separate from plaintext port). (2) Configure `tls-cert-file` and `tls-key-file` for the server certificate. (3) For mTLS, set `tls-client-cert-file` and `tls-auth-clients yes` (or `optional` for mixed mode). (4) Set `tls-ca-cert-file` for the CA that signed client certificates. (5) Disable non-TLS port (`port 0`) for security. (6) Performance: TLS adds 5-15% CPU overhead -- use hardware acceleration (AES-NI) and TLS 1.3 for reduced latency. (7) For Redis Cluster, configure TLS on all nodes; inter-node replication and cluster bus also use TLS. (8) Client configuration: `redis-cli --tls --cert client.crt --key client.key --cacert ca.crt`. (9) Monitor TLS handshake failures via `INFO TLS`.

## Q23: How do you implement Redis ACLs with command and key pattern restrictions?
**A:** Redis 6+ ACLs with granular permissions: (1) Create a user: `ACL SETUSER alice on >password_here ~app:* +GET +SET +DEL`. (2) `~app:*` restricts key access to keys matching the glob pattern `app:*`. (3) `+GET +SET +DEL` allows only those commands; `+@read` allows all read commands; `-@dangerous` disables dangerous commands. (4) Categories: `@admin`, `@dangerous`, `@keyspace`, `@read`, `@write`, `@set`, `@sortedset`, `@list`, `@hash`, `@string`, `@stream`, `@pubsub`, `@slow`. (5) `-@all +@read +@write` -- allow all read/write, deny everything else. (6) `ACL SAVE` persists the ACL configuration. (7) `ACL LOG` shows failed authentication attempts and command rejections. (8) For multi-tenant Redis, create per-tenant users with key pattern restrictions: `ACL SETUSER tenant1 on ~tenant1:* +@all`.

## Q24: How do you implement Redis persistent connections with connection pooling for high throughput?
**A:** Redis connection pooling best practices: (1) Pool size = number of concurrent operations. Start with `min_idle=5, max_idle=20, max_total=50` and adjust based on load testing. (2) Too many connections waste Redis memory (each connection uses ~16KB). Too few cause queueing. (3) Configure connection timeout (2-5 seconds) and socket timeout (based on operation latency expectations). (4) Use `JedisPoolConfig` or equivalent: `setTestOnBorrow(true)`, `setTestOnReturn(false)`, `setTestWhileIdle(true)`. (5) Implement circuit breaker: when the pool is exhausted or Redis is down, fail fast rather than queuing indefinitely. (6) Monitor pool metrics: active connections, idle connections, wait times, blocked threads. (7) For Cluster mode, maintain a pool per-node. (8) For Sentinel/Cluster, use connection pooling that automatically reconnects on failover.

## Q25: How do you implement Redis as a distributed counter for rate limiting across multiple services?
**A:** Distributed counter with Redis: (1) `INCR key` (atomic increment, returns new value). (2) For rate limiting: check the INCR result, reject if over limit. (3) Set expiry: `INCR key; EXPIRE key TTL`. (4) Atomic alternative: `EVAL "local c = redis.call('INCR', KEYS[1]); if c == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end; return c" 1 key ttl`. (5) Use Lua for conditional increments: increment only if under limit, otherwise return error. (6) For multi-window rate limits (10 req/sec + 100 req/min), maintain separate keys with different scopes. (7) For global rate limits across service instances, all instances share the same Redis keys. (8) Consider Redis Cluster slot affinity -- all keys for the same limiter must be on the same node (use hash tags: `rate_limit:{user_id}:1s`).

## Q26: How do you implement Redis-backed leaderboards for gaming applications?
**A:** Gaming leaderboard with sorted sets: (1) `ZADD leaderboard:{game_id}:{period} score player_id`. (2) For real-time: `ZINCRBY leaderboard:daily:2025-01-15 100 player_123`. (3) Retrieve top N: `ZREVRANGE leaderboard:daily:2025-01-15 0 99 WITHSCORES`. (4) Get player rank: `ZREVRANK leaderboard:daily:2025-01-15 player_123`. (5) Get player score: `ZSCORE leaderboard:daily:2025-01-15 player_123`. (6) For ties (same score), encode timestamp in the fractional part of the score: `score = actual_score * 10^10 + (max_timestamp - entry_timestamp)`. (7) Periodic cleanup: delete old period leaderboards using `DEL` or `UNLINK`. (8) For millions of players, consider sharding by player ID range to avoid hot key issues. (9) Combine with Redis Streams for async score submission processing.

## Q27: How do you implement Redis for real-time fraud detection?
**A:** Fraud detection with Redis: (1) **Transaction velocity** -- `INCR user:{id}:tx_count EX 60` (transactions per minute). Flag if > threshold. (2) **Geographic velocity** -- maintain a sorted set of locations with timestamps as scores. Detect impossible travel (login from New York, 5 minutes later from Tokyo). (3) **Device fingerprint** -- store device IDs associated with each user. Flag if too many devices per user in a short window. (4) **Card testing detection** -- track `INCR ip:{ip}:bins_checked EX 300`. Flag if BINs checked exceeds threshold. (5) **Blacklisted patterns** -- `SISMEMBER blacklist:emails email@example.com`. (6) **Behavioral patterns** -- store user behavior profiles (amount ranges, typical times, typical devices) as hashes. (7) All checks must complete in <10ms -- Redis is ideal for this. (8) Use Lua for multi-condition evaluation in a single round trip.

## Q28: How do you migrate data between Redis instances using replication?
**A:** Redis replication-based migration: (1) Set up the target instance as a replica of the source: `REPLICAOF source_host source_port`. (2) Wait for replication to catch up: compare `master_repl_offset` and `slave_repl_offset` from `INFO replication`. (3) Verify data consistency by sampling keys on both instances. (4) When replication is fully synced, switch the application to the target instance. (5) Make the target a master: `REPLICAOF NO ONE`. (6) If you need to keep the source running, reconfigure it as a replica of the target. (7) This approach has near-zero downtime -- the switch is just changing the connection string. (8) For cross-version migrations, check compatibility: a newer version can replicate from an older version, but not always vice versa. (9) For Cluster to Cluster migration, use Redis Enterprise's built-in tools.

## Q29: How do you handle hot keys in Redis at the application level?
**A:** Hot key mitigation strategies: (1) **Local cache** -- add a client-side cache (Caffeine, Guava) in front of Redis for the hot key. (2) **Key replication** -- store the hot key on multiple shards. The client hashes on a randomized component: `hotkey:{shard_id}:{actual_key}` and ORs the results. (3) **Read replicas** -- distribute the hot key reads across replicas using `READROBE` or client-side replica selection. (4) **Shard splitting** -- split the hot key into N sub-keys (e.g., `leaderboard:1`, `leaderboard:2`, ...). The client reads all and merges. (5) **Cache-aside with short TTL** -- very short TTL (1-5 seconds) forces frequent refresh but prevents long-term hot spotting. (6) **Backpressure** -- when a shard is overloaded, return rate limit responses to clients to reduce load. (7) Use `redis-cli --hotkeys` to identify hot keys. (8) In Cluster mode, a hot key can bottleneck a single node -- the local cache approach is most effective.

## Q30: How do you implement Redis for time-series data with downsampling?
**A:** Time-series with Redis: (1) Raw data points stored in sorted sets with timestamp as score: `ZADD ts:sensor:temp <timestamp> <value>`. (2) For downsampling (e.g., 1-second -> 1-minute averages), use a scheduled task: `ZRANGEBYSCORE ts:sensor:temp -inf <one_minute_ago>`. (3) Compute aggregate (avg, min, max, count) and store in a separate key: `ZADD ts:sensor:temp:1m <minute_timestamp> <avg_value>`. (4) Delete raw data older than the retention period: `ZREMRANGEBYSCORE ts:sensor:temp -inf <retention_boundary>`. (5) For high-ingestion scenarios, use Redis Streams for the raw data and sorted sets for the aggregated views. (6) RedisTimeSeries module automates this with `TS.CREATE`, `TS.ADD`, and `TS.RANGE` with aggregation parameters. (7) Retention policies are configured per time-series, and downsampling rules (`TS.CREATERULE`) automate the aggregation pipeline.

## Q31: How does Redis Cluster handle read-replica scalability for read-heavy workloads?
**A:** Read scaling with Cluster replicas: (1) Client libraries support `READONLY` mode -- the client sends read commands to replicas and write commands to masters. (2) `READONLY` command must be sent to each replica connection. (3) `WAIT` command can be used to ensure replica consistency before reads: `WAIT numreplicas timeout` after a write. (4) Replication is asynchronous -- there can be staleness. Applications must tolerate eventually consistent reads from replicas. (5) For strongly consistent reads, read from the master. (6) Load balance reads across replicas using client-side round-robin or random selection. (7) Monitor replica lag via `INFO replication` (`master_repl_offset` vs `slave_repl_offset`). (8) If a replica falls too far behind (lag > threshold), stop routing reads to it until it catches up.

## Q32: How do you implement Redis for feature flag evaluation at scale?
**A:** Feature flag system with Redis: (1) Store feature flags as hashes: `HSET feature:flags dark_mode 1 new_checkout 0`. (2) Cache flags in application memory with short TTL (5-30 seconds). Use Redis Pub/Sub to invalidate the local cache when flags change. (3) For user-level targeting, store user-flag overrides in a hash: `HSET user:flag_overrides:{user_id} dark_mode 1`. (5) Use Redis Streams for flag change audit log: `XADD audit:feature_flags * timestamp now user admin action toggle flag dark_mode`. ## Q72: How do you implement Redis for distributed caching with probabilistic early expiration?
**A:** Probabilistic early expiration (aka "XFetch"): (1) Cache entries store `value`, `created_at`, and `ttl`. (2) On each cache read, compute a delta: `delta = now - created_at`. (3) Compute an early refresh probability: `prob = delta / ttl * 2` (bounded by [0, 1]). (4) Generate a random number `r = random(0, 1)`. (5) If `r < prob`, asynchronously refresh the cache. (6) The current request still gets the (possibly stale) value from the cache. (7) The refresh happens in the background -- the requesting thread doesn't wait. (8) This prevents cache stampedes without locks: at high request rates, approximately one request in N will trigger a refresh (where N = 1/prob). (9) The probability increases as the entry ages, ensuring eventual refresh even at low request rates. (10) Use Redis Time to get current timestamp efficiently: `local now = redis.call('TIME')[1]`.

**A:** Fencing leader election: (1) Traditional Redis lock-based election has a flaw: the leader might stop (GC pause) and another leader gets elected while the first still thinks it's leader. (3) Implementation: `local token = redis.call('INCR', 'election:token'); local ok = redis.call('SET', 'leader:lock', candidate_id, 'NX', 'EX', 30); if ok then return token else return nil end`. (4) The acquired token is included in all write operations to external systems (database, message queue). (5) The external system rejects writes with stale tokens (lower than the last seen token). (6) This prevents split-brain writes even if the old leader is still running. (7) Fencing is critical for systems where dual-writes cause data corruption (filesystems, databases, queues). (8) For systems that don't support token validation (most databases), use lease-based approaches where the leader must periodically assert its leadership with external heartbeats.

## Q90: How do you implement Redis for a distributed system with exactly-once message processing?
**A:** Exactly-once with Redis: (1) Idempotent producer: generate a unique message ID (UUID) and include it in the message body. (2) Deduplication at consumer: `SETNX msg:dedup:{message_id} 1` before processing. If returns 0, skip (already processed). (3) Set TTL on the dedup key: `EXPIRE msg:dedup:{message_id} 86400` (retention for dedup window). (4) Consumer processes the message using Redis Streams with consumer groups: `XREADGROUP GROUP mygroup consumer1 STREAMS mystream >`. (5) After processing: `XACK mystream mygroup message_id`. (6) On consumer crash before ack: message remains pending. Other consumers can claim via `XAUTOCLAIM`. (7) The combination of dedup + consumer groups provides exactly-once semantics (at-least-once delivery + dedup = exactly-once). (8) Store processing results to ensure exactly-once output: `SETEX output:{message_id} 86400 result`. (9) Use Redis transaction (MULTI/EXEC) for atomic dedup check + processing.

## Q93: How do you implement Redis for a distributed system with eventual consistency and conflict resolution?
**A:** Eventual consistency with conflict resolution: (1) Last-Writer-Wins (LWW): store a timestamp with each value: `HSET doc:{id} content <value> updated_at <timestamp>`. On conflict, highest timestamp wins. (2) Multi-Value Register: store all conflicting values: `SADD doc:{id}:conflicts userA:content userB:content`. Application resolves by merging or choosing. (3) CRDT-based: Redis Enterprise's Active-Active uses CRDTs (Conflict-free Replicated Data Types). Each data type has a merge function: LWW-Register, GCounter (grow-only counter), OR-Set (observed-remove set). (4) Application-level merge: store vector clocks: `HSET doc:{id} content <value> vector_clock '{"node1": 5, "node2": 3}'`. On conflict, merge or pick latest by wall clock. (5) TTL-based reconciliation: if a value hasn't been updated for > 5 minutes, it's considered stable and accepted. (6) Application-level CRDT: use Redis sets for add-only, use sorted sets for LWW-element sets. (7) For DNS-like records (simple overwrite), LWW is sufficient. For collaborative editing (OT/CRDT), use specialized data types.

## Q95: How do you implement Redis for a distributed system with causal consistency?
**A:** Causal consistency with Redis: (1) Each client maintains a vector clock: `{node1: version, node2: version}`. (2) On write: `HSET data:{key} value <value> vector_clock <client_vector_clock>`. The write propagates the client's vector clock. (3) On read: the server returns the latest vector clock. The client updates its vector clock: `max(client_clock, server_clock)`. (4) For cross-shard consistency: use Redis Streams to track causal dependencies. Each message carries the vector clock. (5) Read-your-writes: after a write, store the version in the session: `SET session:{id}:version <vector_clock>`. Reads check that they see a version >= session version. (6) Monotonic reads: track the highest version seen by a client session. Don't serve reads that return a lower version. (7) For replication: replicas must apply writes in causal order. Use the vector clock to order writes from different sources. (8) Redis doesn't natively support causal consistency across keys. You must implement it at the application layer or use Redis Enterprise's Active-Active with CRDTs. (9) For simple read-your-writes, use `WAIT` to ensure the write is replicated before responding.

## Q96: How do you implement Redis for a distributed messaging system with priority and delayed delivery?
**A:** Priority + delayed messaging: (1) **Priority queues**: use sorted sets with priority encoded in the score: `ZADD msg:priority <priority> <message_id>`. A sweeper moves due messages to the priority queue: `ZRANGEBYSCORE msg:delayed -inf <now>`. (3) **Priority + delay combined**: `score = priority * 10^15 + (max_time - delivery_time)`. This prioritizes by priority first, then by delivery time within the same priority. (4) **Dead-letter after max retries**: `SADD msg:dead_letter message_id`. (5) **Scheduled delivery**: `XADD msg:schedule * deliver_at <timestamp> message {...}`. A scheduler consumes from the stream and moves to the delivery queue at the scheduled time. (6) **Rate-limited delivery**: limit how many messages per second a consumer receives from a priority queue. (7) Use Lua for atomic dequeue to prevent two consumers from getting the same message.

## Q99: How do you implement Redis for distributed caching with read-through pattern?
**A:** Read-through cache: (1) On cache miss, the application doesn't directly call the database. Instead, the cache itself (or a cache library) loads the data. (2) Using Jedis with read-through: implement a custom `CacheLoader` that fetches from DB and populates Redis. ## Q100: How do you implement Redis for a distributed caching system with cross-region replication?
**A:** Cross-region cache replication: (1) **Active-Passive**: primary region handles writes, asynchronously replicated to secondary regions via Redis replication. Reads from secondary use potentially stale data. (2) **Active-Active**: Redis Enterprise's CRDT-based Active-Active geo-distribution. Writes in any region are replicated to all regions with automatic conflict resolution. (3) **Write-through**: writes go to all regions synchronously (adds latency equal to farthest region). (4) **Hybrid**: write to local + async replicate. On read, check local first, then fall back to the primary region (cross-region read). (5) **Consistency levels**: strong (read from primary), eventual (read from local), read-your-writes (track last write region). (6) **Cache warming across regions**: when a new region comes online, pre-populate from primary region's cache dump. (7) **Conflict resolution**: CRDT-based (Redis Enterprise) handles concurrent writes. For custom implementations, use LWW (last-writer-wins) with timestamps. (8) **Routing**: DNS-based routing (Route53, Cloudflare) directs users to the nearest region. (9) **Monitoring**: track cross-region replication lag (`INFO replication`, `master_repl_offset` differences). Alert on lag exceeding thresholds. (10) **Disaster recovery**: on primary region failure, promote a secondary region to primary (change DNS, update replication orientation).
