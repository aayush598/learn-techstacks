# MySQL Interview Questions and Answers - Part 2

## Q1: How do you implement MySQL replication lag monitoring and alerting?
**A:** Monitor replication lag: `SHOW SLAVE STATUS\G` (or `SHOW REPLICA STATUS` in 8.0.22+). Key fields: `Seconds_Behind_Master` (lag in seconds), `Read_Master_Log_Pos` (binlog position read), `Exec_Master_Log_Pos` (binlog position executed). For more detail: `SELECT NOW() - (SELECT MAX(ts) FROM mydb.heartbeat)` using a heartbeat table updated on the source. Set alerts: lag > 30 seconds = warning, > 300 seconds = critical. Use `pt-heartbeat` from Percona Toolkit for precise millisecond-level lag measurement: `pt-heartbeat -D percona --update --daemonize; pt-heartbeat -D percona --check --master-server-id=1`. Monitor `Replica_IO_Running` and `Replica_SQL_Running` for YES to detect replication failures.

## Q2: How do you implement MySQL advanced indexing with descending indexes?
**A:** MySQL 8.0+ supports descending indexes for better ORDER BY DESC performance: `CREATE INDEX idx_created_desc ON users (created_at DESC); CREATE INDEX idx_multi ON users (status ASC, created_at DESC)`. Descending indexes store keys in reverse order, eliminating the need for backward scans (which are less efficient). Query optimizer uses them for: `SELECT * FROM users WHERE status = 'active' ORDER BY created_at DESC LIMIT 10`. Check usage with `EXPLAIN`: look for `Backward index scan` vs `Index scan`. For compound indexes, mixing ASC and DESC matches mixed ORDER BY: `ORDER BY status ASC, created_at DESC`.

## Q3: How do you implement MySQL connection pooling with ProxySQL advanced features?
**A:** ProxySQL offers advanced connection pooling: Configure `admin> INSERT INTO mysql_servers (hostgroup_id, hostname, port) VALUES (0, 'primary', 3306), (1, 'replica1', 3306), (1, 'replica2', 3306); INSERT INTO mysql_query_rules (active, match_pattern, destination_hostgroup) VALUES (1, '^SELECT.*', 1), (1, '^INSERT|UPDATE|DELETE', 0); LOAD MYSQL SERVERS TO RUNTIME; LOAD MYSQL QUERY RULES TO RUNTIME`. Features: read/write splitting, connection multiplexing (share connections across transactions), query caching (`mysql_query_rules.cache_ttl`), query rewriting, query timeout (`mysql-default_query_timeout`), and firewall (block queries matching patterns). ProxySQL uses a management interface on port 6032 and proxy on 6033.

## Q4: How do you implement MySQL JSON data type with generated columns for indexing?
**A:** Create virtual generated columns to index JSON keys: `CREATE TABLE products (id INT PRIMARY KEY, data JSON, price DECIMAL(10,2) GENERATED ALWAYS AS (data->>'$.price') STORED, sku VARCHAR(50) GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(data, '$.sku'))) STORED, INDEX idx_price (price), UNIQUE INDEX idx_sku (sku));`. STORED columns persist data (uses disk). VIRTUAL columns (default) compute on read. Query: `SELECT name, price FROM products WHERE sku = 'ABC-123' AND price BETWEEN 10 AND 50`. Multi-column indexes work on generated columns: `INDEX idx_category_price ((CAST(data->>'$.category' AS CHAR(20))), price)`. JSON path expressions with `CAST()` in functional key parts (MySQL 8.0.13+).

## Q5: How do you implement MySQL table partitioning with pruning optimization?
**A:** Partition pruning eliminates irrelevant partitions: `CREATE TABLE logs (id INT, ts DATETIME, message TEXT) PARTITION BY RANGE (TO_DAYS(ts)) (PARTITION p_jan VALUES LESS THAN (TO_DAYS('2024-02-01')), PARTITION p_feb VALUES LESS THAN (TO_DAYS('2024-03-01')), PARTITION p_future VALUES LESS THAN MAXVALUE)`. Query: `SELECT * FROM logs WHERE ts BETWEEN '2024-01-15' AND '2024-01-31'` scans only p_jan. Check pruning: `EXPLAIN SELECT ...` — look for `partitions` column showing only relevant partitions. For hash partitioning: `PARTITION BY HASH (YEAR(ts)) PARTITIONS 4`. For list partitioning: `PARTITION BY LIST (region_id) (PARTITION p_north VALUES IN (1,2), PARTITION p_south VALUES IN (3,4))`. Subpartitioning: `PARTITION BY RANGE (YEAR(ts)) SUBPARTITION BY HASH (MONTH(ts))`.

## Q6: How do you implement MySQL point-in-time recovery from binary logs?
**A:** PITR restores to a specific transaction: 1) Restore the full backup: `mysql -u root -p < full_backup.sql`. 2) Identify target position: `mysqlbinlog binlog.000001 binlog.000002 | grep -i "DROP TABLE"` to find problematic statement. 3) Restore binlogs up to just before the incident: `mysqlbinlog --stop-datetime="2024-06-15 14:29:59" binlog.000001 binlog.000002 | mysql -u root -p`. Or use position: `mysqlbinlog --stop-position=1234567 binlog.000001 | mysql -u root -p`. 4) Skip the problematic event: `mysqlbinlog --start-position=1234567 binlog.000001 binlog.000002 | mysql -u root -p`. Use `mysqlbinlog --verbose` for human-readable output. For large binlogs, use `mysqlbinlog -j` for parallel processing.

## Q7: How do you implement MySQL user-defined functions (UDFs) and stored functions?
**A:** Stored functions: `DELIMITER $$; CREATE FUNCTION get_full_name(first_name VARCHAR(50), last_name VARCHAR(50)) RETURNS VARCHAR(101) DETERMINISTIC READS SQL DATA BEGIN DECLARE full_name VARCHAR(101); IF first_name IS NULL AND last_name IS NULL THEN RETURN NULL; END IF; SET full_name = CONCAT(COALESCE(first_name, ''), ' ', COALESCE(last_name, '')); RETURN TRIM(full_name); END$$; DELIMITER ;`. Use: `SELECT get_full_name('John', 'Doe')`. DETERMINISTIC (same inputs always return same output) enables function result caching. NOT DETERMINISTIC functions with READS SQL DATA can be used in queries but not for indexed generated columns. UDFs written in C/C++: `CREATE FUNCTION metaphon RETURNS STRING SONAME 'metaphon.so'`. For performance, avoid stored functions in WHERE clauses on large tables — they prevent index usage.

## Q8: How do you implement MySQL full-text search with Boolean mode and query expansion?
**A:** Full-text search modes: `-- Natural Language (default): SELECT * FROM articles WHERE MATCH(title, body) AGAINST('database optimization'); -- Boolean mode (operators): SELECT * FROM articles WHERE MATCH(title, body) AGAINST('+database -nosql' IN BOOLEAN MODE); -- Boolean operators: + (must have), - (must not), > (increase rank), < (decrease rank), * (wildcard), "" (phrase), () (grouping); -- Wildcard: SELECT * FROM articles WHERE MATCH(title) AGAINST('optim*' IN BOOLEAN MODE); -- Query expansion (finds related content): SELECT * FROM articles WHERE MATCH(title, body) AGAINST('database' WITH QUERY EXPANSION)`. Query expansion performs a two-pass search: first finds top matches, then extracts significant words and searches again. This is effective for finding conceptually related content.

## Q9: How do you implement MySQL CTE (Common Table Expressions) for hierarchical data?
**A:** Recursive CTE for org hierarchy: `WITH RECURSIVE org AS (SELECT id, name, manager_id, 1 AS level FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.id, e.name, e.manager_id, org.level + 1 FROM employees e JOIN org ON e.manager_id = org.id) SELECT id, CONCAT(REPEAT('--', level - 1), name) AS hierarchy FROM org ORDER BY id`. For limiting recursion depth: `... WHERE level < 10`. For finding all subordinates: `WITH RECURSIVE subordinates AS (SELECT id FROM employees WHERE id = 5 UNION ALL SELECT e.id FROM employees e JOIN subordinates s ON e.manager_id = s.id) SELECT * FROM employees WHERE id IN (SELECT id FROM subordinates)`. Non-recursive CTEs: `WITH sales AS (SELECT region, SUM(amount) AS total FROM orders GROUP BY region), ranked AS (SELECT region, total, RANK() OVER (ORDER BY total DESC) AS r FROM sales) SELECT region, total FROM ranked WHERE r <= 3`.

## Q10: How do you implement MySQL InnoDB buffer pool optimization and monitoring?
**A:** Monitor and optimize buffer pool: `SHOW ENGINE INNODB STATUS\G` — look at "Buffer pool hit rate" (>99% is good). Query: `SELECT (1 - (SUM(innodb_buffer_pool_reads) / (SUM(innodb_buffer_pool_read_requests) + SUM(innodb_buffer_pool_reads))) * 100 AS hit_ratio FROM performance_schema.global_status WHERE variable_name IN ('Innodb_buffer_pool_read_requests', 'Innodb_buffer_pool_reads')`. Tune `innodb_buffer_pool_size` to 70-80% of dedicated RAM. Enable `innodb_buffer_pool_instances` (multiple pool instances reduce contention). Preload warm buffer pool: `SELECT * FROM information_schema.tables WHERE table_schema = 'mydb'` to populate the buffer pool. Use `innodb_buffer_pool_dump_at_shutdown=ON` and `innodb_buffer_pool_load_at_startup=ON` to persist buffer pool across restarts.

## Q13: How do you implement MySQL table checksum validation for replication consistency?
**A:** Verify data consistency between source and replica: 1) `pt-table-checksum` (Percona Toolkit): `pt-table-checksum h=source,u=checksum,p=pass,P=3306 --databases=mydb --replicate=percona.checksums; pt-table-sync --execute --replicate=percona.checksums h=source,u=checksum,p=pass h=replica,P=3306`. 2) Manual: `SELECT @c := MD5(GROUP_CONCAT(id, name, email ORDER BY id)) FROM users` on both servers and compare. 3) `CHECKSUM TABLE users` (slower, locks table). 4) For large tables, checksum in chunks: `SELECT CONCAT('CHUNK', CEILING(id / 1000)), COUNT(*), BIT_XOR(CRC32(CONCAT(id, name))) FROM users GROUP BY CEILING(id / 1000)`. Regular checksums detect silent data corruption that SHOW SLAVE STATUS might miss.

## Q14: How do you implement MySQL spatial queries with geographic distance calculations?
**A:** Spatial data and distance queries: `CREATE TABLE locations (id INT PRIMARY KEY, name VARCHAR(100), coord POINT NOT NULL SRID 4326, SPATIAL INDEX idx_coord (coord)); INSERT INTO locations VALUES (1, 'Office', ST_SRID(POINT(-73.9857, 40.7484), 4326)); -- Distance in meters: SELECT name, ST_Distance_Sphere(coord, ST_SRID(POINT(-73.9857, 40.7484), 4326)) AS distance_m FROM locations ORDER BY distance_m LIMIT 10; -- Within bounding box: SELECT * FROM locations WHERE MBRContains(ST_Envelope(ST_GeomFromText('LINESTRING(-74.0 40.5, -73.5 41.0)', 4326)), coord); -- Within radius: SELECT * FROM locations WHERE ST_Distance_Sphere(coord, ST_SRID(POINT(-73.9857, 40.7484), 4326)) < 1000`. For performance, use spatial indexes on POINT/GEOMETRY columns. The `ST_Distance_Sphere` function (MySQL 5.7+) uses the Earth as a sphere for accurate distance calculations.

## Q15: How do you implement MySQL online DDL using pt-online-schema-change?
**A:** Percona's pt-online-schema-change alters tables without blocking writes: `pt-online-schema-change --alter "ADD COLUMN last_login DATETIME, ADD INDEX idx_status (status)" D=mydb,t=users --execute`. How it works: 1) Creates a shadow table with the new schema. 2) Creates triggers on the original table to sync changes. 3) Chunks through the table copying rows incrementally. 4) Swaps tables atomically with RENAME TABLE. 5) Drops the old table and triggers. Options: `--chunk-size=1000` (rows per chunk), `--max-lag=5` (pause if replica lag exceeds), `--check-interval=2`, `--dry-run` (test without running). For large tables (>100GB), use `gh-ost` (GitHub's tool) which uses binlog streaming instead of triggers for less load.

## Q16: How do you implement MySQL GROUP_CONCAT limitations and workarounds?
**A:** `GROUP_CONCAT` has a 1024-byte default limit. Increase: `SET SESSION group_concat_max_len = 1000000;` or globally: `SET GLOBAL group_concat_max_len = 1000000;`. For ordered concatenation: `SELECT dept_id, GROUP_CONCAT(name ORDER BY salary DESC SEPARATOR ' | ') AS employees FROM employees GROUP BY dept_id`. For distinct: `GROUP_CONCAT(DISTINCT name ORDER BY name)`. For large results, GROUP_CONCAT produces a LONGTEXT value. Memory usage: the entire result is built in memory. If the result exceeds `group_concat_max_len`, it truncates silently. For extremely large aggregations, consider application-side processing or storing intermediate results in a temporary table.

## Q19: How do you implement MySQL deadlock detection and resolution strategies?
**A:** Deadlocks occur when two transactions hold locks each other needs. Detection: `SHOW ENGINE INNODB STATUS\G` — search for "LATEST DETECTED DEADLOCK". InnoDB automatically rolls back the smallest transaction. Strategies to minimize: 1) Access tables in the same order in all transactions. 2) Keep transactions short. 3) Use READ COMMITTED isolation to reduce gap locks: `SET TRANSACTION ISOLATION LEVEL READ COMMITTED`. 4) Use `SELECT ... FOR UPDATE NOWAIT` or `SKIP LOCKED` to avoid waiting. 5) Add indexes to reduce row lock ranges. 6) Avoid user interaction within transactions. 7) Implement retry logic at the application level: catch error 1213 (ER_LOCK_DEADLOCK) and retry the transaction. Monitor: `SELECT * FROM performance_schema.events_statements_current WHERE SQL_TEXT LIKE '%FOR UPDATE%'`.

## Q20: How do you implement MySQL WITH ROLLUP for hierarchical aggregation?
**A:** ROLLUP generates subtotal and grand total rows: `SELECT COALESCE(region, 'All Regions') AS region, COALESCE(product_category, 'All Products') AS category, SUM(sales) AS total_sales FROM sales_data GROUP BY region, product_category WITH ROLLUP`. Output includes: rows per region+category, per-region subtotals (category = NULL), and grand total (both NULL). For multi-level ROLLUP: `GROUP BY YEAR(order_date), QUARTER(order_date), MONTH(order_date) WITH ROLLUP` gives yearly, quarterly, and monthly subtotals. Use `COALESCE` or `IFNULL` for display formatting. Alternative: `GROUPING()` function distinguishes NULL from actual NULL values: `SELECT IF(GROUPING(region), 'All', region) AS region`. Performance: ROLLUP is calculated during aggregation with minimal overhead.

## Q21: How do you implement MySQL with Redis caching strategies for read-heavy workloads?
**A:** Cache MySQL query results in Redis: 1) Cache-aside (lazy loading): `function getUser(id) { const cached = redis.get(`user:${id}`); if (cached) return JSON.parse(cached); const user = db.query('SELECT * FROM users WHERE id = ?', [id]); redis.setex(`user:${id}`, 3600, JSON.stringify(user)); return user }`. 2) Write-through: on update, write to MySQL and Redis: `UPDATE users SET name = ? WHERE id = ?; redis.set(`user:${id}`, JSON.stringify(updatedUser))`. 3) Cache invalidation: delete cache keys on UPDATE/DELETE: `redis.del(`user:${id}`)`. 4) Use Redis for session storage, rate limiting counters, and leaderboards. 5) Use MySQL `GROUP_CONCAT` to batch cache warming queries. 6) For query result caching: cache the hash of the query + parameters as the key. TTL should match data volatility.

## Q22: How do you implement MySQL query optimization with index merge?
**A:** Index merge uses multiple indexes on a single table: `EXPLAIN SELECT * FROM users WHERE status = 'active' OR created_at > '2024-01-01'`. Look for `type: index_merge`. MySQL combines results from multiple indexes (union, intersection, sort-union). Optimize: `CREATE INDEX idx_status ON users (status); CREATE INDEX idx_created ON users (created_at)`. For AND conditions, intersection merge is efficient. For OR conditions, MySQL uses union merge — but it's often better to create a compound index: `CREATE INDEX idx_status_created ON users (status, created_at)`. Index merge has limitations: results from each index are fetched (row pointers), then merged — can be slower than a full table scan for large result sets. Test both approaches with `EXPLAIN ANALYZE`.

## Q24: How do you implement MySQL with read replicas and load balancing?
**A:** Distribute reads across replicas: 1) Application-level: `class Database { static $primary = new PDO(DSN_PRIMARY); static $replicas = [new PDO(DSN_REPLICA1), new PDO(DSN_REPLICA2)]; static function read($sql) { $replica = self::$replicas[array_rand(self::$replicas)]; return $replica->query($sql) } static function write($sql) { return self::$primary->exec($sql) } }`. 2) Using HAProxy: `frontend mysql bind *:3306 default_backend mysql_servers; backend mysql_servers server primary 10.0.1.1:3306 check port 3306 inter 5s fall 3 rise 2, server replica1 10.0.1.2:3306 check port 3306 inter 5s fall 3 rise 2; use_backend mysql_read if { path /read }`. 3) Using ProxySQL (recommended): define hostgroups and query rules to route SELECTs to replicas and writes to primary. Monitor replica lag and remove lagging replicas from rotation.

## Q25: How do you implement MySQL full outer join emulation?
**A:** MySQL doesn't support FULL OUTER JOIN natively. Emulate with LEFT JOIN + RIGHT JOIN + UNION: `SELECT a.*, b.* FROM users a LEFT JOIN orders b ON a.id = b.user_id UNION SELECT a.*, b.* FROM users a RIGHT JOIN orders b ON a.id = b.user_id`. Alternatively: `SELECT a.*, b.* FROM users a LEFT JOIN orders b ON a.id = b.user_id UNION ALL SELECT a.*, b.* FROM users a RIGHT JOIN orders b ON a.id = b.user_id WHERE a.id IS NULL`. The WHERE clause removes the duplicate row from LEFT JOIN (the row where both match). For better performance, use: `SELECT * FROM (SELECT a.id AS a_id, b.id AS b_id FROM users a LEFT JOIN orders b ON a.id = b.user_id UNION ALL SELECT a.id, b.id FROM users a RIGHT JOIN orders b ON a.id = b.user_id WHERE a.id IS NULL) AS full_join`.

## Q26: How do you implement MySQL query performance monitoring with Performance Schema?
**A:** Performance Schema provides detailed query metrics: `-- Top queries by total latency: SELECT DIGEST_TEXT, COUNT_STAR, SUM_TIMER_WAIT/1000000000 AS total_ms, AVG_TIMER_WAIT/1000000000 AS avg_ms, SUM_ROWS_EXAMINED/SUM_ROWS_SENT AS rows_ratio FROM performance_schema.events_statements_summary_by_digest ORDER BY SUM_TIMER_WAIT DESC LIMIT 20; -- Queries with full table scans: SELECT DIGEST_TEXT, COUNT_STAR FROM performance_schema.events_statements_summary_by_digest WHERE SUM_NO_INDEX_USED > 0 ORDER BY COUNT_STAR DESC; -- Lock analysis: SELECT * FROM performance_schema.events_waits_current WHERE EVENT_NAME LIKE '%lock%' ORDER BY TIMER_WAIT DESC; -- Enable consumers: UPDATE performance_schema.setup_consumers SET ENABLED = 'YES' WHERE NAME LIKE '%events_statements%'; -- Enable instruments: UPDATE performance_schema.setup_instruments SET ENABLED = 'YES', TIMED = 'YES' WHERE NAME LIKE '%statement/%'`. Performance Schema has minimal overhead (~5%) and is enabled by default in MySQL 5.7+.

## Q27: How do you implement MySQL with transactions and savepoints for partial rollbacks?
**A:** Savepoints enable partial rollback within a transaction: `START TRANSACTION; INSERT INTO orders (user_id, total) VALUES (1, 100.00); SET @order_id = LAST_INSERT_ID(); SAVEPOINT sp_items; INSERT INTO order_items (order_id, product_id, qty) VALUES (@order_id, 10, 1); -- Error occurs; ROLLBACK TO SAVEPOINT sp_items; -- Only rolls back the items insert; INSERT INTO order_items (order_id, product_id, qty, price) VALUES (@order_id, 10, 1, 100.00); RELEASE SAVEPOINT sp_items; -- Removes savepoint; COMMIT; -- Commits the order and corrected items`. Savepoints are useful for batch processing where one failed item shouldn't abort the entire batch. Nested savepoints: `SAVEPOINT sp1; ... SAVEPOINT sp2; ... ROLLBACK TO sp2; ... RELEASE sp1`. Savepoints are automatically released on COMMIT or ROLLBACK.

## Q29: How do you implement MySQL constraint deferring (or lack thereof)?
**A:** MySQL (InnoDB) does NOT support deferrable constraints — they are checked immediately on each row, not at transaction commit. Workarounds: 1) Set `foreign_key_checks = 0` before bulk operations: `SET foreign_key_checks = 0; -- Bulk load data; SET foreign_key_checks = 1; -- Check at re-enable`. 2) Use `CHECK` constraints (MySQL 8.0.16+) which are also immediate but can be turned off: `SET check_constraint_checks = 0`. 3) Restructure application logic to insert data in dependency order. 4) Use application-level validation before database operations. 5) For complex cross-table validation, use stored procedures that validate before inserting: call validate + insert in sequence within a transaction.

## Q33: How do you implement MySQL with hash sharding for horizontal scaling?

## Q36: How do you implement MySQL with InnoDB doublewrite buffer optimization for SSDs?
**A:** The doublewrite buffer prevents partial page writes. On modern SSDs with atomic write size of 16KB (matching InnoDB page size), the doublewrite buffer adds overhead. ## Q63: How do you implement MySQL with NOLOCK (READ UNCOMMITTED) and its risks?
**A:** MySQL doesn't support `WITH (NOLOCK)` like SQL Server. Equivalent: `SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED; SELECT * FROM users WHERE id = 1`. Risks: dirty reads (read uncommitted data from other transactions), non-repeatable reads, phantom reads. Use cases: approximate counts, non-critical reporting, schema introspection. Better alternatives: `SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE` (shared lock, prevents DELETE/UPDATE), or `SELECT ... FOR UPDATE` (exclusive lock). For reporting: use a read replica or `SET TRANSACTION ISOLATION LEVEL READ COMMITTED` (prevents dirty reads). For MyISAM tables: READ UNCOMMITTED is irrelevant since MyISAM uses table-level locking. Never use READ UNCOMMITTED for financial or critical operations.

## Q66: How do you implement MySQL advanced locking with advisory locks?

## Q82: How do you implement MySQL with consistent non-locking reads (MVCC)?

**A:** Filesort occurs when ORDER BY can't use an index: `EXPLAIN SELECT * FROM users WHERE status = 'active' ORDER BY created_at DESC; -- "Using filesort"`. Optimize: 1) Create composite index matching WHERE + ORDER BY: `CREATE INDEX idx_status_created ON users (status, created_at DESC)`. 2) Increase `sort_buffer_size` (but per-session, too high causes memory waste): `SET SESSION sort_buffer_size = 2 * 1024 * 1024; -- 2MB`. 3) Increase `read_rnd_buffer_size`. 4) Reduce columns in ORDER BY (limit to indexed columns). 5) If sorting a small result set, filesort in memory is fast. 6) For large sorts, ensure `tmp_table_size`/`max_heap_size` are adequate (else spills to disk). 7) Check `SHOW STATUS LIKE 'Sort%'`: `Sort_merge_passes` > 0 indicates disk sorts. 8) Avoid `ORDER BY RAND()` — use application-level random selection. 9) Use indexed ORDER BY with LIMIT for top-N queries.

## Q90: How do you implement MySQL with the CONNECT storage engine for external data?
**A:** CONNECT engine (MariaDB, available as MySQL plugin via Spider) accesses external data: `CREATE TABLE csv_data ENGINE=CONNECT TABLE_TYPE=CSV FILE_NAME='/data/import.csv' HDR=1; -- SOURCE type: SELECT * FROM csv_data; -- Access remote MySQL: CREATE TABLE remote_users ENGINE=CONNECT TABLE_TYPE=MYSQL CONNECTION='mysql://user:pass@remote_host:3306/mydb/users'; -- Access XML: CREATE TABLE xml_data ENGINE=CONNECT TABLE_TYPE=XML FILE_NAME='/data/data.xml'`. CONNECT supports CSV, XML, JSON, ODBC, JDBC, MongoDB, and other sources. Use cases: ETL without loading data, real-time access to external systems, integrating legacy data. Performance depends on external source speed. For production, consider extracting data to InnoDB tables. CONNECT engine is not included in Oracle MySQL — use the FEDERATED engine or Oracle's MySQL Enterprise (ODBC).

## Q91: How do you implement MySQL with XA transactions for distributed transactions?
**A:** XA transactions coordinate across multiple databases: `XA START 'xid1'; INSERT INTO users (name) VALUES ('Alice'); XA END 'xid1'; XA PREPARE 'xid1'; XA COMMIT 'xid1'; -- Or rollback: XA ROLLBACK 'xid1'`. Two-phase commit: Phase 1 (prepare — all participants must agree), Phase 2 (commit). In application (Spring): `@Transactional @Bean public PlatformTransactionManager transactionManager() { return new JtaTransactionManager(); }`. XA limitations: higher overhead (fsync per prepare), longer lock holding, recovery complexity (in-doubt transactions). Monitore in-doubt transactions: `XA RECOVER;` — shows prepared but uncommitted transactions. For in-doubt recovery: manually `XA COMMIT` or `XA ROLLBACK` based on business logic. For modern distributed systems, consider Sagas pattern instead of XA.

## Q92: How do you implement MySQL with the ROLE feature for privilege management?
**A:** Roles simplify privilege management (MySQL 8.0+): `CREATE ROLE 'app_readonly', 'app_readwrite', 'app_admin'; GRANT SELECT ON mydb.* TO 'app_readonly'; GRANT INSERT, UPDATE, DELETE ON mydb.* TO 'app_readwrite'; GRANT ALL ON mydb.* TO 'app_admin'; -- Grant role inheritance: GRANT 'app_readonly' TO 'app_readwrite'; -- Assign roles to users: CREATE USER 'alice'@'%' IDENTIFIED BY 'secret'; GRANT 'app_readonly' TO 'alice'@'%'; CREATE USER 'bob'@'%' IDENTIFIED BY 'secret'; GRANT 'app_readwrite' TO 'bob'@'%'; -- Set default roles (active on login): SET DEFAULT ROLE 'app_readonly' TO 'alice'@'%'; -- Activate role: SET ROLE 'app_readwrite'; -- Current roles: SELECT CURRENT_ROLE(); -- Manage: DROP ROLE 'app_readonly'; REVOKE 'app_readonly' FROM 'alice'@'%'`. Roles can be global, database, or function-level. Use roles to enforce least privilege and simplify auditing.

## Q93: How do you implement MySQL with InnoDB page compression for storage savings?
**A:** InnoDB transparent page compression: `CREATE TABLE users (id INT, name VARCHAR(255), email VARCHAR(255)) COMPRESSION='zlib'; -- Or lz4; ALTER TABLE users COMPRESSION='lz4'; -- Check compression: SELECT SPACE, COMPRESSED_SIZE, UNCOMPRESSED_SIZE FROM information_schema.INNODB_CMP_PER_INDEX`. Requirements: `innodb_file_per_table=ON`, MySQL 5.7+ with transparent page compression (not the deprecated InnoDB table compression). Benefits: 30-50% storage savings for text-heavy data. Performance trade-off: slightly higher CPU for compression/decompression, but reduced I/O (fewer pages read from disk). For SSD-heavy workloads, compression reduces storage costs meaningfully. Not all data compresses well — test with your data: `CREATE TABLE test_compress (id INT, data TEXT) COMPRESSION='zlib'; INSERT INTO test_compress SELECT * FROM your_table; SELECT (SUM(COMPRESSED_DATA_SIZE) / SUM(DATA_SIZE)) * 100 AS ratio FROM information_schema.INNODB_CMP_PER_INDEX`. LZ4 uses less CPU than zlib with good compression.

## Q94: How do you implement MySQL with the FLOAT and DOUBLE precision issues?
**A:** Floating-point precision: `-- FLOAT (4 bytes): ~7 digits precision; -- DOUBLE (8 bytes): ~15 digits precision; -- Problem: SELECT 0.1 + 0.2; -- Returns 0.30000000000000004; -- Comparison fails: SELECT * FROM prices WHERE price = 0.1; -- Might not match`. Solutions: 1) Use `DECIMAL` for exact numeric values (money, quantities): `price DECIMAL(10,2)`. 2) Compare with tolerance: `WHERE ABS(price - 0.1) < 0.0001`. 3) Round explicitly: `WHERE ROUND(price, 2) = 0.1`. 4) Store as integer (cents): `price_cents INT`, divide by 100 for display. 5) Use `BINARY` comparison: `WHERE price = CAST(0.1 AS BINARY)`. 6) For aggregations: `ROUND(SUM(price), 2)`. Best practice: use DECIMAL for financial data, DOUBLE for scientific calculations where precision tolerance is acceptable.

## Q95: How do you implement MySQL with the GREATEST and LEAST functions?
**A:** GREATEST/LEAST return max/min from a list: `SELECT GREATEST(10, 20, 30); -- 30; SELECT LEAST(10, 20, 30); -- 10; -- With columns: SELECT id, GREATEST(col1, col2, col3) AS max_value, LEAST(col1, col2, col3) AS min_value FROM data; -- Date comparison: SELECT GREATEST(start_date, COALESCE(end_date, '9999-12-31')) AS effective_end FROM projects; -- String: SELECT LEAST('apple', 'banana', 'cherry'); -- Returns 'apple' (alphabetically first)`. NULL handling: GREATEST/LEAST return NULL if any argument is NULL. Workaround: `GREATEST(COALESCE(a, 0), COALESCE(b, 0))`. Use cases: finding boundary values across columns, determining effective date ranges, implementing business rules that need max/min across multiple fields. For row-based max across columns vs MAX aggregate across rows.

## Q96: How do you implement MySQL lock contention monitoring and resolution?
**A:** Monitor lock contention: `-- Current locks: SELECT * FROM performance_schema.data_locks; -- Lock waits: SELECT * FROM performance_schema.data_lock_waits; -- Blocked queries: SELECT p.id AS blocking_pid, p.user AS blocking_user, p.query AS blocking_query, p2.id AS blocked_pid, p2.user AS blocked_user, p2.query AS blocked_query, lw.wait_started FROM performance_schema.data_lock_waits lw JOIN performance_schema.threads t ON lw.REQUESTING_THREAD_ID = t.THREAD_ID JOIN information_schema.processlist p ON t.PROCESSLIST_ID = p.id JOIN performance_schema.threads t2 ON lw.BLOCKING_THREAD_ID = t2.THREAD_ID JOIN information_schema.processlist p2 ON t2.PROCESSLIST_ID = p2.id; -- Resolve: SELECT CONCAT('KILL ', p.id) FROM ... WHERE blocking_query IS NOT NULL`. Prevent: 1) Keep transactions short. 2) Access tables in consistent order. 3) Use appropriate isolation level (READ COMMITTED reduces gap locks). 4) Add proper indexes (reduce row lock ranges). 5) Use `NOWAIT` or `SKIP LOCKED` where acceptable. 6) Avoid user interaction within transactions.

## Q97: How do you implement MySQL with the FORMAT and locale-aware number formatting?

MySQL's locale support is limited — for full internationalization, format in the application layer.

## Q99: How do you implement MySQL with the DATA DIRECTORY and INDEX DIRECTORY for storage management?
**A:** Separate data and index storage: `CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(100), email VARCHAR(255)) DATA DIRECTORY = '/mnt/fast_ssd/data' INDEX DIRECTORY = '/mnt/fast_ssd/index'; -- For existing table: ALTER TABLE users DATA DIRECTORY = '/mnt/slow_hdd/data' INDEX DIRECTORY = '/mnt/fast_ssd/index'; -- Or move tablespace: ALTER TABLE users TABLESPACE = innodb_file_per_table`. Considerations: 1) `DATA DIRECTORY` works per-table for InnoDB file-per-table mode. 2) Path must be absolute, directory must exist. 3) MySQL must have write permissions. 4) Use for tiered storage: fast SSDs for indexes, slower HDDs for data. 5) Partitioned tables can have individual partition directories: `CREATE TABLE logs (id INT, ts DATE) PARTITION BY RANGE (YEAR(ts)) (PARTITION p2023 VALUES LESS THAN (2024) DATA DIRECTORY = '/mnt/archive/', PARTITION p2024 VALUES LESS THAN (2025) DATA DIRECTORY = '/mnt/fast_ssd/')`. 6) Not supported on all filesystems — test before production use.

## Q100: How do you implement MySQL best practices for high-availability architecture?
**A:** Production HA architecture: 1) InnoDB Cluster: 3 nodes with Group Replication + MySQL Router for automatic failover and read/write splitting. 2) Semi-sync replication for zero data loss. 3) ProxySQL for query routing, connection pooling, and read/write splitting with automatic health checks. 4) Automated failover: Orchestrator (or MySQL Shell AdminAPI) for master election. 5) Backup strategy: daily physical backup (xtrabackup) + continuous binlog backup for PITR. 6) Monitoring: Prometheus + mysqld_exporter + Grafana. 7) Alerting: replication lag > 30s, connections > 80%, disk > 85%, buffer pool hit ratio < 95%, slow queries > 100 per min. 8) Regular DR testing: failover drills, backup restoration tests. 9) Security: SSL connections, firewall rules, least-privilege users, audit logging. 10) Performance: regular `ANALYZE TABLE`, slow query review, index usage analysis, schema review process.
