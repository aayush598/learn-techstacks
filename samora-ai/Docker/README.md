# Docker Interview Questions and Answers

## Q1: What is Docker?
**A:** Docker is an open-source platform for developing, shipping, and running applications inside lightweight, portable containers. Containers package an application with its dependencies, providing consistency across different environments (development, staging, production).

## Q2: What is the difference between a Docker container and a virtual machine?
**A:** Containers share the host OS kernel and run as isolated processes, while VMs include a full guest OS running on a hypervisor. Containers are lightweight (MBs vs GBs), start in seconds, and have less overhead. VMs provide stronger isolation with their own kernel.

## Q3: What is a Docker image?
**A:** A Docker image is an immutable, read-only template that contains the application code, runtime, libraries, environment variables, and configuration needed to run a container. Images are built using Dockerfiles and stored in registries (like Docker Hub).

## Q4: What is a Dockerfile?
**A:** A Dockerfile is a text file with instructions for building a Docker image. It specifies a base image, copies files, runs commands, sets environment variables, and defines the startup command. Each instruction creates a layer in the image.

## Q5: What is the difference between `CMD` and `ENTRYPOINT` in a Dockerfile?
**A:** `CMD` provides default arguments for the ENTRYPOINT or a default command that can be overridden. `ENTRYPOINT` defines the executable that always runs. When combined, ENTRYPOINT is the command and CMD provides default arguments. `CMD` can be overridden at runtime, while `ENTRYPOINT` cannot (without `--entrypoint` flag).

## Q6: What is the difference between `COPY` and `ADD` in a Dockerfile?
**A:** `COPY` copies files from the build context into the image. `ADD` does everything `COPY` does plus supports remote URLs and automatic tar extraction. Best practice is to use `COPY` for local files and reserve `ADD` for when you need URL downloads or tar extraction.

## Q7: What is Docker Compose?
**A:** Docker Compose is a tool for defining and running multi-container Docker applications. It uses a YAML file (`docker-compose.yml`) to define services, networks, and volumes. A single `docker-compose up` command starts all services.

## Q8: What is a Docker volume?
**A:** A Docker volume is a persistent data storage mechanism that survives container restarts and removals. Volumes are managed by Docker and stored in the Docker host's filesystem (`/var/lib/docker/volumes/`). They are the preferred way to persist data generated by containers.

## Q9: What is the difference between a bind mount and a volume?
**A:** Volumes are fully managed by Docker and stored in Docker's storage directory. Bind mounts map a host filesystem path directly into a container. Volumes are easier to back up, portable across hosts, and have better performance. Bind mounts are useful for development (live code reloading).

## Q10: What is a Docker network?
**A:** Docker networks enable communication between containers and between containers and the host. Types include: bridge (default, isolated network for containers on the same host), host (container shares host's network stack), overlay (multi-host networking for Swarm), and macvlan (assigns MAC addresses to containers).

## Q11: What is the default Docker network driver?
**A:** The default network driver is `bridge`. On a single host, containers connected to the same bridge network can communicate with each other. Each container gets an IP address on the bridge network.

## Q12: What is Docker Hub?
**A:** Docker Hub is a cloud-based registry service for sharing Docker images. It hosts public and private repositories, official base images, automated builds, and webhooks. It is the default registry for Docker.

## Q13: What is a multi-stage build in Docker?
**A:** Multi-stage builds use multiple `FROM` statements in a single Dockerfile to separate build-time and runtime dependencies. The final image copies only the necessary artifacts from earlier stages, resulting in significantly smaller images.

## Q14: What is the difference between `docker run` and `docker start`?
**A:** `docker run` creates and starts a new container from an image. `docker start` starts an existing, stopped container (by name or ID). `docker run` is for new containers; `docker start` is for restarting previously created containers.

## Q15: What is `docker exec`?
**A:** `docker exec` runs a new command in an already running container. It is commonly used to open an interactive shell (`docker exec -it <container> /bin/bash`) or to execute diagnostic commands inside a running container.

## Q16: What is the difference between `docker save` and `docker export`?
**A:** `docker save` saves one or more images (including their history and metadata) to a tar archive. `docker export` exports a container's filesystem as a tar archive (without image history or metadata). `docker save` preserves layers; `docker export` flattens them.

## Q17: What is the difference between `docker load` and `docker import`?
**A:** `docker load` loads images saved with `docker save` (preserving history and metadata). `docker import` creates a filesystem image from a tar archive (exported with `docker export`), creating a new image without layer history.

## Q18: What is Docker Swarm?
**A:** Docker Swarm is Docker's native clustering and orchestration solution. It groups multiple Docker hosts into a single virtual host, providing service discovery, load balancing, scaling, rolling updates, and declarative service configuration.

## Q19: What is a Docker service in Swarm mode?
**A:** A service is the definition of tasks to execute on worker nodes in a Swarm cluster. It specifies the image, replicas, ports, networks, environment variables, and update policies. Swarm ensures the desired state of the service is maintained.

## Q20: What is Docker Stack?
**A:** A Docker Stack is a group of services defined in a Compose file, deployed and managed as a single application in a Swarm cluster. `docker stack deploy` uses the Compose file to deploy the entire application with its services, networks, and volumes to the Swarm.

## Q21: What is the difference between a Docker image layer and an image?
**A:** An image consists of multiple read-only layers. Each layer corresponds to an instruction in the Dockerfile (e.g., `RUN`, `COPY`, `ADD`). Layers are stacked, and only the topmost layer is writable in a running container. Layers are cached and shared across images.

## Q22: How does Docker's layer caching work?
**A:** Docker caches each layer after building it. If a Dockerfile instruction and its context haven't changed, Docker reuses the cached layer instead of rebuilding it. This significantly speeds up builds. To maximize caching, put instructions that change less frequently (like installing dependencies) earlier in the Dockerfile.

## Q23: What is the build cache, and how can you invalidate it?
**A:** The build cache stores intermediate layers from previous builds. A layer is invalidated (and must be rebuilt) when: (1) the instruction changes, (2) files copied by `COPY`/`ADD` change, (3) `ARG` values change, or (4) `--no-cache` is used. Use `--no-cache` to force a full rebuild.

## Q24: What is a `.dockerignore` file?
**A:** `.dockerignore` specifies files and directories to exclude from the Docker build context, similar to `.gitignore`. It reduces build context size, speeds up builds, and prevents sensitive files from being copied into images.

## Q25: How do you reduce Docker image size?
**A:** Techniques: (1) Use minimal base images (Alpine, distroless), (2) Multi-stage builds, (3) Combine `RUN` commands to reduce layers (`&&` chaining), (4) Remove package manager caches in the same layer, (5) Use `.dockerignore`, (6) Avoid installing unnecessary packages.

## Q26: What is the Alpine Linux Docker image?
**A:** Alpine Linux is a security-oriented, lightweight Linux distribution based on musl libc and busybox. The Docker Alpine image is about 5 MB, making it popular for creating minimal Docker images. Alpine uses `apk` as its package manager.

## Q27: What is a distroless Docker image?
**A:** Distroless images contain only the application and its runtime dependencies, without package managers, shells, or other OS utilities. They reduce the attack surface and image size significantly. Debugging is harder since there is no shell.

## Q28: What is the `HEALTHCHECK` instruction in Dockerfile?
**A:** `HEALTHCHECK` tells Docker how to test if a container is still working. Docker runs the specified command periodically. If it fails, the container's status becomes "unhealthy." Swarm and orchestration tools can then restart unhealthy containers.

## Q29: What is the `USER` instruction in a Dockerfile?
**A:** `USER` sets the user (by UID or username) for subsequent `RUN`, `CMD`, and `ENTRYPOINT` instructions. It is a security best practice to avoid running containers as root. You should create a dedicated user in the Dockerfile.

## Q30: What is the `WORKDIR` instruction?
**A:** `WORKDIR` sets the working directory for subsequent `RUN`, `CMD`, `ENTRYPOINT`, `COPY`, and `ADD` instructions. It creates the directory if it does not exist. It is preferred over `RUN mkdir -p` + `cd` for clarity.

## Q31: What is the `EXPOSE` instruction?
**A:** `EXPOSE` documents which ports a container listens on at runtime. It is purely informational — it does not publish the port. To actually expose ports to the host, use `-p` or `--publish` with `docker run`.

## Q32: What is the difference between `ENV` and `ARG` in a Dockerfile?
**A:** `ARG` defines build-time variables that are only available during image build (not in running containers). `ENV` defines environment variables available both during build and in running containers. `ARG` values can be set via `--build-arg` flag.

## Q33: How do you pass environment variables at runtime?
**A:** Use `-e` or `--env` flag with `docker run`: `docker run -e MY_VAR=value my_image`. Or use an env file: `docker run --env-file my.env my_image`. Environment variables can also be set in Docker Compose under the `environment` key.

## Q34: What is the `ONBUILD` instruction?
**A:** `ONBUILD` registers a trigger instruction that executes when the image is used as a base image for another build. It is used in builder patterns to create generic images that customize themselves in downstream Dockerfiles.

## Q35: What is the `SHELL` instruction?
**A:** `SHELL` changes the default shell used for `RUN`, `CMD`, and `ENTRYPOINT` instructions (default `/bin/sh -c` on Linux). It is useful when using alternative shells like PowerShell on Windows.

## Q36: What is the difference between `docker-compose up` and `docker-compose start`?
**A:** `docker-compose up` creates and starts containers. `docker-compose start` only starts existing containers that were previously created (e.g., with `docker-compose create` or stopped). `docker-compose up` is more common as it handles both creation and startup.

## Q37: What is the difference between `docker-compose` and `docker stack`?
**A:** Both use Compose files. `docker-compose` deploys to a single Docker host. `docker stack` deploys to a Docker Swarm cluster, supporting features like rolling updates, service scaling, and multi-host networking. `docker stack` ignores some Compose fields like `build`.

## Q38: What is a Docker registry?
**A:** A Docker registry is a storage and distribution system for Docker images. Docker Hub is a public registry. Private registries can be self-hosted (using the open-source Registry image) or cloud-based (AWS ECR, GCR, Azure ACR, GitLab Container Registry).

## Q39: What is Docker Trusted Registry?
**A:** Docker Trusted Registry (DTR) was Docker Enterprise's private registry with image signing, vulnerability scanning, role-based access control, and integration with LDAP/AD. It has been superseded by Mirantis Container Registry.

## Q40: How do you secure a Docker container?
**A:** (1) Run as non-root user (`USER`), (2) Use minimal base images, (3) Scan images for vulnerabilities, (4) Limit capabilities (`--cap-drop=ALL`), (5) Enable seccomp and AppArmor profiles, (6) Use read-only root filesystem, (7) Set resource limits, (8) Sign images with Docker Content Trust.

## Q41: What are Linux capabilities in Docker?
**A:** Capabilities are fine-grained privileges that can be granted to processes (e.g., `CAP_NET_BIND_SERVICE`, `CAP_SYS_ADMIN`). Docker drops all capabilities by default and allows selective addition. Use `--cap-add` and `--cap-drop` to control capabilities for containers.

## Q42: What is seccomp in Docker?
**A:** Seccomp (secure computing mode) restricts system calls a container can make. Docker ships with a default seccomp profile that blocks dangerous syscalls (~44 blocked out of ~300). Custom profiles can be defined for stricter security.

## Q43: What is Docker Content Trust?
**A:** Docker Content Trust (DCT) enables image signing and verification using digital signatures. When enabled (`DOCKER_CONTENT_TRUST=1`), Docker only pulls, pushes, or runs images that are signed by trusted publishers, ensuring image integrity and authenticity.

## Q44: What is Docker Bench Security?
**A:** Docker Bench Security is an open-source script that checks Docker deployments against CIS (Center for Internet Security) benchmarks. It identifies misconfigurations in host configuration, daemon settings, container images, and runtime security.

## Q45: What is the difference between `docker rm` and `docker rmi`?
**A:** `docker rm` removes one or more containers (must be stopped unless `-f` is used). `docker rmi` removes one or more images. `docker system prune` removes unused containers, images, networks, and volumes.

## Q46: What is `docker system prune`?
**A:** `docker system prune` removes unused Docker objects: stopped containers, dangling images, unused networks, and build cache. Use `docker system prune -a` to remove all unused images (not just dangling). Use `--volumes` to also remove unused volumes.

## Q47: What is `docker stats`?
**A:** `docker stats` displays real-time resource usage metrics for running containers, including CPU, memory, network I/O, and block I/O. It is useful for monitoring container performance and debugging resource issues.

## Q48: What is `docker top`?
**A:** `docker top` displays the running processes inside a container. It is similar to the Linux `top` command but scoped to a specific container. Useful for debugging what processes are running within a container.

## Q49: What is `docker inspect`?
**A:** `docker inspect` returns detailed metadata about Docker objects (containers, images, volumes, networks) in JSON format. It includes configuration, network settings, mount points, state information, and more.

## Q50: What is `docker logs`?
**A:** `docker logs` retrieves logs from a container's stdout/stderr. It supports `-f` (follow), `--tail` (show last N lines), `--since` (filter by timestamp), and `--timestamps` options. Logging drivers control where logs are sent (json-file, syslog, journald, etc.).

## Q51: What are Docker logging drivers?
**A:** Logging drivers determine how container logs are handled. Options include: `json-file` (default, writes to JSON files), `syslog`, `journald`, `gelf`, `fluentd`, `awslogs`, `splunk`, and `gcplogs`. Use `docker info` to see the configured logging driver.

## Q52: What is the difference between `json-file` and `journald` logging drivers?
**A:** `json-file` writes logs to JSON files on disk (supports `docker logs`). `journald` sends logs to the systemd journal (accessible via `journalctl`). `journald` integrates better with systemd-based systems but `docker logs` may not work directly.

## Q53: What is `docker cp`?
**A:** `docker cp` copies files or directories between a container and the local filesystem. Syntax: `docker cp <container>:<path> <host_path>` (from container to host), `docker cp <host_path> <container>:<path>` (from host to container).

## Q54: What is a Docker bridge network?
**A:** The bridge network driver creates a private internal network on the host. Containers on the same bridge network can communicate using IP addresses or container names (with embedded DNS). By default, containers communicate via a virtual ethernet bridge.

## Q55: What is the difference between `bridge` and `host` network?
**A:** Bridge network isolates containers in their own network namespace with port mapping required for external access. Host network removes network isolation — the container uses the host's network stack directly (no port mapping), offering better performance but less isolation.

## Q56: What is the overlay network in Docker?
**A:** The overlay network driver creates a distributed network across multiple Docker hosts in a Swarm cluster. It enables containers on different hosts to communicate securely using VXLAN encapsulation. It is the foundation for multi-host container networking.

## Q57: What is the macvlan network in Docker?
**A:** The macvlan driver assigns a MAC address to each container, making them appear as physical devices on the network. Containers can communicate directly with the physical network, bypassing Docker's port mapping. Useful for legacy applications that need direct network access.

## Q58: What is Docker's embedded DNS?
**A:** Docker has an internal DNS server (127.0.0.11) that provides name resolution for containers. On user-defined bridge networks, containers can resolve other containers by name. Docker Compose uses service names for DNS resolution.

## Q59: How do you set resource limits for a container?
**A:** Use runtime flags: `--memory` (max memory), `--memory-swap` (total memory+swap), `--cpus` (CPU count), `--cpu-shares` (CPU weight), `--pids-limit` (max PIDs). In Docker Compose, use `deploy.resources.limits` (Swarm) or `mem_limit` (Compose).

## Q60: What is OOM killer in Docker?
**A:** OOM (Out Of Memory) killer is the Linux kernel mechanism that kills processes when the system runs out of memory. If a container exceeds its memory limit, Docker's OOM killer terminates the container's process. Adjust with `--memory` and `--oom-kill-disable` (not recommended).

## Q61: What are restart policies in Docker?
**A:** Restart policies control whether a container restarts after exit: `no` (never, default), `always` (always restart), `on-failure[:max-retries]` (restart on non-zero exit), `unless-stopped` (always except when manually stopped). In Compose: `restart: always`.

## Q62: What is the `--init` flag for Docker?
**A:** `--init` runs an init process (tini) as PID 1 inside the container. This ensures proper signal handling and reaping of zombie processes. It is recommended for containers that run complex applications that may spawn child processes.

## Q63: What is the difference between `docker pause` and `docker stop`?
**A:** `docker pause` suspends all processes in a container using cgroups freezer (SIGSTOP). The container's state is preserved in memory. `docker stop` sends SIGTERM (and after timeout SIGKILL) to the main process, transitioning the container to "exited" state.

## Q64: How do you debug a container that exits immediately?
**A:** (1) Check logs with `docker logs <container>`, (2) Use `docker run -it <image> /bin/bash` to enter the container without running CMD, (3) Override CMD: `docker run --entrypoint /bin/bash <image>`, (4) Check the container's exit code, (5) Inspect with `docker inspect`.

## Q65: What is the difference between `Dockerfile` and `docker-compose.yml`?
**A:** A `Dockerfile` defines how to build a single Docker image. `docker-compose.yml` defines how to run multiple services (using existing images or building from Dockerfiles), including networks, volumes, environment variables, and dependencies between services.

## Q66: How do you run multiple commands in a single `RUN` statement?
**A:** Chain commands with `&&` to create a single layer: `RUN apt-get update && apt-get install -y package && rm -rf /var/lib/apt/lists/*`. This reduces image layers and prevents caching issues.

## Q67: What is the purpose of `docker-compose` overrides?
**A:** Docker Compose supports multiple compose files (e.g., `docker-compose.yml` + `docker-compose.override.yml`). The override file extends or overrides services defined in the base file. This is useful for development-specific overrides (different ports, volume mounts, debug modes).

## Q68: What is the `depends_on` directive in Docker Compose?
**A:** `depends_on` expresses startup and shutdown dependencies between services. It starts services in dependency order and stops them in reverse order. However, it does not wait for the service to be "ready" — only for the container to start. Use health checks or wait scripts for readiness.

## Q69: How do you implement health checks in Docker Compose?
**A:** Use the `healthcheck` directive: `healthcheck: test: ["CMD", "curl", "-f", "http://localhost"]; interval: 30s; timeout: 10s; retries: 3`. Compose can use `depends_on.condition: service_healthy` to wait for health checks before starting dependent services.

## Q70: What is the difference between `volumes` and `bind mounts` in Docker Compose?
**A:** In Compose, `volumes:` at the top level define named volumes managed by Docker. `bind mounts` use host paths directly in the service's volumes section. Example: `volumes: - ./src:/app/src` (bind mount) vs `volumes: - mydata:/data` with `volumes: mydata:` top-level.

## Q71: What is the `tmpfs` mount in Docker?
**A:** A `tmpfs` mount stores data in the host's memory (RAM), not on disk. It is fast and ephemeral — data is lost when the container stops. Useful for sensitive data (e.g., secrets, tokens) or cache that does not need persistence.

## Q72: What is Docker Security Scanning?
**A:** Docker Security Scanning (Docker Scan / Snyk integration) scans images for known vulnerabilities against CVE databases. It checks OS packages and application dependencies. It should be integrated into CI/CD pipelines to prevent deploying vulnerable images.

## Q73: What is the difference between a Docker tag and a Docker digest?
**A:** A tag (e.g., `myimage:latest`) is a human-readable label that can be reassigned. A digest (e.g., `myimage@sha256:abc123...`) is a content-addressable identifier based on the image manifest's SHA256 hash. Digests are immutable and uniquely identify a specific image version.

## Q74: What is Docker Manifest?
**A:** A Docker manifest is a JSON file that describes the contents of an image, including layers, architecture, and OS. Multi-architecture manifests (`docker manifest create`) allow a single tag to serve images for different platforms (linux/amd64, linux/arm64, windows/amd64).

## Q75: How do you build multi-architecture Docker images?
**A:** Use `docker buildx build --platform linux/amd64,linux/arm64 -t user/image:latest --push .`. Docker Buildx creates and pushes a multi-architecture manifest that enables automatic platform selection when pulling the image.

## Q76: What is Docker Buildx?
**A:** Docker Buildx is a CLI plugin that extends Docker's build capabilities. It supports multi-platform builds, advanced build cache backends, building in parallel, and custom build drivers (docker, kubernetes, remote). It is the successor to the legacy Docker build.

## Q77: What is the difference between `docker build` and `docker buildx`?
**A:** `docker build` uses the legacy builder. `docker buildx` uses BuildKit, offering parallel builds, better caching (inline, registry, S3, GCS), multi-platform builds, and custom output types. `docker buildx` is recommended for all new projects.

## Q78: What is a cache mount in BuildKit?
**A:** Cache mounts (`--mount=type=cache,target=/root/.cache`) allow BuildKit to persist directories between builds without including them in the final image. This speeds up repeated operations like `npm install`, `apt-get`, or `pip install`.

## Q79: What is a bind mount in BuildKit?
**A:** Bind mounts (`--mount=type=bind,source=.,target=/src`) in Dockerfiles (via `RUN --mount=type=bind`) mount build context files without copying them into the image layer. This is used for multi-stage builds where source code is needed during build but not in the final image.

## Q80: How do you handle secrets in Docker builds?
**A:** Use BuildKit's secret mounts: `RUN --mount=type=secret,id=mysecret cat /run/secrets/mysecret`. Secrets are passed at build time with `--secret id=mysecret,src=./secret.txt`. Secrets are never stored in the final image layers.

## Q81: What is Docker Compose profiles?
**A:** Profiles allow selective activation of services in Docker Compose. Services with `profiles: ["debug"]` are only started when `--profile debug` is specified. This allows different service sets for development, testing, and production in a single compose file.

## Q82: What is the difference between `ports` and `expose` in Docker Compose?
**A:** `ports` maps host ports to container ports (e.g., `"8080:80"`), making the service accessible externally. `expose` only documents ports without publishing them — used for inter-service communication within the compose network.

## Q83: How do you scale services in Docker Compose?
**A:** Use `docker-compose up --scale web=3` or define replicas under `deploy.replicas` (Swarm mode). Note: scaling requires services to be stateless or use shared volumes. Compose V2 also supports `--scale` for development.

## Q84: What is Docker's image pull policy?
**A:** Pull policy (`imagePullPolicy` in Kubernetes, not directly in Docker) determines when to pull images. Docker always pulls `:latest` tags (unless the image exists locally), pulls specific tags only if missing, and never pulls `:latest` with `image:` reference to a digest.

## Q85: What is the difference between `Docker Swarm` and `Kubernetes`?
**A:** Swarm is Docker's native orchestrator (simpler, tighter Docker integration, less steep learning curve). Kubernetes is the industry standard with more features (auto-scaling, self-healing, service mesh, extensive ecosystem, rolling updates with health checks). Swarm is easier to start; Kubernetes is more powerful for complex deployments.

## Q86: How do you convert Docker commands to Kubernetes?
**A:** `docker run` maps to a Pod, `docker-compose up` maps to multiple Deployments + Services, Docker Compose networks map to Kubernetes network policies, volumes map to PersistentVolumeClaims. Tools like `kompose` can auto-convert Docker Compose files to Kubernetes manifests.

## Q87: What is a Docker Node in Swarm?
**A:** A node is an instance of the Docker engine participating in a Swarm cluster. Manager nodes handle cluster management (scheduling, service state, API). Worker nodes run containers. A node can be both manager and worker.

## Q88: How do you promote a Swarm worker to a manager?
**A:** Use `docker node promote <node-name>`. To demote: `docker node demote <node-name>`. Manager nodes should be odd-numbered (1, 3, 5) for consensus (Raft) — never deploy an even number of managers.

## Q89: What is Docker Swarm's Raft consensus?
**A:** Swarm uses the Raft consensus algorithm to maintain a consistent state across manager nodes. It tolerates up to (N-1)/2 manager failures. For example, 3 managers tolerate 1 failure; 5 managers tolerate 2 failures. Raft ensures data consistency even with network partitions.

## Q90: What is rolling update in Docker Swarm?
**A:** A rolling update gradually updates service containers to a new version without downtime. Configured with `--update-parallelism` (how many to update at once), `--update-delay` (pause between updates), `--update-order` (start-first or stop-first), and `--update-failure-action` (pause or continue).

## Q91: What is the difference between `docker service create` and `docker run`?
**A:** `docker service create` creates a service in Swarm mode (distributed, replicated across nodes, with rolling updates). `docker run` creates a single container on the local host. Services are declarative and self-healing; containers are ephemeral.

## Q92: What is secret management in Docker Swarm?
**A:** Docker Swarm secrets are encrypted during transit and at rest. They are stored in the Swarm's Raft log and mounted as files (in-memory tmpfs) in containers at `/run/secrets/`. Use `docker secret create` to manage secrets and `docker service create --secret` to use them.

## Q93: What is Docker Config?
**A:** Docker Config manages non-sensitive configuration data (YAML, JSON, etc.) for Swarm services. Unlike secrets, configs are not encrypted at rest by default. They are mounted as files inside containers, similar to secrets.

## Q94: How do you debug network issues between containers?
**A:** (1) `docker network inspect <network>` to check container connectivity, (2) `docker exec -it <container> ping <other-container>`, (3) `docker exec -it <container> nslookup <service-name>`, (4) Check if containers are on the same network, (5) Verify firewall/iptables rules.

## Q95: What are Docker labels?
**A:** Labels are key-value metadata that can be attached to Docker objects (containers, images, networks, volumes). They are used for organization, automation (e.g., `docker ps --filter "label=env=prod"`), and integration with tools like Traefik and Prometheus.

## Q96: How does Docker handle log rotation?
**A:** Docker's json-file logging driver supports log rotation via options: `--log-opt max-size=10m` (max file size) and `--log-opt max-file=3` (max number of files). In daemon.json, set `"log-driver": "json-file"` with `"log-opts": {"max-size": "10m", "max-file": "3"}`.

## Q97: What is the difference between `docker attach` and `docker exec -it`?
**A:** `docker attach` connects your terminal to a running container's main process (PID 1) stdout/stderr/stdin. `docker exec -it` creates a new process inside the container. `attach` is for interacting with the container's main process; `exec` is for running additional commands.

## Q98: What is `docker update`?
**A:** `docker update` changes resource constraints (CPU, memory, restart policy) of one or more running containers without stopping them. Example: `docker update --memory 512m --cpus 0.5 mycontainer`.

## Q99: What is the Docker Registry Garbage Collection?
**A:** The Docker Registry stores layers and manifests. When images are deleted, layers remain on disk until garbage collection runs. Use `registry garbage-collect /etc/docker/registry/config.yml` to reclaim disk space by removing unreferenced blobs.

## Q100: How do you migrate a Docker installation to a new host?
**A:** (1) Save images: `docker save -o images.tar $(docker images -q)`, (2) Copy volumes from `/var/lib/docker/volumes/`, (3) Copy data from bind mounts, (4) Export container configurations (docker-compose files, run commands), (5) On new host: load images (`docker load -i images.tar`), restore volumes, and restart containers.
