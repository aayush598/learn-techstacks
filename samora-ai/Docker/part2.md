# Docker Interview Questions and Answers - Part 2

## Q1: How does Docker's `overlay2` storage driver handle copy-on-write, and what are the performance implications of deep layer hierarchies?
**A:** overlay2 uses the OverlayFS union filesystem. Each image layer is a separate directory on the host. When a container modifies a file from a lower layer, overlay2 performs a copy-up operation: copying the file from the lower layer to the container's upper layer and then modifying it. Deep hierarchies (>100 layers) cause: (1) increased copy-up latency during writes, (2) more inode usage, (3) longer image pull/push times. Best practice: keep layers under 50, combine RUN commands, and put frequently changed layers at the top.

## Q2: What happens when you run out of inodes on the Docker filesystem due to overlay2 layers?
**A:** overlay2 uses one inode per layer per file. A large number of small files across many layers can exhaust the filesystem's inode limit, even when disk space is available. Symptoms: `mkdir` fails with `ENOSPC`, Docker operations fail with "no space left on device" despite `df -h` showing free space. Check with `df -i`. Mitigation: (1) use a filesystem with higher inode count (XFS), (2) clean up unused images/containers, (3) merge layers in Dockerfile, (4) use `docker system prune -a`, (5) move Docker's data directory to a larger partition.

## Q3: How does Docker handle `ENTRYPOINT` and `CMD` interaction when using `docker run` with arguments?
**A:** The interaction: (1) `ENTRYPOINT` is the executable that always runs, (2) `CMD` provides default arguments to ENTRYPOINT, (3) `docker run myimage arg1` appends `arg1` to the ENTRYPOINT, overriding CMD, (4) `docker run --entrypoint /bin/bash myimage` overrides ENTRYPOINT. For flexible images: use ENTRYPOINT for the binary and CMD for default flags. For pure executables: use ENTRYPOINT with `[]` exec form. The exec form (`["executable", "arg"]`) avoids shell string munging; shell form (`executable arg`) spawns `/bin/sh -c`.

## Q4: How does Docker's `user-namespace remapping` (`--userns-remap`) work, and what are its limitations?
**A:** User namespace remapping maps the container's root user (UID 0) to a non-root UID on the host. Docker creates subordinate UID/GID ranges in `/etc/subuid` and `/etc/subgid`. The host sees the container's UID 0 as, e.g., UID 100000. Benefits: containers cannot escalate to root on the host. Limitations: (1) cannot use host networking, (2) cannot mount bind mounts owned by host UIDs outside the mapped range, (3) some Docker features break (privileged containers, certain volume plugins), (4) performance overhead from UID mapping on file operations.

## Q5: What causes `container exit code 137`, and how do you distinguish between OOM kill and manual `docker kill`?
**A:** Exit code 137 = 128 + 9 (SIGKILL). Both OOM kill and `docker kill -s 9` produce this exit code. To distinguish: (1) check `docker inspect` for `State.OOMKilled: true`, (2) check `dmesg | grep -i oom` for kernel OOM messages, (3) If `docker event` shows `die` event with `oomKilled=true`, (4) Monitor container memory usage before the kill. If OOMKilled is false but exit code is 137, external SIGKILL was sent. For `docker stop` (default SIGTERM), exit code is 143 (128+15) unless it times out and Docker sends SIGKILL.

## Q6: How do you implement Docker container health checks that depend on other services?
**A:** Docker Compose's `depends_on` with `condition: service_healthy` ensures ordering. For complex dependencies, implement a custom entrypoint script that: (1) polls dependent services via TCP (`/dev/tcp` or `nc -z`), (2) waits for database readiness (pg_isready, mysqladmin ping), (3) retries with backoff, (4) times out and fails. Best practice: each service's health check should only check its own process, not its dependencies. Use the wait-for-it.sh pattern: `./wait-for-it.sh db:5432 -- ./myapp`.

## Q7: How does Docker Swarm's `mesh routing mesh` handle incoming traffic, and what are its limitations for sticky sessions?
**A:** Swarm's routing mesh uses IPVS to load-balance incoming requests on any node's published port across all service replicas. When a request arrives on node A for a service on node B, the packet is forwarded via VXLAN overlay. Limitations: (1) source IP is lost (replaced by the overlay network gateway IP) unless using `host` mode, (2) session stickiness is not built in (use an external load balancer with cookie-based stickiness), (3) adds latency for cross-node routing, (4) each node must run the service or accept routing mesh traffic. Use `mode: host` to bypass the mesh and let the load balancer handle routing.

## Q8: How does Docker's `build cache` invalidation work with `ARG` instructions, and what is the `--cache-from` strategy?
**A:** ARG declarations trigger cache invalidation: (1) Changing `ARG` value invalidates all subsequent layers (ARG persists but its value change causes re-evaluation), (2) `--build-arg` with different values breaks cache from that point, (3) `--cache-from` allows using a previously pushed image as a cache source (CI/CD optimization): `docker build --cache-from myimage:latest -t myimage:new .`. This is critical for CI where the build environment is fresh. The cache-from image must be pulled locally first.

## Q9: How does the `--restart=always` policy interact with `docker stop` and `docker kill`?
**A:** `--restart=always` restarts the container on any exit regardless of exit code, EXCEPT when the container is explicitly stopped via `docker stop` or `docker kill`. Docker tracks whether the stop was intentional: (1) `docker stop` sends SIGTERM, sets the restart policy to not restart, (2) `docker kill` sends SIGKILL (or custom signal), also prevents restart, (3) If the container exits due to its own crash (internal error), it restarts, (4) `docker restart` ensures the container stops and starts cleanly. `unless-stopped` differs only in that it also does NOT restart after daemon restart if the container was stopped before the daemon restarted.

## Q10: How does Docker's `storage driver` selection (overlay2 vs devicemapper vs aufs) affect `fsync` behavior and database performance inside containers?
**A:** overlay2 and aufs add a copy-on-write layer that can interfere with database `fsync` calls. When a database inside a container calls `fsync()`, overlay2 must ensure data reaches the underlying filesystem. Performance: (1) overlay2 on XFS with `ftype=1` offers near-native fsync performance, (2) devicemapper (loop-lvm) adds significant fsync overhead due to double-writes (data -> thin pool -> filesystem), (3) aufs has known fsync issues with databases (inconsistent behavior on crash). For databases, always use bind mounts (bypass COW) or volumes (direct filesystem access). Never run databases with devicemapper loop-lvm.

## Q11: How does Docker's `iptables` manipulation work, and what happens when you restart Docker on a system with custom iptables rules?
**A:** Docker manages iptables rules for: (1) container network isolation (bridge networks), (2) port publishing (DNAT rules), (3) inter-container communication on the same network. Docker inserts rules at specific positions in the FORWARD and NAT tables. On restart: (1) Docker flushes its managed chains (DOCKER, DOCKER-USER), (2) recreates rules based on current container state, (3) Custom rules in DOCKER-USER chain are preserved (this chain is designed for user additions). Always add custom firewall rules to the DOCKER-USER chain to persist across Docker restarts.

## Q12: How do you handle Docker secrets rotation without container restart?
**A:** Docker secrets are mounted as files at `/run/secrets/<name>`. By default, secrets are not updated without container restart. For rotation without restart: (1) The secret file is a symlink to the actual secret in the Swarm's encrypted store, (2) Docker Swarm can update the secret file content while the container runs (but the container must re-read the file), (3) Applications must watch for file changes or periodically re-read the secret from the filesystem, (4) For automatic rotation, use `docker service update --secret-rm` and `--secret-add` to swap secrets, (5) Configure the container's entrypoint to SIGHUP on secret change. Without re-reading, old secrets persist until container restart.

## Q13: What is `Docker socket` binding, and what are the security implications of mounting `/var/run/docker.sock` into a container?
**A:** Mounting the Docker socket (`/var/run/docker.sock`) into a container gives that container root-level access to the Docker daemon. The container can: (1) start new containers (including privileged ones), (2) access all volumes and networks, (3) execute commands in any container, (4) potentially escape to the host. This is equivalent to giving root access. Never mount the Docker socket in production containers. Alternatives: (1) use Docker's API proxy with auth, (2) use the Docker SDK for gRPC/proxy, (3) use Kubernetes podman or rootless podman, (4) use SSH execution for specific tasks.

## Q14: How does Docker's `logger` performance affect high-throughput applications, and which log driver has the least overhead?
**A:** Logging drivers impact container throughput: (1) `json-file` (default) writes each log line as a JSON object to a file with `write()` system call per line, (2) `journald` sends logs to systemd's journal via `sd_journal_send()`, (3) `syslog` sends to syslog daemon over UDP (least overhead), (4) `fluentd` has protocol overhead but enables structured log shipping, (5) `none` (no logging) eliminates all overhead but loses all logs. For high-throughput apps (10K+ log lines/sec), use `gelf` with UDP, or `fluentd` with async. For `json-file`, set `max-size` and `max-file` to avoid filling disk.

## Q15: How does Docker's `--read-only` root filesystem interact with applications that need temporary writes?
**A:** `--read-only` mounts the container's root filesystem as read-only. Applications that need temporary writes (e.g., `/tmp`, `/var/run`, log directories) will fail. Solution: mount specific writable directories as tmpfs: `docker run --read-only --tmpfs /tmp:rw,noexec,nosuid,size=64m --tmpfs /var/run:rw,noexec,nosuid,size=32m myimage`. This makes the root filesystem immutable (security) while allowing controlled temporary writes in memory. For persistent writes, use volumes. The tmpfs mount is RAM-backed and lost on container stop.

## Q16: How do you debug a container that exits immediately with no logs?
**A:** Steps: (1) Override the entrypoint: `docker run --entrypoint /bin/sh myimage -c "echo running"`, (2) Check exit code with `docker inspect --format '{{.State.ExitCode}}' container`, (3) If the container exits 0 but immediately, the app may be detaching to background (use `-d` or don't daemonize), (4) Run interactively: `docker run -it myimage /bin/bash`, (5) Check for missing shared libraries: `ldd /path/to/binary` inside container, (6) Check environment variables: `env`, (7) Test with minimum config, (8) Use `docker export` to inspect the filesystem.

## Q17: How does Docker's `multistage build` interact with the build cache, and when should you use `--target` for development?
**A:** Multi-stage builds use multiple FROM statements. Cache behavior: (1) Each stage is cached independently, (2) Changing the base image (`FROM`) invalidates that stage's entire cache, (3) The `COPY --from=prev_stage` copies files, not layers — so a change in the source stage invalidates only the specific COPY layer in the target stage. For development: use `docker build --target dev-stage` to build only the development stage (with all dev dependencies) while the production stage remains separate. This allows fast dev iterations with full tooling while keeping production images small.

## Q18: What is Docker `rootless` mode, and what are its limitations compared to rootful Docker?
**A:** Rootless mode runs the Docker daemon and containers without root privileges, using: (1) `newuidmap`/`newgidmap` for UID/GID mapping, (2) `slirp4netns` for network (userspace networking), (3) `fuse-overlayfs` for storage. Limitations: (1) no `--privileged` containers, (2) no port mapping to ports <1024 (use `authbind` or proxy), (3) limited cgroup support (no memory/CPU limits on older kernels), (4) overlay2 network performance is slower (userspace networking), (5) cannot use host networking mode, (6) limited to a single user. Benefits: eliminates container escape to root on host, no daemon socket privilege.

## Q19: How does Docker's `buildx` with `kubernetes` driver work, and what are the benefits for CI/CD?
**A:** The buildx kubernetes driver creates build pods on a Kubernetes cluster for building images. Benefits: (1) dynamic scaling: builds run on cluster nodes instead of local machine, (2) parallel builds: multiple builds across nodes, (3) native multi-platform builds without emulation if cluster has mixed architectures, (4) shared build cache across CI runners. Setup: `docker buildx create --driver kubernetes --driver-opt namespace=buildkit,nodeselector="role=build" --name k8s-builder --use`. The driver interacts with the Kubernetes API to schedule build pods; BuildKit runs inside the pod.

## Q20: How does Docker handle `SIGTERM` vs `SIGKILL` during `docker stop`, and what is the `stop-grace-period`?
**A:** `docker stop` sends SIGTERM to the container's PID 1 process. Docker waits for `stop-grace-period` (default 10 seconds) for the process to exit cleanly. If the process does not exit, Docker sends SIGKILL. Key behaviors: (1) PID 1 inside the container receives SIGTERM — if the app does not handle it, the default action is termination, (2) the container's entrypoint or init process must properly forward signals to child processes, (3) use `--init` to include tini as PID 1 for proper signal handling, (4) `stop-grace-period` can be set per-container (e.g., `--stop-grace-period=30s` for apps needing longer shutdown), (5) `docker kill -s SIGTERM` sends any signal without timeout.

## Q21: How does Docker's `network` MTU affect container throughput, and how do you tune it?
**A:** Docker's default bridge network MTU is 1500 bytes. If the host's physical network uses jumbo frames (MTU 9000), Docker containers default to 1500, causing packet fragmentation and reduced throughput. Tuning: (1) Set daemon-wide MTU in `/etc/docker/daemon.json`: `{ "mtu": 9000 }`, (2) For specific networks: `docker network create --opt com.docker.network.driver.mtu=9000 mynet`, (3) Verify with `ip link show` inside the container, (4) MTU mismatch between container and upstream switch causes 5-10% throughput loss for large transfers. For cloud environments, match the instance's network MTU (typically 1500 or 9001 on AWS).

## Q22: How does Docker's `--cpuset-cpus` work for CPU pinning, and what NUMA considerations exist?
**A:** `--cpuset-cpus` pins a container to specific CPU cores (e.g., `--cpuset-cpus=0-3,8-11`). Benefits: (1) eliminates CPU cache thrashing from context switching between unrelated workloads, (2) improves performance predictability for latency-sensitive apps. NUMA considerations: (1) pinning CPUs from multiple NUMA nodes increases cross-node memory access latency, (2) ideally, all pinned CPUs should be on the same NUMA node, (3) use `numactl --hardware` to check topology, (4) combine with `--cpuset-mems` to pin memory nodes, (5) for high-performance apps, pair CPU pinning with `--cpu-shares` and `--cpu-quota` for fine-grained control.

## Q23: What happens to container network connections when Docker daemon restarts?
**A:** When the Docker daemon restarts: (1) running containers (with `restart: always` or `unless-stopped`) continue running but lose connectivity until the daemon is back, (2) network interfaces are not managed during the restart window, (3) iptables rules are recreated, potentially causing a brief network disruption, (4) containers connected to user-defined networks may need reconnection, (5) the daemon's absence is undetected by clients — `docker ps` hangs until timeout or daemon returns. For production, use `live-restore: true` in daemon.json to keep containers alive during daemon upgrades.

## Q24: How does Docker's `live-restore` feature work, and when is it insufficient for zero-downtime daemon restarts?
**A:** `live-restore` (enabled in daemon.json: `{ "live-restore": true }`) keeps containers running during Docker daemon restarts (e.g., version upgrades). How: the daemon passes container state to the new daemon instance via files in `/var/run/docker`. Insufficient when: (1) the host system needs reboot (kernel update), (2) the storage driver configuration changes, (3) network configuration changes require iptables rules to be reloaded, (4) the daemon crashes and state files are lost, (5) Swarm services are restarted (orchestrator re-scheduling). For true zero-downtime, use Swarm services with rolling updates or Kubernetes.

## Q25: How does Docker's `buildx` `--load` vs `--push` work with multi-platform images?
**A:** When building multi-platform images with buildx: (1) `--load` loads the result into the local Docker image store — but only the native platform variant is loaded (others are dropped), (2) `--push` pushes the multi-architecture manifest to a registry — all platforms are included, (3) If you need multi-platform images locally, use `docker buildx build --platform linux/amd64,linux/arm64 --output type=docker,dest=./output.tar .` for a combined archive, or run `docker buildx build --platform linux/amd64 -t image:amd64 --load` per platform. The `--load` option cannot store multiple platform variants in the local store.

## Q26: How does Docker's `resource reservation` (`--reserve-memory`, `--reserve-cpus`) differ from limits?
**A:** Resource limits (`--memory`, `--cpus`) cap resource usage — the container cannot exceed these. Resource reservations (`--reserve-memory`, `--reserve-cpus`) guarantee minimum resources — Docker ensures these resources are available when the container starts, preventing overcommit. In Compose: `deploy.resources.reservations.memory: 256M`. Reservations are only enforced in Swarm mode; in standalone mode, they are ignored. If a host cannot meet reservations, the container is not scheduled (Swarm) or the container may compete for resources (standalone).

## Q27: How does Docker's `tini` init process handle zombie reaping and signal forwarding?
**A:** The `tini` init process (`docker run --init`) runs as PID 1 inside the container. Functions: (1) reaps zombie processes (child processes that have exited but not been waited on), (2) forwards signals (SIGTERM, SIGINT, etc.) to the child process, (3) ensures the child receives the signal even if the child ignores it (tini waits and re-sends). Without an init process: (1) orphaned zombies accumulate (PID 1 in the container ignores SIGCHLD), (2) signals sent by Docker (via `docker stop`) may not reach child processes, (3) containers may not shut down cleanly when the application spawns children.

## Q28: How does Docker handle DNS resolution for containers on custom bridge networks, and what is embedded DNS?
**A:** Docker runs an embedded DNS server at 127.0.0.11 inside each container. On custom bridge networks: (1) containers can resolve each other by container name, (2) Docker automatically updates DNS records when containers start/stop, (3) the embedded DNS forwards external requests to the host's DNS or configured `--dns`. Limitations: (1) DNS resolution only works within the same network, (2) default bridge network does NOT support name resolution by container name (must use `--link`), (3) DNS caching inside containers may cause stale records (set `ndots:0` to avoid search domain expansion).

## Q29: How do you handle Docker volume migration between hosts with different filesystem layouts?
**A:** Options: (1) `docker volume create --driver local --opt type=nfs --opt o=addr=... --opt device=:/path/nfs` for shared NFS volumes, (2) use `docker run --rm -v myvolume:/source -v $(pwd):/destination alpine tar czf /destination/backup.tar.gz -C /source .` to tar the volume, then restore on target host, (3) use volume drivers like REX-Ray, Portworx, or Cloudstor for cross-host volume management, (4) for Docker Swarm, use `--mount type=volume,source=mydata,target=/data,volume-driver=rexray` with the same driver. Important: bind mounts (absolute paths) are NOT portable across hosts; named volumes are preferred for portable configurations.

## Q30: How does Docker's `cap-drop=ALL --cap-add=NET_BIND_SERVICE` impact application compatibility?
**A:** Dropping all capabilities and adding back only specific ones implements least-privilege security. `NET_BIND_SERVICE` allows binding to ports <1024. This pattern breaks applications that need: (1) raw sockets (ping with `NET_RAW`), (2) `ptrace` (debuggers with `SYS_PTRACE`), (3) changing time (with `SYS_TIME`), (4) mounting filesystems (with `SYS_ADMIN`), (5) `chown` (with `CHOWN`). Test your application's capability requirements with `docker run --rm --cap-drop=ALL myapp` and add capabilities gradually based on error logs. Most web applications only need `NET_BIND_SERVICE` and `NET_RAW`.

## Q31: How does Docker's `--shm-size` affect shared memory allocation for applications like PostgreSQL or Python multiprocessing?
**A:** Docker allocates 64MB of `/dev/shm` by default (tmpfs). Applications that use POSIX shared memory or System V shared memory: (1) PostgreSQL uses shared memory for shared buffers (SET shared_buffers) — insufficient `/dev/shm` causes "could not create shared memory segment" errors, (2) Python multiprocessing uses `/dev/shm` for shared state by default, (3) machine learning frameworks use shared memory for tensor passing. Increase with `--shm-size=2g` or mount a dedicated tmpfs: `--tmpfs /dev/shm:rw,noexec,nosuid,size=2g`. For Kubernetes, set `emptyDir.medium: Memory`.

## Q32: How does Docker's `--add-host` work, and what are the implications for DNS resolution ordering?
**A:** `--add-host` adds entries to the container's `/etc/hosts` file. Docker manages this file (you should not mount a custom `/etc/hosts`). Entries added this way: (1) take precedence over DNS resolution, (2) are managed by Docker's embedded DNS server (127.0.0.11), (3) persist across container restarts, (4) are unaffected by DNS resolver changes. In Docker Compose: `extra_hosts: - "host.docker.internal:host-gateway"` maps to the host's gateway IP. Limitations: (1) `/etc/hosts` is read-only after container start, (2) entries cannot be removed without restarting the container, (3) large numbers of entries slow down all DNS lookups.

## Q33: How does Docker's `docker cp` handle symbolic links and hard links?
**A:** `docker cp` copies from container to host or vice versa. Behavior: (1) from container to host: symbolic links are followed (the target file is copied, not the link itself), (2) from host to container: symbolic links are copied as-is (not followed), (3) hard links are not preserved — copied as separate files, (4) device files and special files may not copy correctly, (5) the format follows tar semantics. For preserving symlinks as symlinks when copying from container, use `docker cp --follow-link=false container:path host_path`. The `--archive` flag preserves permissions.

## Q34: How does Docker's `layer` size affect image pull performance across slow network connections?
**A:** Image pull time = sum of per-layer download times + extraction time. Each layer is downloaded, verified (checksum), decompressed, and extracted sequentially (with parallel decompression in newer Docker versions). For slow connections: (1) larger layers increase the chance of timeout failures, (2) `--platform` ensures only the needed platform's layers are pulled, (3) compressed size matters more than uncompressed — use `docker save` + `gzip` to compare, (4) layer reuse across images means shared layers only need to be pulled once, (5) configure registry mirrors (`--registry-mirror`) for caching, (6) use `docker pull --quiet` to reduce output overhead. Each layer has a fixed overhead for manifest parsing and checksum verification.

## Q35: How does Docker's `build history` pruning work, and why do dangling images accumulate?
**A:** Dangling images (tagged as `<none>:<none>`) accumulate when: (1) a new build with the same tag replaces an old one, (2) `docker rmi` removes a tag but leaves the underlying layer, (3) intermediate layers are not cleaned after a build. Cleanup: (1) `docker image prune` removes dangling images, (2) `docker system prune -a` removes ALL unused images (not just dangling), (3) `docker image prune --filter "until=24h"` removes images older than 24h. Layers may persist when referenced by another image — SQLite dependency chains prevent cleanup. Use `docker system df` to see space usage by type.

## Q36: How does Docker Compose's `extends` directive work for sharing configuration across services?
**A:** The `extends` directive in Docker Compose V1/V2 allows a service to inherit configuration from another service or a separate compose file: `services: web: extends: file: common.yml service: base-web`. The extending service can override any inherited field. Limitations: (1) `extends` cannot chain more than one level, (2) `depends_on` from the base service is copied but relationships to other services in the extending file must be re-declared, (3) not supported in Compose V3 for Swarm mode, (4) removed in Compose specification (use YAML anchors instead: `&base` and `<<: *base`). YAML anchors are more flexible and portable.

## Q37: How does Docker handle `IPv6` networking, and what configuration is required?
**A:** Docker supports IPv6 with specific daemon configuration: (1) Enable IPv6: `{ "ipv6": true, "fixed-cidr-v6": "2001:db8:1::/64" }` in daemon.json, (2) Docker automatically adds IPv6 NAT (using `ip6tables`), (3) Containers get IPv6 addresses from the configured subnet, (4) Default bridge does not support IPv6; use user-defined networks: `docker network create --ipv6 --subnet "2001:db8:1::/64" mynet`, (5) Port publishing maps both IPv4 and IPv6 by default. If the host kernel lacks IPv6 support or `ip6tables`, Docker disables IPv6 features. Test with `docker run --rm busybox ip -6 addr`.

## Q38: How do you configure Docker's `log` driver to avoid disk space exhaustion from log files?
**A:** Strategies: (1) Set per-container limits: `--log-opt max-size=10m --log-opt max-file=3` (rotates at 10MB, keeps 3 files), (2) Set daemon-wide defaults in daemon.json: `{ "log-driver": "json-file", "log-opts": { "max-size": "10m", "max-file": "3" } }`, (3) Use the `local` log driver (more efficient than `json-file`): `--log-driver local --log-opt max-size=10m`, (4) For high-volume logging, use `syslog` or `journald` to offload to system logging, (5) Configure logrotate for `/var/lib/docker/containers/` if not using log options, (6) Use `docker system prune` to clean up stopped container logs.

## Q39: How does Docker's `buildx` cache from `type=gha` work for GitHub Actions CI/CD?
**A:** `type=gha` (GitHub Actions cache backend) stores BuildKit cache in GitHub Actions' built-in caching layer. Setup: `docker buildx build --cache-from type=gha --cache-to type=gha,mode=max`. How it works: (1) cache is stored as a blob in GitHub's cache service, (2) keyed on the cache scope, (3) `mode=max` caches all intermediate layers (not just exported layers), (4) cache is scoped per-branch by default, (5) `--cache-from` pulls cache at the start of the build. Benefits: faster builds across workflow runs, no external cache storage needed. Limitations: GitHub's cache has a 10GB limit per repository, cache eviction is LRU.

## Q40: How does Docker handle overlapping CNI plugins (Docker's built-in vs Kubernetes)?
**A:** Docker and Kubernetes both manage container networking but are independent. When Docker is used as the container runtime for Kubernetes (deprecated as of k8s 1.24): (1) Docker creates the network namespace, (2) the CRI (containerd/cri-o) calls CNI plugins to configure networking, (3) Docker's own network configuration is ignored by Kubernetes, (4) containers started directly with Docker (outside k8s) use Docker's network; those started via k8s use CNI networks. This dual-network setup can cause confusion: containers on the same host may not reach each other if one uses Docker networks and the other uses CNI.

## Q41: How does Docker's `--network=host` affect container security and port conflicts?
**A:** `--network=host` removes network isolation — the container shares the host's network stack. Implications: (1) all host ports are directly accessible to the container (no port mapping needed), (2) the container can bind to any host port, causing conflicts with other services, (3) the container can sniff or interfere with host network traffic, (4) security: any process in the container sees all host network interfaces, (5) performance: zero network overhead, best for high-throughput applications, (6) DNS is fully host-dependent, (7) no container-level firewall isolation. Only use for applications that require raw network access (e.g., packet capture, performance-critical services).

## Q42: How does Docker's `container` network mode work for sidecar containers?
**A:** `--network=container:container_name` makes the new container share the network stack of an existing container. Both containers share: (1) the same IP address, (2) the same port mappings, (3) the same network interfaces, (4) the same /etc/hosts and DNS config, (5) they communicate via localhost. Common use cases: (a) sidecar proxy container sharing the app container's network, (b) network debugging containers without exposing ports, (c) service mesh sidecars (Envoy, Linkerd). The sidecar container does not need its own port mapping — it connects through the parent container's network.

## Q43: How do you perform a Docker engine version upgrade on a production Swarm cluster without downtime?
**A:** Rolling Docker engine upgrade on Swarm: (1) Drain the first manager node: `docker node update --availability drain manager1`, (2) Stop Docker on the drained node: `systemctl stop docker`, (3) Upgrade Docker packages: `apt-get update && apt-get install docker-ce` (or yum), (4) Restart Docker: `systemctl start docker`, (5) Wait for the node to rejoin the Swarm (show as Ready), (6) Set availability back: `docker node update --availability active manager1`, (7) Repeat for each manager node, (8) After all managers are upgraded, upgrade worker nodes. For Raft consistency, ensure one manager is always available. Never upgrade all managers simultaneously.

## Q44: How does Docker's `docker scan` (Snyk integration) detect vulnerabilities, and what are its limitations?
**A:** `docker scan` uses Snyk's vulnerability database to scan: (1) OS packages (Alpine apk, Debian apt, RHEL yum), (2) application dependencies (npm, pip, gem, maven, etc.), (3) the base image's CVE database. It compares installed package versions against known CVEs. Limitations: (1) only scans the final image (not intermediate layers), (2) requires internet access to Snyk's API, (3) false positives for CVEs not applicable to the image's runtime context, (4) no fix information for some transitive dependencies, (5) only free tier: limited scans per day, (6) does not detect misconfigurations or secrets in images. For comprehensive scanning, use Trivy or Grype (open-source, offline).

## Q45: How does Docker's `checkpoint` and `restore` feature work with CRIU?
**A:** `docker checkpoint` uses CRIU (Checkpoint/Restore In Userspace) to save and restore container state as a checkpoint. Capabilities: (1) saves full process state (CPU registers, memory pages, open file descriptors), (2) allows live migration of containers between hosts, (3) enables snapshot-based debugging. Requirements: (1) CRIU must be installed on the host, (2) kernel must support `CONFIG_CHECKPOINT_RESTORE`, (3) container must not have: host devices, GPU passthrough, or complex network configurations, (4) checkpoints are stored in Docker's data directory. Limitations: (1) not supported with runc on all kernels, (2) large checkpoints for memory-heavy containers, (3) TCP connections are not preserved, (4) experimental feature.

## Q46: How does Docker's `buildx` `--set` flag modify build arguments across multi-stage builds?
**A:** `docker buildx build --set "*.args.MYARG=value"` sets build arguments across all stages. The `--set` flag can modify: (1) `target.<stage>.args.<arg>=<value>` — per-stage args, (2) `target.<stage>.output=<type>` — per-stage output, (3) `name-pattern.args.<arg>=<value>` — by name pattern. This is useful for CI/CD where different stages need different configurations. Example: `--set "target.production.args.VERSION=1.0"` sets the VERSION arg only in the production stage. Without `--set`, all ARGs must be specified at the Dockerfile level or with `--build-arg`.

## Q47: How does Docker handle `oom_score_adj` and `--oom-kill-disable` for prioritizing container termination?
**A:** Docker sets `oom_score_adj` for containers based on memory limits. Default: (1) containers without memory limit: `oom_score_adj = 0`, (2) containers with memory limit: `oom_score_adj = -998` for the container's init process, (3) lower values mean the process is less likely to be killed by OOM killer. `--oom-kill-disable` prevents the kernel from killing the container's process when memory is exhausted — but only works with `--memory` set (otherwise, the kernel cannot constrain memory). If OOM kill is disabled and the container uses all memory, the system runs out of memory and the kernel eventually kills other processes or hangs. Use with extreme caution.

## Q48: How does Docker's `--device` flag expose host devices, and what security risks exist?
**A:** `--device=/dev/snd:/dev/snd` maps host devices into the container. Risks: (1) `/dev/sda` gives the container raw disk access (can read/write all partitions), (2) `/dev/mem` gives access to physical memory, (3) `/dev/dri` allows GPU access but also potential DMA attacks, (4) `/dev/net/tun` allows creating TUN/TAP network interfaces. Best practices: (1) always use `--device-cgroup-rule` to restrict access types (read/write/mknod), (2) avoid exposing block devices to containers, (3) for GPU access, prefer NVIDIA Container Toolkit (`--gpus`) which isolates the GPU device, (4) add `--security-opt no-new-privileges:true` to prevent privilege escalation.

## Q49: How does Docker's `docker build` with `--squash` (experimental) reduce image size, and what are the trade-offs?
**A:** `--squash` merges all filesystem layers into a single layer. Benefits: (1) eliminates intermediate layer artifacts (deleted files still occupy space in layers without squash), (2) smaller final image size, (3) reduced pull time. Trade-offs: (1) no layer caching for subsequent builds (entire image rebuild), (2) cannot push individual layers (no partial updates), (3) eliminates the ability to share layers with other images, (4) still experimental (not recommended for production). Alternative: use multi-stage builds with careful `COPY --from` to achieve the same result without squashing.

## Q50: How does Docker Swarm's `rolling update` order affect service availability?
**A:** Swarm supports two update orders: (1) `stop-first` (default): old task is stopped, then new task is started. Zero-downtime requires at least 2 replicas and a load balancer that detects the stopped task. (2) `start-first`: new task is started first, then the old task is stopped after the new one passes health checks. Requires more resources (extra replica during update) but better availability. Configure: `docker service create --update-order start-first myimage`. For zero-downtime with stop-first, set `--update-parallelism 1` and ensure the service has at least 2 replicas. Health checks ensure traffic is only routed to the new task after it passes.

## Q51: How does Docker's `--sysctl` flag modify kernel parameters inside a container?
**A:** `--sysctl` modifies kernel parameters in the container's network namespace. Settable parameters: (1) `net.*` — network settings (net.core.somaxconn, net.ipv4.tcp_tw_reuse), (2) `kernel.*` — kernel settings (not all, only namespaced ones), (3) `vm.*` — rarely namespaced. Parameters that cannot be namespaced (require `--privileged`): kernel.hostname (only in new PID ns), kernel.msg*, kernel.sem, fs.*, vm.* (on most kernels). For non-namespaced sysctls, consider if `--sysctl` is appropriate or if the default value suffices. In Docker Compose V2: `sysctls: net.core.somaxconn: 1024`.

## Q52: How does Docker's `buildx` `--provenance` flag work for SLSA compliance?
**A:** `docker buildx build --provenance=true` attaches provenance attestation (SLSA Provenance) to the image. The attestation is stored as an image manifest annotation and contains: (1) build metadata (builder ID, build config, build start/finish times), (2) materials (source repository, commit SHA), (3) builder identification (BuildKit version). Use `docker buildx imagetools inspect <image>:<tag> --format '{{ json .Provenance }}'` to view. SLSA Level 1 is achieved with source provenance; Level 2 requires signed provenance and a hosted builder. Combine with `--attest type=sbom` for SBOM generation.

## Q53: How does Docker's `--pids-limit` prevent fork bombs, and what happens when the limit is reached?
**A:** `--pids-limit=100` limits the number of processes (PIDs) a container can create. When the limit is reached: (1) `fork()`/`clone()` syscalls return `EAGAIN`, (2) the application receives "Resource temporarily unavailable", (3) the container cannot create new processes/threads, (4) existing processes continue running. This prevents fork bomb attacks and limits zombie accumulation. Default: unlimited (except with cgroup v2 where it's max by default). For thread-heavy applications (e.g., Apache with many worker threads), ensure the limit is high enough. In Kubernetes, pod-level PID limits prevent noisy-neighbor PID exhaustion.

## Q54: How does Docker's `docker tag` work with multi-architecture images?
**A:** `docker tag` modifies the image reference (repository:tag) pointing to a specific image manifest or index. For multi-architecture images: (1) the `latest` tag typically points to an image index (manifest list), (2) `docker tag source:latest target:latest` creates a new tag pointing to the same multi-arch index, (3) `docker tag source:linux-amd64 target:linux-amd64` tags a specific platform image, (4) tagging a multi-arch image preserves all platforms. However, `docker pull` by tag fetches the native platform variant; `docker pull --platform linux/arm64` fetches a specific platform. Use `docker buildx imagetools create --tag user/image:new user/image:old` to create multi-arch tags properly.

## Q55: How does Docker's `buildx` `--iidfile` help with CI/CD workflows?
**A:** `--iidfile=image-id.txt` writes the full image digest (sha256:...) to a file after the build. This is essential for CI/CD: (1) the digest uniquely identifies the built image, (2) subsequent jobs can reference the exact image by digest rather than tag (immutable reference), (3) avoids tag race conditions (pushing a new image with the same tag while another job reads the tag), (4) the iidfile contains the manifest digest for multi-arch images or single-arch digest. Usage: `docker buildx build --iidfile /tmp/iid . && docker push $(cat /tmp/iid)`.

## Q56: How does Docker's `docker history` show the true size impact of each layer?
**A:** `docker history --no-trunc myimage` shows each layer with: (1) CREATED BY (the command), (2) SIZE (the layer's actual size, not cumulative). The SIZE column shows the unique bytes added by that layer. A layer that deletes files may have a negative size (if the deletion removes more bytes than the layer adds). The `--no-trunc` flag shows full command text. The `--human` flag shows sizes in human-readable format. Note: `docker history` does NOT show the compressed size (which matters for network transfer). For compressed size, inspect the image manifest.

## Q57: How does Docker's `docker system df` report detailed disk usage?
**A:** `docker system df --verbose` reports: (1) Images: count, active, size, reclaimable, (2) Containers: count, size, reclaimable, (3) Local Volumes: count, size, reclaimable, (4) Build Cache: count, size, reclaimable. `--verbose` breaks down by image and shows shared vs unique sizes. Shared = 0 means the image shares layers with other images. Reclaimable = the space freed if all unused images/containers are removed. `docker system df --format '{{.Type}}\t{{.TotalCount}}'` provides machine-readable output for monitoring.

## Q58: How does Docker's `--cgroup-parent` affect resource accounting in systemd-based systems?
**A:** `--cgroup-parent` sets the parent cgroup for the container. In systemd systems: (1) the default parent is `/docker`, (2) setting `--cgroup-parent=/myslice` places the container under a custom cgroup slice, (3) this integrates with systemd's resource control (CPU, memory, I/O via `systemctl set-property`), (4) systemd slices allow hierarchical resource limits. Use case: running Docker containers alongside systemd services with coordinated resource allocation. For example: `docker run --cgroup-parent=/system.slice/docker-custom.slice mycontainer` allows `systemctl set-property docker-custom.slice MemoryLimit=1G`.

## Q59: How does Docker's `DOCKER_BUILDKIT=0` disable BuildKit, and why would you use the legacy builder?
**A:** `DOCKER_BUILDKIT=0 docker build` uses the legacy builder instead of BuildKit. Reasons to use legacy builder: (1) compatibility with older Docker versions (BuildKit requires Docker 18.09+), (2) some third-party Docker build integrations do not support BuildKit, (3) known issues with specific proxy configurations, (4) simpler debugging (single-threaded). Disadvantages: no parallel builds, no cache mounts, no secret mounts, slower builds, no --no-cache-filter. Always prefer BuildKit for new projects. Set `{ "features": { "buildkit": true } }` in daemon.json to make BuildKit the default.

## Q60: How does Docker's `network` `--internal` flag isolate containers from external networks?
**A:** `docker network create --internal mynet` creates a network with no external connectivity. Containers on this network: (1) can communicate with each other, (2) CANNOT access the internet or host network, (3) CANNOT be accessed from outside (even with port publishing), (4) DNS only resolves internal names. Useful for: (a) database networks that should not be exposed, (b) processing containers that should not exfiltrate data, (c) isolated testing environments. Combine with a proxy container on another network for controlled external access.

## Q61: How does Docker's `bridge` network `iptables` rules change when containers are started/stopped?
**A:** Docker dynamically updates iptables rules when containers start/stop. On container start: (1) a DNAT rule is added to the PREROUTING chain (for port publishing), (2) a rule is added to the FORWARD chain allowing traffic to the container's IP, (3) a rule is added to the DOCKER chain for the container. On container stop: all related rules are removed. Race condition: during rapid start/stop cycles, old rules may persist briefly, causing port conflicts. The `--iptables=false` daemon option disables automatic iptables management, requiring manual configuration (not recommended for most setups).

## Q62: How does Docker's `buildx` `--no-cache-filter` work for selective cache invalidation?
**A:** `--no-cache-filter=stage1,stage2` invalidates the build cache for specific stages while preserving cache for others. Without this flag, `--no-cache` invalidates ALL stages. Use case: invalidate the `npm-install` stage to force fresh dependency downloads while keeping the `base` stage cached. Syntax: `docker buildx build --no-cache-filter="install-deps" -t myimage .`. The filter targets stage names (with `AS stage_name` in Dockerfile). This is a BuildKit-specific feature not available with the legacy builder.

## Q63: How does Docker's `pause` and `unpause` affect process states and network connections?
**A:** `docker pause` uses cgroups freezer to suspend all processes in the container (SIGSTOP equivalent). All processes freeze: (1) no CPU consumption, (2) memory is preserved, (3) open network connections remain open but data stops flowing — TCP connections may time out from the remote side, (4) container is not accessible via published ports. `docker unpause` sends SIGCONT, resuming all processes from their frozen state. Long pauses (minutes+) will cause TCP keepalive timeouts and connection drops. Pause is useful for: (a) taking snapshots, (b) investigating state without process interference, (c) resource management (pausing idle containers).

## Q64: How does Docker's `--blkio-weight` and `--device-write-bps` control disk I/O fairness?
**A:** `--blkio-weight=500` (range 10-1000) sets proportional I/O weight relative to other containers (higher = more I/O bandwidth). `--device-write-bps=/dev/sda:10mb` limits write throughput to a specific device. `--device-read-iops` and `--device-write-iops` limit IOPS. These use the CFQ (or BFQ) I/O scheduler on the host. If the host uses the none/deadline scheduler (common on SSDs), blkio-weight has NO effect. For SSDs, use the `bfq` scheduler: `echo bfq > /sys/block/sda/queue/scheduler`. I/O limits work at the block layer and cannot distinguish between random vs sequential I/O.

## Q65: How does Docker's `--restart=on-failure:N` handle restart loops, and what is the exponential backoff?
**A:** `--restart=on-failure:3` restarts the container up to 3 times after it exits with non-zero status. Between restarts, Docker applies exponential backoff: 100ms, 200ms, 400ms, 800ms, up to a max of 5 minutes. If the container remains running for more than 10 seconds between failures, the retry counter resets. After exhausting max retries, the container stays stopped. Docker uses `dockerd`'s restart manager; if the daemon restarts, the restart policy is re-evaluated. For `always`, the container restarts indefinitely with backoff.

## Q66: How does Docker's `docker export` vs `docker save` handle image metadata differently?
**A:** `docker save myimage:tag -o image.tar` saves the full image with layers, metadata, and tag history. It produces a tar that can be loaded back with `docker load`, preserving the exact image. `docker export mycontainer -o container.tar` exports the container's filesystem as a flat tar, losing: (1) all layer history, (2) image metadata (EXPOSE, ENV, CMD, etc.), (3) only the merged filesystem (one layer). Export is useful for: (a) inspecting the filesystem, (b) migrating data between systems, (c) creating filesystem snapshots. Exported tars cannot be `docker load`ed — they must be `docker import`ed, creating a new image with no history.

## Q67: How does Docker's `docker commit` create images from containers, and what are the best practices?
**A:** `docker commit container_id myimage:tag` creates a new image from a container's current state (including all filesystem changes). Best practice: (1) prefer Dockerfiles (repeatable, documented), (2) use commit only for debugging or ephemeral snapshots, (3) `--change` flag can add Dockerfile instructions: `--change='CMD ["app"]'`, `--change='ENV MYVAR=value'`, (4) commit pauses the container briefly, (5) the commit includes all mounted volumes by default (use `--include-volumes=false` to exclude), (6) committing a running container creates an inconsistent image if files are being modified. Never use commit for production CI/CD; always use Dockerfiles.

## Q68: How does Docker's `docker context` manage multiple Docker endpoints?
**A:** `docker context` allows switching between multiple Docker endpoints (local, remote, cloud). Commands: (1) `docker context create myremote --docker "host=ssh://user@host"`, (2) `docker context use myremote`, (3) `docker context ls`, (4) `docker context inspect`. Contexts can point to: (a) local socket (`unix:///var/run/docker.sock`), (b) remote TCP (`tcp://host:2375`), (c) SSH (`ssh://user@host`), (d) AWS ECS, (e) Azure ACI. Use `DOCKER_CONTEXT=myremote docker ps` for per-command switches. Contexts store TLS certificates and endpoint configuration. This is the recommended way to manage multiple Docker hosts without wrapper scripts.

## Q69: How does Docker's `--security-opt seccomp=unconfined` affect container security?
**A:** `--security-opt seccomp=unconfined` disables seccomp filtering, allowing the container to make ALL system calls. Security implications: (1) the container can use syscalls that are normally blocked (e.g., `ptrace`, `perf_event_open`, `kexec`, `bpf`), (2) this reduces the barrier for kernel privilege escalation attacks, (3) Docker's default seccomp profile blocks about 44 dangerous syscalls out of ~300, (4) unconfined seccomp should only be used for applications that need rare syscalls (certain debuggers, performance tools, custom kernel modules). Better approach: create a custom seccomp profile that only allows needed syscalls.

## Q70: How does Docker's `docker buildx build --output type=local` generate filesystem output?
**A:** `--output type=local,dest=./output` writes the built filesystem to a local directory instead of creating a Docker image. This produces: (1) the final stage's filesystem as plain files/directories, (2) no Docker metadata (not importable), (3) good for generating static sites, binaries, or artifacts without creating a Docker image. Combined with multi-stage builds: (a) the build stage compiles code, (b) the output is extracted to host, (c) no runtime Docker image is needed. Use case: CI/CD where you want to build artifacts using Docker's reproducible environment but deploy them outside Docker.

## Q71: How does Docker's `--env-file` handle variable expansion and special characters?
**A:** `--env-file` reads environment variables from a file (one KEY=value per line). Rules: (1) lines starting with `#` are comments, (2) blank lines are ignored, (3) `KEY` without `=value` is considered unset, (4) no variable expansion — `KEY=$OTHER` is literal, (5) values can contain quotes — they are part of the value, (6) leading/trailing whitespace in values is trimmed unless quoted, (7) binary values are not supported. For expansion, use shell: `env $(cat file.env | xargs) docker run ...`. Protect secrets by ensuring the env file has restricted permissions (`chmod 600`).

## Q72: How does Docker's `swarm` `--replicas-max-per-node` limit service distribution?
**A:** `docker service create --replicas-max-per-node 2 myimage` limits how many replicas of a service can run on a single node. This ensures: (1) service placement across multiple nodes for fault tolerance, (2) no single node runs all replicas, (3) during rolling updates, at most 2 new tasks per node. Default: unlimited. Use this to enforce: (a) spreading load across a cluster, (b) reducing blast radius (node failure affects fewer replicas), (c) resource fairness across nodes. Combine with `--constraint node.labels.zone==us-east` for zone-aware placement.

## Q73: How does Docker's `docker scout` (Docker Scout) analyze image vulnerabilities differently from traditional scanners?
**A:** Docker Scout (successor to Docker Scan) uses a different approach: (1) analyzes the full dependency tree including transitive dependencies, (2) evaluates whether a vulnerable function is actually callable from the application (reachability analysis), (3) provides policy-based governance (fail builds based on policy), (4) continuously monitors images even after deployment, (5) integrates with the Docker CLI, Docker Hub, and Docker Desktop. It uses provenance data (SBOM) attached to images. The key difference: traditional scanners report all CVEs; Scout prioritizes CVEs that are actually exploitable in your image's context.

## Q74: How does Docker's `--init` flag handle zombie processes that are not direct children of PID 1?
**A:** `--init` runs tini as PID 1. Tini handles: (1) `SIGCHLD` — when a child process exits, tini's signal handler catches it and reaps zombies by calling `waitpid()`, (2) processes that reparent to init (when their parent dies before they do) are automatically reaped, (3) orphaned process groups are cleaned up when the main process exits. Without an init: (1) Docker runs the application as PID 1, (2) Linux PID 1 has a special property: orphaned processes are automatically reparented to it, (3) but if the application does not `wait()` for these orphans, they become zombies in the container, (4) zombie accumulation eventually exhausts the PID limit.

## Q75: How does Docker's `--add-host` interact with Docker's embedded DNS?
**A:** `--add-host` entries are written to `/etc/hosts` inside the container. Docker's embedded DNS (at 127.0.0.11) resolves names in this order: (1) `/etc/hosts` is checked first (via `nsswitch.conf`), (2) if not found, the embedded DNS resolver checks Docker's internal DNS (container names, service names), (3) if still not found, it queries the external DNS servers. The embedded DNS does NOT serve the `/etc/hosts` entries to other containers. For service discovery, use Docker's DNS (container/service names) instead of `/etc/hosts`. Mixing `/etc/hosts` entries with Docker DNS can cause confusion because `/etc/hosts` is static.

## Q76: How does Docker's `audit` logging for Docker daemon activities work?
**A:** Docker daemon audit logging requires configuring the system's audit daemon (auditd). Rules example: `-w /usr/bin/docker -p rwxa -k docker` (audit Docker binary), `-w /var/lib/docker -p rwxa -k docker-lib` (audit Docker data directory). Audit logs capture: (1) all Docker CLI invocations, (2) container file modifications, (3) configuration changes. Docker does not have native audit logging of API calls (unlike Kubernetes audit logs). For API audit: (1) use a reverse proxy that logs all requests to the Docker socket/API, (2) use Docker Enterprise's built-in auditing (if applicable), (3) configure dockerd with `--log-level=debug` and parse logs externally.

## Q77: How does Docker's `buildx` cache from `type=s3` work for shared CI/CD cache?
**A:** `--cache-to type=s3,bucket=my-bucket,region=us-east-1,prefix=buildkit-cache/` stores BuildKit cache in an S3 bucket. `--cache-from type=s3,bucket=my-bucket,region=us-east-1,prefix=buildkit-cache/` retrieves it. Benefits: (1) shared cache across all CI runners, (2) persists beyond GitHub Actions cache limits, (3) cache is incremental (only changed layers are uploaded), (4) supports `mode=max` for full cache. Requires AWS credentials configured in the environment. The cache is stored as individual layer blobs — S3 costs are minimal for most projects. Cache keys can include branch or build ID for isolation.

## Q78: How does Docker's `--pid=host` mode affect container process visibility?
**A:** `--pid=host` makes the container share the host's PID namespace. The container: (1) can see all host processes (`ps aux` shows all), (2) can signal any host process (kill, stop), (3) can access `/proc/<pid>/` of all processes (including reading environment variables with secrets). Security: (1) extremely dangerous — gives the container full visibility into all processes, (2) enables process injection attacks, (3) breaks tenancy between containers. Only use for: (a) debugging tools (htop, strace), (b) monitoring agents that need to observe all processes, (c) container runtimes that need to manage child containers.

## Q79: How does Docker's `docker compose` V2 YAML type resolution differ from V1 for environment variables?
**A:** Docker Compose V2 (docker compose, no hyphen) handles environment variable types differently: (1) V1 always treated environment values as strings, (2) V2 preserves types: `key: 123` becomes integer, `key: true` becomes boolean, (3) This causes issues if the container expects a string — `MYVAR=true` becomes boolean `true` in YAML parsing, passed as `MYVAR: True` (capitalized). Fix: explicitly use quotes in Compose file: `key: "123"`, `key: "true"`. The `--no-ansi` flag does not affect this. Use consistent quoting in env definitions to avoid type coercion surprises.

## Q80: How does Docker's `docker build` `--network=host` allow building images with localhost dependencies?
**A:** `--network=host` during build gives the build container access to the host's network stack. Use cases: (1) connecting to a local database or service on `localhost`, (2) accessing private repositories on the host's VPN, (3) running integration tests during build that need `localhost` services. Risks: (1) the build can access any host service, (2) the build can potentially interfere with host networking, (3) network isolation is removed for the entire build. Only use in trusted CI environments. In BuildKit: use `--network host` docker/build-push-action@v5 in GitHub Actions.

## Q81: How does Docker's `healthcheck` `--interval` and `--timeout` interaction affect container readiness reporting?
**A:** The health check lifecycle: (1) Docker runs the health command every `--interval` (default 30s), (2) each run gets `--timeout` (default 30s) to complete — if it exceeds, the check is considered failed, (3) after `--retries` (default 3) consecutive failures, the container is marked `unhealthy`, (4) `--start-period` (default 0s) gives the container time to start before health checks begin. A health check that takes longer than `--timeout` is killed. For slow-starting apps, set `--start-period=60s` and `--retries=6`. For quick checks (e.g., HTTP ping), reduce `--timeout=5s` to surface failures faster.

## Q82: How does Docker's `docker swarm init --force-new-cluster` recover from Raft consensus loss?
**A:** `--force-new-cluster` creates a new Swarm cluster from a single manager node's state, discarding all other manager nodes' data. This is used when: (1) a majority of manager nodes are permanently lost, (2) Raft consensus cannot be reached (cluster is stuck), (3) manager nodes have conflicting logs. The command: (a) initializes a new Swarm with just the local node as manager, (b) preserves existing services and containers on this node, (c) discards all other managers' state. After recovery: (1) add new manager nodes: `docker swarm join --token <manager-token>`, (2) services will be re-scheduled but only on available nodes. Services will lose state about tasks on lost nodes.

## Q83: How does Docker's `data` directory (`/var/lib/docker`) structure differ between storage drivers?
**A:** The directory structure depends on the storage driver:
- overlay2: `/var/lib/docker/overlay2/<layer-hash>/` (layers as dirs with `diff`, `link`, `lower` files)
- aufs: `/var/lib/docker/aufs/<layer-hash>/` and `/var/lib/docker/aufs/mnt/<container-id>/`
- devicemapper: `/var/lib/docker/devicemapper/mnt/<container-id>/` (thin pool metadata in JSON)
- btrfs: `/var/lib/docker/btrfs/subvolumes/`
All drivers share: `containers/` (container metadata), `image/` (image metadata), `volumes/` (named volumes), `network/` (network config), `swarm/` (Swarm state), `tmp/` (temp files), `buildkit/` (BuildKit cache).

## Q84: How does Docker's `--security-opt label=disable` work with SELinux?
**A:** `--security-opt label=disable` disables SELinux labeling for the container. Without this: (1) Docker applies SELinux labels to container processes (`svirt_lxc_net_t`), (2) container content is labeled with `svirt_sandbox_file_t`, (3) SELinux blocks container processes from accessing host resources with different labels, (4) even as root inside the container, the process is restricted by SELinux policy. Disabling SELinux labels removes this protection layer. Alternative: `--security-opt label=type:my_custom_t` uses a custom SELinux type. Proper SELinux + Docker is recommended for multi-tenant hosts.

## Q85: How does Docker's `docker image prune -a` interact with images referenced by running containers?
**A:** `docker image prune -a` removes ALL images not referenced by ANY container (including stopped containers). It will NOT remove: (1) images currently in use by running containers, (2) images used by paused containers, (3) images that are tagged but the tag points to an image layer that another image depends on. After pruning: (1) stopped containers may become unable to start (their image is gone), (2) `docker history` shows `<missing>` for removed intermediate layers, (3) rebuilding from Dockerfile re-fetches the base image. Use `docker container prune` first to identify containers that can safely be removed.

## Q86: How does Docker's `--security-opt no-new-privileges:true` prevent privilege escalation?
**A:** `no-new-privileges:true` sets the `NO_NEW_PRIVS` process flag, which prevents: (1) `setuid` binary execution from gaining the binary's owner privileges, (2) `setgid` binary execution from gaining group privileges, (3) Linux capabilities from being gained via `capset()` syscall, (4) `user namespaces` from escalating (e.g., via bubblewrap or similar), (5) any future privilege escalation vector based on credential changes. This is one of the most important security hardening options. Combined with `--cap-drop=ALL` and `--read-only`, it creates a highly restricted container. Most applications work fine with this flag.

## Q87: How does Docker's `buildx` `--cache-to type=registry`, `--cache-from type=registry` affect image layer publishing?
**A:** Registry cache mode stores cache alongside the image in the container registry. `--cache-to type=registry,ref=myimage:cache,mode=max` pushes build cache layers to the registry. `--cache-from type=registry,ref=myimage:cache` pulls them at the start. Benefits: (1) cache is automatically distributed across all CI runners, (2) no external cache storage needed, (3) works with any OCI-compatible registry. Trade-offs: (1) increases image push/pull time (cache layers are additional), (2) increases registry storage costs, (3) `mode=max` pushes ALL intermediate layers (larger cache), `mode=min` pushes only exported layers (smaller cache, less effective). Registry cache is the simplest shared cache option.

## Q88: How does Docker's `docker network connect` and `docker network disconnect` affect running containers?
**A:** `docker network connect mynet container` attaches a network interface to a running container. This: (1) creates a new virtual ethernet (veth) pair, (2) attaches one end to the container's network namespace, (3) attaches the other end to the bridge, (4) the container gets a new IP on the network, (5) the container can now communicate on both original and new networks. `docker network disconnect mynet container` removes the interface. Limitations: (1) you cannot disconnect the last network (container must have at least one network), (2) disconnecting the default bridge or host network leaves the container with no networking, (3) container's `/etc/hosts` is NOT updated with new network names.

## Q89: How does Docker's `--dns-opt` flag affect DNS resolution behavior?
**A:** `--dns-opt` passes options to the container's `/etc/resolv.conf`. Common options: (1) `ndots:0` — don't append search domains for names with at least one dot, (2) `ndots:5` — default, append search domains for names with fewer than 5 dots, (3) `timeout:2` — DNS query timeout in seconds, (4) `attempts:3` — number of DNS query attempts, (5) `rotate` — rotate DNS servers, (6) `single-request` — use single-socket UDP for A and AAAA queries (fixes some DNS timeout issues on IPv6 networks). Example: `docker run --dns-opt ndots:0 --dns-opt timeout:3`. Docker's embedded DNS (127.0.0.11) handles these options transparently.

## Q90: How does Docker's `docker buildx build --compression` flag control layer compression?
**A:** `--compression=zstd` sets the compression algorithm for exported layers. Options: (1) `gzip` (default) — widely compatible, good compression, (2) `zstd` — better compression ratio and faster decompression than gzip, but requires BuildKit 0.10+ on the pulling side, (3) `estargz` — seekable gzip for lazy pulling (stargz), (4) `uncompressed` — no compression (faster build but larger). `--compression-level=N` controls compression level (0-22, higher = better compression but slower). For CI/CD, `zstd` significantly reduces push/pull time. For registry compatibility, `gzip` is safest.

## Q91: How does Docker's `docker run --entrypoint` interact with the SHELL instruction in Dockerfile?
**A:** `--entrypoint` overrides the Dockerfile's ENTRYPOINT. The SHELL instruction (default: `/bin/sh -c`) affects how `RUN`, `CMD`, and `ENTRYPOINT` are executed when using shell form. When you use `--entrypoint /bin/bash` (shell form), the SHELL is not prepended because exec form is used. When using `--entrypoint` with shell form text (no brackets): `--entrypoint "myapp --flag"`, Docker runs it via the shell: `/bin/sh -c "myapp --flag"`. To bypass the shell, always use JSON/exec form for entrypoint. The SHELL instruction only affects Dockerfile operations, not runtime overrides.

## Q92: How does Docker's `--cpus` flag convert to cgroup CPU quotas (CFS)?
**A:** `--cpus=1.5` sets: (1) `cpu.cfs_period_us = 100000` (default 100ms), (2) `cpu.cfs_quota_us = 150000` (150ms per 100ms period = 1.5 CPUs). The quota determines how much CPU time the container gets per period. If the container uses its quota before the period ends, it is throttled until the next period. For latency-sensitive workloads: (1) increase the period (`--cpu-period=50000`) for finer-grained scheduling, (2) `--cpu-quota` alone (without `--cpus`) allows custom period, (3) CFS bandwidth control is per-CPU group, not per-core. On cgroup v2: `cpu.max` file stores `$MAX $PERIOD`.

## Q93: How does Docker's `docker stack deploy` handle configuration changes for existing services?
**A:** `docker stack deploy` with an updated compose file: (1) detects changes by comparing the desired state (compose file) with the current state (running services), (2) creates new services that don't exist, (3) updates existing services with changed configuration, (4) removes services no longer in the compose file, (5) performs rolling updates for changed services. Configuration diff: Docker does NOT show a diff; it applies the entire new configuration. If you change only one service's image, only that service is updated. Stack name is derived from the compose file's directory name or `-p` flag.

## Q94: How does Docker's `--device-cgroup-rule` allow granular device access without --privileged?
**A:** `--device-cgroup-rule` adds rules to the device cgroup for a container without giving full device access. Format: `--device-cgroup-rule="b 8:0 rwm"` (type: block, major: 8, minor: 0, permissions: read/write/mknod). This allows: (1) access to specific device numbers, (2) limited to allowed operations (r=read, w=write, m=mknod), (3) no other devices beyond the whitelist are accessible. This is more secure than `--privileged` (which allows all devices) and more flexible than `--device` (which only maps specific device paths). Use to give a container access to a specific partition or device without full device access.

## Q95: How does Docker's `docker logs --since` and `--until` filter log output efficiently?
**A:** `docker logs --since 2024-01-01T00:00:00Z --until 2024-01-02T00:00:00Z` filters logs by timestamp. For the `json-file` driver: (1) Docker reads the log file sequentially, parsing each JSON line's timestamp, (2) it skips lines until `--since` is reached, (3) stops when `--until` is passed, (4) for large log files, this is O(N) scan from the beginning. For efficient filtering: (1) use `--tail N` to read only the last N lines, (2) use Docker's log driver that supports indexing (journald, splunk), (3) for production log analysis, use external log shippers (Fluentd, Logstash) that index logs, (4) the `--timestamps` flag adds timestamps to each output line.

## Q96: How does Docker's `docker build` `--build-context` allow accessing external directories during build?
**A:** `--build-context mycontext=../path` makes a directory available during build as a named context. In Dockerfile: `COPY --from=mycontext /source /target`. Unlike `COPY .`, this: (1) allows access to files outside the build context, (2) does not invalidate Dockerfile cache when the external directory changes, (3) can point to local dirs, git repos (`--build-context repo=git@github.com:user/repo.git`), or S3 buckets (`--build-context s3=my-bucket/path`). Available only with BuildKit. This replaces the need to copy files from the host into the build context for multi-project builds.

## Q97: How does Docker's `docker compose up --quiet-pull` affect the output, and when should you use it?
**A:** `--quiet-pull` suppresses the pull progress output, showing only errors. Use cases: (1) CI/CD pipelines where log noise is unwanted, (2) production deployments where only errors matter, (3) automated scripts where clean output is needed. Without `--quiet-pull`, `docker compose up` shows detailed per-layer pull progress for each image. With it, only service name and pull result are shown. Combine with `--wait` to wait for containers to become healthy before returning. The `--no-ansi` flag further cleans output for log aggregation systems.

## Q98: How does Docker's `--oom-score-adj` prioritize OOM killer selection between containers?
**A:** `--oom-score-adj` (range -1000 to 1000) adjusts the OOM score adjustment for the container's processes. The kernel OOM killer selects the process with the highest OOM score (based on memory usage + score adjustment). Default: (1) `docker run` without memory limit: `oom_score_adj = 0`, (2) `docker run` with memory limit: `oom_score_adj = -998` for the init process. Setting `--oom-score-adj=-500` makes the container LESS likely to be killed; `500` makes it MORE likely. Use for: (a) critical services that should not be OOM-killed (set low/negative), (b) non-essential services that can be sacrificially killed first.

## Q99: How does Docker's `docker buildx build --sbom` generate Software Bill of Materials?
**A:** `--sbom=true` generates an SBOM (Software Bill of Materials) in SPDX format during the build. The SBOM: (1) lists all packages installed in the image (OS packages, language dependencies), (2) includes versions and licenses, (3) is stored as an attestation attached to the image manifest, (4) can be viewed with `docker buildx imagetools inspect <image> --format '{{ json .SBOM }}'`. Generation uses Syft internally. Combine with `--provenance` for a complete supply chain artifact. The SBOM is separate from the image layers — it does not increase image size.

## Q100: How does Docker's `docker context create --docker host=unix:///run/user/1000/docker.sock` support rootless Docker integration?
**A:** Rootless Docker runs the daemon as a non-root user and exposes the socket at `/run/user/$UID/docker.sock`. Creating a context pointing to this socket: (1) allows rootless Docker access from a rootful Docker client, (2) enables mixed workflows where some contexts use rootful and others rootless, (3) requires the user to have proper socket permissions. Use case: running a rootless Docker daemon for unprivileged builds while the system Docker runs as root for production. Context switching: `docker context use rootless` or per-command `--context rootless`.
