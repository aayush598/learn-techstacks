# Git Interview Questions and Answers - Part 2

## Q1: How does Git's packfile compression work with delta chains?
**A:** Git packfiles use delta compression to store objects efficiently. When creating a packfile, Git groups similar objects together (based on heuristics like file extension and name), then computes binary deltas between them. Each delta stores the difference between a newer version and a base object. The base object is stored as-is, and subsequent versions are stored as deltas. The `git repack` command controls how deep delta chains can go (`--depth=50` default) and how far Git searches for delta candidates (`--window=10`). Shallow delta chains improve read performance (less chain traversal) at the cost of packfile size. Deep chains save space but slow down object access. Use `git verify-pack -v` to inspect delta chains.

## Q2: How do you use `git rev-parse` for advanced commit resolution?
**A:** `git rev-parse` resolves Git references to commit hashes. Advanced uses: (1) `git rev-parse HEAD` -- current commit hash. (2) `git rev-parse --abbrev-ref HEAD` -- current branch name. (3) `git rev-parse --symbolic-full-name @{upstream}` -- upstream branch ref. (4) `git rev-parse --verify HEAD^{tree}` -- tree object of HEAD. (5) `git rev-parse --git-dir` -- path to .git directory. (6) `git rev-parse --is-inside-work-tree` -- check if in a Git repo. (7) `git rev-parse --show-toplevel` -- root of the repo. (8) `git rev-parse --verify --quiet master^{commit}` -- validate that master is a valid commit. (9) Critical for scripting: parsing revisions, detecting repo state, generating dynamic refs.

## Q3: How does `git bisect` work internally with the skiplist feature?
**A:** `git bisect` uses binary search to find the commit that introduced a bug. Internally: (1) Git marks good and bad commits. (2) It creates a graph and computes the midpoint commit that splits the remaining commits as evenly as possible. (3) The user tests that commit. (4) Git narrows the search to the remaining half. (5) The `--first-parent` flag follows only the first parent of merge commits (useful for trunk-based development where merge commits represent feature integrations). (6) The `--skiplist` feature accepts a file of commits to skip (flaky commits that can't be tested). (7) `git bisect run <script>` automates the process: the script should return 0 (good) or 1-127 (bad). (8) `git bisect visualize` shows the remaining range. (9) At the end, Git shows the first bad commit with full details.

## Q4: How do you implement a custom Git merge driver for conflict resolution?
**A:** Custom merge drivers handle specific file types during merges: (1) Configure the driver in `.gitconfig`: `[merge "custom"]; name = custom merge driver; driver = custom-merge %O %A %B %L %P`. (2) Assign the driver in `.gitattributes`: `*.json merge=custom`. (3) The driver receives: `%O` (ancestor), `%A` (current branch), `%B` (other branch), `%L` (conflict marker size), `%P` (path). (4) The driver must exit with 0 if the merge was resolved, non-zero if conflicts remain. (5) Use cases: JSON merge (merge object keys rather than line-based), XML merge (element-aware), lock file merge (PNPM, Yarn), changelog merge (append entries rather than conflict). (6) Example: a JSON merge driver uses `jq -s add` to merge two JSON objects at the top level. (7) Custom drivers dramatically reduce manual conflict resolution for structured file formats.

## Q5: How do you implement `git diff` with external diff tools (custom diff drivers)?
**A:** Custom diff drivers: (1) Configure in `.gitconfig`: `[diff "csv"]; textconv = csv-formatter; cachetextconv = true`. (2) Assign in `.gitattributes`: `*.csv diff=csv`. (3) `textconv` transforms the file before diffing -- CSV formatter normalizes whitespace, sorts rows, etc. (4) For binary files: `[diff "pdf"]; textconv = pdftotext %F; cachetextconv = true`. (5) Binary diff drivers convert binary to text for meaningful diffs. (6) Cache text conversions with `cachetextconv = true` to avoid re-converting on each diff. (7) Advanced: custom diff algorithm via `[diff "custom"]; command = custom-diff-tool %O %A %B`. (8) Use `GIT_EXTERNAL_DIFF` environment variable for a global external diff tool. (9) `git difftool` with `--tool=your-tool` (meld, beyondcompare, vimdiff) for visual diffing.

## Q6: How does `git rebase --onto` work and when would you use it?
**A:** `git rebase --onto` allows moving a series of commits from one base to another. Syntax: `git rebase --onto <newbase> <upstream> <branch>`. (1) It takes commits from `<upstream>..<branch>` and applies them onto `<newbase>`. (2) Use case: you branched from `master` at commit A, but you want to rebase onto `release` instead: `git rebase --onto release master feature`. (3) Another use: you applied hotfix commits on `master` instead of the hotfix branch. Move them: `git rebase --onto hotfix master~3 hotfix`. (4) Splitting a branch: take the last N commits from `feature` and rebase them onto a new branch: `git rebase --onto new-base feature~N feature`. (5) Cherry-picking a range: similar to cherry-pick but rebase-style. (6) The key difference from regular rebase: you explicitly specify the target base, not just the upstream.

## Q7: How does `git replace` enable history grafting and object replacement?
**A:** `git replace` allows replacing one Git object with another without rewriting history. (1) Creates a replace ref in `.git/refs/replace/`. (2) Git transparently uses the replacement when reading the object. (3) Use cases: (a) Grafting histories -- connect a repository to a new root commit: `git replace <new_root_commit> <original_orphan_commit>`. (b) Fixing corrupted commits without rewriting history for collaborators. (c) Simulating merges: create a merge commit that connects two histories. (4) `git replace --list` shows active replacements. (5) `git replace --delete` removes a replacement. (6) Replacements are local-only by default -- use `--push` and `--fetch` to share. (7) Clone with `--reference-if-able` to apply replacements. (8) Replacements can be used to consolidate multiple repositories into a unified history view.

## Q8: How do you use `git worktree` for parallel development workflows?
**A:** `git worktree` allows checking out multiple branches simultaneously: (1) `git worktree add ../feature-branch feature-branch` -- creates a new working directory linked to the same repository. (2) Each worktree has its own working directory and index but shares the Git object database. (3) Use cases: (a) Work on a hotfix without stashing current changes. (b) Run tests on a different branch while coding on the current one. (c) Review PRs by checking out the PR branch in a separate worktree. (d) Maintain a `gh-pages` build output in a separate worktree. (4) `git worktree list` shows all worktrees. (5) `git worktree prune` cleans up worktree references after deletion. (6) `git worktree remove <path>` removes a worktree. (7) Each worktree has its own `HEAD` and `refs/worktree/` namespace for per-worktree branches. (8) IMPORTANT: you cannot have the same branch checked out in two worktrees simultaneously.

## Q9: How do you use `git range-diff` to compare two series of commits?
**A:** `git range-diff` compares two commit ranges, showing how commits changed between them. (1) `git range-diff A...B C...D` -- compares the range A..B with C..D. (2) `git range-diff origin/main...feature origin/main...feature-rebased` -- shows what changed during rebase. (3) Output: for each commit, it shows if it was added, removed, or changed (with diffstat). (4) `git range-diff --creation-factor=100` adjusts how aggressively Git matches commits. (5) Use cases: (a) Reviewing a rebase -- verify that only intended changes occurred. (b) Reviewing a PR iteration -- compare v1 and v2 of a feature branch. (c) Auditing cherry-picks -- verify that cherry-picked commits were applied correctly. (6) `git range-diff` is the definitive tool for reviewing history rewrites -- it's far better than visual inspection of individual commits.

## Q10: How do you configure Git hooks for a monorepo with multiple projects?
**A:** Monorepo Git hooks: (1) Store hooks in a version-controlled directory: `.githooks/` instead of `.git/hooks/`. (2) Configure: `git config core.hooksPath .githooks`. (3) Each hook determines which projects are affected by the commit: `git diff --cached --name-only | grep "^packages/backend/"` to check if backend files changed. (4) Pre-commit hook runs linters only for changed projects: `if git diff --cached --name-only | grep -q "^packages/frontend/"; then (cd packages/frontend && npm run lint)`. (5) Commit-msg hook validates commit message format with project prefix: `fix(backend): ...`, `feat(frontend): ...`. (6) Pre-push hook runs tests for all affected projects. (7) Use `husky` with `lint-staged` for staged-file-only operations. (8) For CI integration, the same hooks can be used in CI pipelines for consistency.

## Q11: How do you configure `git rerere` (Reuse Recorded Resolution) for merge conflicts?
**A:** `rerere` (Reuse Recorded Resolution) remembers how you resolved merge conflicts and auto-applies them: (1) Enable: `git config --global rerere.enabled true`. (2) Git stores resolved conflict resolutions in `.git/rr-cache/`. (3) When the same conflict occurs again (e.g., rebasing the same branch multiple times), Git automatically applies the previous resolution. (4) `git rerere status` shows recorded resolutions. (5) `git rerere diff` shows the recorded resolution. (6) `git rerere forget <path>` clears a specific resolution. (7) `git rerere gc` cleans up old cached resolutions. (8) Extremely useful for: (a) Rebasing long-lived feature branches that hit the same conflicts repeatedly. (b) Cherry-picking between branches with overlapping changes. (c) Frequent merges between stable branches (release -> master -> develop). (9) Caveat: review auto-resolved conflicts -- the cached resolution may not be correct for the new context.

## Q12: How does `git notes` work for attaching metadata to commits?
**A:** `git notes` adds metadata to commits without changing the commit SHA: (1) `git notes add -m "Reviewed-by: Alice <alice@example.com>" HEAD` -- attaches a note. (2) Notes are stored as separate objects in `refs/notes/commits`. (3) `git log --show-notes` displays notes alongside commits. (4) Use cases: (a) Code review annotations. (b) CI/CD status (test results, build artifacts). (c) Post-commit metadata (JIRA ticket status). (d) Signatures (additional sign-offs). (5) `git notes append -m "Approved by Bob" HEAD` -- appends to existing notes. (6) Notes can be fetched and pushed: `git push origin refs/notes/*`. (7) Notes are not transferred by default in clones -- use `--fetch-notes` or configure `fetch.notes`. (8) `git notes merge` handles conflicts when multiple note sources exist. (9) Notes live outside the commit DAG, so rebasing/filter-branch doesn't affect them.

## Q13: How do you implement a Git pre-push hook that enforces code review requirements?
**A:** Pre-push hook for review enforcement: (1) The hook receives: remote name, URL, and list of refs being pushed (local_ref local_sha remote_ref remote_sha). (2) For each ref being pushed, check if the commits have been reviewed: `git log --format="%H" origin/main..HEAD | while read commit; do git notes --ref=review show $commit 2>/dev/null || { echo "Commit $commit has no review"; exit 1; }; done`. (3) Check that commits are signed: `git verify-commit $commit` or check `%G?` format. (4) Enforce that the branch name matches conventions: `feature/JIRA-123-description`. (5) For protected branches (main, release), require the push to come via a merge request (check that the ref has a specific prefix like `merge-requests/`). (6) Run a final security scan on the diff before push: `git diff --cached --name-only | xargs secret-scanner`. (7) Exit with non-zero to reject the push. (8) Store the hook in `.githooks/pre-push` with `core.hooksPath`.

## Q14: How do you use `git interpret-trailers` for structured commit metadata?
**A:** `git interpret-trailers` adds structured key-value metadata to commit messages: (1) Trailers are `Key: Value` pairs at the end of commit messages (e.g., `Reviewed-by: Alice`, `Fixes: JIRA-123`). (2) Use: `git interpret-trailers --trailer "Signed-off-by: Alice <alice@example.com>" --trailer "Fixes: JIRA-456" < commit_msg`. (3) Configure in `.gitconfig`: `[trailer "fixes"]; key = "Fixes"; ifexists = "replace"`. (4) Automatically add via `commit-msg` hook: `git interpret-trailers --trailer "Signed-off-by: $(git config user.name)" < $1 > temp && mv temp $1`. (5) `git log --format="%(trailers:key=Fixes,valueonly)"` extracts specific trailers. (6) Trailers enable automation: CI systems parse `Fixes`, `Closes`, `Breaking` to auto-link issues, generate changelogs, and enforce policies. (7) GitHub/GitLab recognize trailers like `Co-authored-by`, `Signed-off-by`, `Fixes`. (8) Unlike `git notes`, trailers are in the commit object itself (immutable).

## Q15: How do you implement `git bisect` with a custom test script for performance regressions?
**A:** Performance bisect: (1) `git bisect start --term-old=fast --term-new=slow` -- custom terminology. (2) Mark a known-fast commit: `git bisect fast v1.0` and a known-slow commit: `git bisect slow main`. (3) Run a custom benchmark script: `git bisect run ./benchmark.sh`. (4) The benchmark script should return 0 (fast) or 1 (slow). Exit code 125 skips the commit (unbuildable). (5) For flaky performance tests, use `git bisect run --no-checkout` with a script that checks out the tree only once and rebuilds incrementally. (6) For flaky benchmarks (varying by a few percent), use `for i in {1..5}; do result=$(./bench); sum=$((sum + result)); done; average=$((sum/5)); if [ $average -gt $threshold ]; then exit 1; else exit 0; fi`. (7) Use `git bisect skip` when a commit can't be tested (build failure, environment issue). (8) Git bisect finds the exact commit where performance changed, enabling targeted optimization or reversion.

## Q16: How does Git's multi-pack index (MIDX) improve repository performance?
**A:** The multi-pack index (`.git/objects/pack/multi-pack-index`) enables efficient object lookups across multiple packfiles: (1) Without MIDX, Git searches each packfile individually (O(num_packs) lookup). (2) With MIDX, Git has a single index covering all packfiles (O(1) lookup). (3) MIDX also enables delta objects across packfiles (previously deltas only existed within the same packfile). (4) Create/update: `git multi-pack-index write`. (5) MIDX reduces memory usage for large repositories with many packfiles. (6) It also enables object pruning across packfiles: `git multi-pack-index expire` removes objects from old packfiles that are now in new packfiles. (7) `git multi-pack-index repack` creates a new packfile containing objects from multiple packfiles, reducing packfile count. (8) For monorepos with millions of objects, MIDX is critical for maintaining acceptable performance. (9) Recommended: run `git multi-pack-index write` after `git gc`.

## Q17: How do you implement `git archive` with custom export-ignore and export-subst attributes?
**A:** `git archive` creates a tar/zip of a Git tree, respecting `.gitattributes` directives: (1) `export-ignore` -- excludes files/directories from the archive: `tests export-ignore; .gitattributes export-ignore`. (2) `export-subst` -- substitutes Git placeholder strings in files: `version.txt export-subst`. Placeholders in the file: `$Format:%H$` (commit hash), `$Format:%an$` (author), `$Format:%ai$` (author date). (3) Create archive: `git archive --format=zip --output=project-1.0.zip v1.0`. (4) Use `--prefix=project-1.0/` to add a top-level directory. (5) Works without .git directory -- ideal for build artifacts and deployments. (6) For custom export filtering, use `git archive --remote=<url> HEAD` to generate archives from remote repos. (7) Combine with custom `tar` command in `.gitattributes`: `*.svg filter=lfs diff=lfs merge=lfs -text` -- LFS files are replaced with pointer text in archives.

## Q18: How do you use `git stash push --staged` for selective stashing?
**A:** `git stash push --staged` (Git 2.35+) stashes only staged changes: (1) `git stash push --staged -m "WIP: api changes"` -- stashes staged changes, leaving unstaged changes in the working directory. (2) `git stash push --staged -- <path>` -- stash staged changes for specific files. (3) Useful for: (a) Creating a focused commit: stash all changes, unstash just the staged portion, commit. (b) Splitting work: stage the portion you want to commit, stash the rest. (c) Emergency fix: stage the bug fix, stash other changes, commit the fix. (4) `git stash push --keep-index` is the opposite: stashes unstaged changes, leaving staged changes intact. (5) Use `-p` (patch mode) to interactively select hunks to stash: `git stash push -p`. (6) `git stash show -p` shows the diff of the latest stash for review before applying.

## Q19: How does Git's commit-graph (also known as commit-graph file) accelerate Git operations?
**A:** The commit-graph file (`.git/objects/info/commit-graph`) stores pre-computed commit metadata for faster traversal: (1) Stores: commit SHA, root tree, parents, generation number, commit date. (2) Accelerates: `git log --graph`, `git merge-base`, `git rev-list`, branch comparison operations. (3) Generation numbers enable Git to terminate graph walks early (commits with gen < boundary can be ignored). (4) Create: `git commit-graph write --reachable` (or `--changed-paths` for Bloom filter support). (5) Changed-path Bloom filters accelerate `git log -- <path>` queries -- Git can skip commits that don't touch the path. (6) Without commit-graph, `git log -- <path>` reads every commit. With Bloom filters, it skips most commits. (7) `git maintenance` automatically updates the commit-graph. (8) For huge repositories, commit-graph reduces `git log` time from minutes to seconds. (9) Verify with `git commit-graph verify`.

## Q20: How do you use `git --no-optional-locks` to prevent Git operations from causing conflicts?
**A:** `git --no-optional-locks` disables optional locking during read-only operations: (1) Without this flag, commands like `git status`, `git log`, `git diff` can acquire locks to update the index, which can conflict with concurrent Git operations. (2) Use in scripts: `GIT_OPTIONAL_LOCKS=0 git status` to prevent locks. (3) The no-optional-locks mode skips: index refresh, commit-graph write, multi-pack-index write, background maintenance. (4) Critical for CI/CD scripts running in parallel: prevents "Unable to create '.git/index.lock': File exists" errors. (5) Also useful in IDE integrations where Git operations run frequently in the background. (6) Use when: (a) Running `git status` for prompt decorations. (b) CI jobs that only read Git data. (c) Batch processing of many repositories. (7) Not suitable: when you need the index to be up-to-date (e.g., before committing).

## Q21: How do you implement a Git pre-commit hook using lint-staged?
**A:** `lint-staged` runs linters only on staged files: (1) Install: `npm install --save-dev lint-staged husky`. (2) Configure in `package.json`: `"lint-staged": { "*.{js,ts}": ["eslint --fix", "prettier --write"], "*.md": "prettier --check" }`. (3) Husky pre-commit hook: `npx lint-staged`. (4) `lint-staged` passes staged file paths to the configured linter. (5) If any linter exits with non-zero, the commit is aborted. (6) `lint-staged` only lints the staged version of files (not the working directory version), preventing false positives from unstaged changes. (7) For monorepos, `lint-staged` detects the project for each file from its path and runs the correct linter. (8) Advanced: `"*.ts": ["bash -c 'tsc --noEmit'", "eslint --fix"]` -- run TypeScript type-checking on all files (not just staged). (9) Performance: runs linters in parallel for different file types.

## Q22: How do you implement Git hooks using a hook manager (husky, pre-commit)?
**A:** Hook managers simplify hook distribution: (1) **Husky** (npm): `npx husky init` creates `.husky/` directory. `npx husky add .husky/pre-commit "npm test"`. Hooks are version-controlled and auto-installed via `npm prepare` script. (2) **pre-commit** (Python): `.pre-commit-config.yaml` defines hooks from a registry. `repos: - repo: https://github.com/pre-commit/pre-commit-hooks; rev: v4.0.1; hooks: - id: trailing-whitespace`. (3) **lefthook** (Go): `lefthook.yml` config. `pre-commit: commands: eslint: glob: "*.{js,ts}" run: npx eslint {staged_files}`. (4) Benefits: (a) Hooks are version-controlled and shared across the team. (b) Auto-installed on clone. (c) No editing `.git/hooks/` manually. (d) Rich plugin ecosystem. (5) Best practice: use a hook manager, commit the hook configuration, and document the required tools in the README.

## Q23: How do you use `git update-ref` for low-level branch manipulation?
**A:** `git update-ref` updates a ref directly without safety checks: (1) `git update-ref refs/heads/my-branch <new-hash>` -- moves my-branch to a specific commit. (2) `git update-ref -d refs/heads/my-branch` -- deletes the branch. (3) `git update-ref refs/heads/my-branch <new-hash> <old-hash>` -- only update if the ref still points to old-hash (compare-and-swap). (4) Use cases: (a) Atomic operations in scripts: update multiple refs as a batch. (b) Creating signed tags programmatically. (c) Recovering branches from the reflog: `git update-ref refs/heads/recovered <reflog-hash>`. (d) Batch branch operations in Git hooks. (5) Unlike `git branch -f`, `update-ref` doesn't check that the new hash is an ancestor. (6) `git symbolic-ref` is similar but for symbolic refs (like HEAD): `git symbolic-ref HEAD refs/heads/main`.

## Q24: How do you use `git replace` for transparently splitting a repository?
**A:** Split a repo with `git replace`: (1) Extract subdirectory into its own repo: `git filter-repo --path src/lib/ --target my-lib`. (2) In the original repository: `git replace <original-lib-commit> <new-lib-root-commit>`. (3) Now the original repo shows the subdirectory as a grafted external history. (4) `git pull --recurse-submodules` can be used alongside. (5) Benefits: (a) No history rewrite needed in the original repo. (b) Sub-team works in the extracted repo with full history. (c) The replacement is transparent -- all Git operations work as if the history is native. (6) Share replacements: `git push origin refs/replace/*` and collaborators `git fetch origin refs/replace/*:refs/replace/*`. (7) This technique enables gradual extraction of components from monorepos without breaking existing clones.

## Q25: How do you configure Git for signed commits using SSH keys?
**A:** Git SSH signing (Git 2.34+): (1) Generate an SSH key: `ssh-keygen -t ed25519 -C "your.email@example.com"`. (2) Configure Git: `git config --global gpg.format ssh; git config --global user.signingkey ~/.ssh/id_ed25519.pub`. (3) Sign commits: `git commit -S -m "feat: new feature"`. (4) Verify: `git log --show-signature`. (5) Configure allowed signers: `git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers`. Format: `email@example.com ssh-ed25519 AAAAC3...`. (6) Auto-sign all commits: `git config --global commit.gpgsign true`. (7) SSH signing is simpler than GPG: no key servers, no web of trust. (8) GitHub supports SSH commit signing verification. (9) For CI/CD, store the signing key as a secret and configure the CI environment to sign tags and releases.

## Q26: How does `git maintenance` automate repository optimization?
**A:** `git maintenance` (Git 2.30+) handles background optimization: (1) `git maintenance start` registers the repository for scheduled maintenance (via cron/systemd timers). Tasks run: hourly, daily, weekly. (2) Tasks: (a) `prefetch` -- fetches from remotes to reduce later fetch times. (b) `loose-objects` -- consolidates loose objects into packfiles. (c) `incremental-repack` -- repacks using multi-pack-index. (d) `gc` -- full garbage collection (weekly). (e) `commit-graph` -- updates commit-graph. (3) `git maintenance run --task=gc --task=commit-graph` runs specific tasks immediately. (4) Configure task schedules: `git config maintenance.gc.interval daily`. (5) Benefits: automatic, incremental, low-impact (runs during idle time). (6) Critical for long-lived repositories that grow over time. (7) `git maintenance` replaces manual `git gc` scheduling. (8) For huge repositories, adjust the `--schedule` to run more frequently.

## Q27: How do you use `git show-bisect` to analyze the bisect result?
**A:** After `git bisect` completes, `git bisect log` shows the full bisect history. `git bisect visualize` uses `gitk` or `git log` to show the remaining range. Advanced analysis: (1) `git bisect log > bisect_log.txt` -- save the log for debugging. (2) `git bisect replay bisect_log.txt` -- replay a saved bisect session. (3) `git bisect run` automatically records the exit codes of the test script. (4) After finding the bad commit, analyze it: `git show <bad-commit>` to see the diff. (5) Use `git log --graph --oneline <bad-commit>^..<bad-commit> --first-parent` to see the commit's direct ancestry. (6) For flaky bisect runs (test script has false positives), `git bisect skip` marks untestable commits. (7) `git bisect reset` ends the bisect session. (8) The found commit can be reverted with `git revert <bad-commit>` if it's a simple bug.

## Q28: How do you implement a Git commit-msg hook to enforce conventional commits?
**A:** Conventional commits enforcement: (1) `commit-msg` hook reads `.git/COMMIT_EDITMSG`. (2) Regex validation: `^(build|chore|ci|docs|feat|fix|perf|refactor|revert|style|test)(\(.+\))?: .{1,72}$`. (3) Check commit message length (first line < 72 chars). (4) Check that body lines wrap at 72 chars. (5) Validate breaking change marker: `!` after type/scope or `BREAKING CHANGE:` in footer. (6) Auto-append issue reference from branch name: `if branch =~ /feature/JIRA-(\d+)/; echo "Refs: JIRA-$1" >> $1`. (7) Validate footer format: `Signed-off-by:`, `Co-authored-by:`, `Reviewed-by:`. (8) Example: `#!/bin/sh; commit_msg=$1; if ! head -1 "$commit_msg" | grep -qE "^(feat|fix|chore)(\(.+\))?:\s.+"; then echo "Invalid commit message format"; exit 1; fi`. (9) Integrate with commitlint: `npx commitlint --edit $1`.

## Q29: How do you implement Git submodule workflows for dependency management?
**A:** Submodule advanced workflows: (1) `git submodule add https://github.com/lib/lib.git lib/lib` -- add a dependency. (2) `git submodule update --init --recursive --remote` -- update all submodules to their latest remote default branch. (3) For pinning: `git submodule update --init --recursive` (uses the commit recorded in the parent repo). (4) `git submodule foreach 'git checkout main && git pull'` -- update all submodules to latest. (5) Nested submodules: `--recursive` for depth. (6) Submodule workflow best practices: (a) Always record submodule URL with the canonical URL (not relative). (b) Document which version each submodule should be at. (c) Use `git config submodule.recurse true` to automate submodule operations. (d) In CI, `git submodule update --init --depth 1` for shallow clones (faster). (7) Drawbacks: submodule operations require explicit commands; detached HEAD state is confusing. (8) Alternatives: subtrees or package managers.

## Q30: How do you use `git subtree` as an alternative to submodules?
**A:** `git subtree` merges an external repository into a subdirectory: (1) Add: `git subtree add --prefix=lib/lib https://github.com/lib/lib.git main --squash`. (2) Update: `git subtree pull --prefix=lib/lib https://github.com/lib/lib.git main --squash`. (3) Push changes back: `git subtree push --prefix=lib/lib https://github.com/lib/lib.git feature-branch`. (4) Vs submodules: (a) No `.gitmodules` file. (b) All code is in the parent repo -- no detached HEAD, no extra commands for consumers. ## Q98: How do you use `git revert` to undo a merge commit?
**A:** Reverting merge commits: (1) `git revert -m 1 <merge-commit-hash>` -- revert a merge, keeping mainline (parent 1). (3) After revert, the branch can be re-merged later if needed: `git revert <revert-commit>` -- this reverts the revert, effectively re-applying the merge.
