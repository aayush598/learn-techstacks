# STT Interview Questions and Answers - Part 2

## Q1: How does CTC handle the alignment problem between input audio frames and output text sequences in practice?
**A:** CTC introduces a blank token (∅) that allows the model to output repeated blanks between non-blank labels, effectively letting the model decide the alignment. During training, CTC marginalizes over all possible alignments using dynamic programming (forward-backward algorithm). The loss is the negative log probability summed over all valid alignment paths. In practice, the model outputs a probability distribution over vocabulary+blank at each timestep, and CTC sums probabilities of all paths that collapse to the target sequence via:
1. Merging consecutive identical non-blank tokens
2. Removing all blank tokens

## Q2: What is the mathematical formulation of RNN-T loss and how does it differ from CTC?
**A:** RNN-T loss operates on a lattice defined by input timesteps (T) and output labels (U). It uses a transcription network (encoder) producing frames h_t, a prediction network (decoder) producing label embeddings g_u, and a joint network combining them as z_{t,u} = f(W_h h_t + W_g g_u + b). The loss sums probabilities over all possible alignments (paths) through the T×U lattice, where each step either emits a label (moving right) or advances time (moving down). Unlike CTC which assumes conditional independence per frame given the input, RNN-T models label dependencies through the prediction network, making it more powerful for streaming ASR.

## Q3: In Conformer-based acoustic models, how do convolution modules and self-attention interact to capture both local and global speech patterns?
**A:** The Conformer encoder block uses a "macaron" structure with two half-step feed-forward modules sandwiching the self-attention and convolution modules. The convolution module consists of: pointwise convolution → GLU activation → depthwise convolution (kernel ~31) → BatchNorm → Swish activation → pointwise convolution. This captures local patterns (formant transitions, coarticulation) via the depthwise conv's limited receptive field. Self-attention captures long-range dependencies (prosody, sentence structure). The multi-headed self-attention uses relative positional encoding to model temporal distances. The combined output is: y = x + 0.5×FFN(x) + SA(x) + Conv(x) + 0.5×FFN(x), where each submodule has residual connections and layer normalization.

## Q4: How does shallow fusion work for integrating an external language model into a CTC/RNN-T decoder during beam search?
**A:** Shallow fusion combines scores from the acoustic model and an external LM during beam search decoding. The total score for a hypothesis is: S_total = log P_am(hyp|x) + λ * log P_lm(hyp) + μ * |hyp|, where λ is the LM weight and μ is a coverage/length penalty. The LM score is computed by feeding the partial hypothesis tokens into a separately trained neural LM (e.g., Transformer LM) or n-gram LM. At each beam expansion step, both the AM token probability and LM probability are combined. The LM is typically much larger than the internal prediction network, providing better linguistic context. Gradient-based shallow fusion variants also exist where λ is dynamically predicted per timestep.

## Q5: What is deep fusion and how does it differ from shallow fusion in neural ASR?
**A:** Deep fusion integrates the external language model directly into the neural network architecture rather than combining scores at the decoding level. In deep fusion, the LM's hidden states are fused with the acoustic model's hidden states through a learned gating mechanism before the output projection. For example, in RNN-T deep fusion, the prediction network's output and an external LM's output are combined via g = σ(W_g[h_pred; h_lm] + b_g), h_fused = g ⊙ h_pred + (1-g) ⊙ h_lm. This allows the model to learn when to trust the LM versus the acoustic input. Deep fusion requires joint fine-tuning of the entire model with the LM, making it more computationally expensive than shallow fusion but potentially more accurate.

## Q6: Explain the cold fusion approach for LM integration in end-to-end ASR.
**A:** Cold fusion pre-integrates a pre-trained language model into the ASR model before training begins. The LM is frozen, and its representations are projected through a learned fusion layer to combine with the acoustic encoder outputs. The fusion layer uses a gating mechanism: g = σ(W_1 h_enc + W_2 h_lm), h_fused = g ⊙ h_enc + (1-g) ⊙ h_lm. The entire model (encoder + fusion + decoder) is then trained jointly. Cold fusion allows the LM to guide training from the start, reducing the acoustic model's burden to learn linguistic patterns. It converges faster than shallow fusion and typically outperforms it because the acoustic model learns to leverage the LM's linguistic knowledge during training.

## Q7: How would you design a streaming RNN-T system that processes audio chunks of 320ms with 160ms left context?
**A:** The audio stream is divided into overlapping chunks: each chunk contains 320ms of new audio plus 160ms of left context from the previous chunk (480ms total per chunk). The encoder processes each chunk independently but uses a causal Conformer or Emformer with cached states (key-value pairs from self-attention) carried across chunks. The RNN-T decoder maintains a prediction network state that persists across chunks. As each chunk is encoded, the joint network aligns encoder outputs with prediction network outputs. Partial transcripts are emitted with a delay of ~chunk_size. The key challenge is maintaining consistency at chunk boundaries — using look-ahead only within the chunk (never future chunks) preserves causality. This enables real-time streaming with ~500ms end-to-end latency.

## Q8: What techniques allow a streaming ASR model to achieve accuracy close to a non-streaming (batch) model?
**A:** Key techniques include: (1) **Causal convolutions with right-context blocks** — small look-ahead (e.g., 10-30ms) within the causal constraint, (2) **Chunk-wise processing with context inheritance** — carrying encoder states (key-value caches) across chunks, (3) **Delayed decoding** — emitting tokens with a fixed delay (e.g., 500ms) after the corresponding audio, (4) **Dynamic chunk training** — training with varying chunk sizes to generalize to different latency budgets, (5) **Knowledge distillation** — training the streaming student model to mimic a non-streaming teacher's representations, (6) **Two-pass decoding** — a lightweight streaming pass produces fast hypotheses while a second pass with full context refines them.

## Q9: How does speaker diarization using x-vectors with spectral clustering actually work under the hood?
**A:** The pipeline: (1) Voice Activity Detection (VAD) segments audio into speech intervals. (2) Each segment is divided into fixed-length windows (~1.5s). (3) An x-vector extractor (TDNN with statistics pooling) processes each window, producing a fixed-dimensional embedding. (4) Standardization (mean/variance normalization) and dimensionality reduction (LDA or PCA) are applied. (5) Spectral clustering: construct an affinity matrix A where A_{ij} = cos(x_i, x_j) or RBF similarity, compute the normalized Laplacian L = I - D^{-1/2} A D^{-1/2}, find the k smallest eigenvectors (k estimated via eigengap heuristic), cluster rows via k-means. (6) Each cluster corresponds to a speaker. For online diarization, incremental clustering with online x-vector extraction and iterative centroid updates is used.

## Q10: How does end-to-end neural diarization (EEND) differ from the traditional clustering-based approach?
**A:** EEND treats diarization as a multi-label classification problem directly — instead of extracting embeddings and clustering separately, EEND uses a transformer/encoder that processes audio frames and outputs per-frame speaker activity vectors (e.g., [P(speaker1), P(speaker2), ...]). It handles overlapping speech naturally since each output dimension is independent. EEND uses permutation invariant training (PIT) to handle the unknown speaker-label assignment. Unlike clustering, EEND is fully supervised, jointly optimized, and handles variable numbers of speakers. The main limitation: it requires fixed maximum speakers during training and doesn't scale to very long recordings without chunking. Hybrid EEND-vector clustering combines both approaches.

## Q11: How do you implement custom vocabulary biasing in an RNN-T model without retraining?
**A:** Custom vocabulary biasing (hot-listing) in RNN-T can be implemented via: (1) **On-the-fly rescoring** — maintain a trie of bias phrases, during beam search, detect when a partial hypothesis matches a bias prefix and boost its score (e.g., +5.0 log-prob bonus), (2) **Dynamic prediction network** — inject bias embeddings into the prediction network's output, (3) **Contextual adapters** — small neural network adapters that take bias list embeddings as side input, (4) **Bias language model** — a small n-gram LM built from the bias list and combined via shallow fusion. For streaming, the bias trie is updated in real-time as new context becomes available (e.g., user's contact list loaded at call start).

## Q12: Describe how to fine-tune OpenAI Whisper for a specific domain like medical transcription.
**A:** Fine-tuning Whisper for medical transcription: (1) Prepare training data — medical dictation recordings with verbatim transcripts, ensure HIPAA compliance, aim for 100+ hours. (2) Preprocess — resample to 16kHz, segment into 30-second chunks (Whisper's context window), normalize audio volume. (3) Format — use HuggingFace datasets format with 'audio' and 'text' columns. (4) Load Whisper weights (base/small/medium depending on compute). (5) Apply LoRA (Low-Rank Adaptation) — add trainable rank-1 matrices to attention layers, keeping most weights frozen (reduces trainable params by 99%). (6) Training — use Seq2SeqTrainer with gradient checkpointing, per-device batch size 8-16, learning rate 1e-4 with cosine schedule, train for 5-10 epochs. (7) Evaluate — compute WER on medical test set. (8) Post-processing — add medical inverse text normalization for terms like "mg" → "mg", "BID" → "BID".

## Q13: What is the impact of LoRA rank on Whisper fine-tuning quality and how do you choose it?
**A:** LoRA decomposes weight updates as ΔW = BA where B∈R^{d×r}, A∈R^{r×k}, and r is the rank. Higher rank (r=32-64) captures more domain-specific patterns but increases parameter count and risk of overfitting with small data. Lower rank (r=4-8) acts as stronger regularization and trains faster. For medical/financial domains with specialized vocabularies, r=16-32 is typical. The rank affects which layers are adapted — for Whisper, applying LoRA to cross-attention (encoder-decoder) and feed-forward layers yields better domain adaptation than self-attention alone. Empirical evaluation: r=16 typically captures 90%+ of full fine-tuning performance while using only 0.1-0.5% of trainable parameters.

## Q14: How does SpecAugment's time warping parameter affect CTC training convergence?
**A:** SpecAugment's time warping (W parameter) applies a random smooth warping along the time axis using an affine transformation sampled from a normal distribution. W=10 means the warping field is sampled with max displacement of 10 time steps. For CTC training, time warping is particularly beneficial because CTC is invariant to small time shifts (due to the blank token mechanism). However, excessive warping (W>40) can distort phonetic boundaries and harm convergence. The optimal W depends on audio length — shorter utterances (<5s) benefit from W=5-10, longer utterances from W=10-20. Time warping combined with frequency masking (F=30) and time masking (T=40) yields the best results for CTC-based models, reducing WER by 15-25% on noisy benchmarks.

## Q15: How do you compute WER when the reference contains notations like numbers, abbreviations, or punctuation?
**A:** Before computing WER, both hypothesis and reference must be normalized through a standardized process: (1) Inverse Text Normalization (ITN) — convert spoken-form to written-form (e.g., "one hundred" → "100"), (2) Remove punctuation, (3) Lowercase everything, (4) Collapse whitespace, (5) Expand abbreviations (both hyp and ref consistently). This preprocessing is critical because the WER metric counts substitutions, insertions, deletions literally. For example, if the reference is "$50.30" and the ASR outputs "fifty dollars and thirty cents", the literal WER would be very high despite semantic correctness. A standardized normalization pipeline ensures the WER reflects actual recognition errors rather than formatting differences.

## Q16: What is Speaker Error Rate (SER) and how is it calculated for diarization evaluation?
**A:** Speaker Error Rate (SER), also called Diarization Error Rate (DER), measures diarization accuracy. DER = (False Alarm + Missed Speech + Speaker Confusion) / Total Reference Speech Duration. False Alarm: time classified as speech when reference has silence. Missed Speech: time classified as silence when reference has speech. Speaker Confusion: time assigned to wrong speaker (with optimal one-to-one mapping between hypothesis and reference speakers). The optimal mapping is found via the Hungarian algorithm that minimizes total speaker confusion time. A collar tolerance (usually 0.25s) is applied around speaker boundaries to avoid penalizing small timing differences. State-of-the-art DER is ~3-8% on conversational telephone speech datasets.

## Q17: How does punctuation restoration using BERT differ from using the ASR model's internal language model?
**A:** BERT-based punctuation restoration treats it as a sequence tagging task: take the ASR transcript text, tokenize with BPE, pass through BERT encoder, and predict punctuation label (COMMA, PERIOD, QUESTION_MARK, O) per token via a linear classification head. This approach benefits from bidirectional context (looking at words before AND after each position). Using the ASR model's internal LM for punctuation is limited because most ASR decoders are left-to-right (unidirectional) and optimized for word prediction, not punctuation. Furthermore, BERT-based models trained on punctuated text (e.g., book corpora) generalize better across domains. Hybrid approaches use the ASR model's encoder representations (audio features) concatenated with BERT's text embeddings for the best of both worlds.

## Q18: How do you handle code-switching between English and Hindi in an ASR system without separate language models?
**A:** Approaches for code-switching ASR: (1) **Multilingual joint training** — train a single model on mixed English-Hindi data with a shared BPE vocabulary covering both languages, allowing the model to learn mixed-language phonetics and language transitions. (2) **Language-agnostic phoneme inventory** — map both languages to a shared IPA phoneme set, training the acoustic model on phonemes rather than graphemes. (3) **Language ID conditioning** — inject a language identification vector (from a separate LID model or the encoder) into the decoder, enabling language-aware decoding. (4) **Code-switching data augmentation** — create synthetic code-switched training data by randomly replacing words/phrases with their translation equivalents. (5) **Adapter modules** — train language-specific adapters for each language and switch dynamically based on LID. The main challenge is the scarcity of transcribed code-switched speech data.

## Q19: What is the architecture of Deepgram's Nova-2 model and what makes it different from standard end-to-end ASR?
**A:** Deepgram Nova-2 uses a custom architecture built on a streaming-capable RNN-T variant with a Conformer encoder. Key differentiators: (1) **Mesh learning** — a multi-objective training approach where the model simultaneously optimizes for transcription, punctuation, capitalization, and inverse text normalization within a single unified model (no separate post-processing pipeline). (2) **Self-supervised pre-training** — uses large-scale unsupervised audio pre-training on 100K+ hours before fine-tuning on labeled data. (3) **Custom data curation** — emphasis on diverse acoustic conditions (noise, reverberation, accents) during training. (4) **Hardware-optimized inference** — uses custom CUDA kernels and tensor core optimizations for low-latency GPU inference. Nova-2 reportedly achieves ~5-6% WER on general English benchmarks.

## Q20: How does AssemblyAI's Conformer-1 architecture differ from standard Conformer implementations?
**A:** AssemblyAI's Conformer-1 introduces several innovations over the standard Conformer: (1) **Unified streaming and batch model** — a single model that can run in either streaming or batch mode by controlling the look-ahead context, (2) **Multi-scale self-attention** — uses different attention span patterns in different layers (short spans in lower layers for local patterns, longer spans in higher layers for global context), (3) **Input-level dynamic resolution** — the model dynamically adjusts the temporal resolution based on speaking rate, using a downsampling controller that varies the subsampling rate through the encoder, (4) **Jointly trained punctuation** — punctuation is integrated directly into the output vocabulary rather than using a separate model, (5) **Custom loss function** — combines RNN-T loss with auxiliary CTC loss for faster convergence (CTC-acting).

## Q21: How does Google's USM (Universal Speech Model) achieve zero-shot cross-lingual transfer?
**A:** Google's Universal Speech Model (USM) uses a multi-stage training pipeline: (1) **Self-supervised pre-training** on 12M+ hours of unlabeled audio across 100+ languages using BEST-RQ (Bootstrap Your Own Latent) with a Conformer encoder, (2) **Multilingual supervised fine-tuning** on labeled data from ~73 languages, learning language-shared representations, (3) **Language-specific adapter fine-tuning** for target languages with limited data. Zero-shot transfer works because the encoder learns language-agnostic phonetic representations during Stage 1 — the model identifies phonemes and prosodic patterns that generalize across languages. For a new language, the encoder's representations are combined with the closest-language's decoder head, often achieving competitive WER with only a few hours of target language data.

## Q22: What quantization techniques are most effective for ASR models on edge devices, and what accuracy trade-offs exist?
**A:** Key quantization techniques for ASR: (1) **Post-training INT8 quantization** — quantizes weights to 8-bit integers using per-tensor or per-channel scaling factors, reduces model size by 4× and latency by 2-3× on CPUs/NPUs. Typical WER increase: 0.5-1.5%. (2) **Post-training FP16 quantization** — half-precision floating point, reduces model size by 2×, minimal WER degradation (<0.1%), works well on GPUs with tensor cores. (3) **Quantization-aware training (QAT)** — simulates quantization during training (fake quantization nodes), yielding much better accuracy (WER loss <0.3%) at INT8. (4) **Mixed-precision** — quantize only certain layers (e.g., Conv layers to INT8, attention to FP16) to balance accuracy and performance. (5) **Weight-only quantization** — used for memory-bound scenarios, maintains FP16 activations while storing weights in INT4 (GPTQ-like), reduces model size by 4-8× with ~1-2% WER increase.

## Q23: How does Whisper's multitask training format work, and what are its advantages?
**A:** Whisper uses a unified multitask format where the decoder is prompted with special tokens that specify the task: `<|startoftranscript|>` starts decoding, optionally preceded by `<|lang_en|>` for language and `<|transcribe|>` for transcription or `<|translate|>` for translation to English. The output includes `<|timestamps|>` tokens for word-level timestamps when enabled. Advantages: (1) **Single model, multiple tasks** — the same model handles transcription, translation, language identification, and timestamping without architectural changes. (2) **Conditional computation** — the prompt tokens guide the decoder's behavior, effectively creating task-specific sub-networks within the shared parameters. (3) **Robust language ID** — the forced language token during training teaches the model to identify the language from audio. (4) **Simple deployment** — one model file serves all use cases.

## Q24: How do you optimize the RTF (Real-Time Factor) of a streaming ASR system to 0.3 or lower?
**A:** Achieving RTF < 0.3 requires: (1) **Model architecture** — use a causal Conformer with reduced attention dimension (d_model=256 instead of 512), fewer encoder layers (12→8), and smaller feed-forward dimension (1024→2048). (2) **Chunk-based processing** — 320ms chunks with state caching eliminates redundant computation on overlapping context. (3) **Batch processing within channels** — process multiple audio chunks from different channels as a single batch on GPU. (4) **ONNX Runtime / TensorRT** — convert model to ONNX with INT8 QAT, use TensorRT with dynamic shapes for GPU inference. (5) **KV-cache reuse** — cache attention key-value states between chunks to avoid recomputing past context. (6) **Efficient beam search** — beam width of 4 with a lightweight prediction network. (7) **GPU inference with CUDA graphs** — capture model execution as a graph to reduce kernel launch overhead. (8) **Model parallelism** — split encoder across multiple GPUs for very large models.

## Q25: How does beam search with length normalization and coverage penalty work for ASR decoding?
**A:** The beam search scoring function: score(hyp) = (1/|hyp|^α) * [log P(hyp|x) + λ_lm * log P_lm(hyp) + λ_cov * coverage_penalty(hyp)]. The length normalization factor |hyp|^α (α typically 0.6-1.0) prevents shorter hypotheses from being unfairly favored (since log-probabilities are negative and accumulate additively over tokens). The coverage penalty discourages the model from ignoring parts of the input: coverage_penalty = Σ_t min(c_t, 1) - total_tokens_covered, where c_t is the cumulative attention weight on encoder position t. A negative λ_cov penalizes under-coverage (ignoring audio). In attention-based models, the coverage vector is updated at each decoder step: c_t = c_t + a_{t,decoder_step}. For RNN-T, coverage mechanisms aren't typically needed since the monotonic alignment naturally covers all input.

## Q26: What techniques reduce the cold-start latency (first-word latency) in streaming ASR below 150ms?
**A:** Techniques to achieve <150ms first-word latency: (1) **Audio pre-buffering** — start buffering audio before VAD triggers, using a ring buffer of 100-200ms, (2) **Speculative decoding** — run inference on incomplete audio chunks and update as more audio arrives, (3) **VAD sensitivity tuning** — lower VAD threshold to detect speech onset earlier, with post-processing to filter false accepts, (4) **Encoder streaming mode** — process audio frames as they arrive without waiting for chunk boundaries, using a causal model with frame-synchronous output, (5) **Pre-computed look-up table** — for embedded systems, pre-encode common acoustic contexts to avoid cold-start encoder computation, (6) **Model warm-start** — keep the model loaded and GPU/accelerator initialized before the user starts speaking, (7) **Network latency optimization** — use WebSocket with no HTTP overhead, deploy inference close to the user (edge), (8) **Audio frontend optimization** — use fixed-point FFT implementation on DSP for faster feature extraction.

## Q27: How do you implement an ASR system that respects GDPR and CCPA data privacy requirements?
**A:** GDPR/CCPA-compliant ASR requires: (1) **On-device processing** — perform ASR on the user's device whenever possible, avoiding audio transmission to servers (e.g., whisper.cpp, Android on-device speech). (2) **Audio anonymization** — if cloud processing is needed, strip PII before sending: apply voice conversion to remove speaker identity, remove silent gaps that could reveal room acoustics, hash speaker IDs. (3) **Data minimization** — only process the minimum audio needed for the task; discard raw audio immediately after transcription. (4) **Right to deletion** — implement APIs to delete specific user's audio and transcript data from all storage systems. (5) **Consent management** — clear opt-in with granular choices (recording, storage, sharing). (6) **Encryption** — end-to-end encryption for audio in transit and at rest, with per-user encryption keys. (7) **Audit logging** — log all data access with timestamps and user IDs for compliance reporting. (8) **Data residency** — process and store data within specified geographic boundaries (EU, US, etc.).

## Q28: How does the integration of an external n-gram language model via shallow fusion differ when using CTC vs RNN-T decoders?
**A:** For **CTC**, shallow fusion is applied during beam search by adding the LM score to the CTC prefix score at each step. The CTC prefix score is computed by summing probabilities of all paths that lead to the prefix, which is computationally expensive (O(T×U) for each hypothesis). For **RNN-T**, shallow fusion is more natural — at each (t,u) step, the joint network outputs P(y_u | t, u, y_<u, x) and the external LM provides P_lm(y_u | y_<u). These are combined as: log P_combined = log P_rnnt + λ * log P_lm, applied at the output projection level. RNN-T's frame-synchronous nature means the LM score is incorporated once per output token, not per frame (as in CTC's prefix beam search). Also, RNN-T's internal prediction network already captures label dependencies, so the external LM weight (λ) is typically smaller (0.1-0.3) compared to CTC (0.5-1.0).

## Q29: How do you use the "draft model" or "speculative decoding" technique to accelerate ASR inference?
**A:** Speculative decoding for ASR uses a small, fast "draft" model (e.g., a 2-layer LSTM) to generate candidate token sequences, which are then verified by the large "target" model (e.g., Conformer-Transducer). The process: (1) The draft model generates k candidate tokens quickly (autoregressively). (2) The target model processes the candidate sequence in parallel (non-autoregressively) to compute log-probabilities. (3) Accept tokens where the target model agrees with the draft; reject and resample at the first disagreement. This works because the verification step can be parallelized (one forward pass for k tokens instead of k sequential passes). Speedups of 2-3× are common with negligible accuracy loss, as the verification preserves the target model's output distribution. For ASR, the draft model can be a lightweight RNN-T or even a simple n-gram.

## Q30: How does the Emformer architecture enable efficient streaming ASR on resource-constrained devices?
**A:** Emformer (Memory-efficient Conformer) introduces: (1) **Segment-level processing** — audio is partitioned into non-overlapping segments processed independently with cross-segment attention, (2) **Memory bank** — a fixed-size cache of representations from previous segments that serves as a summary of past context, used for cross-attention instead of attending to all past frames, (3) **Stride-aware downsampling** — applies different downsampling rates to the current segment (full rate) vs. memory bank (reduced rate) to balance accuracy and efficiency. On a mobile CPU, Emformer achieves RTF <0.5 with ~60M parameters (vs. ~600M for large models) while maintaining competitive WER. The memory bank mechanism limits the attention complexity from O(T^2) to O(S^2 + S×M) where S is segment size and M is memory size (typically 128 frames).

## Q31: How do you evaluate ASR system accuracy for numbers with inverse text normalization (ITN) in the pipeline?
**A:** Evaluation of ASR with ITN uses an end-to-end metric: (1) The raw ASR output is the "spoken form" transcript. (2) ITN converts to "written form" (e.g., "five hundred dollars and thirty cents" → "$530.30"). (3) Compare ITN output directly against the reference ITN text using standard WER. However, this conflates ASR errors and ITN errors. For diagnostic evaluation, compute: (a) **Spoken-form WER** — compare ASR raw output against spoken-form reference (requires spoken-form reference annotations), (b) **ITN-specific accuracy** — measure ITN errors separately by comparing ITN output with ground-truth ITN for the ASR hypothesis, (c) **Semantic WER** — use task-specific metrics (e.g., for dates: exact date match accuracy; for currency: amount deviation). For production monitoring, track "semantic error rate" — the fraction of utterances where the final ITN output would cause a downstream task failure.

## Q32: What is the role of the alignment head in a two-pass ASR system (e.g., Google's RNN-T + LAS)?
**A:** In two-pass ASR, the first pass (RNN-T) produces fast hypotheses with low latency, and the second pass (LAS — Listen, Attend, and Spell) refines them using full audio context. The alignment head aligns RNN-T's output token timestamps with audio frames to provide LAS with the audio segment corresponding to each hypothesis token. Specifically: (1) RNN-T's joint network outputs include both token probabilities and time boundaries (t,u coordinates). (2) The alignment head extracts, for each output token, the encoder frame range that contributed most to its prediction. (3) LAS receives these aligned segments as "hard attention" initialization, so it can re-score and potentially correct early hypotheses by attending to the specific audio regions. This alignment enables the second pass to improve accuracy without recomputing full attention over the entire audio, reducing computational cost.

## Q33: How do you handle time-stamp prediction for word-level alignment in a CTC-based model?
**A:** CTC models naturally produce frame-level alignments because each output frame either emits a token or a blank. To get word-level timestamps: (1) Run CTC decoding to get the token sequence (including blanks). (2) Force-align by finding, for each non-blank token, its first and last frame occurrence using the CTC alignment path. (3) Map BPE tokens to words by merging subword timestamps (first frame of first subword to last frame of last subword). (4) For increased precision, use a second-pass alignment model (e.g., a lightweight forced aligner like Montreal Forced Aligner or a CTC segmentation algorithm). The challenge is that CTC doesn't explicitly model word boundaries. Techniques like "CTC segmentation" (Luo et al., 2021) compute time boundaries by analyzing the CTC posterior distribution — the probability of a token peaks near its center and drops to blank between words.

## Q34: How does the "Deliberation" network improve accuracy in streaming ASR?
**A:** The Deliberation network (used in production by Google) is a second-pass model that refines the first-pass output using full audio context. It combines three inputs: (1) the first-pass hypothesis (e.g., from RNN-T), (2) encoded audio features, and (3) the alignment between them. The deliberation decoder uses cross-attention to both the audio encoder and the first-pass hypothesis encoder. Unlike LAS which generates from scratch, deliberation re-reads and corrects the first-pass output. It can overturn initial mistakes by looking at the complete audio context (which the streaming first-pass model couldn't see). The deliberation network is typically a small Transformer (2-4 layers) running after the utterance ends, adding ~100ms latency. WER improvements of 5-15% over first-pass alone are common.

## Q35: What techniques improve ASR accuracy for children's speech (age 3-12)?
**A:** Children's speech differs acoustically (higher F0, formant frequencies) and linguistically. Effective techniques: (1) **Pitch normalization** — use VTLN (Vocal Tract Length Normalization) to warp frequency axes, making children's acoustics closer to adult patterns. (2) **Age-specific fine-tuning** — fine-tune on pediatric speech corpora (MyST, OGI Kids). (3) **Data augmentation** — shift formant frequencies upward via spectral augmentation to simulate child-like speech. (4) **Separate acoustic model** — train a dedicated child ASR model with architecture adapted for higher pitch resolution. (5) **Modified feature extraction** — extend MFCC filterbank range (up to 8kHz instead of 4kHz for adults) to capture children's higher formants. (6) **Language model adaptation** — use children's literature and age-appropriate vocabulary for LM training. WER for children's speech is typically 2-3× higher than adults without these adaptations.

## Q36: How do you build a code-switching ASR system that handles intra-sentential switching (switch within a sentence)?
**A:** Intra-sentential code-switching requires: (1) **Phoneme-shared modeling** — create a combined phoneme inventory that covers both languages' phones, training the acoustic model to output a shared phoneme set. (2) **Code-switching lexicon** — build a pronunciation dictionary that includes both languages' words in the same phonetic space. (3) **Dual-encoder approach** — two language-specific encoders with a shared fusion layer that combines representations; a language ID vector controls the fusion weight per frame. (4) **Mixed BPE vocabulary** — train SentencePiece on mixed-language text, creating subword units that can represent both languages. (5) **Switching penalty** — during beam search, add a small penalty for language switches to prevent spurious switching (e.g., -0.5 log-prob for each language boundary crossing). (6) **Training data** — collect or synthesize code-switching speech (using text-to-speech with code-switching text data). Without sufficient training data, zero-shot code-switching using multilingual pre-trained models (Whisper, USM) is the most practical approach.

## Q37: How do you implement an ASR confidence score that is well-calibrated (reflects actual accuracy)?
**A:** Well-calibrated confidence scores require: (1) **Temperature scaling** — after training, calibrate the softmax temperature T: P_calibrated = softmax(logits/T), optimizing T to minimize Expected Calibration Error (ECE) on a held-out set. (2) **Confidence = P(tokens | audio) from beam search** — the normalized log-probability of the top hypothesis (after length normalization). (3) **Word-level confidence** from RNN-T — the joint network's probability for each emitted token, optionally smoothed with neighbors. (4) **External confidence model** — train a separate classifier (e.g., gradient-boosted tree) using features: ASR probability, language model score, acoustic likelihood, speaking rate, signal-to-noise ratio, and duration. (5) **Monotonic regression** — calibrate raw scores using isotonic regression on a validation set, mapping scores to empirical accuracy. (6) **Auxiliary confidence head** — during training, add an MLP head to the encoder that directly predicts per-token correctness (binary classification).

## Q38: What is Flashlight and how does it compare to Kaldi for ASR decoding?
**A:** Flashlight (formerly Facebook's Wav2letter++) is a fast, lightweight ASR decoding library written in C++. Key differences from Kaldi: (1) **Architecture** — Flashlight uses beam search decoding optimized for neural models (CTC, RNN-T, seq2seq); Kaldi was designed for HMM-GMM and hybrid DNN-HMM systems. (2) **Speed** — Flashlight's decoder is 2-10× faster than Kaldi's due to vectorized operations and optimized data structures for neural LM rescoring. (3) **LM support** — Flashlight supports neural LM integration (Transformer, LSTM, ConvLM) natively; Kaldi mainly supports n-gram LMs (though neural LM integration is possible through custom extensions). (4) **Lexicon** — Flashlight uses a trie-based lexicon with token indexing; Kaldi uses FST (Finite State Transducer) composition. (5) **Modern features** — Flashlight supports SpecAugment, RNN-T loss, and streaming; Kaldi's last major release was 2021. For production neural ASR, Flashlight is the modern choice. However, Kaldi's feature extraction (MFCC, fMLLR) and alignment tools remain widely used.

## Q39: How does the Levenshtein distance used in WER calculation work with language-specific tokenization?
**A:** WER calculation uses word-level Levenshtein distance: minimum number of word insertions, deletions, and substitutions to transform the hypothesis into the reference. For languages without spaces (Chinese, Japanese, Thai): (1) Tokenize both reference and hypothesis using the same segmentation algorithm (usually a lexicon-based or BPE-based tokenizer). (2) Treat each character/ subword as a "word" unit for WER, which can inflate error rates because a single character error splits differently. The alternative is CER (Character Error Rate) which operates on individual characters/graphemes. For Japanese, character-level WER (combining kanji, hiragana, katakana) with proper segmentation is standard. For Chinese, CER at the character (hanzi) level is preferred since Chinese characters are morphemes. Using BPE-based WER (subword error rate) is also common with modern neural ASR.

## Q40: How do you design a Voice Activity Detection (VAD) system that works reliably in noisy environments (SNR < 5dB)?
**A:** Robust VAD for low SNR: (1) **Feature extraction** — use robust features beyond simple energy: pitch harmonicity, spectral flatness, modulation spectrum features (4Hz modulation is speech-specific), and phase-based features. (2) **DNN-based VAD** — train a small CNN or CRNN on noisy data, outputting per-frame speech/non-speech probability. Use data augmentation with diverse noise profiles (SNR -5 to 20dB). (3) **Spectral subtraction pre-processing** — estimate noise profile during non-speech segments and subtract before feature extraction. (4) **Multi-microphone VAD** — use beamforming + coherence analysis between mics to distinguish speech from noise. (5) **Hangover smoothing** — extend VAD decisions with hangover (50-200ms) to avoid clipping word endings. (6) **Adaptive threshold** — dynamically adjust the VAD threshold based on estimated noise level from recent non-speech frames. (7) **Context-aware VAD** — use a bidirectional RNN that considers past and future frames (works offline) for higher accuracy.

## Q41: How does Gaussian approximation work for integrating a neural language model during first-pass decoding?
**A:** Gaussian approximation (or "density ratio" method) integrates a neural LM without full computation at each beam step: (1) Train a small, fast "background" LM (e.g., an n-gram). (2) At each decoding step, compute: score(prefix + w) = score(prefix) + log P_acoustic(w|...) + λ * [log P_neural(w|prefix) - log P_background(w|prefix)]. The term log P_neural - log P_background represents the "gain" from using the neural LM over the background LM. (3) Approximate this gain using a pre-computed look-up table or a lightweight regression model trained on the difference between neural and n-gram LM scores. (4) Only compute the full neural LM when the gain is uncertain (or at the end for n-best rescoring). This reduces neural LM computations by 80-90% during first-pass decoding while maintaining most of the accuracy benefit. The background LM can also be used as the main decoder LM for speed.

## Q42: What is the "teacher-student" training approach in ASR and how does it enable model compression?
**A:** Teacher-student (knowledge distillation) training: (1) A large, accurate "teacher" model (e.g., Conformer-XXL) generates soft targets (logits) on a large unlabeled audio dataset. (2) A smaller "student" model (e.g., Conformer-S or even LSTM-based) is trained to match the teacher's output distributions. The student loss combines: L = α * L_CE(student_output, hard_labels) + (1-α) * L_KL(student_logits || teacher_logits), where L_KL is the KL divergence between teacher and student probability distributions. (3) The temperature parameter T controls softness: logits_soft = logits / T, higher T produces softer distributions that teach the student about class relationships. Benefits: student models achieve 80-95% of teacher accuracy with 5-10% of the parameters. For streaming models, the teacher is often a non-streaming model and the student is streaming — the teacher's "future context" knowledge is distilled into the causal student.

## Q43: How do you implement a custom "bias list" or "boosted LM" for domain-specific terms without retraining?
**A:** Server-side bias list implementation: (1) Build a trie (prefix tree) from the bias phrase list. (2) During beam search, at each decoding step, check if the current prefix matches any bias phrase prefix. (3) Apply a contextual bonus: boost tokens that continue a matched bias phrase, e.g., add β * depth_of_match to the token log-probability (β=2-5). (4) Handle overlapping bias phrases (e.g., "machine learning" and "machine learning model") by tracking all active matches. (5) For RNN-T, inject bias scores at the joint network output: P_bias(y|context) = softmax(original_logits + bias_vector), where bias_vector has higher values for tokens in active bias phrases. (6) Decay the bias after the phrase is completed to prevent biasing subsequent unrelated speech. This approach requires no model retraining and can be updated dynamically (e.g., load user-specific contacts at conversation start).

## Q44: How does the joint network in RNN-T work and what is its output dimension?
**A:** The RNN-T joint network combines the encoder output (h_t) and prediction network output (g_u) into a single representation for output token prediction. The standard formulation: z_{t,u} = f(W_h h_t + W_g g_u + b), where f is typically tanh or ReLU activation. This is followed by a linear projection to vocabulary size V: output(z_{t,u}) = W_out z_{t,u} + b_out, producing logits over V tokens (including blank). The joint network's output dimension is typically 512-1024 for medium models. The key property: z_{t,u} captures the interaction between acoustic context (from h_t) and linguistic context (from g_u) at each possible alignment point (t,u). The joint operation is computationally expensive because it must be computed at every (t,u) pair in the alignment lattice during training. During inference, efficient implementations avoid computing all pairs by exploiting the monotonic alignment.

## Q45: What is the difference between "frame-synchronous" and "label-synchronous" decoding in ASR?
**A:** Frame-synchronous decoding: the model makes a prediction at every input frame. CTC and RNN-T are frame-synchronous — they emit a token (or blank) for each time step. This means the number of output steps equals the number of input frames (plus blank emissions), naturally supporting streaming. Label-synchronous decoding: the model generates one output per label, continuing until an end-of-sentence token is emitted. Attention-based encoder-decoder (AED/LAS) models are label-synchronous — the number of output steps equals the transcript length. Label-synchronous models require the full input before generating output (non-streaming). A hybrid approach: use frame-synchronous (RNN-T) for streaming first-pass, then label-synchronous (LAS) for second-pass refinement with full context. Frame-synchronous models have simpler streaming but may produce empty/incomplete prefixes; label-synchronous models produce more coherent outputs.

## Q46: How do you implement ASR keyword spotting (KWS) using a CTC-based model?
**A:** CTC-based keyword spotting: (1) Train a CTC model with a vocabulary that includes keyword tokens (e.g., "hey_jarvis", "stop", "play"). (2) During inference, run the CTC encoder on streaming audio frames. (3) For each keyword, maintain a "keyword prefix score" using CTC prefix beam search, but only tracking hypotheses that match keyword prefixes. (4) A keyword is detected when the probability of the full keyword sequence exceeds a threshold (e.g., P(keyword|audio) > 0.5). (5) For background (non-keyword), use a "filler" model — the probability of non-keyword tokens plus blank must be lower than the keyword probability. (6) Alternatively, use a dedicated KWS model (smaller, optimized for few keywords) that shares the same CTC framework but uses a single "target" output node per keyword. The advantage of CTC-based KWS over traditional HMM-based KWS is better noise robustness and easier keyword addition (fine-tune only).

## Q47: How does the Connectionist Temporal Classification (CTC) algorithm handle the conditional independence assumption and what are its practical limitations?
**A:** CTC assumes that output labels at each timestep are conditionally independent given the input: P(y|x) = Π_t P(y_t|x, y_{<t}) where the assumption is that y_t depends only on x and not on previous outputs y_{<t}. This is a strong assumption because speech has clear linguistic dependencies (a word influences the next word). Practical limitations: (1) CTC models tend to produce over-confident but incorrect predictions because they cannot model label dependencies. (2) CTC cannot capture language model patterns inherently — it needs an external LM for good accuracy. (3) CTC produces many redundant blanks between tokens. (4) Prons to "peakier" distributions where most probability mass is on blank, making the model less confident about non-blank tokens. Mitigations: (1) Use RNN-T instead (models label dependencies through prediction network), (2) Use CTC with an external LM via shallow fusion, (3) Use joint CTC/attention training (CTC loss + cross-entropy loss for decoder).

## Q48: How does the minimum word error rate (MWER) training differ from maximum likelihood training for ASR?
**A:** MWER training directly optimizes the word error rate (or a differentiable approximation). Instead of maximizing P(correct_transcript | audio) as in maximum likelihood, MWER minimizes the expected WER under the model's distribution: L = E_{y~p(y|x)}[WER(y, y_gt)]. Since this expectation is intractable, MWER uses the following approach: (1) Generate an n-best list of hypotheses using beam search (n=4-8). (2) Compute the "reward" for each hypothesis as 1 - normalized_WER(hyp, ref). (3) Optimize: L = Σ_{hyp in n-best} [exp(score(hyp)/τ) / Σ exp(score/τ)] * WER(hyp, ref), where τ is a temperature. This is a form of policy gradient. MWER trains the model to reduce its expected error rate rather than just matching the reference, which often yields 10-20% relative WER reduction over cross-entropy training, especially for challenging cases.

## Q49: How do you handle "open microphone" scenarios where background speech (babble) is present during ASR?
**A:** Babble noise is particularly challenging because it's non-stationary and speech-like. Mitigations: (1) **Multi-channel processing** — use microphone arrays with beamforming (MVDR, delay-and-sum) to spatially filter the target speaker. (2) **Target speaker extraction** — use a speaker extraction model (SpEx+) that takes a reference enrollment utterance (a few seconds of target speaker) and extracts that speaker's voice from babble. (3) **Continuous speech separation** (CSS) — separate overlapping speakers into distinct streams before ASR. (4) **Perceptual weighting** — modify the ASR loss to emphasize frequencies where target speech dominates (using ratio mask as weights). (5) **Babble augmentation** — during training, mix target speech with multi-talker babble at various SNRs. (6) **Confidence-based filtering** — for word-level ASR output, filter words with confidence below a threshold, as babble-corrupted words typically have lower confidence scores.

## Q50: How does the "Conformer" architecture's convolution module actually work and what kernel size is typically used?
**A:** The Conformer convolution module has this exact structure: (1) Layer normalization on the input, (2) Pointwise convolution (kernel=1) projecting from d_model to 2*d_model, (3) GLU (Gated Linear Unit) activation — splits into two halves, applies sigmoid to one half and element-wise multiplies, reducing back to d_model, (4) Depthwise convolution (kernel=31, groups=d_model) — each input channel is convolved independently with a learned 1D kernel of size 31, (5) Batch normalization, (6) Swish activation (x * sigmoid(x)), (7) Pointwise convolution (kernel=1) projecting back to d_model, (8) Dropout. The kernel size of 31 is a well-tuned value — large enough to capture local phonetic patterns (formant transitions spanning ~310ms at 10ms frame shift) but small enough to be computationally efficient. Some variants use kernel sizes of 15-65 depending on the model scale. The depthwise convolution is the key: it provides position-aware local pattern detection (lacking in self-attention) with O(k×d) complexity instead of O(d²).

## Q51: How do you implement streaming batch inference where multiple audio streams are processed simultaneously on GPU?
**A:** Streaming batch inference: (1) **Dynamic batching** — collect audio chunks from multiple streams arriving within a small time window (e.g., 10ms), pad to the same length within the batch. (2) **Sequence length masking** — use attention masks and padding masks to ensure each stream processes only its valid frames. (3) **Stream state management** — each stream maintains its own KV-cache (for cross-attention), prediction network state, and decoding state. These are stored in a dictionary indexed by stream ID and passed through the batch. (4) **Chunk boundary alignment** — align chunk boundaries across streams so they can be processed together (e.g., wait 320ms or until all streams in the batch have data). (5) **Priority scheduling** — prioritize streams near their latency budget to avoid deadline violations. (6) **Variable batch size** — start inference as soon as the batch fills or a timeout expires (like NVIDIA Triton's dynamic batching). Typical batch sizes: 4-32 streams, achieving 2-5× throughput improvement over sequential stream processing.

## Q52: How does "semi-supervised learning" with noisy student training work for ASR?
**A:** Noisy student training for ASR: (1) Train a teacher model on labeled data (e.g., 10K hours). (2) Use the teacher to generate pseudo-labels on unlabeled data (e.g., 100K hours). (3) Train a student model (same architecture as teacher or larger) on the combined labeled + pseudo-labeled data. Crucially, add noise to the student during training: SpecAugment, dropout, stochastic depth, and random feature masking. (4) Optionally, iterate — the student becomes the new teacher for the next round. The noise is essential: it forces the student to learn beyond mere memorization of teacher outputs, improving generalization. Results: noisy student training can reduce WER by 10-25% over teacher-only training. The pseudo-labels should be filtered by confidence (e.g., keep only utterances where teacher confidence > 0.8) and by WER against itself (to avoid reinforcing teacher errors). Multiple iterations (2-3) provide diminishing returns.

## Q53: How do you perform ASR model comparison between different vendors (Whisper, Deepgram, Azure, Google) in a production setting?

Minimum 2-3 hours per category. (2) **Standardized pre-processing** — ensure consistent audio format (16kHz, mono, 16-bit PCM) across vendors. (3) **Metrics** — compute WER, SER (semantic error rate), latency (p50/p95/p99), RTF, and cost-per-hour. (4) **Bake-off framework** — send all test audio to each vendor's API simultaneously, collect results, run evaluation pipeline. (5) **A/B test in production** — route 5% of production traffic to each vendor during low-traffic periods, collect real-world metrics. (6) **Evaluate qualitative aspects** — punctuation quality, capitalization, inverse text normalization, speaker diarization (if applicable). (7) **Consider non-functional factors** — latency stability (not just average), availability SLA, data privacy compliance, rate limits, and pricing model (per-audio-second vs. per-character).

The system uses the uploaded data to fine-tune a base model through transfer learning. ## Q55: How does Google Cloud Speech-to-Text's "Chirp" model differ from the standard model?

Differences from standard Google STT: (1) **Multi-modal pre-training** — uses both audio-only and audio-text joint pre-training (contrastive loss between audio and text embeddings). (2) **Universal model** — Chirp is a single model supporting 100+ languages without separate per-language models, achieving competitive WER across languages. (4) **Adaptive computation** — automatically adjusts the computational budget based on audio complexity (simpler audio uses fewer layers). (5) **API features** — Chirp supports streaming with word-level timestamps, automatic punctuation, profanity filtering, and speaker diarization in a single model. ## Q56: How does AssemblyAI's "Conformer-2" improve upon Conformer-1?
**A:** AssemblyAI's Conformer-2 introduces: (1) **Unified streaming and batch** — same model architecture can switch between streaming (chunked causal) and batch (full context) modes by controlling the attention mask, (2) **Rotary Position Embeddings (RoPE)** — replaces absolute positional encoding, improving generalization to different audio lengths and providing better relative position modeling, (3) **SwiGLU activation** — replaces standard Swish in feed-forward layers, providing better gradient flow, (4) **Pre-normalization** — applies layer norm before each sublayer (Pre-LN) instead of after, stabilizing training of deeper models, (5) **Larger context** — extended attention span (30s vs 20s) for better long-range modeling, (6) **Audio-text joint training** — uses text embeddings to condition the audio encoder during training (text knowledge distilled into audio encoder). These improvements reportedly yield ~15% relative WER reduction over Conformer-1 while maintaining the same inference speed.

## Q57: How do you implement a custom "end-of-utterance" detection using ASR decoder confidence?
**A:** End-of-utterance detection using decoder confidence: (1) During streaming ASR, maintain a sliding window of the last N (e.g., 20) output tokens' confidence scores. (2) After each new token, compute: avg_confidence = mean of last N confidences. (3) If avg_confidence drops below a threshold (e.g., 0.3) AND silence from VAD exceeds a time threshold (e.g., 500ms), signal end-of-utterance. (4) Additionally, track "improbable tokens" — if the model outputs a high-entropy prediction (uncertain across many candidates), it likely indicates the end of speech. (5) Use the RNN-T blank token probability — if blank probability dominates for K consecutive frames (K = expected gap between words × frame_rate), end-of-utterance is likely. (6) Combine VAD + decoder confidence + blank probability into a weighted decision: EOU_score = w1*VAD_silence_prob + w2*(1-avg_confidence) + w3*blank_prob. Optimize weights on a held-out set to minimize false accepts (cutting off mid-speech) and false rejects (waiting too long).

## Q58: How does the combination of CTC and attention (hybrid CTC/attention) work for ASR?
**A:** Hybrid CTC/Attention jointly trains CTC and attention-based decoder losses. The total loss: L = λ * L_ctc + (1-λ) * L_att, where λ is typically 0.1-0.3. During decoding, both CTC and attention scores are combined: y* = argmax [λ * log P_ctc(y|x) + (1-λ) * log P_att(y|x)]. The CTC branch operates on encoder outputs and provides: (1) **Regularization** — encourages monotonic alignment and prevents the attention decoder from skipping parts of the input (attention collapse). (2) **Streaming capability** — the CTC branch can produce streaming outputs while the attention branch refines them. (3) **Joint decoding** — during inference, one-pass decoding uses both scores: CTC provides a strong left-to-right bias, and attention provides context-aware refinement. (4) **Alignment** — CTC alignments can initialize attention for faster convergence. ESPnet implements hybrid CTC/Attention, where CTC loss is computed on encoder outputs and cross-entropy loss is computed on decoder outputs, with shared encoder parameters.

## Q59: How do you handle regional accents and dialects in a deployed ASR system without per-dialect models?
**A:** Accent-robust ASR without per-dialect models: (1) **Accent augmentation during training** — use voice conversion (e.g., VC-GAN, FreeVC) to convert existing training data into different accents, effectively expanding the training distribution. (2) **Speaker-adversarial training** — train the encoder with a gradient reversal layer that tries to predict speaker/dialect identity — the encoder learns accent-invariant features. (3) **Accent embedding conditioning** — extract an accent embedding (from a pre-trained accent classifier) and concatenate it with encoder features, allowing the model to adapt per-utterance. (4) **Multi-accent fine-tuning** — fine-tune the base model on a curated multi-accent dataset (e.g., Common Voice, L2-ARCTIC) with accent labels. (5) **Domain-adversarial loss** — add an auxiliary domain classifier with gradient reversal during training, forcing the encoder to produce representations that are indistinguishable across accents. (6) **Self-supervised pre-training** — models like Wav2Vec 2.0 pre-trained on diverse unlabeled speech capture accent-robust representations. WER gaps between standard and regional accents can be reduced from 15-20% to 3-5% with these techniques.

## Q60: How does the "Emformer" architecture achieve constant memory usage for long-form streaming ASR?
**A:** Emformer's constant memory comes from its segment-based processing with a fixed-size memory bank: (1) Audio is partitioned into equal-length segments (e.g., 4 seconds = 400 frames at 10ms per frame). (2) Each segment is processed independently, with attention only within the segment plus a memory bank of K previous segment summaries (K=128-256 frames). (3) The memory bank is implemented as a fixed-size FIFO queue — oldest entries are dropped as new segment summaries are added. (4) Segment summaries are generated by pooling the encoder output of the right half of each segment (avg pooling), compressing ~200 frames into a single summary vector. (5) The total memory = O(segment_size² + segment_size × memory_size), which is constant since segment_size and memory_size are fixed. (6) This means Emformer can process arbitrarily long audio streams without growing memory, unlike standard Transformers whose memory grows quadratically with input length. Emformer achieves this with <5% WER degradation compared to full-context Transformers.

## Q61: How do you integrate a pre-trained language model (e.g., GPT, LLaMA) with ASR for context-aware transcription?
**A:** Integrating LLMs with ASR: (1) **LLM rescoring** — generate an n-best list from the ASR decoder, then score each hypothesis with the LLM: P_llm(hyp | context_prompt), where context_prompt includes conversation history, user preferences, and domain descriptions. Rank by combined ASR + LLM score. (2) **LLM post-editing** — pass the ASR transcript with a prompt like "Fix transcription errors in this medical dictation: {transcript}." The LLM corrects errors using world knowledge. (3) **LLM as predictor** — use the LLM's hidden states as the prediction network in RNN-T (replacing the standard LSTM prediction network), providing richer linguistic context. (4) **Audio-conditioned LLM** — project audio encoder outputs into the LLM's embedding space and feed them as prefix tokens, treating ASR as a language modeling problem. (5) **In-context learning** — provide few-shot examples of corrected transcripts in the prompt to adapt the LLM to the specific domain on-the-fly. Latency is the main challenge — LLM inference adds 100-1000ms. Practical approaches use small LLMs (1-7B parameters) with KV-cache optimization.

## Q62: How does "acoustic echo cancellation" interact with ASR in a voice-calling scenario?
**A:** In voice calling (full-duplex), AEC removes the far-end audio from the near-end microphone signal before ASR. The interaction: (1) AEC uses adaptive filtering (NLMS or Kalman filter) to model the echo path from speaker to microphone, subtracts the estimated echo from the mic signal. (2) Residual echo suppressor (RES) removes non-linear echo components (speaker distortion, harmonic artifacts). (3) The cleaned signal is fed to ASR. Common issues: (a) **Double-talk** — both parties speak simultaneously, causing AEC divergence. Solutions: double-talk detectors freeze filter adaptation during double-talk. (b) **Non-linearities** — clipping or speaker saturation creates harmonic echo components that linear AEC cannot remove. Solutions: use neural AEC (DeepAEC) with LSTM-based echo suppression. (c) **ASR degradation** — if AEC over-suppresses, the near-end speech can be distorted, increasing WER by 10-30%. (d) **Delay** — AEC processing adds 5-20ms latency, which can be critical for real-time ASR. Modern AEC uses deep neural networks for full-band echo suppression, achieving 20-30dB echo return loss enhancement (ERLE).

## Q63: What is "multi-channel ASR" and how does beamforming improve accuracy?
**A:** Multi-channel ASR uses multiple microphones (array) to improve robustness to noise and reverberation. Beamforming is the core technique: (1) **Delay-and-sum beamforming** — time-align signals from each microphone based on the direction of arrival (DOA) and sum them, creating constructive interference for the target direction and destructive for others. (2) **MVDR (Minimum Variance Distortionless Response)** — adaptive beamformer that minimizes output power while maintaining unity gain in the target direction. (3) **WAV (Weighted Prediction Error)** — jointly performs dereverberation and beamforming. (4) **Neural beamforming** — use a DNN to estimate time-frequency masks for each channel, then compute beamforming filters from the masked covariance matrices. Multi-channel ASR typically reduces WER by 20-40% relative in noisy conditions compared to single-channel, especially in far-field scenarios (3-5 meters distance). The array geometry (linear, circular, spherical) affects spatial resolution and frequency coverage.

## Q64: How does "voice activity detection" (VAD) affect ASR latency in a streaming system?
**A:** VAD directly impacts streaming ASR latency in several ways: (1) **Pre-roll delay** — most VAD systems require a detection window (20-50ms) before deciding "speech detected." Some VADs accumulate frames before classification, adding latency. (2) **Sensitivity vs. false accepts** — a more sensitive VAD detects speech faster but may trigger ASR on noise (false accept), wasting computation. A less sensitive VAD is more accurate but may clip the first 50-200ms of speech. (3) **End-of-speech delay** — VAD must wait for a silence period (200-800ms) before deciding "speech ended." Longer waits increase latency but reduce premature cutoff. (4) **Hangover time** — VAD extends speech detection for 100-200ms after apparent silence to avoid clipping word-final consonants and filled pauses. (5) **Barge-in detection** — in conversational AI, VAD must detect when the user interrupts, requiring re-decision within 100ms. Optimized VAD (e.g., WebRTC VAD with tuned mode) achieves 20-50ms pre-roll latency for 95th percentile cases. Some systems use VAD-less streaming (always processing audio with a low-noise model) to eliminate VAD latency.

## Q65: How do you implement "prosody detection" (emphasis, tone, question rise) alongside ASR?
**A:** Prosody detection with ASR: (1) **Joint model** — add an auxiliary prosody head to the ASR encoder that predicts per-frame prosodic features: pitch (F0 contour), energy, speaking rate, and spectral tilt. These can be trained with supervision from manually annotated prosodic events. (2) **Post-processing** — extract F0 contour from the audio using a pitch tracker (PDAO, CREPE, or PyWorld), compute prosodic features within word boundaries (using ASR word timestamps), and classify prosodic events (question rise, emphasis, topic boundary) using a separate classifier (SVM or small Transformer). (3) **Multi-task learning** — train the ASR model jointly to predict both text and prosody labels: L = L_asr + λ * L_prosody, where prosody labels include accent (primary, secondary, none) and boundary (major, minor, none) from the ToBI (Tones and Break Indices) annotation system. (4) **Uses** — question detection (final rise), emphasis detection (pitch accent + duration lengthening), and topic segmentation. Prosody-aware ASR improves downstream tasks like sentiment analysis and intent detection by 5-10%.

## Q66: How does the "lattice-free maximum mutual information" (LF-MMI) criterion compare to CTC for ASR training?
**A:** LF-MMI is a sequence discriminative training criterion that maximizes the mutual information between the acoustic observation and the word sequence. Unlike CTC which uses a fixed blank-based topology, LF-MMI uses a numerator graph (exact word sequence) and a denominator graph (all possible word sequences, typically a phone n-gram language model). Key differences: (1) **Training signal** — LF-MMI provides per-frame gradients even for non-target paths because the denominator graph includes alternative word sequences, while CTC only considers the target sequence. (2) **Language model integration** — LF-MMI inherently incorporates the denominator LM during training, making it better at discriminating confusable words. (3) **Acoustic model** — LF-MMI was designed for hybrid DNN-HMM models with context-dependent phones; CTC works with any output units. (4) **Convergence** — LF-MMI converges faster in low-resource setups because the denominator graph provides richer training signal. (5) **WER** — LF-MMI typically achieves 5-10% relative WER improvement over CTC for the same acoustic model, especially on challenging acoustic conditions. However, LF-MMI is more complex to set up (requires phone-level alignments, decision tree, and denominator graph construction).

## Q67: How do you handle "channel mismatch" between training and inference ASR data (e.g., telephone vs. studio microphone)?
**A:** Channel mismatch mitigation: (1) **Multi-condition training** — train on data from diverse channels (telephone, VoIP, mobile, studio mic, webcam). Augment training data with random channel characteristics: band-pass filtering, equalization, codec compression (G.711, G.729, Opus). (2) **Channel normalization** — apply feature-space normalization: CMVN (cepstral mean and variance normalization) per utterance, VTLN (vocal tract length normalization) to warp frequency axis, or fMLLR (feature-space Maximum Likelihood Linear Regression) to adapt features. (3) **Front-end adaptation** — estimate a linear transform of the input features to match training conditions using a small amount of adaptation data (e.g., 5-10 seconds from the target channel). (4) **Invariant representation learning** — train a domain-adversarial model (gradient reversal layer with channel classifier) to produce channel-invariant encoder representations. (5) **Channel embedding** — extract a channel embedding (using a small classifier head) and concatenate with acoustic features, allowing the model to learn channel-specific adjustments. (6) **Test-time adaptation** — use test-time normalization or few-shot adaptation to adjust batch-normalization statistics to match the target channel.

## Q68: What is "on-device ASR" and how does it differ architecturally from cloud ASR?
**A:** On-device ASR processes audio entirely on the user's device (phone, edge device). Architectural differences from cloud ASR: (1) **Model size** — on-device models are 10-100MB vs. 1-5GB for cloud, achieved via quantization (INT8), pruning, and knowledge distillation. (2) **Architecture** — on-device uses memory-efficient architectures like Emformer, depthwise separable Conformers, or RNN-T with compact prediction networks (2-3 LSTM layers). (3) **Streaming focus** — on-device ASR is always streaming (no batch processing), with RTF < 0.3 on device CPU/NPU. (4) **Vocabulary** — on-device typically has smaller vocabulary (~50K vs. 200K+ for cloud) with dynamic bias lists for OOV words. (5) **LM** — on-device uses small n-gram or distilled neural LMs (10-50MB) vs. large Transformer LMs (GBs) in cloud. (6) **Offline** — no network dependency, but no access to server-side models for error correction. (7) **Privacy** — audio never leaves the device. Leading on-device ASR solutions: Google's Gboard ASR (RNN-T on Pixel Neural Core), Apple's Siri on-device ASR, and whisper.cpp with optimized small models.

## Q69: How does "progressive learning" or "curriculum learning" improve ASR training convergence?
**A:** Curriculum learning for ASR presents training examples in order of increasing difficulty: (1) **Stage 1 (Easy)** — clean speech (SNR > 20dB), short utterances (2-5s), slow speaking rate, restricted vocabulary (common words). (2) **Stage 2 (Medium)** — moderate noise (SNR 10-20dB), longer utterances (5-15s), normal speaking rate, full vocabulary. (3) **Stage 3 (Hard)** — noisy speech (SNR 0-10dB), very long utterances (15-30s), fast speech, code-switching, disfluencies. Difficulty scoring: use a pre-trained ASR model's WER to rank training examples (higher WER = harder). Alternatively, use heuristics: utterance length, SNR, word frequency diversity. Benefits: (1) Faster initial convergence (2-3× speedup in early epochs), (2) Better final WER (2-5% relative improvement), (3) Avoids early overfitting to easy examples. The curriculum schedule (when to transition between stages) can be fixed (e.g., epoch 1-3 easy, 4-6 medium, 7-10 hard) or adaptive (based on validation WR plateau detection).

## Q70: How does the "stacked sub-sampling" technique reduce computational cost in ASR encoders?
**A:** Stacked sub-sampling reduces the frame rate as audio progresses through the encoder, decreasing sequence length and computational cost: (1) **Time stride convolution** — first two encoder layers use convolutions with stride 2, reducing 100Hz → 50Hz → 25Hz frame rate. (2) **Stacking** — in input pre-processing, stack 3 consecutive frames (80-dim → 240-dim), then apply a linear projection, reducing effective frame rate from 10ms → 30ms. (3) **VGG-style sub-sampling** — use VGG blocks with max-pooling to reduce time resolution by 4× before the main encoder (used in Listen, Attend, Spell). (4) **Convolutional sub-sampling with residual connections** — downsample time dimension while preserving information via residual connections around the strided convolution. Benefits: 4-8× reduction in encoder sequence length, which reduces self-attention complexity from O(T²) to O((T/4)²) — a 16× reduction. The accuracy impact is minimal (0.5-1% WER increase) if sub-sampling is well-designed. Modern Conformers use stacking or strided convolutions to achieve ~25 Hz frame rate from 100 Hz input.

## Q71: How do you build a custom ASR model for a language with only 10 hours of transcribed data?
**A:** Low-resource ASR (10 hours): (1) **Transfer learning** — start from a pre-trained model (e.g., Wav2Vec 2.0 XLSR-53, Whisper, or HuBERT pre-trained on 100+ languages). (2) **Self-supervised pre-training** — if unlabeled audio is available (100+ hours), pre-train using Wav2Vec 2.0 on target language unlabeled data → fine-tune on 10 hours labeled data. (3) **Cross-lingual transfer** — use XLSR-53 (pre-trained on 53 languages) and fine-tune on target language. (4) **Data augmentation** — augment the 10 hours with: speed perturbation (0.9x, 1.0x, 1.1x), noise addition, SpecAugment, and back-translation (synthesize speech from text using TTS). (5) **Semi-supervised learning** — train a teacher model on labeled 10h, generate pseudo-labels on unlabeled data, train student on combined data. (6) **Subword modeling** — use a shared BPE vocabulary related languages (if available) for cross-lingual transfer. (7) **Language model** — build an n-gram LM from target language text (can be obtained from web crawl, does not require transcribed audio). With these techniques, a 10-hour language can achieve 15-25% WER (vs. 40-60% without these techniques).

## Q72: How does the "noise robust loss" (e.g., SNR-Invariant Minimum Word Rate) improve ASR in noisy conditions?
**A:** Noise-robust training objectives modify the ASR loss to emphasize invariance to noise: (1) **SNR-weighted loss** — weight the per-utterance loss by 1/SNR or use a sigmoid weight that upweights noisy utterances: L = Σ_i w(SNR_i) * L_i where w(SNR) = 1 - sigmoid((SNR - SNR0)/τ). This makes the model focus more on difficult noisy utterances. (2) **Invariant representation loss** — add a term that minimizes the distance between representations of clean and noisy versions of the same utterance: L_inv = ||f(x_clean) - f(x_noisy)||², where f is the encoder output. (3) **Adversarial noise robustness** — add a discriminator that tries to predict the noise type from encoder outputs, and train the encoder adversarially to fool it. (4) **SNR-aware training** — condition the model on estimated SNR (as an auxiliary input feature or embedding), then during inference set SNR to a high value to get clean-output behavior. (5) **Mix-up training** — for each batch, mix clean and noisy speech (interpolation in input or feature space): x_mix = α*x_clean + (1-α)*x_noisy, y_mix = y_clean (label mixing optional). These techniques reduce WER in low-SNR conditions by 10-30% relative.

## Q73: How does "Rescore-by-ASR" (self-correcting) decoding work and when is it beneficial?
**A:** Rescore-by-ASR is a technique where an ASR model re-scores its own top hypotheses: (1) First pass: run beam search (beam width = 8-16) to generate an n-best list of hypotheses. (2) For each hypothesis, compute a "reverse" score: pass the hypothesis text through a text encoder (or use the ASR decoder) conditioned on the audio to compute P(audio | hypothesis). (3) Re-rank: combined_score = log P(hypothesis | audio) + λ * log P(audio | hypothesis). (4) Alternatively, use a confidence estimation model per hypothesis and select the one with the best confidence-calibrated score. The intuition: if the ASR is confident about a hypothesis (high P(hyp|audio)), it's probably correct. But if P(audio|hypothesis) is low, the hypothesis may not explain the audio well. Self-correction is beneficial when beam search produces competing hypotheses with similar scores — rescoring with a different perspective (or the same model in a different mode) resolves ambiguity. WER reduction of 5-15% is typical for ambiguous utterances (homophones, proper names).

## Q74: How do you deploy ASR on a Raspberry Pi (or similar low-power device) with <500MB RAM?
**A:** ASR on Raspberry Pi constraints (500MB RAM, ARM CPU, no GPU): (1) **Model selection** — use Whisper "tiny" (~39M params, ~6MB) or a distilled small Conformer-Transducer (~10M params, ~5MB INT8). (2) **Quantization** — INT8 quantization via ONNX Runtime or TFLite. Quantize both weights and activations. (3) **Architecture optimizations** — use depthwise separable convolutions, avoid self-attention on long sequences (use chunked/streaming mode). (4) **Feature extraction on NPU** — offload MFCC/spectrogram computation to the Raspberry Pi's VideoCore GPU. (5) **Streaming inference** — process audio in 320ms chunks with state caching. (6) **Threading** — use 4 CPU cores: 1 for audio capture, 1 for feature extraction, 2 for model inference (pipeline parallelism). (7) **Memory management** — use memory-mapped model files (mmap) to avoid loading the entire model into RAM. (8) **VAD-based scheduling** — only run ASR inference during speech segments. Results: whisper.cpp with tiny model achieves RTF ~1.0-1.5 on Pi 4 (slower than real-time but usable for non-streaming). For real-time, a distilled Conformer-Transducer with INT8 achieves RTF ~0.5. WER is typically 10-15% on clean speech.

## Q75: How does "phoneme-based ASR" differ from "grapheme-based ASR" and when would you use each?
**A:** Phoneme-based ASR: (1) Output units are phonemes (40-60 for English, 30-200 depending on language). (2) Requires a pronunciation dictionary (lexicon) to map words to phoneme sequences (e.g., "cat" → K AE1 T). (3) Training requires phoneme-level alignments or CTC over phonemes. (4) Better for languages with irregular spelling (English, French) because phonemes abstract away spelling irregularities. (5) More parameter-efficient — small output vocabulary. Grapheme-based ASR: (1) Output units are characters/graphemes (26 for English, thousands for Chinese). (2) No pronunciation dictionary needed — text is used directly as output. (3) Simpler pipeline (no lexicon management). (4) Better for languages with transparent spelling (Spanish, Hindi, Korean). (5) More data-hungry — needs more training data to learn spelling variations. If you have <100 hours of training data and a language with irregular spelling, phoneme-based ASR (with a good lexicon) typically outperforms grapheme-based by 5-15% relative WER. With >1000 hours, grapheme-based can match or exceed phoneme-based.

## Q76: How do you measure and optimize the "memory bandwidth" of ASR model inference on edge devices?
**A:** Memory bandwidth optimization for edge ASR: (1) **Measure bandwidth** — use profiling tools (Arm Streamline, perf stat) to identify memory-bound vs. compute-bound operations. ASR inference is typically memory-bound (model weights > cache). (2) **Weight quantization** — INT8 reduces memory traffic by 4× vs. FP32, INT4 by 8×. Combined with activation quantization, this is the most impactful optimization. (3) **Operator fusion** — fuse adjacent operations (e.g., conv + batch_norm + activation) into a single kernel to reduce intermediate memory reads/writes. (4) **Memory layout** — use NHWC layout on ARM CPUs (vs. NCHW on GPUs) to enable vectorized memory access. (5) **Cache-friendly architecture** — design smaller models that fit in L2 cache (typical edge device L2: 256KB-1MB). At 4-8 bytes per weight, a 10M param model is 40-80MB — must be in DRAM. (6) **Weight pruning** — unstructured pruning (remove 50-80% of weights) converts to sparse format, but needs sparse hardware support for bandwidth benefit. Structured pruning (remove channels/filters) is more effective for bandwidth. (7) **Memory pooling** — pre-allocate all intermediate tensor buffers to avoid runtime allocation overhead.

## Q77: What is "ASR with multi-turn dialogue context" and how do you implement it?
**A:** Multi-turn ASR leverages the conversation history to improve current-utterance transcription: (1) **Context encoding** — encode previous utterances using a text encoder (e.g., BERT) and concatenate with the audio encoder features. (2) **Biased decoding** — from the conversation history, extract topic-specific terms and proper names, add them as a dynamic bias list for the current utterance. (3) **Contrastive decoding** — if the conversation history contains entities, contrast the ASR model's predictions with and without context to bias towards contextually relevant entities. (4) **LM adaptation** — maintain a per-session LM that adapts from the conversation history (cache LM: boost n-grams seen in history). (5) **Prompt engineering** — prepend conversation context as text prompt to the ASR decoder (e.g., Whisper-style: "<|previous: Hello, I'm calling about my account #12345|> <|current:|>"). (6) **Query reformulation** — use the conversation history to determine that the current utterance likely contains an account number, and boost numeric token probabilities. Multi-turn ASR reduces WER by 10-20% for entity-rich conversational domains (e.g., customer support) by leveraging topic continuity and entity re-mention.

## Q78: How does "self-training with noisy student" compare to "knowledge distillation" for ASR model improvement?
**A:** Both improve models but with key differences:

| Aspect | Noisy Student Training | Knowledge Distillation |
|--------|----------------------|----------------------|
| Teacher size | Same architecture or slightly larger | Typically much larger (2-10×) |
| Student size | Same or larger | Smaller (compression) |
| Data | Uses UNLABELED data (pseudo-labeled) | Uses labeled data + teacher logits |
| Noise | Essential — SpecAugment, dropout, masking | Not required, but helps |
| Iterations | Multiple rounds (student becomes teacher) | Single round typically |
| Goal | Leverage more data | Compress model |

In practice: use noisy student when you have abundant unlabeled audio but limited labeled data. Use knowledge distillation when you need to deploy a smaller model while retaining accuracy. They can be combined: noisy student on unlabeled data creates better pseudo-labels → distillation to compress. For a production ASR system, the sequence would be: labeled training → teacher model → noisy student on unlabeled (creates better teacher) → distill to small model for deployment.

## Q79: How do you implement "streaming language model biasing" without rebuilding the LM graph for each user?
**A:** Streaming LM biasing without per-user LM rebuild: (1) **Trie-based biasing** — maintain a global LM (n-gram). For per-user bias phrases, build a small trie (prefix tree) that is loaded at runtime. During beam search, at each step, check if the current prefix matches any bias phrase in the trie. Boost the bias continuation tokens by adding a bias score: boosted_score = original_score + λ * match_depth. (2) **Per-user context embeddings** — train a small context encoder that converts the bias phrase list (e.g., user's contacts, recent topics) into a fixed-size embedding vector. Inject this embedding into the RNN-T prediction network or as an additional encoder conditioning. (3) **Neural cache model** — maintain a cache of recently recognized phrases (from the current conversation). Boost n-grams that match cache entries using a small cache-based LM: P_cache(phrase) = (count_in_cache + ε) / total_cache. (4) **FST composition** — compose the main LM FST with a small per-user bias FST (built from bias phrases) at runtime. FST composition is fast (O(n) in practice) and doesn't require rebuilding the main LM. The bias FST is built once per session from the user's bias list. This approach supports 10K+ bias phrases and 1000+ concurrent users with negligible overhead.

## Q80: How does "quantization-aware training" (QAT) differ from post-training quantization (PTQ) for ASR models?
**A:** QAT vs PTQ for ASR:
- **PTQ**: After training, quantize weights by computing scale and zero-point per tensor/channel. Feed a calibration dataset (~100-500 utterances) to determine activation ranges. Simple, fast, no retraining. But: WER degradation of 1-3% at INT8 for ASR models, especially for attention layers.
- **QAT**: During training, insert "fake quantization" nodes that simulate INT8 quantization effects: Q(x) = round(x / scale + zero_point) - zero_point) * scale. Forward pass uses quantized values; backward pass uses straight-through estimator (STE) — gradient bypasses the quantizer. The model learns to produce activations and weights that survive quantization. WER degradation at INT8: 0.1-0.5%. Training requires: (1) Pre-trained FP32 model, (2) QAT for 5-20 epochs with learning rate 1/10 of original, (3) Same training data (or a subset). Special considerations for ASR: (1) Attention softmax is sensitive to quantization — keep in FP16 or use per-layer mixed-precision. (2) Convolution weights need per-channel quantization, not per-tensor. (3) Batch normalization folding before QAT to avoid incorrect quantization statistics. For production ASR, QAT is strongly recommended over PTQ for INT8 deployment.

## Q81: How do you implement "real-time speaker diarization" (who spoke when) during streaming ASR?
**A:** Streaming diarization: (1) **Online VAD** — detect speech segments in real-time. (2) **Speaker embedding extraction** — for each detected speech segment (~1s), extract a speaker embedding (x-vector or d-vector). Use a lightweight embedding extractor (TDNN or ResNet) that runs faster than real-time. (3) **Online clustering** — maintain speaker clusters (centroids) incrementally. When a new embedding arrives: assign to nearest cluster if cosine similarity > threshold (0.5-0.7); else create new cluster. (4) **Update strategy** — update cluster centroid as weighted average of all assigned embeddings (online k-means with forgetting factor). (5) **Overlap handling** — for overlapping speech, parallel embedding extractors process each detected stream separately (requires source separation front-end or multi-channel input). (6) **ASR association** — align diarization output with ASR timestamps: each ASR word gets the speaker label from the closest diarization segment. (7) **Delay** — streaming diarization has inherent delay (1-2s) because you need a minimum segment duration (~500ms) for reliable embedding extraction. (8) **Key challenge** — maintaining consistent speaker labels when the same speaker leaves and returns (global speaker clustering vs. session-based clusters). Streaming DER of 10-15% is typical vs. 5-8% for offline diarization.

## Q82: What is "end-to-end ASR with transducer and attention" (E2E T-A) and how does it combine both approaches?
**A:** E2E T-A (Transducer + Attention) combines RNN-T's streaming capability with attention-based refinement: (1) **Shared encoder** — single Conformer encoder processes audio. (2) **RNN-T decoder** — lightweight prediction + joint network that operates frame-synchronously for streaming first-pass decoding. (3) **Attention decoder** — separate LAS-style decoder that consumes encoder outputs and RNN-T hypotheses (or encoder states) for second-pass refinement. The attention decoder uses cross-attention over the full encoder sequence and receives the RNN-T hypothesis as additional context (via learned embeddings). (4) **Training** — joint loss: L = λ * L_rnnt + (1-λ) * L_att, where L_att is the cross-entropy of the attention decoder given ground-truth text. (5) **Inference** — RNN-T produces streaming hypotheses with low latency (e.g., <500ms); when the utterance ends, the attention decoder refines the hypothesis using full audio context (adds 100-200ms). The combined model achieves WER within 1-3% of non-streaming models while supporting streaming. Google's universal speech model uses this approach, enabling flexible latency vs. accuracy trade-offs.

## Q83: How does "noise-robust feature extraction" work for ASR in extreme noise conditions (SNR < 0dB)?
**A:** Noise-robust features for very low SNR: (1) **Power-normalized cepstral coefficients (PNCC)** — replaces log nonlinearity with power-law nonlinearity and includes medium-time power subtraction, suppressing noise more effectively than MFCCs. (2) **Gammatone frequency cepstral coefficients (GFCC)** — uses gammatone filterbanks that model human auditory filters more accurately, providing better noise robustness. (3) **RASTA-PLP** — Relative Spectral PLP applies a band-pass filter to each spectral channel, removing slow (channel) and fast (noise) variations. (4) **Time-frequency masking** — use an enhancement front-end (e.g., DCCRN or DeepFilterNet) that estimates a spectral mask from noisy audio and applies it before feature extraction. At -5dB SNR, DCCRN improves SI-SNR by 15-20dB. (5) **Multi-resolution features** — combine features at different time scales (10ms, 30ms, 100ms) to capture both phonetic detail and prosodic context. (6) **Learned front-end** — replace hand-crafted features with a learnable filterbank (SincNet, LEAF) trained jointly with the ASR model. At SNR < 0dB, learned front-ends significantly outperform MFCCs (10-20% relative WER reduction), and a DNN enhancement front-end is essential.

## Q84: How do you implement "speculative decoding" for RNN-T to reduce inference latency?
**A:** Speculative decoding for RNN-T: (1) **Draft RNN-T** — train a small, fast draft model (e.g., 2-layer LSTM prediction + joint network, tiny encoder). This draft runs every N frames (e.g., every 320ms) to predict the next K tokens (K=5-10) autoregressively. (2) **Verification** — the full target RNN-T model processes the draft's predicted sequence in a single forward pass through the joint network, computing P_draft(tokens) and P_target(tokens) for the candidate sequence. (3) **Acceptance/rejection** — greedily accept prefix where P_target matches P_draft (argmax agreement). Stop at the first disagreement and emit tokens up to that point. (4) **Fallback** — if the draft predicts incorrectly, the target model takes over from the mistake point. (5) **Speedup mechanism** — verification processes K tokens in one joint network pass (parallel), while the draft generates K tokens autoregressively. Since the draft is much faster (1/10th the size), the overall latency is reduced by 2-3×. Crucially, the output distribution matches the target model's distribution exactly (no accuracy loss). For RNN-T, speculative decoding works best when the blank token is common — the draft's blank predictions speed up frame processing. Implementation complexity: requires managing two models pipelined; errors must be handled gracefully.

## Q85: How does "model parallelism" work for training large Conformer models across multiple GPUs?
**A:** Training large Conformer ASR models (1B+ params) across multiple GPUs: (1) **Data parallelism** — each GPU has model copy, processes different audio batches. All-reduce gradients after backward pass. Simple but limited by per-GPU memory for large models. (2) **Model parallelism** — split the model across GPUs. For Conformer: (a) **Layer-wise** — layers 1-6 on GPU 0, layers 7-12 on GPU 1. Forward/backward passes communicate outputs between GPUs (pipeline parallelism). (b) **Tensor parallelism** — split self-attention heads across GPUs (each GPU computes a subset of heads), all-reduce after. (3) **Hybrid parallelism** — combine data + model parallelism. Use 3D parallelism (data × model × pipeline) with 64-512 GPUs. (4) **Communication optimization** — use NCCL for GPU communication, overlap communication with computation (compute next layer while all-reducing current layer). (5) **Activation checkpointing** — don't store all intermediate activations; recompute during backward pass. Saves 50-80% memory but adds ~30% compute overhead. (6) **Micro-batching** — split each GPU's batch into micro-batches, pipeline through model-parallel groups. (7) **Mixed precision** — FP16/BF16 forward/backward with FP32 master weights. Layout: for a 1B parameter Conformer, use 16 GPUs: 4 pipeline stages × 4 tensor parallelism, batch size 128 per GPU, micro-batch 4. Training throughput: ~200-500 audio-hours/second on 128 A100 GPUs.

## Q86: How does "continual learning" or "lifelong learning" work for ASR models that need to adapt to new domains?
**A:** Continual learning for ASR prevents catastrophic forgetting when adapting to new domains: (1) **Elastic Weight Consolidation (EWC)** — after training on domain A, identify important parameters (using Fisher information). When training on domain B, penalize changes to important parameters: L = L_B + λ * Σ_i F_i * (θ_i - θ_A,i)², where F_i is Fisher information for parameter i. (2) **Replay buffers** — store a subset of training examples (e.g., 5-10% per domain) in a memory buffer. When training on new domain, interleave replay data: L = L_new + L_replay. (3) **Progressive neural networks** — freeze the base model, add new "adapter" layers per domain (small residual adapters). Only adapter weights are trained for new domains. (4) **Knowledge distillation** — when adapting to new domain, use the old model as teacher: L = L_new + λ * KL(p_old || p_new) on old domain data (or just on random samples). (5) **Sub-network isolation** — train a binary mask per domain that selects a subset of weights to modify (Lottery Ticket Hypothesis, SupSup). For ASR, the replay buffer approach is most practical: store 1000 hours of representative data per domain (or use generated pseudo-data from a TTS system). Adapter-based approaches are gaining popularity as they separate domain-specific knowledge without forgetting.

## Q87: How do you implement "ASR with style transfer" (e.g., converting informal speech to formal text)?
**A:** ASR style transfer (e.g., "gonna" → "going to", "ain't" → "is not"): (1) **Joint model** — train an end-to-end model that outputs formal text directly from audio. Requires paired data (informal speech audio + formal transcript). Create synthetic pairs by converting formal transcripts to informal form (rule-based) and using TTS. (2) **Two-stage pipeline** — ASR produces verbatim transcript → style transfer model (T5 or BART fine-tuned on formal/informal text pairs) converts to formal text. (3) **Prompt engineering** — use a large language model with a prompt like "Convert this speech transcript to formal text, expanding contractions and correcting grammar: {transcript}". (4) **Gradient reversal** — train the ASR encoder to be invariant to formality: a formality classifier head with gradient reversal pushes the encoder to produce formality-independent representations. (5) **Multi-task learning** — ASR decoder is trained to output both verbatim and formal transcripts (separate output heads or tokens). The formal output head receives additional supervision from the style transfer objective. Style transfer ASR is particularly useful for meeting transcription, medical documentation (where verbatim disfluencies should be removed), and legal transcription (filtering casual language).

## Q88: How does "confidence-based semi-supervised learning" work when the teacher ASR model has systematic biases?
**A:** Addressing teacher bias in semi-supervised ASR: (1) **Calibrated confidence** — apply temperature scaling to the teacher's outputs so confidence scores are well-calibrated (confidence ~= accuracy). (2) **Confidence thresholding with bias awareness** — use different confidence thresholds per subgroup (e.g., lower threshold for underrepresented accents to avoid filtering them out disproportionately). (3) **Uncertainty-aware filtering** — instead of hard thresholding, use Monte Carlo dropout to estimate model uncertainty. Keep pseudo-labels where the teacher is confident AND certain. (4) **Model ensemble** — use an ensemble of teachers (different architectures, random seeds, training data subsets). Consensus pseudo-labels (where all teachers agree) are more reliable. (5) **Round-trip consistency** — for pseudo-labeled audio that has no reference text, convert pseudo-label to speech using TTS, run ASR on TTS output, compare. Low WER on this round-trip indicates reliable pseudo-label. (6) **Bias detection** — monitor pseudo-label acceptance rates across demographic groups; if one group has significantly lower acceptance, adjust thresholds. (7) **Progressive confidence** — start with a high confidence threshold, gradually reduce as training progresses, allowing more pseudo-labels while the model has more knowledge to resist teacher bias.

## Q89: How do you implement "page-level ASR" for reading documents aloud, with word-by-word highlighting?
**A:** Page-level ASR with word highlighting: (1) **Forced alignment** — use a CTC-based forced aligner (e.g., Montreal Forced Aligner, CTC segmentation) to align each word in a known text with audio timestamps. (2) **Known text** — when the text is known (e.g., e-book, script), use the text to guide alignment: forced alignment is much more accurate than free decoding because it only needs to find the best timing for known words, not recognize them. (3) **Real-time tracking** — during reading, run streaming ASR with VAD. When a sentence is completed, match the ASR output to the known text (using dynamic programming for fuzzy alignment), then highlight matching words. (4) **Cross-word boundaries** — handle line breaks and page turns. When the reader finishes a page, quickly align the cumulative transcript to the page text. (5) **Error recovery** — if the reader skips a word or mispronounces, use edit distance alignment to skip the missed word in highlighting. (6) **Speed adaptation** — track reading speed and adjust highlighting pace. Use VAD timing for inter-word gaps to keep highlighting synchronized even with pauses. (7) **Multi-modal** — combine audio features with visual features from the document image for robust page tracking. The key insight: with a known reference text, "recognition" becomes a much simpler alignment problem, enabling near-perfect word-level localization.

## Q90: What is "ASR with lexical access" and how does it differ from standard language model integration?
**A:** Lexical access ASR uses a phonetically-indexed mental lexicon (like human speech processing) instead of a word-level LM. Key differences from standard LM: (1) **Structure** — a large vocabulary (500K+ words) organized by phonetic similarity. Each word has multiple pronunciation variants, each with a phoneme sequence and word-level prior probability. (2) **Decoding process** — the acoustic model produces phoneme posteriors. The decoder activates words in the lexicon that match the phoneme sequence (with tolerance for phonetic variation). Word candidates are combined into phrase hypotheses. (3) **Activation spreading** — when a word is activated, it boosts phonetically similar words (neighbors), modeling the human "neighborhood activation" effect. (4) **Advantages** — better OOV generalization (can recognize words not in training data if they're in the lexicon), handles proper names better, provides faster lexical access for known words. (5) **Disadvantages** — requires a large, well-maintained pronunciation dictionary, struggles with phonetic ambiguity, computationally expensive (searching 500K+ entries). Lexical access ASR is primarily used in enterprise settings with custom vocabularies (legal, medical) where precision on proper nouns is critical. The architecture resembles the original "HMM with dictionary" approach but with neural acoustic models.

## Q91: How does "codec-aware ASR" improve accuracy for audio compressed with modern codecs (Opus, AAC)?
**A:** Codec-aware ASR addresses artifacts introduced by audio compression: (1) **Codec-specific augmentation** — during training, augment audio with random codec compression: apply Opus at various bitrates (6-128 kbps), AAC (32-256 kbps), MP3 (64-320 kbps). (2) **Bandwidth-aware conditioning** — estimate the effective bandwidth of the audio (e.g., Opus narrowband 8kHz vs. fullband 48kHz) and pass as a conditioning vector to the encoder. (3) **Codec embedding** — train a small network to predict the codec type and bitrate from audio, use the embedding to condition the ASR encoder (FiLM layers). (4) **Enhanced frontend** — use a speech enhancement network (e.g., DCCRN) pre-trained to restore full-bandwidth audio from compressed input. (5) **Codec reconstruction loss** — add an auxiliary loss to the ASR model to predict the original uncompressed audio features from compressed input (self-supervised pretraining). Codec-aware ASR reduces WER by 15-25% on Opus-compressed audio (at 24 kbps) compared to a model trained only on uncompressed audio. With VoIP and streaming services increasingly using Opus, codec awareness is becoming essential for production ASR.

## Q92: How do you implement "bandwidth extension" for narrowband (8kHz) telephone speech to improve ASR accuracy?
**A:** Bandwidth extension (BWE) for ASR: (1) **Feature-based extension** — train a neural network to predict wideband (16kHz) log-mel features from narrowband (8kHz) features. The extended features are fed to the ASR model: L = ||f_wb_pred - f_wb_true||² + α * L_asr(f_wb_pred). (2) **GAN-based extension** — use a generator to produce wideband spectrograms and a discriminator to distinguish from real wideband. The generator is trained adversarially, and the ASR model is fine-tuned on generated wideband features. (3) **Direct extension** — train a small upsampling network (1D residual CNN) that processes narrowband features and outputs wideband frames. Insert before the ASR encoder. The whole system (BWE + ASR) is fine-tuned jointly. (4) **Implicit extension** — do not explicitly extend bandwidth. Instead, augment narrowband ASR training with bandwidth-broadened features: take wideband training data, apply telephony bandpass filter (300-3400 Hz), train ASR on narrowband. The model learns to recognize speech from limited bandwidth. BWE typically reduces WER by 5-15% on 8kHz telephone speech, with GAN-based approaches yielding the best perceptual quality but not necessarily the best ASR accuracy (feature-based extension often works better for ASR since it optimizes for feature reconstruction, not human perception).

## Q93: How does "emotional speech recognition" work and how do emotions affect ASR accuracy?
**A:** Emotional speech recognition: (1) **Separate emotion recognition model** — a classifier (CNN-LSTM, wav2vec 2.0 fine-tuned on emotion datasets like IEMOCAP, CREMA-D) predicts emotion from speech: happy, sad, angry, neutral, fearful, surprised. The classifier uses prosodic features (pitch, energy, speaking rate) and spectral features. (2) **Emotion-aware ASR** — the ASR encoder is conditioned on the emotion label (via emotion embedding concatenated with encoder features). Multi-task learning: L = L_asr + λ * L_emotion. (3) **Emotion impact on ASR accuracy** — angry and happy speech typically have 15-30% higher WER than neutral due to: faster speaking rate (angry), wider pitch variation (happy, angry), non-standard articulation (laughing while speaking), breathiness (sad). Neutral speech has the lowest WER. (4) **Adaptation** — for emotional speech, use emotion-specific adapters (small residual networks inserted into the Conformer encoder, one per emotion). Test-time: predict emotion → activate corresponding adapter. (5) **Augmentation** — synthesize emotional speech from neutral using voice conversion with emotion transfer. Augment training data 3-5× with emotional variants. Emotion-aware ASR can reduce the WER gap between neutral and emotional speech from 20% to 5%.

## Q94: How does "session-based ASR adaptation" personalize recognition for a specific user during a conversation?
**A:** Session-based ASR adaptation personalizes the model incrementally: (1) **Bias list updating** — as the conversation progresses, extract named entities (names, places, account numbers) from recognized text and add them to a session-specific bias list. This list is merged with the decoding beam. (2) **Acoustic adaptation** — after the user speaks for 30-60 seconds, extract i-vectors or speaker embeddings and use them to adapt the encoder via feature-wise linear modulation (FiLM) or speaker-adversarial training. (3) **Language model adaptation** — maintain a session-specific cache LM: n-grams observed in the session are boosted. For neural LM, maintain a small prefix-level hidden state that captures the session's linguistic style. (4) **Confidence-based correction** — track words with low confidence; when the user repeats a low-confidence word (and it's recognized with high confidence), update the model to prefer the verified pronunciation (online learning). (5) **Context injection** — the session context (CRM data, conversation topic, previous utterances) is compressed into a context embedding and fed to the ASR decoder at each step. (6) **Practical implementation** — session adaptation runs on a server per session. After each utterance (or every 5-10 seconds), the adaptation module updates bias lists and LM state. Session setup time: <100ms. Recovery on session end: adaptation state is discarded (session-specific). Adaptation reduces WER by 10-30% within-session, especially for proper nouns and domain-specific terms.

## Q95: How do you handle "ASR for children's speech" with age-appropriate vocabulary and grammar?
**A:** Children's ASR (age 4-12): (1) **Acoustic differences** — children have higher F0 (200-400Hz for children vs. 80-180Hz for adults), shorter vocal tracts (different formant locations), and more variable pronunciation. Adapt by: pitch-normalized features (VTLN), using higher-resolution filterbanks in the 2-8kHz range, and age-dependent CMVN. (2) **Language model** — train on children's literature, educational materials, age-appropriate conversations. Use a smaller vocabulary (5K-15K vs. 200K for adults). (3) **Joint age-ASR model** — include age as an input feature. Train on age-annotated children's speech. The encoder conditions on age embedding to produce age-appropriate representations. (4) **Data augmentation** — use voice conversion (e.g., SVC) to convert adult speech to child-like speech (pitch shift + formant shift). Augment training data with converted samples. (5) **Error patterns** — children make different errors: shorter utterances, more disfluencies, higher pronunciation variability. A model trained on adults will have 30-50% higher WER on children (e.g., 5% → 15%). With proper adaptation: WER gap reduces to 5-7%. (6) **Age-dependent decoding** — adjust beam search parameters for children: lower LM weight (children use less standard grammar), higher insertion penalty (children speak less fluently with more pauses).

## Q96: How does "wake word detection" integrate with a full ASR system and what models are used?
**A:** Wake word (hotword) detection + ASR: (1) **Two-stage architecture** — (a) small, always-on wake word detector (e.g., 500KB model) running on-device. (b) Full ASR activated only after wake word detection. (2) **Wake word models** — (a) Single-keyword: small-footprint CNN or depthwise separable ConvNet trained with keyword/non-keyword classification on ~50ms frames. (b) Multi-keyword: CTC-based model with the wake word as a token (e.g., "hey_siri"). (c) Streaming: RNN-T or CTC with blank token, monitoring probability of the wake word sequence. (3) **Integration** — wake word detector outputs probability score. Above threshold (e.g., 0.8): trigger full ASR. Below threshold: continue monitoring. (4) **False accept management** — if ASR starts and the first ~500ms has very low confidence (no speech-like content), cancel the session. (5) **Barge-in** — after wake word, the user can speak immediately without waiting for a beep. The ASR system uses voice activity detection to segment the command from the wake word. (6) **On-device optimization** — wake word detection must consume <10mW power (DSP or NPU), process audio in <100ms, and trigger with <500ms latency. State-of-the-art: 1M-parameter models achieve <1% false reject rate at <0.5 false accepts per hour. Embedded implementations use INT8 quantized models on DSPs with ring buffer processing.

## Q97: How does "ASR with noise profile estimation" improve accuracy in non-stationary noise (e.g., cafe, traffic)?
**A:** Noise profile estimation for non-stationary noise: (1) **Online noise estimation** — during non-speech intervals (detected by VAD), estimate a noise power spectral density (PSD) using recursive averaging: P_noise(t) = α * P_noise(t-1) + (1-α) * P_audio(t) during non-speech; during speech, freeze P_noise. (2) **Noise embedding** — the estimated noise PSD (or log-mel noise features from recent non-speech) is encoded into a noise embedding and concatenated with speech features, conditioning the ASR encoder on current noise conditions. (3) **Noise-aware training (NAT)** — during training, pair each utterance with a noise profile (extracted from the same recording before the utterance). Train the model with noise embedding as side input. (4) **Dynamic noise tracking** — for rapidly changing noise (e.g., door slamming), use a minimum statistics tracker: track the minimum power in each frequency bin over a 1-2s window; the minima represent the noise floor. (5) **Spectral gating** — use the estimated noise profile to compute a spectral mask: G(t,f) = max(0, 1 - β * (P_noise(f) / P_speech(t,f))), apply to features. (6) **Results** — noise-adapted ASR reduces WER in non-stationary noise by 20-40% relative compared to noise-unaware models. The key advantage over fixed noise suppression: the noise profile adapts to the current acoustic environment continuously.

## Q98: How does "confidence calibration" work for ASR models with temperature scaling and what validation metric is used?
**A:** ASR confidence calibration: (1) **Temperature scaling** — after training, compute calibration on a held-out set: T* = argmin_T ECE(T), where ECE(T) = Σ_m |B_m|/N * |acc(B_m) - conf(B_m)|. B_m are bins of model confidence (0-0.1, 0.1-0.2, ..., 0.9-1.0). acc is the fraction correct (word-level match), conf is the average confidence in that bin. T is the softmax temperature: P_calibrated = softmax(logits/T). (2) **Word-level vs. sequence-level** — word-level calibration: for each output word, confidence = max softmax probability for that token. Sequence-level calibration: confidence = normalized log-probability of entire hypothesis. Word-level calibration is more useful for downstream filtering. (3) **Calibration metrics** — (a) Expected Calibration Error (ECE) — average gap between confidence and accuracy across bins. (b) Maximum Calibration Error (MCE) — worst-case bin gap. (c) Brier Score — mean squared error between confidence and correctness. (4) **Platt scaling** — replace softmax temperature with logistic regression on logits: P(y=1|x) = 1/(1+exp(α*logit + β)). More flexible than single temperature because it can also shift. (5) **Validation** — good calibration: ECE < 0.05, meaning confidence estimates are within 5% of true accuracy. For a model with 95% accuracy, well-calibrated confidence should also average ~95%.

## Q99: How does "server-side ASR with prefix caching" reduce latency for repeated queries?
**A:** Prefix caching for ASR: (1) **Cache structure** — a dictionary (e.g., LRU cache) mapping audio hashes or prefixes to decoded text. Keys: the first N frames (e.g., 1 second = 100 frames) of an utterance, hashed using perceptual hashing (pHash on log-mel features). (2) **Cache hit** — when a new utterance arrives, compute the fingerprint of its first 1 second. If found in cache, use the cached transcript as a hypothesis with a high prior (e.g., boost its probability by 3.0 in beam search). Complete decoding continues from this strong initial hypothesis. (3) **Cache warmup** — frequently used utterances (e.g., "show me my account balance", "what's the weather") are pre-populated in the cache at server startup. (4) **Cache types** — (a) Exact prefix: same audio start → same transcript prefix → reduce beam width for faster decoding. (b) Partial match: similar audio prefix → similar transcript → use as initial beam fill. (c) Semantic match: previous utterance in same session → same user's likely requests. (5) **Performance** — prefix caching reduces average decoding time by 20-40% for repetitive queries (common in IVR, customer support). Cache hit rate: 30-50% for tasks with predictable patterns. (6) **Challenges** — perceptual hashing must be robust to minor variations (different recording device, background noise). Use hashing on normalized features (CMVN-applied log-mel). Cache invalidation: LRU eviction (max 10K entries) or TTL-based expiration (5 minutes).

## Q100: How would you design a large-scale ASR training pipeline that processes 1M hours of audio with automatic quality filtering?
**A:** Large-scale ASR training pipeline (1M hours): (1) **Data ingestion** — parallel data reading from distributed file system (HDFS/S3) using petabyte-scale storage. Audio formats: WAV, FLAC, Opus, MP3. Transcoding pipeline converts all to 16kHz mono FLAC. (2) **Quality filtering** — automated pipeline: (a) Audio quality: SNR estimation (discard <5dB), clipping detection, duration check (min 0.5s, max 30s). (b) Transcript quality: check alignment confidence (forced alignment WER < 30%), language ID match, profanity/NSFW filtering, PII detection and redaction. (c) Deduplication: near-duplicate audio detection using audio fingerprinting (Dejavu/PANAKO). (3) **Data balancing** — stratified sampling to balance: language, accent, gender, SNR range, speaking rate, domain (read, conversational, spontaneous). (4) **Compute infrastructure** — (a) GPU cluster: 128-512 GPUs (A100/H100), using NCCL with GPUDirect RDMA for inter-GPU communication. (b) Distributed training: DeepSpeed or PyTorch FSDP with 3D parallelism (data × pipeline × tensor). (c) Mixed precision: BF16 for forward/backward, FP32 master weights. (5) **Training loop** — (a) Dynamic batch sizing: aggregate gradients over micro-batches until target effective batch size (e.g., 4096 utterances). (b) Learning rate schedule: 25K steps linear warmup, cosine decay over 500K steps. (c) Evaluation every 10K steps on held-out sets (clean, noisy, domain-specific). (6) **Monitoring** — (a) GPU utilization, memory bandwidth, I/O throughput. (b) Training metrics: loss, WER on eval sets, gradient norm. (c) Automated failure recovery: checkpoint every 1K steps, auto-restart on node failure with distributed checkpoint. (7) **Output** — model sharded for inference (64 shards for 1B param model). Training time: ~10-20 days for 1M hours with 512 GPUs. Typical iteration: data collection (3 months), curation (1 month), training (2-3 weeks), evaluation (1 week).
