# Python Interview Questions and Answers - Part 2

## Q1: How does Python's garbage collector handle circular references?
**A:** Python uses reference counting as its primary memory management mechanism, but reference counting alone cannot handle circular references (where two or more objects reference each other). To solve this, Python includes a cyclic garbage collector (gc module) that periodically detects and collects cycles. The gc module works by tracking all container objects (objects that can hold references to other objects) and using a generational approach with three generations. Objects are moved to older generations if they survive collection cycles. The gc can be controlled via gc.disable(), gc.enable(), and gc.collect() for manual collection. You can inspect unreachable objects with gc.garbage and set debug flags with gc.set_debug().

## Q2: What is the difference between a function decorator and a class decorator?
**A:** Function decorators are callables that take a function as input and return a modified function or callable object. They use the @decorator syntax and are applied at function definition time. Class decorators work similarly but operate on classes instead of functions. A function decorator wraps or modifies function behavior, while a class decorator modifies or enhances class behavior (e.g., adding methods, attributes, or registering the class). Class decorators were introduced in Python 3.0 and are more flexible than metaclasses for simple class modifications. Function decorators use functools.wraps to preserve metadata, while class decorators typically return a modified version of the original class or a wrapper class.

## Q3: How does the `__slots__` mechanism work and when should it be used?
**A:** `__slots__` is a class attribute that defines a fixed set of instance attributes, preventing the creation of a per-instance `__dict__`. This reduces memory overhead significantly, especially when creating many instances of a class. When `__slots__` is defined, Python reserves space only for the listed attributes in a compact, array-like structure. However, using `__slots__` prevents adding new attributes dynamically and breaks code that relies on `__dict__`. It also cannot be combined with inheritance unless all parent classes also define `__slots__`. It's best used for classes with many instances (like data classes or value objects) where memory savings matter and the attribute set is fixed.

## Q4: Explain Python's descriptor protocol and its use cases.
**A:** The descriptor protocol defines how attribute access works in Python through `__get__`, `__set__`, and `__delete__` methods. A descriptor is any object that implements at least one of these methods. When an attribute is accessed on an object, Python checks if the attribute's class has a descriptor protocol; if so, the descriptor methods are invoked. There are two types: data descriptors (implement `__set__` or `__delete__`) and non-data descriptors (only `__get__`). Data descriptors take precedence over instance `__dict__` lookups. Common use cases include properties, classmethods, staticmethods, ORM field definitions (like Django models), and validation/serialization frameworks. The `property` built-in is implemented using the descriptor protocol.

## Q5: What is the difference between `copy.copy()` and `copy.deepcopy()`?
**A:** `copy.copy()` creates a shallow copy of an object, meaning it creates a new object but inserts references to the objects found in the original. If the original object contains references to mutable objects, those references are shared between the original and the copy. `copy.deepcopy()` creates a deep copy, recursively copying all objects found within the original, creating fully independent copies. Deep copy also handles circular references by keeping a memo dictionary that tracks already-copied objects. Deep copy can be customized via `__copy__()` and `__deepcopy__()` methods, and some objects like module objects, method objects, and file handles cannot be deep-copied. Deep copy is significantly slower and uses more memory than shallow copy.

## Q6: How does Python's `asyncio` event loop work under the hood?
**A:** The asyncio event loop is the core execution engine for asynchronous Python code. It maintains a queue of callbacks and coroutines, running them cooperatively via a single thread. The event loop uses an epoll (Linux) or kqueue (macOS) selector to monitor file descriptors for I/O readiness. When a coroutine awaits an operation, it yields control back to the event loop, which suspends the coroutine's execution and registers a callback. Once the I/O completes, the callback reschedules the coroutine. The event loop also handles timers, signal handlers, and subprocesses. Internally, it alternates between running callbacks and polling for I/O events. The loop iterates through ready callbacks, runs them, checks for timed-out timers, and then polls for new I/O events.

## Q7: What is the purpose of `__enter__` and `__exit__` methods?
**A:** `__enter__` and `__exit__` are the context manager protocol methods that enable the `with` statement. `__enter__(self)` is called when entering the `with` block and should return the context manager's resource (often `self`). `__exit__(self, exc_type, exc_val, exc_tb)` is called when leaving the `with` block, regardless of whether an exception occurred. If `__exit__` returns `True`, any exception raised in the `with` block is suppressed. The method receives the exception type, value, and traceback if an exception occurred, or all `None` if no exception. This protocol is used for resource management like file handles, locks, database connections, and network sockets, ensuring proper cleanup even when exceptions occur.

## Q8: Explain method resolution order (MRO) and C3 linearization in detail.
**A:** MRO determines the order in which base classes are searched when looking up a method or attribute. Python uses the C3 linearization algorithm, which ensures three properties: consistency (a class always appears before its parents), monotonicity (MRO respects the order of bases as defined), and local precedence order (parents are searched in the order they're listed). The MRO can be viewed via `ClassName.__mro__` or `ClassName.mro()`. C3 linearization works by merging the linearizations of parent classes while maintaining consistency. Diamond inheritance patterns are handled gracefully — a common ancestor class appears only once in the MRO. If C3 cannot produce a consistent linearization (e.g., due to incompatible parent orderings), Python raises a `TypeError`.

## Q9: How does the `functools.lru_cache` decorator work and what are its limitations?
**A:** `functools.lru_cache` is a decorator that caches function results based on the arguments using a Least Recently Used (LRU) eviction policy. It uses a dictionary mapping arguments to results and a doubly-linked list tracking access order. When the cache exceeds `maxsize`, the least recently used entry is evicted. With `maxsize=None`, the cache is unbounded. The decorator is thread-safe for reads but not writes. Limitations include: arguments must be hashable, the function must be deterministic (same arguments always same result), cache grows with `maxsize` (memory concern), and methods on mutable objects can produce incorrect results if object state changes. The cache_info() method returns hit/miss statistics, and cache_clear() clears the cache. It's particularly useful for recursive functions like Fibonacci and for memoizing expensive pure functions.

## Q10: What is the difference between `threading` and `multiprocessing` modules?
**A:** `threading` provides concurrency within a single process using OS threads, sharing the same memory space. Due to the Global Interpreter Lock (GIL), only one thread executes Python bytecode at a time, making threading suitable for I/O-bound tasks but not CPU-bound tasks. `multiprocessing` spawns separate processes, each with its own Python interpreter and GIL, enabling true parallel execution across CPU cores. Multiprocessing uses mechanisms like pipes, queues, and shared memory for inter-process communication. Threads share memory naturally, making data sharing easier but requiring locks. Processes have separate memory spaces, making data sharing more complex but avoiding GIL contention. Multiprocessing has higher overhead due to process creation, while threading is more lightweight. For CPU-bound work, use `multiprocessing`; for I/O-bound work, use `threading` or `asyncio`.

## Q11: How does Python handle multiple inheritance and what are common pitfalls?
**A:** Python supports multiple inheritance where a class inherits from multiple base classes. The MRO (C3 linearization) determines method lookup order. Common pitfalls include: diamond problem (same ancestor class appearing multiple times), unexpected method resolution order, name clashes between parent classes, complex `__init__` chains requiring cooperative super() calls, and fragile base class problems where changes in one parent affect child behavior. To handle cooperative inheritance, use `super()` consistently throughout the hierarchy and ensure all `__init__` methods follow the same signature pattern (using `*args, **kwargs`). Misko Hevery's advice applies: prefer composition over inheritance for complex hierarchies, and consider using mixin classes that are designed for multiple inheritance.

## Q12: What is the Python GIL and why hasn't it been removed?
**A:** The Global Interpreter Lock (GIL) is a mutex in CPython that protects access to Python objects, preventing multiple threads from executing Python bytecode simultaneously. It simplifies the interpreter implementation (especially memory management and reference counting) and makes C extensions easier to write. The GIL hasn't been removed because: removing it would significantly degrade single-threaded performance (estimates suggest 15-30% slowdown), it would require making all CPython internals thread-safe (extremely complex and error-prone), many C extensions rely on the GIL for thread safety, and there are alternative approaches (asyncio, multiprocessing) for parallelism. Projects like `nogil`, PyPy STM, and efforts like PEP 703 aim to make the GIL optional, with PEP 703 being accepted for Python 3.13 as a compile-time flag.

## Q13: Explain the difference between `@classmethod`, `@staticmethod`, and instance methods.
**A:** Instance methods receive `self` (the instance) as the first argument and can access and modify instance state. `@classmethod` receives `cls` (the class) as the first argument and can access/modify class state but not instance state. Class methods are used for alternative constructors (like `dict.fromkeys()`). `@staticmethod` receives neither `self` nor `cls` — it's essentially a regular function placed inside a class for organizational purposes. It cannot access class or instance state. Instance methods are the most common; class methods are used for factory patterns; static methods are used for utility functions logically grouped with a class. The `@classmethod` decorator uses the descriptor protocol to bind the method to the class, while `@staticmethod` bypasses the binding entirely.

## Q14: What is the purpose of `__new__` vs `__init__`?
**A:** `__new__(cls, *args, **kwargs)` is a static method responsible for creating and returning a new instance of the class. It receives the class as the first argument and is called before `__init__`. `__init__(self, *args, **kwargs)` is the initializer that modifies the already-created instance. `__new__` is rarely overridden but is essential for immutable types (like int, tuple, str) where the object must be created before initialization, for implementing singletons, and for metaclass programming. If `__new__` doesn't return an instance of the class, `__init__` won't be called. Most classes only need `__init__`. `__new__` is automatically converted to a static method by Python, even if not decorated.

## Q15: How does Python's import system work?
**A:** Python's import system follows a sequence: (1) check `sys.modules` cache, (2) find the module spec using finders (meta path and path hooks), (3) load the module using the associated loader, (4) execute the module code in the module's namespace, and (5) add the module to `sys.modules`. The process involves `importlib` which replaced the older `imp` module. Module search paths include: current directory, `PYTHONPATH`, standard library directories, and site-packages. Relative imports use dots (`.` for current package, `..` for parent). Python supports three types of finders: path-based finders for filesystem modules, frozen module finders, and custom finders via `sys.meta_path`. The import system can be customized with import hooks.

## Q16: What are Python's abstract base classes (ABCs) and how are they used?
**A:** ABCs defined in the `abc` module provide a way to define abstract interfaces in Python. They use the `ABCMeta` metaclass or `ABC` base class and the `@abstractmethod` decorator to declare methods that subclasses must implement. ABCs can also provide concrete methods. Unlike interfaces in other languages, ABCs can have both abstract and concrete methods. ABCs support virtual subclasses via `register()` — a class can be considered a subclass of an ABC without explicit inheritance. The `collections.abc` module provides many useful ABCs like `Sequence`, `Mapping`, `Iterable`, `Callable`. Duck typing is preferred in Python, but ABCs are useful for formal interface definitions, especially in large frameworks.

## Q17: Explain the difference between `__str__`, `__repr__`, and `__format__`.
**A:** `__repr__` returns an unambiguous string representation of the object, typically used for debugging. Ideally, `eval(repr(obj))` should recreate the object. `__str__` returns a human-readable string representation, used by `print()` and `str()`. If `__str__` is not defined, Python falls back to `__repr__`. `__format__` controls how the object appears in format strings and f-strings. It receives a format spec string. The default `__format__` calls `__str__` for empty format specs, but custom implementations can handle format specs like `:.2f`. The rule: `__repr__` for developers, `__str__` for users. Django models demonstrate this distinction well.

## Q18: What is the Python buffer protocol and how is it used?
**A:** The buffer protocol allows objects to expose their raw memory buffers for direct access without copying. Objects like `bytes`, `bytearray`, `memoryview`, and `array.array` implement this protocol. `memoryview` objects allow slicing and accessing the underlying buffer without copying, making them efficient for large data operations. The protocol is defined in `PEP 3118` and uses `Py_buffer` struct in C API. It's crucial for numerical computing (NumPy uses it extensively), image processing, and any application needing zero-copy data access. The buffer protocol supports multi-dimensional arrays, different data formats, and strided access. Python 3 introduced `__buffer__` and `__release_buffer__` protocol methods for custom objects.

## Q19: How do you implement a custom iterator and what is the iterator protocol?
**A:** The iterator protocol requires implementing two methods: `__iter__(self)` which returns the iterator object, and `__next__(self)` which returns the next value or raises `StopIteration`. An object that implements both is an iterator. An iterable (object that can be iterated over) only needs `__iter__` to return an iterator. Built-in iterables like lists return an iterator object when `iter()` is called. Generator functions automatically create iterators. Custom iterators can maintain state across iterations. The classic example is a range-like iterator or a tree traversal. After `StopIteration` is raised, calling `__next__` again should continue to raise `StopIteration`. Protocols like `__getitem__` can also make an object iterable (Python creates an iterator that indexes sequentially until `IndexError`).

## Q20: What is the difference between `raise`, `raise e`, and `raise ... from ...`?
**A:** Bare `raise` reraises the current exception, preserving the original traceback. `raise e` raises the exception instance `e` (typically from an `except` block) with the current stack frame as origin. `raise ... from ...` chains exceptions using `__cause__` — `raise B from A` sets `B.__cause__` to `A`, showing "The above exception was the direct cause of the following exception" in traceback. `raise ... from None` suppresses the chain, showing only the new exception. Bare `raise` is used in exception handler re-raising. `raise e` is useful when creating new exceptions. Exception chaining with `from` helps maintain context when one exception leads to another.

## Q21: How does Python's `super()` work with multiple inheritance?
**A:** `super()` returns a proxy object that delegates method calls following the MRO. In single inheritance, it simply calls the parent class method. In multiple inheritance, `super()` ensures that all ancestor classes' methods are called in MRO order, avoiding duplicate calls to the same class. `super()` uses two arguments: the current class and the instance (or subclass). When called without arguments (Python 3 only), it automatically infers them. For cooperative multiple inheritance, all `__init__` methods in the MRO chain should use `super().__init__()` with `**kwargs` to pass unused arguments to the next class in the chain. This pattern is essential for mixins and cooperative diamond inheritance.

## Q22: What are Python's `enum` capabilities and when should they be used?
**A:** Python's `enum` module (Python 3.4+) provides `Enum`, `IntEnum`, `Flag`, `IntFlag`, and `auto()`. Enums have named values with `.name` and `.value` attributes. They support iteration, comparison, and hashing. `Enum` ensures values are unique (use `@unique` decorator). `IntEnum` inherits from `int`, allowing integer comparisons. `Flag` supports bitwise operations for flags. `StrEnum` (3.11+) inherits from `str`. Enums should be used for fixed sets of constants (status codes, directions, days of the week). They provide type safety, better readability, and prevent sentinel value bugs. Enum members are singletons. Enum creation can be customized via `__init__`, `__new__`, or `_generate_next_value_`.

## Q23: Explain the Global Interpreter Lock's impact on multi-threaded Python programs.
**A:** In CPU-bound programs, the GIL ensures only one thread executes Python bytecode at a time, effectively serializing computation across threads. This means CPU-intensive tasks cannot benefit from multiple cores via threads. I/O-bound programs, however, are largely unaffected because the GIL is released during blocking I/O operations (sleep, network read, file read), allowing other threads to run. The GIL is released in C extensions for expensive operations (like NumPy). To achieve parallelism for CPU-bound work, use `multiprocessing`, `concurrent.futures.ProcessPoolExecutor`, or write C extensions. Newer approaches include `asyncio` for I/O concurrency and using NumPy/PyTorch which release the GIL for heavy computation.

## Q24: How do you profile and optimize Python code?
**A:** Python profiling tools include `cProfile` (deterministic profiling), `profile` (pure Python profiler), `timeit` (micro-benchmarking), `py-spy` (sampling profiler for production), and `memory_profiler` (memory usage). The process: (1) identify bottlenecks with `cProfile` run via `python -m cProfile script.py`, (2) analyze with `pstats` or visualization tools like `snakeviz`, (3) optimize hotspots using algorithm improvements, (4) use `functools.lru_cache` for memoization, (5) use `__slots__` for memory-intensive classes, (6) consider C extensions with Cython or cffi, (7) use NumPy for array operations, (8) use `asyncio` for I/O-bound concurrent tasks, and (9) use `multiprocessing` for CPU-bound parallel tasks. `dis` module helps inspect bytecode. `perf` (Linux) can profile at the system level.

## Q25: What is the purpose of `__post_init__` in dataclasses?
**A:** `__post_init__` is a special method in Python dataclasses (Python 3.7+) that's automatically called after `__init__` completes. It allows post-initialization processing and validation that can't be done with field defaults or `field()` alone. Common uses: derived/calculated fields, validation of combinations of fields, initializing non-dataclass attributes (like mutable defaults via `field(default_factory=...)`), converting field types, and setting up internal state. The method is not called automatically for `InitVar` processing unless explicitly handled. It's a cleaner alternative to overriding `__init__` in dataclasses.

## Q26: How does Python handle string interning?
**A:** String interning is an optimization where Python reuses string objects for identical string literals to save memory. Short strings that look like identifiers (alphanumeric with underscores) are automatically interned. Strings longer than 4096 bytes or containing spaces/control characters are not interned. String literals in source code are interned at compile time. Strings created dynamically (e.g., via `str()`, `join()`, or concatenation) are typically not interned. The `sys.intern()` function allows explicit interning, which can be useful when many identical string values are compared repeatedly (string identity checks with `is` are faster than equality checks with `==`). Interning is CPython-specific behavior and not guaranteed by the Python language specification.

## Q27: What are Python's `__getattr__` and `__getattribute__` and how are they different?
**A:** `__getattribute__(self, name)` is called unconditionally for every attribute access on an object. It must return the attribute or raise `AttributeError`. `__getattr__(self, name)` is only called when normal attribute lookup fails (i.e., `__getattribute__` raised `AttributeError`). `__getattribute__` should always call `super().__getattribute__()` to avoid infinite recursion. `__getattr__` is used for dynamic attribute handling (like delegating to internal objects, lazy loading, or default values). Using `__getattr__` is safer and more common than overriding `__getattribute__`. `__setattr__` controls attribute setting and must use `object.__setattr__()` or `super().__setattr__()` to avoid recursion when setting instance attributes.

## Q28: What is the difference between a coroutine and a generator in Python?
**A:** Generators produce values using `yield` and can be iterated over. They maintain state between yields but primarily generate sequences. Coroutines (async def functions) use `await` and `yield` (in the context of `asyncio`) for cooperative multitasking. Coroutines are executed by an event loop and represent potentially long-running operations. Generators are synchronous by default; coroutines are asynchronous. Generators use `yield` to produce values, while coroutines use `await` to wait for results. Since Python 3.5, `async def` defines coroutines. With `@asyncio.coroutine` decorator (deprecated) and `yield from`, older coroutines shared syntax with generators but served different purposes. Generators can be used to implement coroutines (as in Python 3.4 and earlier), but native coroutines are now preferred.

## Q29: How do you implement a singleton pattern in Python?
**A:** Several approaches exist: (1) Module-level singleton — Python modules are naturally singletons since they're only imported once. (2) Metaclass approach — override `__call__` to control instance creation. (3) Decorator approach — a decorator that maintains a mapping of classes to instances. (4) Shared-state approach — use a shared `__dict__` via `__new__`. (5) Borg pattern — all instances share state but are separate objects. The metaclass approach is the most "Pythonic" way:

```python
class SingletonMeta(type):
    _instances = {}
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]
```

Singletons in Python are often considered anti-patterns; dependency injection is preferred for testability.

## Q30: Explain Python's `__name__ == '__main__'` idiom.
**A:** `__name__` is a special variable set by Python. When a script is run directly, `__name__` is set to `'__main__'`. When the script is imported as a module, `__name__` is set to the module name. The idiom `if __name__ == '__main__':` allows code blocks to execute only when the file is run directly, not when imported. This is essential for making Python files both importable (as libraries) and executable (as scripts). The idiom typically wraps main logic, argument parsing, or testing code. It provides clean module design, separating importable functionality from script behavior.

## Q31: How does Python's `pickle` serialization work and what are its security implications?
**A:** `pickle` serializes Python object hierarchies to byte streams and deserializes them back. It supports most Python types natively. When unpickling, `pickle` reconstructs objects by importing modules and calling constructors with `__reduce__`. This makes `pickle` dangerous: malicious pickle data can execute arbitrary code during unpickling because `__reduce__` can return any callable with arbitrary arguments. Never unpickle data from untrusted sources. `pickle` is protocol-based (protocol 0-5), with higher protocols being more efficient. Alternatives: `json` (safe, text-based), `msgpack` (binary, efficient), or project-specific serialization. Python 3.8+ added `pickle` protocol 5 with out-of-band data support for efficient array serialization in NumPy.

## Q32: What is the purpose of `weakref` in Python?
**A:** The `weakref` module allows creating weak references to objects — references that don't increase the reference count and don't prevent garbage collection. When the referenced object is garbage collected, the weak reference returns `None` or the callback is invoked. Weak references are useful for: (1) implementing caches (like `functools.lru_cache` uses weakref internally), (2) avoiding circular references that prevent garbage collection, (3) observer patterns where observers shouldn't keep subjects alive, (4) weak value dictionaries (`WeakValueDictionary`), (5) weak key dictionaries and sets (`WeakKeyDictionary`, `WeakSet`). A weak reference is created with `weakref.ref(obj, callback)`. Not all objects support weak references; lists, dicts, and ints don't by default, but can be made to support them via `__weakref__` attribute.

## Q33: How do you achieve thread safety in Python?
**A:** Thread safety in Python is achieved through: (1) `threading.Lock` — basic mutual exclusion lock, (2) `threading.RLock` — reentrant lock (same thread can acquire multiple times), (3) `threading.Semaphore` — limits concurrent access, (4) `threading.Event` — signaling between threads, (5) `threading.Condition` — combined lock and wait/notify pattern, (6) `queue.Queue` — thread-safe FIFO queue with blocking operations, (7) `concurrent.futures.ThreadPoolExecutor` — high-level thread management, (8) atomic operations for simple operations on primitive types (due to GIL), (9) `threading.local()` — thread-local storage. Best practices: minimize shared state, use `Queue` for producer-consumer, use `Lock` as context manager, avoid deadlocks by acquiring locks in consistent order, and prefer `concurrent.futures` over raw threads.

## Q34: What is the Python `typing` module and how does it help with type hints?
**A:** The `typing` module (Python 3.5+) provides runtime support for type hints specified in PEP 484 and related PEPs. Key components include: `List[T]`, `Dict[K,V]`, `Tuple[T,...]`, `Set[T]`, `Optional[T]` (equivalent to `Union[T, None]`), `Union[T, U]`, `Any`, `Callable[[Args], Return]`, `Type[T]`, `Generic[T]`, `Protocol` (structural subtyping, PEP 544), `Literal` (PEP 586), `TypedDict` (PEP 589), `Final`, `overload`, and `TypeVar`. The module also includes `cast()` for explicit type casts, `NewType` for distinct types, and `TYPE_CHECKING` for conditional imports only used during type checking. Tools like mypy, pyright, and pytype analyze these annotations without requiring runtime overhead.

## Q35: How do you handle circular imports in Python?
**A:** Circular imports occur when two or more modules depend on each other. Solutions include: (1) restructure code to eliminate circular dependencies — extract shared dependencies to a third module, (2) use lazy imports inside functions/methods instead of top-level imports, (3) use `TYPE_CHECKING` from `typing` for type-only imports that are only needed for annotation (these imports are not executed at runtime), (4) use the `import` statement at the end of a module after other imports are resolved, (5) use absolute imports instead of relative imports, (6) merge the interdependent modules into a single module. The best long-term solution is architectural — following a layered dependency structure (e.g., clean architecture) where lower layers never import from higher layers.

## Q36: What is the difference between `deepcopy` and `copy` with custom objects?
**A:** Both `copy.copy()` and `copy.deepcopy()` respect the `__copy__()` and `__deepcopy__()` custom methods. `__copy__()` should return a new shallow copy of the object. `__deepcopy__(memo)` should return a new deep copy, using the `memo` dict to track already-copied objects for handling circular references. `__getstate__()` and `__setstate__()` are also used during copying if defined. For objects without these methods, the `copy` module uses `__reduce_ex__` (used by pickle). Overriding these methods allows custom copy behavior, including intentionally sharing certain sub-objects between copies or performing validation during copying.

## Q37: How does Python's `__init_subclass__` work?
**A:** `__init_subclass__(cls, **kwargs)` is a hook introduced in Python 3.6 that's called whenever a class is subclassed. It receives the new subclass as the first argument and any keyword arguments passed in the class definition. It allows parent classes to customize subclass creation without metaclasses. Common uses: (1) registering subclasses automatically, (2) enforcing subclass constraints, (3) modifying subclass attributes, (4) injecting methods or descriptors into subclasses. Keyword arguments in class definition (`class Child(Parent, kwarg=value)`) are passed to `__init_subclass__`. This method is simpler than metaclasses for many use cases and works well with multiple inheritance if all parents call `super().__init_subclass__()`.

## Q38: Explain Python's `__del__` method and its pitfalls.
**A:** `__del__(self)` is the destructor method called when an object is about to be destroyed (reference count reaches zero or object is garbage collected). Pitfalls include: (1) `__del__` is not guaranteed to be called at program exit (modules may already be cleaned up), (2) exceptions in `__del__` are ignored (only a warning is printed), (3) circular references involving `__del__` prevent garbage collection in Python versions before 3.4 (the objects end up in `gc.garbage`), (4) global state accessed in `__del__` may already be destroyed, (5) `__del__` can be called in any thread. Python 3.4+ improved circular reference handling for objects with `__del__`. The context manager protocol (`__enter__`/`__exit__`) is preferred for cleanup over `__del__`.

## Q39: What are Python's data classes and how are they different from namedtuples?
**A:** Dataclasses (`@dataclass` decorator, Python 3.7+) generate `__init__`, `__repr__`, `__eq__`, and optionally `__hash__`, `__lt__`, `__le__`, `__gt__`, `__ge__`, `__str__` methods automatically. They support mutable instances by default, type annotations, default values, default factories, `__post_init__` processing, `InitVar` variables, field metadata, and inheritance. Namedtuples (`collections.namedtuple`) are immutable, tuple-based, lighter-weight, and support tuple indexing and unpacking. Neither is strictly better: dataclasses are more flexible and feature-rich, while namedtuples are more memory-efficient and tuple-compatible. Dataclasses can be ordered, immutable (via `frozen=True`), and customized extensively. Namedtuples are better for simple, immutable data containers that need to work as dictionary keys.

## Q40: How does Python's `asyncio.gather` differ from `asyncio.wait`?
**A:** `asyncio.gather(*coros, return_exceptions=False)` runs multiple awaitables concurrently and returns a list of results in the same order as the input. If any awaitable raises an exception, `gather` propagates it immediately (unless `return_exceptions=True`, in which case exceptions are returned as results). `gather` is higher-level and simpler for parallel execution. `asyncio.wait(aws, timeout=None, return_when=ALL_COMPLETED)` is lower-level, accepting a set of awaitables and returning two sets: done and pending. It supports `FIRST_COMPLETED`, `FIRST_EXCEPTION`, and `ALL_COMPLETED` conditions and optional timeout. `wait` provides more control for complex scenarios. For simple parallel execution, use `gather`; for timeout control or first-completed patterns, use `wait`.

## Q41: What is the purpose of `__future__` imports in Python?
**A:** `__future__` imports allow using features that will be standard in future Python versions but are not backward-compatible. They're typically used in codebases that need to support multiple Python versions while adopting newer language features.

## Q42: Explain the difference between `async def` and `def` with a generator.
**A:** `def` with `yield` creates a generator function that returns an iterator. When called, it doesn't execute immediately but returns a generator object that can be iterated with `next()` or a `for` loop. `async def` creates a coroutine function that returns a coroutine object when called. Coroutines are executed by an event loop via `await` or `asyncio.run()`. Generators are synchronous and produce values lazily; coroutines are asynchronous and represent potentially blocking operations. Generators use `yield` to produce values while maintaining state; coroutines use `await` to suspend execution pending an async operation. Since Python 3.6, `async generators` (`async def` with `yield`) combine both concepts.

## Q55: Explain the Python buffer protocol and zero-copy operations.
**A:** The buffer protocol allows objects to expose their raw memory buffer for direct access. `memoryview` objects provide Python-level access to the buffer protocol without copying data. Operations like slicing a `memoryview` create new `memoryview` objects referencing the same memory, enabling zero-copy operations on large datasets. The protocol supports multi-dimensional arrays with strides and format codes for different data types. Using `memoryview.cast()` changes the interpretation of data without copying. Zero-copy is critical for performance in: video processing, network packet handling, large file parsing, and numerical computing. `numpy` arrays implement the buffer protocol, allowing zero-copy conversion between `bytes` and `numpy` arrays.

## Q56: What are Python's `__slots__` and `__dict__` and how do they interact?
**A:** By default, Python instances store attributes in a `__dict__` (a per-instance dictionary), which provides flexibility but uses memory. `__slots__` is a class attribute that defines a fixed set of instance attributes stored in a compact tuple-like structure, eliminating `__dict__`. When `__slots__` is defined, Python allocates space for the listed attributes in a descriptor-based layout, reducing memory usage. Objects with `__slots__` cannot have attributes not listed in `__slots__`. If `'__dict__'` is included in `__slots__`, instances retain dynamic attribute capability but lose some memory benefits. Classes with `__slots__` cannot have `__weakref__` unless it's included in `__slots__`. Inheritance requires all parent classes to define `__slots__` for proper behavior.

## Q58: What is the Global Interpreter Lock and how does it affect C extensions?
**A:** The GIL protects CPython internals from race conditions. C extensions can release the GIL during long-running operations to allow Python threads to run concurrently. The Python C API provides `Py_BEGIN_ALLOW_THREADS` and `Py_END_ALLOW_THREADS` macros for this purpose. Extensions that don't release the GIL block all Python threads. NumPy, for example, releases the GIL during array computations, allowing multi-threaded Python programs to achieve parallelism. Extensions that modify Python objects must hold the GIL. The GIL must be re-acquired before calling Python API functions from a released state. Understanding GIL management is crucial for writing performant C extensions.

## Q59: How do you debug memory leaks in Python?

## Q60: Explain Python's `__await__` method.
**A:** `__await__` is a special method that makes an object awaitable (usable with `await`). It must return an iterator. When `await obj` is used, Python calls `obj.__await__()` and iterates over the returned iterator, which should yield coroutine-like objects. This is used to implement custom awaitables without using `async def`. For example, a custom future-like class can implement `__await__` to provide a custom awaiting mechanism. The `__await__` method is how coroutines are implemented internally — `coroutine.__await__()` returns an iterator. The returned iterator should yield `None` (or other values) until the awaited operation completes, then return the result or raise an exception.

## Q65: Explain the difference between `yield from` and `await`.
**A:** `yield from` (Python 3.3+) delegates to a subgenerator, allowing the outer generator to yield values from an inner generator and receive sent values/throw exceptions. It's synchronous. `await` (Python 3.5+) suspends execution of a coroutine until the awaited awaitable completes. It's asynchronous and used in event-loop-driven code. `yield from` is a generator delegation mechanism; `await` is an async suspension mechanism. Historically, `yield from` was used to implement coroutines before native async/await (e.g., `asyncio.coroutine` based on generators). `await` is essentially `yield from` for a restricted set of objects (awaitables), with added semantics for the event loop.

## Q66: How do you implement a thread pool in Python?

## Q67: What are Python `async for` and `async with`?

## Q68: How does Python's `__prepare__` method work in metaclasses?

## Q69: What is the difference between `is` and `==` with custom objects?
**A:** `is` compares identity (whether two references point to the same object in memory). `==` compares equality (whether two objects are considered equal based on `__eq__` implementation). By default, `__eq__` inherits from `object` and also compares identity. Custom classes should implement `__eq__` for meaningful value equality, and should implement `__hash__` if used as dictionary keys or set members. Immutable types (strings, tuples, ints) may have `is` return `True` for equal values due to interning, but this is not guaranteed. Always use `==` for value comparison, `is` for singleton checks (like `is None`, `is True`).

## Q70: How do Python's `__reduce__` and `__reduce_ex__` work?

## Q72: How does Python handle Unicode and string encoding?

## Q74: Explain Python's `sys.settrace` and `sys.setprofile`.
**A:** `sys.settrace(tracefunc)` sets a trace function that's called for every line of Python code execution (frame entry, line event, return, exception). ## Q75: How do you implement a custom context manager without using classes?
**A:** Using `contextlib.contextmanager` decorator on a generator function:

```python
from contextlib import contextmanager

@contextmanager
def managed_resource(*args, **kwargs):
    # Setup (called when entering with block)
    resource = acquire_resource(*args, **kwargs)
    try:
        yield resource  # Value returned by __enter__
    finally:
        # Teardown (called when exiting with block, even on exception)
        release_resource(resource)
```

The generator should yield exactly once. If an exception occurs in the `with` block, it's re-raised at the yield point, allowing the `finally` block to clean up. The `@contextmanager` decorator handles the `__enter__`/`__exit__` protocol. For suppressing exceptions, use `try/except` around the yield.

## Q76: What is the difference between `typing.Protocol` and ABCs?
**A:** `typing.Protocol` (Python 3.8+, PEP 544) implements structural subtyping (duck typing at the type-checking level). A class satisfies a Protocol if it has the required methods/attributes, without explicit inheritance. ABCs require explicit inheritance or `register()`. Protocols are checked statically by type checkers (mypy, pyright). `@runtime_checkable` decorator enables `isinstance()` checks at runtime (limited to checking method existence). Protocols support inheritance, generic parameters, and composition. Protocols are preferred for API definitions where you want to accept any object with certain capabilities, while ABCs are better when you need to provide default implementations or enforce method implementation at class definition time.

## Q77: How does Python's `async generator` work?
**A:** Async generators (Python 3.6+) are defined with `async def` and contain `yield` statements. They produce values asynchronously — the generator is iterated with `async for`. Between yields, `await` can be used for async operations. Async generators implement `__aiter__` and `__anext__`. They can also have `asend`, `athrow`, and `aclose` methods corresponding to the synchronous generator protocol. Cleanup is done with `aclose()`. They're useful for: streaming data from async sources (websockets, database cursors), paginated API responses, and incremental data processing. Async generators must be used within an event loop context.

## Q78: What is the purpose of `__wrapped__` attribute in functools?

Multiple decorators each wrap the previous, creating a chain of `__wrapped__` references.

## Q79: How does Python's `sorted()` function work with custom objects?
**A:** `sorted()` accepts an iterable and returns a sorted list. For custom objects, it uses the `<` operator (`__lt__`) by default (Python 3). The `key` parameter specifies a function that extracts a comparison key from each element. The `reverse` parameter reverses the sort order. The `key` function is applied once per element (decorate-sort-undecorate pattern). For stable sorting (equal elements retain original order), use `key` with `operator.attrgetter` or `operator.itemgetter`. `functools.cmp_to_key` converts legacy comparison functions to key functions. Custom classes should implement `__lt__` and `__eq__` for sorting. Python uses Timsort, a stable O(n log n) algorithm.

## Q80: What are Python's `__getstate__` and `__setstate__` methods?
**A:** `__getstate__(self)` should return a picklable object representing the instance's state. It's called during pickling. `__setstate__(self, state)` restores the instance's state from the pickled state during unpickling. These methods override the default pickling behavior. If `__getstate__` is not defined, `__dict__` is used. If `__getstate__` returns `None`, only `__setstate__` is called (not `__dict__.update`). They're useful for: excluding transient or un-picklable attributes, optimizing pickling by returning a simplified representation, and reconstructing complex state. If `__slots__` is used without `__dict__`, these methods become necessary for custom pickling.

## Q81: How do you implement dependency injection in Python?
**A:** Dependency injection can be implemented in Python through: (1) constructor injection — passing dependencies via `__init__`, (2) setter injection — setting dependencies as attributes after construction, (3) parameter injection — passing dependencies to methods, (4) container-based injection — using a DI container that auto-wires dependencies (like `dependency_injector`, `inject`, or `python-inject`). Python's duck typing makes DI natural — depend on interfaces/Protocols rather than concrete classes. Approaches range from simple manual DI (most common in Python) to full containers with auto-wiring. The simplest Pythonic DI: accept dependencies in `__init__` with default values, allowing easy testing with mocks.

## Q82: What is the Python `__del__` method's interaction with garbage collection?
**A:** Objects with `__del__` methods complicate garbage collection. In Python < 3.4, objects with `__del__` involved in circular references could not be garbage collected — they'd be placed in `gc.garbage`. Python 3.4 (PEP 442) improved this by changing the finalization order: `__del__` is called before the cyclic GC breaks references, and the object's `__del__` may be called even with circular references. However, `__del__` can still resurrect objects (make them reachable again), causing issues. The GC collects unreachable objects, then calls `__del__` on them in a safe order. `weakref.finalize` is preferred over `__del__` for reliable cleanup.

## Q83: How does Python's `asyncio.create_task` differ from `ensure_future`?
**A:** `asyncio.create_task(coro)` (Python 3.7+) creates a Task from a coroutine and schedules it on the current event loop. It's the preferred way to run coroutines concurrently. `ensure_future(coro_or_future)` is more flexible — it accepts futures, coroutines, and awaitables, wrapping them in tasks if needed. `ensure_future` returns the input as-is if it's already a Future, or wraps coroutines in a Task. `create_task` is simpler and more explicit: it always creates a new Task. `create_task` also provides a reference to the task, important for avoiding garbage collection of pending tasks (Python 3.11+ holds weak references, so explicit task tracking is recommended).

## Q84: What are Python's `sys.excepthook` and `sys.unraisablehook`?
**A:** `sys.excepthook(type, value, traceback)` is called when an uncaught exception propagates to the top level. The default handler prints the traceback to stderr. Custom exception hooks can log exceptions, send alerts, or perform cleanup. `sys.unraisablehook` (Python 3.8+) is called when an exception occurs in a context where it can't be propagated (e.g., in `__del__`, weakref callbacks). Custom unraisable hooks can log these suppressed exceptions. Both hooks affect all exceptions in the process. For thread-specific handling, use `threading.excepthook` (Python 3.8+).

## Q85: How do you create immutable objects in Python?
**A:** Approaches to immutability: (1) use `dataclasses(frozen=True)` — generates `__setattr__` and `__delattr__` that raise `FrozenInstanceError`, (2) create a class with `__slots__` and override `__setattr__` to raise `AttributeError`, (3) use `types.MappingProxyType` for dict immutability, (4) use `collections.namedtuple` — tuple-based immutable objects, (5) use `typing.NamedTuple` — typed namedtuples, (6) use frozen dataclasses with `__init_subclass__` to prevent subclassing, (7) return copies from mutating methods. Note that frozen dataclasses and namedtuples don't guarantee deep immutability — mutable attributes (lists, dicts) can still be modified. Deep immutability requires recursive freezing or using immutable collections.

## Q86: What is the Python `__iter__` and `__reversed__` protocol?
**A:** `__iter__(self)` returns an iterator object implementing `__next__`. Sequences can also implement `__getitem__` for iteration (Python falls back to indexing from 0). `__reversed__(self)` (Python 2.3+) returns an iterator that traverses the container in reverse order, used by `reversed()`. If `__reversed__` is not defined but `__len__` and `__getitem__` are, `reversed()` uses them. Both iterators and async iterators are available. `__aiter__` and `__anext__` for async iteration with `async for`. Container types should implement both forward and reverse iteration for complete interface support.

## Q87: How does Python's `map` and `filter` differ from list comprehensions?
**A:** `map(func, iterable)` applies a function to every item and returns an iterator. `filter(func, iterable)` returns items where func returns True. List comprehensions combine both: `[f(x) for x in iterable if condition]`. List comprehensions are generally more Pythonic and readable. `map` and `filter` are lazy (return iterators) while list comprehensions are eager. `map` can be faster in some cases (especially with built-in functions). `map` and `filter` shine when: (1) chaining transformations, (2) working with function composition using `operator` module, (3) processing huge datasets where lazy evaluation matters. Python 3 made `map`/`filter` lazy (they returned lists in Python 2).

## Q88: What is the purpose of `typing.Literal`?
**A:** `typing.Literal` (Python 3.8+, PEP 586) specifies that a type must be one of a specific set of literal values. Examples: `Literal['red', 'green', 'blue']`, `Literal[1, 2, 3]`, `Literal[True]`. Type checkers use Literal for: (1) discriminating unions (narrowing types based on exact values), (2) function overloading (different return types based on argument values), (3) validating configuration values, (4) precise typing of constants and enum alternatives. Literal types are checked at static analysis time and don't provide runtime validation. They're particularly useful with `@overload` decorators and discriminated unions.

## Q89: How does Python's `__pprint__` or pretty-printing work?
**A:** The `pprint` module provides pretty-printing for Python data structures. Objects can customize their pretty-printed representation via `__repr__` (which `pprint` uses by default). For advanced customization, classes can define `__format__` or use `pprint.PrettyPrinter` subclassing with custom `format` methods. `pprint` handles nested structures, indentation, line wrapping, and sorted keys. The `pprint` module is significantly more readable than `print` for complex data. Python 3.9+ added `pprint.pp` as an alias for `pprint.pprint`. The module is useful in debugging, logging structured data, and REPL exploration.

## Q90: What are Python's `importlib.import_module` and dynamic imports?
**A:** `importlib.import_module(name, package=None)` dynamically imports a module at runtime, given the module name as a string. It returns the imported module. Unlike `__import__`, it's a higher-level function. Dynamic imports are useful for: loading plugins, lazy loading, supporting customized import paths, implementing module-based configuration systems. The standard `import` statement is static (known at compile time). Dynamic imports are slower and bypass compile-time optimization. Combined with `sys.meta_path`, custom importers can load from databases, URLs, or custom storage. Security note: dynamic imports of user-provided strings can be dangerous.

## Q91: How does Python's `__init_subclass__` interact with multiple inheritance?
**A:** When a class inherits from multiple parents that define `__init_subclass__`, Python calls each parent's `__init_subclass__` in MRO order, but only if they call `super().__init_subclass__()`. Without `super()` calls, only the first parent's `__init_subclass__` is called (following MRO). This is the same cooperative pattern as `super().__init__()`. Keyword arguments passed in class definition are consumed by the first `__init_subclass__` in MRO. To pass different kwargs to different parents, use `**kwargs` and `pop()` for each parent's parameters. This pattern is essential in frameworks using cooperative multiple inheritance with mixins.

## Q92: What is the Python `os` module's support for file system operations?
**A:** The `os` module provides OS-level file system operations: `os.path` for path manipulation, `os.listdir()`, `os.walk()` for directory traversal, `os.mkdir()`/`os.makedirs()` for directory creation, `os.remove()`/`os.unlink()` for file deletion, `os.rename()`, `os.stat()` for file metadata, `os.chmod()` for permissions, `os.symlink()` for symbolic links. `os.path` provides: `exists()`, `isfile()`, `isdir()`, `join()`, `split()`, `splitext()`, `basename()`, `dirname()`, `abspath()`. Python 3.4+ introduced `pathlib` which provides object-oriented path handling, recommended over `os.path` for new code. `os` module is lower-level and cross-platform, but `pathlib.Path` is more ergonomic.

## Q93: How do you implement a Python decorator for logging function calls?
**A:** A logging decorator:

```python
import functools
import logging

def log_calls(logger=None, level=logging.INFO):
    if logger is None:
        logger = logging.getLogger(__name__)
    
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            args_repr = [repr(a) for a in args]
            kwargs_repr = [f"{k}={v!r}" for k, v in kwargs.items()]
            signature = ", ".join(args_repr + kwargs_repr)
            logger.log(level, f"Calling {func.__name__}({signature})")
            try:
                result = func(*args, **kwargs)
                logger.log(level, f"{func.__name__} returned {result!r}")
                return result
            except Exception as e:
                logger.log(level, f"{func.__name__} raised {e!r}")
                raise
        return wrapper
    return decorator
```

This decorator logs entry, exit (with result), and exceptions. The `@functools.wraps` preserves the original function's metadata. Parameterized decorators use three levels of nesting. For async functions, use `async def wrapper` with `await`.

## Q94: What is the purpose of Python's `typing.Final`?
**A:** `typing.Final` (Python 3.8+) indicates that a variable should not be reassigned, a method should not be overridden, or a class should not be subclassed. It's checked statically by type checkers and has no runtime effect. `Final` can be used: (1) with variable annotations — `MAX_RETRIES: Final = 3`, (2) with method declarations — `def final_method(self) -> None: ...` (though `@final` decorator is more common for methods), (3) with class declarations — `class FinalClass: ...` (again, `@final` decorator is preferred). The `@final` decorator (Python 3.8+) enforces the same at class and method level. These help create clear API contracts by preventing unintended extension or modification.

## Q95: How does Python's `bytes` vs `bytearray` differ?
**A:** `bytes` is an immutable sequence of bytes (0-255). `bytearray` is a mutable sequence of bytes. `bytes` is hashable (can be used as dict keys), `bytearray` is not. Both support indexing, slicing, and iteration. `bytes` can be created from literals (`b'hello'`), from ints (`bytes(10)`), from iterables of ints (`bytes([65, 66, 67])`). `bytearray` has the same creation methods plus `append()`, `extend()`, `insert()`, `pop()`, `remove()`, `reverse()`, and item assignment. Use `bytes` for immutable binary data (like hash digests), `bytearray` when you need to modify binary data in-place (like building network packets).

## Q96: What is the Python `__path__` attribute in packages?
**A:** `__path__` is an attribute of packages that lists the directories where the package's subpackages and submodules are searched. For regular packages (directories with `__init__.py`), `__path__` is a list with one element — the package directory. For namespace packages (PEP 420), `__path__` can contain multiple directories. Modifying `__path__` allows dynamically adding locations where submodules are searched. This is used by plugin systems and namespace packages to extend package contents from multiple locations. `pkgutil.extend_path()` modifies `__path__` for backward compatibility. `importlib.abc.PathEntryFinder` uses `__path__` for subpackage discovery.

## Q97: How do you handle time zones in Python?
**A:** The `datetime` module provides time zone support through `tzinfo` abstract base class and `timezone` (fixed offset) class. For IANA (Olson) time zones, use `pytz` or `zoneinfo` (Python 3.9+, PEP 615). `zoneinfo.ZoneInfo('America/New_York')` provides proper time zone objects. Best practices: (1) store datetimes in UTC internally, (2) convert to local time zones only for display, (3) use `datetime.now(tz=UTC)` or `datetime.now(timezone.utc)` for aware datetimes, (4) avoid naive datetimes, (5) use `astimezone()` for conversions, (6) use `dateutil` library for parsing time zone-aware strings. `datetime.fromtimestamp(ts, tz=UTC)` creates aware datetimes. `pendulum` and `arrow` are popular third-party alternatives.

## Q98: Explain Python's `sys.modules` and module caching.
**A:** `sys.modules` is a dictionary mapping module names to loaded module objects. Python checks this cache first during imports — if a module name is in `sys.modules`, it's reused without loading again. This makes module imports idempotent. Mutating `sys.modules` (adding, removing entries) affects subsequent imports. Removing an entry forces Python to reload the module on next import. `importlib.reload(module)` re-executes module code and updates `sys.modules`. Module caching can cause issues during development (stale modules) — use `importlib.reload` or restart the interpreter. Some frameworks manipulate `sys.modules` for lazy loading or mocking.

## Q99: What is the Python `asyncio.run()` function's lifecycle?
**A:** `asyncio.run(main())` (Python 3.7+) creates a new event loop, runs the coroutine until completion, and closes the loop. It handles: creating the event loop, running the coroutine, canceling remaining tasks, and shutting down async generators. `asyncio.run()` cannot be called from within a running event loop. It should be the entry point for asynchronous programs. The function handles the boilerplate of `loop = asyncio.new_event_loop()` and `loop.run_until_complete()`. Python 3.10+ added the `debug` parameter. Python 3.12+ added the `loop_factory` parameter. In Python 3.11+, `asyncio.run()` also handles KeyboardInterrupt properly.

## Q100: How does Python's `__subclasses__` method work and what are its use cases?
**A:** `__subclasses__()` is a method on class objects that returns a list of immediate subclasses (not deep subclasses). It uses weak references internally to avoid preventing garbage collection of subclasses. Use cases: (1) plugin discovery — finding all available plugin classes, (2) registry patterns — automatically registering all subclasses of a base type, (3) serialization frameworks — determining all possible types for polymorphic deserialization, (4) UI frameworks — listing all widget types, (5) testing — finding all test case subclasses. For deep subclass traversal, recursive walk is needed. `__subclasses__` only returns classes that have been imported and defined, so plugin modules must be explicitly imported first.
