# Regex Interview Questions and Answers - Part 2

## Q1: How do you implement a positive lookbehind in JavaScript and what are its browser compatibility limitations?
**A:** Use `(?<=pattern)` syntax. Example: `(?<=\$)\d+` matches digits preceded by a dollar sign. Lookbehinds were introduced in ES2018 (Chrome 62+, Firefox 78+, Safari 16.4+). They are NOT supported in Internet Explorer or older Safari versions. In lookbehinds, the pattern must be fixed-length in most implementations (though some engines like PCRE2 and Python support variable-length lookbehinds). For browser compatibility, use a capture group workaround: `(?:(\$\d+))` and extract group 1.

## Q2: How do atomic groups prevent catastrophic backtracking and what is the syntax across different engines?
**A:** Atomic groups `(?>...)` tell the engine: "if this group matches, don't backtrack into it." Example: `(?>a+)b` on input "aaaa" — `a+` matches all four `a`s, then `b` fails. Without atomic, the engine would backtrack: try 3 `a`s, fail; 2 `a`s, fail; 1 `a`, fail — catastrophic. With atomic, it fails immediately. Syntax: `(?>...)` in PCRE, Perl, Ruby, Java. Not supported in JavaScript. Python: `(?>...)` (available in `regex` module, not `re`). .NET: `(?>...)`. Alternative in JS: use possessive quantifiers where available.

## Q3: How do recursive regex patterns work in PCRE for matching nested structures?
**A:** Recursive patterns use `(?R)` to recurse into the entire pattern, or `(?n)` to recurse into a specific group. Example for balanced parentheses: `\((?:[^()]|(?R))*\)`. This matches an opening paren, then any non-paren characters or a recursive match of the entire pattern, then a closing paren. In PHP/PCRE: `$pattern = '/\((?:[^()]|(?R))*\)/';`. Perl: same syntax. Python's `regex` module supports `(?R)`. Not supported in JavaScript. For generalized nesting (multiple bracket types): `(?:\(|\[)(?:[^()\[\]]|(?R))*\)|\]` is limited because recursion matches the entire pattern, not specific sub-patterns.

## Q4: What is the `\G` anchor and how is it used for iterative matching?
**A:** `\G` matches at the position where the previous match ended (or at the start of the string for the first match). Useful for tokenizing or parsing. Example: matching all hex color codes in sequence: `\G#([0-9a-f]{6}|[0-9a-f]{3})`. In Perl: `while (/(\G#([0-9a-f]{6}|[0-9a-f]{3}))/g) { ... }`. In PHP's `preg_match_all` with `PREG_SET_ORDER`. `\G` is NOT supported in JavaScript. In Python's `regex` module: yes. In Python's `re` module: only in `scanner` objects. `\G` combined with lookaheads can validate entire structured strings.

## Q5: How does the `\K` match reset operator work and what is its use case?
**A:** `\K` resets the match start point, so everything matched before `\K` is not included in the final match (it acts as a variable-length lookbehind replacement). Example: `\d+\K[a-z]+` on "123abc" matches "abc" while requiring preceding digits. Use cases: replacing only part of a matched pattern without capture groups. In PHP/PCRE: `preg_replace('/foo\Kbar/', 'baz', 'foobar')` yields "foobaz". Perl: same. Not supported in JavaScript or Python's `re` module. Supported in Python's `regex` module. `\K` is more flexible than lookbehinds because it can match variable-length content.

## Q6: What are conditional patterns in regex and how do you implement if-else logic?
**A:** Syntax: `(?(condition)then-pattern|else-pattern)`. The condition can be a backreference `(?(1)...)` or a lookahead `(?(?=...)...)`. Example: match US or UK spellings: `(?(?=.*colour)(?:colour|flavour)|(?:color|flavor))`. More practical: match optional quotes around a value: `(['\"])?([^'\"]+)(?(1)\1)` — if we captured a quote, require it at the end. Another: validate phone numbers with or without area code: `^(\(\)?\d{3}(?(1)\)|-)\d{3}-\d{4}$`. Supported in PCRE, Perl, Python (`regex` module), .NET, not in JavaScript. Conditions enhance regex to near-Turing completeness in some engines.

## Q7: How do you detect and mitigate ReDoS (Regular Expression Denial of Service) attacks in user-supplied regex patterns?
**A:** ReDoS occurs when a regex has exponential worst-case time complexity due to nested quantifiers with overlapping matches. Pattern: `(a+)+b` vs input "aaaaac" (no `b`) — the engine tries all partitionings of `a`s. Mitigations: 1) Set a timeout for regex execution: use `Promise.race` with a timeout. 2) Validate user-supplied patterns against known bad patterns: detect nested quantifiers like `(.+)+`, `(.*)*`, `(a*)*`. 3) Use `re2` library (Google's safe regex engine) which guarantees linear time. 4) In Node.js, use `safe-regex` npm package to test patterns before execution. 5) In Python, use `regex` module's built-in timeouts with `regex.DEFAULT_TIMEOUT`.

## Q8: How do possessive quantifiers differ from greedy quantifiers in backtracking behavior?
**A:** Possessive quantifiers (`*+`, `++`, `?+`, `{n,m}+`) match as much as possible and NEVER give back characters (no backtracking). Greedy quantifiers match as much as possible but will give back characters if the overall pattern fails, trying backtracking. Example: `".++"` on `"abc"x"` fails because `".++` matches the entire string and the engine won't backtrack. Greedy `".+"` on the same input backtracks and succeeds. Possessive quantifiers are supported in PCRE, Perl, Java, Ruby, not in JavaScript. Use them for performance optimization and preventing catastrophic backtracking.

## Q9: What are branch reset groups and how do they differ from alternation in capture groups?
**A:** Branch reset groups `(?|...)` reset the capture group numbering in each alternation branch so that groups in different branches share the same numbers. Example: `(?|(foo)|(bar))` — both `foo` and `bar` are captured in group 1. Without branch reset: `(foo)|(bar)` — `foo` is in group 1, `bar` in group 2. Useful when you have alternatives that capture the same semantic data in different positions. Syntax: `(?|pattern1|pattern2|...)`. Supported in PCRE, Perl, PHP, R. Not in JavaScript or Python. Example: parsing date formats: `(?|(\d{4})-(\d{2})-(\d{2})|(\d{2})/(\d{2})/(\d{4}))` — group 1=year, 2=month, 3=day regardless of format.

## Q10: How do balancing groups work in .NET regex for matching nested structures?
**A:** Balancing groups (`(?<open>...)` and `(?<close-open>...)`) use a stack-based approach to track nesting depth. Named capturing groups push to a stack; `(?<close-open>)` pops from the "open" stack and captures the popped content into "close". Example matching balanced parentheses: `\([^()]*(?:(?<open>\()[^()]*)+(?:(?<close-open>\))[^()]*)*)*(?(open)(?!))\)`. The `(?(open)(?!))` condition ensures no unclosed open groups remain. Only available in .NET regex. For matching HTML tags: `(?<open><tag[^>]*>)[^<]*(?:(?<open><tag[^>]*>)[^<]*)*(?:(?<-open></tag>)[^<]*)*`. This is the most powerful form of regex recursion available.

## Q11: How do you optimize a regex that matches against millions of lines for maximum throughput?
**A:** Key strategies: 1) Avoid backtracking: use possessive quantifiers (`++`) and atomic groups. 2) Use specific character classes instead of `.*`: `[^,]*` instead of `.*?`. 3) Fail fast: put the most distinguishing part of the pattern early. 4) Unroll loops: replace `(?:[^"]|\\.)*` with `[^"\\]*(?:\\.[^"\\]*)*`. 5) Use regex prefixes: anchor with `^` if possible. 6) Compile once: `re.compile()` in Python, `new RegExp(...)` in JS. 7) Use `re2` or `rust/regex` for guaranteed linear time. 8) For line-by-line matching, use `re.finditer()` instead of `re.findall()` to avoid building a huge list. 9) Use `\A` and `\z` instead of `^` and `$` to avoid multiline overhead.

## Q12: How do Unicode property escapes differ from character classes for matching international text?
**A:** `\p{...}` matches characters by Unicode property rather than by specific code points. `\p{L}` matches any letter in any script (Latin, Cyrillic, Arabic, CJK, etc.), while `[a-zA-Z]` only matches ASCII letters. Key categories: `\p{L}` (letter), `\p{N}` (number), `\p{P}` (punctuation), `\p{Sc}` (currency), `\p{Emoji}` (emoji), `\p{Script=Greek}` (specific scripts). `\P{...}` is negation. Requires the `u` flag in JavaScript. In Python, use `re.UNICODE` flag or the `regex` module. Example: matching any letter in any language: `\p{L}+`. Performance: `\p{...}` is faster than listing ranges.

## Q13: How do you implement regex-based parsing of CSV data that handles quoted fields with embedded commas and newlines?
**A:** Robust CSV regex: `(?:,|\n|^)(?:"((?:[^"]|"")*)"|([^",\n]*))` — matches the delimiter first (comma, newline, or start), then captures either a quoted field (allowing `""` escape) or an unquoted field. Better approach: match each field: `/(?:^|,)(?:"((?:[^"]|"")*)"|([^",\n]*))/g`. Handle multiline quoted fields by using `[\s\S]` instead of `[^\n]`: `/(?:^|,)(?:"((?:[^"]|"")*)"|([^,"\n]*))/gm`. For production, use a dedicated CSV parser. The full RFC 4180 regex is complex and demonstrates the limitations of regex for complex parsing.

## Q14: How do you use regex with the `x` (verbose) flag to write self-documenting patterns?
**A:** The `x` flag (or `re.VERBOSE` in Python) ignores whitespace and allows comments. Example in Python:
```python
pattern = re.compile(r"""
    ^                    # Start of string
    (?=.*[a-z])          # At least one lowercase
    (?=.*[A-Z])          # At least one uppercase
    (?=.*\d)             # At least one digit
    [A-Za-z\d!@#$%^&*]   # Allowed characters
    {8,}                 # Minimum 8 characters
    $                    # End of string
""", re.VERBOSE)
```
In JavaScript: `/pattern/x` (ES2018). In PCRE: `/(?x)pattern/`. Spaces must be escaped with `\ ` or `[ ]` if you want to match literal whitespace. Comments extend to the first newline.

## Q15: How do you implement a regex that matches an IPv6 address in all its compressed forms?
**A:** Full IPv6 regex: `^(?:(?:(?:[0-9a-f]{1,4}:){6}(?:[0-9a-f]{1,4}(?::[0-9a-f]{1,4})?|(?::[0-9a-f]{1,4}){2}))|(?:(?:[0-9a-f]{1,4}:){5}(?::[0-9a-f]{1,4}){1,2})|(?:(?:[0-9a-f]{1,4}:){4}(?::[0-9a-f]{1,4}){1,3})|(?:(?:[0-9a-f]{1,4}:){3}(?::[0-9a-f]{1,4}){1,4})|(?:(?:[0-9a-f]{1,4}:){2}(?::[0-9a-f]{1,4}){1,5})|(?:(?:[0-9a-f]{1,4}:){1}(?::[0-9a-f]{1,4}){1,6})|(?:^:(?::[0-9a-f]{1,4}){1,7}))$`. This covers all forms: full, compressed (::), embedded IPv4 (::ffff:192.168.1.1), and loopback (::1). The pattern uses alternation for each possible number of groups handling the `::` compression in each position.

## Q16: How do you implement regex-based tokenization for a simple programming language lexer?
**A:** Define token patterns and match them sequentially. Example patterns: `WHITESPACE: \s+`, `IDENTIFIER: [a-zA-Z_]\w*`, `NUMBER: \d+(?:\.\d+)?`, `STRING: "(?:[^"\\]|\\.)*"`, `OPERATOR: [+-*/=<>!]=?|&&|\|\|`, `PUNCTUATION: [(){}\[\];,]`, `COMMENT: //.*|/\*[\s\S]*?\*/`. Apply them in priority order: use `re.finditer()` in Python with a combined pattern using alternation. Use named groups to identify token types. For performance, compile a single master regex. Regex-based lexers work well for simple languages but hit limitations with nested comments, here-docs, and context-sensitive tokens.

## Q17: How does regex backtracking work internally and how can you visualize it?
**A:** The regex engine uses a backtracking algorithm (similar to depth-first search). It tries different paths through the pattern, saving states at decision points (quantifiers, alternation). On failure, it backtracks to the last saved state and tries the next alternative. For `a(b|c)d` on "acd": 1) match `a`, 2) try `b` — fail, 3) backtrack to `(b|c)`, 4) try `c` — success, 5) match `d` — done. Tools like regex101.com (step-by-step debugger), RegexBuddy (debug mode), and Python's `re.DEBUG` flag show the internal process.

## Q18: How do you implement a regex that matches valid parentheses of multiple types (round, square, curly)?
**A:** Single-depth validation: `^(?:[^()\[\]{}]*(?:\([^()\[\]{}]*\)|\[[^()\[\]{}]*\]|\{[^()\[\]{}]*\})*[^()\[\]{}]*)*$`. This matches properly nested pairs but does NOT enforce correct ordering (e.g., `([)]` would incorrectly match). For correct ordering with arbitrary nesting, regex cannot solve this (it's context-free). Use a stack-based parser. PCRE recursive patterns can handle it: `(?:\((?:(?>[^()])|(?R))*\)|\[(?:(?>[^()])|(?R))*\]|\{(?:(?>[^()])|(?R))*\})`.

## Q19: How do you use named backreferences in replacement strings across different regex engines?
**A:** Named groups `(?<name>pattern)` can be referenced in replacements: JavaScript: `$<name>` (ES2018). Python: `\g<name>`. PHP: `\k<name>` or `${name}` in replacement. .NET: `${name}`. Perl: `$+{name}`. Ruby: `\k<name>`. Example: convert key=value to key: value: `str.replace(/(?<key>\w+)=(?<value>\w+)/g, '$<key>: $<value>')`. Best practice: use named groups for anything beyond 2-3 groups for readability.

## Q20: How do you implement input sanitization regex to prevent XSS and SQL injection?
**A:** Sanitization approaches: 1) Whitelist validation: `^[a-zA-Z0-9 _-]+$` — only allow known-safe characters. 2) HTML entity encoding: replace `<` with `&lt;`, `>` with `&gt;`. 3) SQL injection prevention: detect SQL keywords: `/\b(?:SELECT|INSERT|UPDATE|DELETE|DROP|UNION|OR|AND)\b/i` — but NEVER rely on regex alone for SQL injection; use parameterized queries. 4) For filenames: `/^[a-zA-Z0-9._-]+$/` to prevent path traversal. Regex-based sanitization is a SECONDARY defense; primary defenses are parameterized queries (SQLi), CSP headers (XSS), and context-aware encoding.

## Q21: How do you implement regex matching for Indian phone numbers with all valid formats?
**A:** Indian phone number regex: `^\+?91[-\s]?[6-9]\d{9}$|^0?[6-9]\d{9}$|^0\d{10,11}$`. Breakdown: `\+?91[-\s]?` — optional +91 country code with optional separator. `[6-9]\d{9}` — first digit must be 6-9, total 10 digits. Alternative formats: landlines with area codes: `^0\d{2,4}[-\s]?\d{6,8}$`. For formats like +91-9876543210, 09876543210, 9876543210. Using lookaheads: `^(?:(?:\+91|0)[-\s]?)?[6-9]\d{9}$`. Indian mobile numbers start with 6, 7, 8, or 9 only.

## Q22: How do you implement a regex-based template engine that replaces `{{variable}}` placeholders with nested object paths?
**A:** Base pattern: `/\{\{([^}]+)\}\}/g`. For nested paths like `{{user.address.city}}`: `const value = path.split('.').reduce((obj, key) => obj?.[key], data)`. Full implementation: `str.replace(/\{\{([^}]+)\}\}/g, (match, path) => { const trimmed = path.trim(); const value = trimmed.split('.').reduce((obj, key) => { if (key.includes('[')) { const [arr, idx] = key.split(/\[(\d+)\]/); return obj?.[arr]?.[parseInt(idx)]; } return obj?.[key]; }, data); return value ?? match; })`. For conditionals and filters, a more complex parser is needed.

## Q23: How do you use the `(*FAIL)` and `(*PRUNE)` backtracking control verbs in PCRE?
**A:** These are PCRE backtracking control verbs: `(*FAIL)` (or `(*F)`) forces the match to fail at the current position, causing the engine to try the next alternative. `(*PRUNE)` tells the engine not to backtrack past this point — if the pattern fails after `(*PRUNE)`, the engine skips all remaining alternatives. `(*SKIP)` advances the match start to the current position on failure. `(*THEN)` backtracks to the nearest alternation. Example: `\d+\+(\d+)(*PRUNE)|` — if the addition pattern fails, don't try other patterns. `(*ACCEPT)` forces an early successful match. Not available in JavaScript or Python's `re` module.

## Q24: How do you implement a regex to parse and validate cron expressions?
**A:** Cron expressions have 5-7 fields: minute, hour, day-of-month, month, day-of-week (optional: year). Pattern: `/^(\*|[0-5]?\d)([-\/,0-9]*)\s+(\*|(?:1?\d|2[0-3]))([-\/,0-9]*)\s+(\*|[12]?\d|3[01])([-\/,0-9]*)\s+(\*|1[0-2]|[1-9])([-\/,0-9]*)\s+(\*|[0-6])([-\/,0-9]*)$/`. For compound expressions like `*/5`, `1,3,5`, `1-10/2`, use separate sub-patterns. For cron with special strings (@yearly, @daily): `/\B@(?:yearly|annually|monthly|weekly|daily|hourly)\b/i`. Full cron validation is complex — combine regex for format with application logic for values.

## Q25: How do you implement regex-based diff/patch parsing for unified diff format?
**A:** Unified diff format regex: `^---\s+(\S+)\s+[\d\s:.-]+$\n^\+\+\+\s+(\S+)\s+[\d\s:.-]+$\n^@@\s+-(\d+)(?:,(\d+))?\s+\+(\d+)(?:,(\d+))?\s+@@`. Capture groups: old file, new file, old start line, old count, new start line, new count. For hunk content: `^-(.*)$` (removed lines), `^\+(.*)$` (added lines), `^\s(.*)$` (context lines). Full diff parser: each hunk is matched then each line within the hunk is processed by `/(^[ +-])(.*)/m`. Useful for building code review tools or automated patch application.

## Q26: How do you implement regex with the `v` flag (unicode sets mode) introduced in ES2024?
**A:** The `v` flag (ES2024) is an upgrade to the `u` flag for character classes. It enables: 1) Set notation: `[\p{ASCII}--\p{P}]` — ASCII characters minus punctuation. 2) Intersection: `[\p{Script=Latin}&&\p{Letter}]` — Latin letters. 3) Subtraction: `[\w--\d]` — word characters except digits. 4) Nested character classes: `[a-z&&[^aeiou]]` — consonants. Syntax uses `--` (difference), `&&` (intersection), and `||` (union). Unlike `u` mode, `v` mode changes escaping rules (e.g., `\` inside character classes is literal). Example: matching ASCII letters only: `/[\p{ASCII}--\p{Alpha}]/v`.

## Q27: How do you implement a regex that matches time durations in various formats (ISO 8601, human-readable)?
**A:** ISO 8601 duration: `^P(?:(\d+)Y)?(?:(\d+)M)?(?:(\d+)D)?(?:T(?:(\d+)H)?(?:(\d+)M)?(?:(\d+(?:\.\d+)?)S)?)?$`. Groups: years, months, days, hours, minutes, seconds. Human-readable: combine patterns for each unit: `(?:\d+\s*(?:years?|yrs?|y)\s*)?(?:\d+\s*(?:months?|mos?)\s*)?` etc. For mixed formats like "1h30m": `/(\d+)\s*(?:h(?:ours?)?|:)\s*(\d+)\s*(?:m(?:in(?:utes?)?)?)?/i`. Always validate ranges (hours 0-23, minutes 0-59) in code after matching.

## Q28: How do you properly escape user input for dynamic regex construction in JavaScript?
**A:** Use a function to escape regex special characters: `function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, '\\$&'); }`. The special characters are: `.`, `*`, `+`, `?`, `^`, `$`, `{`, `}`, `(`, `)`, `|`, `[`, `]`, `\`. Example: `new RegExp(escapeRegex(userInput), 'gi')`. For whole-word search: `new RegExp('\\b' + escapeRegex(word) + '\\b', 'gi')`. For fuzzy search: escape the input, then replace spaces with `.*`. Never interpolate user input directly into a regex string without escaping — this leads to ReDoS vulnerabilities.

## Q29: How do you implement a regex that matches semantic versioning (semver) with pre-release and build metadata?
**A:** Full semver regex (from semver.org): `^(0|[1-9]\d*)\.(0|[1-9]\d*)\.(0|[1-9]\d*)(?:-((?:0|[1-9]\d*|\d*[a-zA-Z-][0-9a-zA-Z-]*)(?:\.(?:0|[1-9]\d*|\d*[a-zA-Z-][0-9a-zA-Z-]*))*))?(?:\+([0-9a-zA-Z-]+(?:\.[0-9a-zA-Z-]+)*))?$`. Groups: 1=major, 2=minor, 3=patch, 4=pre-release, 5=build. Pre-release identifiers: alphanumeric, no leading zero on numeric-only identifiers. Build metadata: alphanumeric with dots. For range matching, use a dedicated library like `semver`.

## Q30: How do you implement regex-based log parsing for common log formats (Apache, JSON, syslog)?
**A:** Apache combined log format: `/^(\S+) (\S+) (\S+) \[([^\]]+)\] "(\S+) (\S+) (\S+)" (\d{3}) (\d+|-) "([^"]*)" "([^"]*)"$/`. Groups: IP, ident, user, datetime, method, path, protocol, status, bytes, referer, user-agent. JSON log lines: parse each line as JSON, then use regex on specific fields. Syslog RFC 5424: `/^<(\d{1,3})>(\d) (\S+) (\S+) (\S+) (\S+) (\S+) (\S+)(?: (\[.*?\]))?(?: (.*))?$/`. For error stack traces: `/at\s+(?:(?:(.+?)\s+)?\(?(?:file:\/\/)?(.+?):(\d+):(\d+)\)?)/`.

## Q31: How do you implement a regex that matches Chinese characters, Japanese kanji, and Korean Hangul?
**A:** Chinese (CJK Unified Ideographs): `\p{Script=Han}` (with `u` flag). Japanese: Hiragana `\p{Script=Hiragana}`, Katakana `\p{Script=Katakana}`, Kanji `\p{Script=Han}`. Korean: Hangul `\p{Script=Hangul}`. Combined: `[\p{Script=Han}\p{Script=Hiragana}\p{Script=Katakana}\p{Script=Hangul}]`. Without Unicode properties: CJK range `[\u4e00-\u9fff]`, Hangul: `[\uac00-\ud7af]`, Hiragana: `[\u3040-\u309f]`, Katakana: `[\u30a0-\u30ff]`. For mixed text: use the `u` flag with script property escapes.

## Q32: How do you implement a regex to parse RFC 5322 email addresses with all valid special cases?
**A:** RFC 5322 is extremely permissive. The full regex handles quoted local parts (e.g., `"john.doe"@example.com`), comments, IP literals, and special characters. In practice, use the simpler RFC 5321 addr-spec: `^[a-zA-Z0-9.!#$%&'*+/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*\.[a-zA-Z]{2,}$`. Validate further by sending a verification email rather than trying to capture every edge case in regex.

## Q33: How do you implement regex for matching code comments across multiple languages?
**A:** Single-line comments: `//.*` (C/JS/Java/Rust), `#.*` (Python/Ruby/Shell), `--.*` (SQL/Lua), `%.*` (MATLAB). Multi-line comments: `/\*[\s\S]*?\*/` (C/JS/Java), `<!--[\s\S]*?-->` (HTML/XML). Combined: `/(\/\/.*)|(\/\*[\s\S]*?\*\/)|(#.*)|(--.*)/gm`. For Python docstrings: `"""[\s\S]*?"""|'''[\s\S]*?'''`. Advanced pattern that skips comments inside strings: `/(?:"(?:[^"\\]|\\.)*"|'(?:[^'\\]|\\.)*'|\/\/.*|\/\*[\s\S]*?\*\/)/g` — match strings first, then replace only comment matches.

## Q34: How do you implement possessive quantifiers in JavaScript using atomic group emulation?
**A:** JavaScript doesn't support possessive quantifiers or atomic groups natively. Emulate them using a pattern that captures in a group and uses a backreference to prevent backtracking. Pattern: `(?=(pattern))\1`. The lookahead `(?=...)` matches without consuming input, then the backreference `\1` consumes the exact characters matched. Since the lookahead matched atomically, the backreference is fixed. Example: `(?=(a+))\1b` emulates `a++b`. For `".++"`: `(?=("[^"]*"))\1`. Limitations: lookahead patterns can't contain backreferences that reference outside the lookahead.

## Q35: How do you implement regex for matching and validating international bank account numbers (IBAN)?
**A:** IBAN structure: 2-letter country code + 2 check digits + BBAN (varies by country). Length varies (15-34 chars). General format: `^([A-Z]{2})(\d{2})([A-Z\d]{1,30})$`. Country-specific examples: UK: `^GB\d{2}[A-Z]{4}\d{14}$`, Germany: `^DE\d{2}\d{18}$`, France: `^FR\d{2}\d{5}\d{5}[A-Z\d]{11}\d{2}$`. More practical: validate format with `^[A-Z]{2}\d{2}[A-Z\d]+$` (length 15-34), then verify check digits using modulo 97 algorithm. IBAN regex cannot validate the check digits — use the modulo 97 algorithm in code.

## Q36: How do you implement a regex for matching DNS hostnames with all valid TLDs?
**A:** Hostname regex: `^(?:(?:[a-zA-Z0-9]|[a-zA-Z0-9][a-zA-Z0-9-]{0,61}[a-zA-Z0-9])\.)*(?:[a-zA-Z0-9]{2,63}\.?)$`. Each label: 1-63 chars, alphanumeric and hyphens (no leading/trailing hyphen). TLD: 2+ chars. For internationalized domain names (IDN): match the ASCII-encoded Punycode form. For matching registered domains (e.g., `example.co.uk`), use the Public Suffix List — regex cannot stay current with all TLDs. Always use a library with the Public Suffix List for accurate domain parsing.

## Q37: How do you implement a regex that matches valid JSON number representation?
**A:** JSON number grammar: optional minus, integer part (0 or digits 1-9 followed by digits), optional fraction, optional exponent. Regex: `^-?(?:0|[1-9]\d*)(?:\.\d+)?(?:[eE][+-]?\d+)?$`. Breakdown: `-?` — optional negative. `(?:0|[1-9]\d*)` — zero OR non-zero starting digit. `(?:\.\d+)?` — optional fraction. `(?:[eE][+-]?\d+)?` — optional exponent. Correctly: allows `0`, `-0`, `0.5`, `1e10`. Correctly rejects: `01` (leading zero), `.5` (no integer part), `1.` (no fraction digits), `+1` (leading plus).

## Q38: How do you implement regex for matching Markdown headers with proper ATX heading levels?
**A:** ATX headers: `^(#{1,6})\s+(.+)$`. Group 1: level, group 2: heading text. For headers with optional closing `#`: `^(#{1,6})\s+(.+?)(?:\s+#+)?\s*$`. ## Q100: How do you implement a regex that matches itself (self-referential/quine regex)?
**A:** A self-referential regex that matches its own source code: this requires the regex to be embedded in the pattern. Example approach: use the pattern as a string, then escape it and include it in the regex. In Perl: `$_ = qr/\Q$_\E/`. In JavaScript: `const r = /\/\/.../`. A known self-referential pattern: `$_ = '$_ = ' . $_ . ';$_;'; $_;`. For a true regex quine: `/^\/\^.*\/$/` — but this only matches regex literals of that form. A more interesting approach: create a pattern that extracts its own definition from a code file. In practice, self-referential regexes are a theoretical curiosity. The concept demonstrates the fixed-point theorem in formal languages: a regex that matches its own representation proves that regex languages can be self-describing to some extent.
