# Authorization Interview Questions and Answers - Part 2

## Q1: How does Google Zanzibar's graph-based authorization model work internally?
**A:** Zanzibar models authorization as a directed acyclic graph of relationships. Nodes are objects (users, documents, teams) and edges are relations (member, owner, viewer). An access check answers: "Is there a path from user U to object O via relation R?" The system uses tuple-based storage: `(object, relation, user)` tuples like `(doc:123, viewer, user:456)`. Zanzibar uses a Spanner-backed global database with versioned snapshots for consistent reads across data centers. Checks traverse the graph using set operations (union, intersection, exclusion). Zanzibar achieves <10ms p99 latency at billions of users by using caching, ordered evaluation, and bloom filters for negative lookups.

## Q2: What is OpenFGA and how does it implement Zanzibar concepts?
**A:** OpenFGA is an open-source authorization system inspired by Zanzibar. It uses Relationship-Based Access Control (ReBAC) with a tuple-based model. Key components: (1) **Authorization Model** -- defines types and relations (e.g., `type document { relations: define viewer: [user] }`). (2) **Tuples** -- stored relationships (`document:1 viewer user:alice`). (3) **Check API** -- `Check(user:alice, viewer, document:1)` returns `{allowed: true}`. (4) **Query API** -- `ListObjects(user:alice, viewer, document)` returns all documents Alice can view. OpenFGA supports conditional relationships, contextual tuples, and reverse expansion queries. It provides SDKs, a REST API, and a gRPC API.

## Q3: How does SpiceDB implement Zanzibar's graph traversal with performance at scale?
**A:** SpiceDB (authzed) uses a PostgreSQL or CockroachDB backend to store relationship tuples. For Check operations, it performs a bidirectional graph traversal: starting from both the subject and the resource, expanding reachable nodes until a connecting path is found. SpiceDB implements: (1) **Dispatch** -- work distribution across goroutines for parallel traversal. (2) **Caching** -- two-level caching (in-memory LRU + distributed cache). (3) **Tuple ordering** -- optimizing for common patterns (expand membership before ownership). (4) **Limit tree depth** -- preventing traversal loops with max depth limits. (5) **Consistency guarantees** -- using SpiceDB's own model (fully consistent, at-least-as-recent, or minimize latency). Performance benchmarks show <50ms p99 for checks with millions of tuples.

## Q4: What is the Rego policy language's `data` and `input` distinction in OPA?
**A:** In OPA/Rego: (1) `input` -- the dynamic request data (who, what, on which resource, under what conditions). It's passed with each authorization query. (2) `data` -- the static or semi-static context (roles, permissions, user attributes, resource metadata). `data` includes both embedded policies and external data loaded via bundles. This separation allows policy authors to write rules like: `allow { input.user.role == "admin" }` or `allow { input.resource.owner == input.user.id }`. External data is loaded into `data` via OPA's bundle mechanism or REST APIs.

## Q5: How do you implement authorization policies in OPA that evaluate against external data?
**A:** OPA can reference external data in two ways: (1) **Bundle loading** -- static data is bundled with the policy at deployment time. Good for role definitions and permission matrices that don't change frequently. (2) **OPA's built-in HTTP calls** -- Rego can call external APIs via `http.send`: `data_from_db = http.send({"method": "POST", "url": "http://db-service/query", "body": json.marshal(query)}).body`. (3) **Partial evaluation** -- OPA can pre-compute parts of a policy that depend only on static data, producing a simplified policy for runtime. For production, use OPA's `data` caching with bundle updates rather than runtime HTTP calls to maintain performance.

## Q6: How do you implement authorization policy testing in Rego?
**A:** Rego test files use the `_test.rego` suffix. Tests are written as rules with the `test_` prefix: `test_user_can_view_own_document { allow with input as {"user": {"id": "alice"}, "action": "view", "resource": {"owner": "alice"}} with data.roles as {"admin": ["view", "edit"]} }`. OPA supports: (1) Mocking -- `with input as {...}` and `with data.path as {...}`. (2) Coverage -- `opa test --coverage` reports which rules were exercised. (3) Benchmarking -- `opa bench` for performance testing. (4) Test coverage thresholds can be enforced in CI. Best practice: test both positive (allow) and negative (deny) cases, covering boundary conditions and edge cases.

## Q7: What is the AWS IAM policy evaluation logic (deny precedence) and how does it work?
**A:** AWS IAM uses a strict evaluation order: (1) All policies are evaluated -- identity-based, resource-based, Organizations SCPs, permissions boundaries, and session policies. (2) If an **explicit deny** is found in any applicable policy, the final decision is **Deny** (deny takes precedence). (3) If there is no explicit deny but there is an **explicit allow**, the decision is **Allow**. (4) If there is neither explicit deny nor explicit allow, the default is **Deny** (implicit deny). This means a single deny in any policy anywhere beats all allows. The deny precedence design ensures that a security admin at one level (SCP, permissions boundary) can always override permissions set at another level.

## Q8: How do you implement authorization with AWS IAM Permissions Boundaries?
**A:** An IAM Permissions Boundary sets the maximum permissions a user or role can have. It is not a grant -- it is a limit. Implementation: (1) Attach a policy (JSON) to a user or role as a permissions boundary. (2) The effective permissions are the intersection of the identity-based policy and the permissions boundary. (3) Example: a boundary that allows only S3 and DynamoDB means even if the user has `"Action": "*"`, they can only access S3 and DynamoDB. (4) Boundaries are commonly used for delegated administration -- allowing teams to create their own IAM roles within set limits. (5) SCPs in Organizations are account-level boundaries; permissions boundaries are within-account.

## Q9: How do you implement attribute-based access control (ABAC) with AWS IAM?
**A:** AWS IAM ABAC uses resource tags, user tags, and session tags for authorization: (1) Define IAM policies with condition blocks: `"Condition": {"StringEquals": {"aws:ResourceTag/Project": "${aws:PrincipalTag/Project}"}}`. (2) Users have tags (e.g., `Project=ProjectX`). (3) Resources have tags (e.g., `Project=ProjectX`). (4) The policy grants access only when the user's tag matches the resource's tag. (5) This scales because you don't need to update policies when users change projects -- just update the tag. (6) Session tags (`aws:PrincipalTag/`) can be passed when assuming a role via STS, enabling temporary scoped-down permissions based on runtime attributes.

## Q10: How does Kubernetes RBAC work with aggregation and why would you use it?
**A:** Kubernetes RBAC aggregation (ClusterRole aggregationRule) combines multiple ClusterRoles into one. Define a parent ClusterRole with `aggregationRule`: `clusterRoleSelectors` that match labels on child ClusterRoles. Example: an `admin` ClusterRole that aggregates `admin-core` and `admin-custom` ClusterRoles. Benefits: (1) Modular permission management -- different teams own their ClusterRoles. (2) Automatic inclusion -- new ClusterRoles with matching labels are automatically added to the aggregate. (3) Third-party extensions -- operators can define ClusterRoles with their labels and they are automatically included in admin roles. (4) Namespace scoping -- subject access reviews can evaluate namespace-level permissions for specific workloads.

## Q11: How do you implement fine-grained authorization for GraphQL using schema directives?
**A:** GraphQL authorization via schema directives: (1) Define a `@auth` directive in the GraphQL schema: `directive @auth(requires: Role = ADMIN) on OBJECT | FIELD_DEFINITION`. (2) Apply directives to fields: `type Query { user(id: ID!): User @auth(requires: USER) }`. (3) In the GraphQL execution pipeline, a middleware reads the directive from the schema. (4) Before resolving a field, the middleware checks the current user's role against the directive's requirement. (5) Unauthorized access returns a GraphQL error (null with errors path). (6) This approach enables field-level authorization -- different fields on the same type can have different requirements. (7) Libraries like `graphql-shield` (Node.js) or `graphql-auth` (Python) implement this pattern.

## Q12: How does PostgreSQL Row-Level Security (RLS) work with policies?
**A:** PostgreSQL RLS enables row-level authorization inside the database: (1) Enable RLS on a table: `ALTER TABLE documents ENABLE ROW LEVEL SECURITY;`. (2) Create policies: `CREATE POLICY user_policy ON documents FOR SELECT USING (owner_id = current_setting('app.current_user_id')::int);`. (3) The USING clause defines which rows are visible; the WITH CHECK clause defines which rows can be inserted/updated. (4) Set the user context: `SELECT set_config('app.current_user_id', '123', false);` at the start of each session (by the application after authentication). (5) RLS policies are automatically applied to all queries -- they cannot be bypassed by the application. (6) Performance: RLS adds overhead to every query -- use indexes on columns referenced in policies. (7) RLS supports policy permissiveness with `PERMISSIVE` (OR logic) and `RESTRICTIVE` (AND logic) policies.

## Q13: How do you implement authorization caching to balance performance with security?
**A:** Authorization caching strategies: (1) In-process cache (Caffeine, Guava) -- cache check results with TTL (5-60 seconds). (2) Distributed cache (Redis) -- cache decisions across service instances. (3) Negative caching -- cache deny results for a shorter period (2-5 seconds) to prevent repeated denied lookups. (4) Hierarchical caching -- coarse roles/permissions cached longer, fine-grained object-level checks cached shorter. (5) Proactive invalidation -- when a user's role changes, publish an invalidation event to clear that user's cache entries. (6) Stale-while-revalidate -- serve cached decisions while asynchronously refreshing. (7) Never cache for high-security operations (financial transfers, admin elevation). (8) Cache key strategy: `auth:check:{userId}:{action}:{resourceId}` with consistent hashing for distribution.

## Q14: How do you implement authorization for an event-driven architecture (Kafka)?
**A:** Event-driven authorization: (1) The producer checks authorization before publishing to a topic (can the user publish to this topic?). (2) The message includes the producer's identity and the authorization context as headers. (3) The consumer validates that the message comes from an authorized producer (by checking the message signature or trusted source). (4) For per-message authorization, the consumer checks: can the intended recipient read this message? (5) Topic-level ACLs control which producers/consumers can access which topics. (6) For sensitive data, encrypt messages with recipient-specific keys -- only authorized consumers can decrypt. (7) Use schema registry authorization -- control who can register, evolve, or read schemas. (8) For Kafka, use Kafka ACLs (ALLOPS, READ, WRITE, CREATE, DELETE, ALTER) with principal-based authorization.

## Q15: How do you implement authorization with Azure RBAC and custom roles?
**A:** Azure RBAC uses role assignments with three elements: security principal (user/group/service principal), role definition (permissions), and scope (management group, subscription, resource group, resource). Custom roles: (1) Defined as JSON with `Actions`, `NotActions`, `DataActions`, `AssignableScopes`. (2) Example: `"Actions": ["Microsoft.Compute/virtualMachines/restart/action"]`. (3) Custom roles are stored in Azure AD and can be assigned at any scope. (4) Unlike built-in roles, custom roles can specify `AssignableScopes` to restrict where they can be assigned. (5) `DataActions` provides data plane permissions (reading blob data vs. managing the storage account). Azure RBAC follows a deny-assignments model where deny rules take precedence over allow.

## Q16: How do you implement authorization with attribute-based conditions in GCP IAM?
**A:** GCP IAM Conditions add attribute-based access control to IAM roles. Conditions are CEL (Common Expression Language) expressions evaluated at access time. Examples: (1) Time-based: `request.time < timestamp("2025-01-01T00:00:00Z")`. (2) Resource-based: `resource.name.startsWith("projects/_/instances/prod-")`. (3) IP-based: `source.ip in ["192.168.0.0/16"]`. (4) Device-based: `access_level in ["REACH_IP"]` (using Access Context Manager). Conditions are evaluated only if the role binding would otherwise grant access -- they narrow but never expand permissions. Performance: condition evaluation adds minimal latency (typically <5ms). Conditions support up to 20 CEL expressions per role binding.

## Q17: How does the Ory Keto authorization system implement relationship tuples?
**A:** Ory Keto implements Zanzibar-style relationship tuples stored in a database (PostgreSQL, MySQL, or in-memory). A tuple is: `namespace:object#relation@subject`. Example: `files:report.txt#viewer@user:alice`. Keto provides: (1) **Check API** -- `GET /relation-tuples/check?namespace=files&object=report.txt&relation=viewer&subject=user:alice`. (2) **Expand API** -- returns the full set of subjects with a given relation on an object. (3) **REST and gRPC APIs**. (4) **Watchers** -- emit change events for cache invalidation. Keto supports subject sets (union, intersection, exclusion) for complex relationships. Performance: uses indexing on (namespace, object, relation) for fast lookups.

## Q18: How do you implement authorization in a multi-tenant SaaS using row-level security?
**A:** Multi-tenant authorization with RLS: (1) Each row has a `tenant_id` column. (2) Enable RLS on all tenant-scoped tables. (3) Create a policy: `CREATE POLICY tenant_isolation ON documents FOR ALL USING (tenant_id = current_setting('app.tenant_id')::int);`. (4) The application sets `app.tenant_id` and `app.user_id` at session start (after authentication). (5) All queries automatically filter by tenant -- the application cannot accidentally leak data across tenants. (6) For cross-tenant admin operations, a separate role bypasses RLS or uses a different session. (7) For shared reference data, use separate tables without tenant isolation or use row-level exceptions. (8) Monitor for RLS policy misconfigurations -- a missing policy on a table could expose all tenants' data.

## Q19: How do you implement authorization in a service mesh (Istio with OPA/SPIFFE)?
**A:** Service mesh authorization with Istio: (1) Istio uses Envoy proxies as Policy Enforcement Points (PEPs). (2) Configure AuthorizationPolicy CRDs: `apiVersion: security.istio.io/v1beta1; kind: AuthorizationPolicy`. (3) Rules specify: `from` (principals, namespaces, IPs), `to` (methods, paths, ports), `when` (conditions like request headers, JWT claims). (4) For fine-grained policy, use OPA as an external authorizer: Istio's `ext_authz` filter calls OPA via gRPC. (5) Workload identity uses SPIFFE -- each pod gets an X.509 certificate (via Istio's mTLS). (6) The AuthorizationPolicy can use `source.principal` (SPIFFE ID) for workload-to-workload authorization. (7) This enables zero-trust network authorization where every service-to-service call is authenticated and authorized.

## Q20: How do you implement authorization delegation with OAuth 2.0 token exchange?
**A:** Token exchange for authorization delegation: (1) Service A receives an access token from the user. (2) Service A needs to call Service B on behalf of the user. (3) Service A calls the authorization server's token exchange endpoint: grants `urn:ietf:params:oauth:grant-type:token-exchange`, with `subject_token` (user's token), `actor_token` (service A's token), and `resource` / `audience` (service B). (4) The authorization server returns a new token for Service B with: the user as the subject, the service as the actor, and scoped-down permissions (impersonation or delegation). (5) Service B receives the delegated token and can verify both the user identity and the calling service identity. This enables security domain boundary crossing without exposing the original user token.

## Q21: How does the AuthZEN standard enable authorization for AI agents?
**A:** AuthZEN is an emerging standard for authorization in AI agent ecosystems. Key concepts: (1) **Agent identity** -- an AI agent (LLM, RPA bot) has its own identity separate from the human who invoked it. (2) **Delegated authority** -- a human grants specific scopes to an agent (e.g., "read calendar, send emails on my behalf"). (3) **Intent verification** -- the authorization service evaluates not just the action but the agent's stated intent. (4) **Human-in-the-loop** -- high-risk agent actions trigger a human authorization step (e.g., "This agent wants to send $10,000 -- confirm?"). (5) **Audit trail** -- all agent actions are logged with both agent and authorizing human identities. AuthZEN defines a common API for authorization services to evaluate agent requests.

## Q22: How do you implement authorization for a headless CMS with fine-grained content permissions?
**A:** Headless CMS authorization: (1) **Content model permissions** -- who can create, read, update, delete entries of each content type. (2) **Field-level permissions** -- restrict access to specific fields (e.g., an author cannot see internal notes or SEO metadata). (3) **Locale-based permissions** -- editors can only edit content in specific languages. (4) **Workflow state permissions** -- who can publish, who can only draft, who can approve. (5) **Environment permissions** -- developers can edit in dev/staging but not production. (6) **Scheduled content permissions** -- only certain roles can schedule content publishing. (7) **Media library permissions** -- folder-level access control for assets. (8) **API key scoping** -- each API key has specific content type and operation restrictions. Implement using a policy engine (OPA) or a dedicated authorization service (SpiceDB) for complex content models.

## Q23: How do you implement authorization with Casbin for multi-language applications?
**A:** Casbin uses a model-configuration-request paradigm that works across languages: (1) **Model file (.conf)** -- defines the access control model (ACL, RBAC, ABAC, or custom). Example: `[request_definition]; r = sub, obj, act; [policy_definition]; p = sub, obj, act; [matchers]; m = r.sub == p.sub && r.obj == p.obj && r.act == p.act`. (2) **Policy file (.csv)** -- defines the actual rules. (3) **Request** -- each authorization check passes (subject, object, action). (4) Casbin supports RBAC with role hierarchies, domain (tenant) isolation, and multiple matching functions (key match, regex, glob). (5) Casbin provides SDKs in Go, Java, Node.js, Python, PHP, .NET, Rust, and C++. (6) The policy can be stored in file, database, or external storage. Casbin's performance: ~10 microseconds per check in local mode.

## Q24: How do you implement authorization for real-time applications (WebSocket, SSE)?
**A:** Real-time authorization: (1) **Connection-time authorization** -- validate the user's token when the WebSocket/SSE connection is established. Reject unauthorized connections. (2) **Per-message authorization** -- for sensitive operations over WebSocket, validate each message against the user's permissions. (3) **Channel-based authorization** -- the client subscribes to specific channels; the server validates that the user can access each channel (like Pusher's private channel auth). (4) **Presence authorization** -- only authorized users can see who else is in a channel. (5) **Rate limiting per connection** -- prevent abuse even from authorized connections. (6) **Re-evaluation** -- if the user's permissions change mid-session, disconnect and force reconnection with new authorization context.

## Q25: How do you implement authorization in a CQRS/Event Sourcing architecture?
**A:** CQRS authorization: (1) **Command side (writes)** -- validate that the user can perform the action before accepting the command. Commands include the user identity. Authorize before enqueuing the command. (2) **Query side (reads)** -- validate that the user can access the data being queried. Use read model projections that already account for user permissions, or filter results post-query. (3) **Event store** -- the event store itself typically authorizes append operations. (4) **Event handlers/projectors** -- these run with system privileges (they process authorized events). (5) **Saga/choreography** -- if a command triggers a saga that performs multiple actions, each action may need to be authorized independently. (6) **View models** -- project authorization-relevant data into read models so queries can filter efficiently.

## Q26: How do you implement authorization for a B2B SaaS with hierarchical org structures?
**A:** Hierarchical org authorization: (1) Organizations have a tree structure (org -> departments -> teams). (2) Permissions can be inherited -- a user who can "view" at the org level can view all departments and teams beneath it. (3) Permission inheritance uses the ReBAC model: `team:backend#member@user:alice`, `org:acme#contains@team:backend`, resolved transitively to `org:acme#member@user:alice`. (4) Support for both upward propagation (manager can view direct reports) and downward propagation (global admin can view all). (5) Break-glass policies -- at any level, an admin can automatically view child resources in emergencies. (6) Role hierarchy -- `ORG_ADMIN` inherits all `DEPARTMENT_ADMIN` and `TEAM_ADMIN` permissions. (7) Deny rules can block inheritance -- a top-level deny blocks all subtree access regardless of inherited allows.

## Q27: How does the Cedar policy language (from Amazon Verified Permissions) work?
**A:** Cedar is a declarative policy language designed for fine-grained authorization. Syntax: `permit(principal in Role::"admin", action in Action::"view", resource in Resource::"document") when { resource.owner == principal };`. Key features: (1) **Principal, Action, Resource** -- each policy specifies who can do what on which. (2) **Conditions** -- `when` (must be true for the policy to apply) and `unless` (must be false). (3) **ABAC support** -- policies can reference any attribute of principals, actions, resources, and context. (4) **Multi-action policies** -- `action in [Action::"read", Action::"write"]`. (5) **Templates** -- reusable policy templates with slots for specific values. (6) **Schema-based** -- Cedar validates policies against a schema that defines entity types and their attributes. (7) **Performance** -- Cedar can evaluate 10,000+ policies per millisecond using efficient trie-based policy indexing.

## Q28: How do you implement authorization testing with property-based testing?
**A:** Property-based testing for authorization: (1) Define properties that must always hold: "No user can access a resource they don't own unless explicitly granted," "An admin can access any resource," "Permissions are never expanded in a deny-override model." (2) Use a property-based testing framework (QuickCheck, Hypothesis, fast-check). (3) The framework generates random authorization states (users, roles, resources, policies) and random access requests. (4) Each generated scenario is evaluated against both the implementation and an oracle (a reference model or simplified version). (5) When mismatches are found, the framework shrinks the scenario to the minimal failing case. (6) This catches edge cases that unit tests miss (overlapping policies, conflicting rules, unexpected attribute combinations). (7) Property-based tests are especially valuable for complex ReBAC and ABAC systems.

## Q29: How do you implement just-in-time (JIT) privilege elevation for break-glass access?
**A:** JIT privilege elevation workflow: (1) User requests elevated access through a dedicated portal or API. (2) The request specifies: which privileges, for which resource, for how long (15 minutes to 24 hours), and a business justification. (3) The request is evaluated against approval policies -- auto-approved for low-risk, requires manager approval for medium-risk, requires multi-party approval for high-risk. (4) If approved, a temporary role or policy is created with the specified time bound and scope. (5) The user's session is updated with the elevated privileges (or a new session is created). (6) All actions during the elevated period are audited in detail. (7) Automatic revocation at expiration -- the temporary policy is deleted, and any active sessions are notified. (8) Post-elevation review -- generate a report of what was done during the elevation window.

## Q30: How do you implement authorization for serverless functions (AWS Lambda)?
**A:** Lambda authorization: (1) **API Gateway Lambda Authorizer** -- a custom function that validates tokens and returns an IAM policy controlling access to API routes. Cache the policy using the authorizer's TTL. (2) **Resource policy** -- control which accounts, IPs, or VPCs can invoke the Lambda function directly. (3) **IAM execution role** -- controls what the Lambda function itself can do (access DynamoDB, S3, etc.). (4) **Event source mapping** -- for SQS/DynamoDB Streams/Kinesis, authorization is done at the source (who can send to the queue/stream). (5) **Lambda@Edge** -- authorize at the CDN edge before requests reach the origin. (6) **Fine-grained function URL authorization** -- Lambda function URLs with `AuthType: AWS_IAM` using SigV4 signing. (7) For multi-tenant serverless, include tenant ID in the JWT and check it in every Lambda function invocation.

## Q31: How do you implement authorization with Google Zanzibar's Check API using SpiceDB?
**A:** SpiceDB implements Zanzibar's Check API: `spicedb.CheckPermission(Subject{sub:"user:alice"}, Permission:"view", Resource{obj:"document:1"})`. Internally: (1) SpiceDB loads the resource's relationships from the database. (2) It expands indirect relationships via the schema's relation definitions. (3) For union relations (e.g., `viewer = owner + member`), it checks if any member of the union matches the subject. (4) For intersection relations (e.g., `editor = owner AND member`), all conditions must be met. (5) For exclusion relations (e.g., `viewer = member NOT blocked`), the excluded set is removed. (6) Results are cached with configurable consistency (fully consistent reads vs. eventually consistent). (7) The response includes `checked_at` timestamp for consistency tracking.

## Q32: How do you implement authorization for a multi-cloud deployment?
**A:** Multi-cloud authorization: (1) **Unified policy language** -- use a cloud-agnostic policy engine (OPA, Cedar) to write policies once and deploy across clouds. (2) **Externalized authorization** -- all cloud deployments check with the same authorization service (deployed in a separate region/cloud). (3) **IAM identity federation** -- use OIDC federation so cross-cloud resources can verify the same identity tokens. (4) **Policy synchronization** -- synchronize authorization policies across clouds using a GitOps approach (bundle distribution). (5) **Cloud-specific wrappers** -- wrap the unified policy engine with cloud-native authorization interfaces (AWS Lambda Authorizer, Google Cloud External Authorizer). (6) **Consistent audit trail** -- aggregate authorization logs from all clouds into a single SIEM. (7) **Failover** -- if one cloud's authorization service is unavailable, fall back to cached decisions.

## Q33: How does authorization work with Apache Kafka's ACLs?
**A:** Kafka ACLs are defined per resource: (1) Resource patterns: TOPIC, GROUP, CLUSTER, TRANSACTIONAL_ID, DEAD_LETTER_QUEUE. (2) Operations: READ, WRITE, CREATE, DELETE, ALTER, DESCRIBE, ALL, DESCRIBE_CONFIGS, ALTER_CONFIGS, IDEMPOTENT_WRITE, CLUSTER_ACTION. (3) Principal format: `User:CN=alice` (TLS) or `User:alice` (SASL). (4) Host-based restrictions: `Host:192.168.*`. (5) ACLs can be applied to literal resource names or prefixed patterns. (6) Kafka's `AclAuthorizer` caches ACLs and checks them for every produce/consume request. (7) Super.users bypass all ACLs. (8) Best practice: use combination of SSL principal extraction and ACLs for topic-level authorization, and add a custom authorizer for additional business logic.

## Q34: How do you implement authorization for a document store (MongoDB, Couchbase)?
**A:** Document DB authorization strategies: (1) **Database user roles** -- MongoDB has built-in roles (`read`, `readWrite`, `dbAdmin`) at database and collection levels. (2) **Field-level redaction** -- MongoDB supports field-level redaction using aggregation pipelines with `$redact` and `$filter`, based on user roles. (3) **Application-level filtering** -- add user context to queries: `db.collection.find({owner: userId})`. (4) **View-based authorization** -- create database views that filter rows based on user permissions. (5) **Proxy layer** -- a middleware service that intercepts queries and injects authorization filters before forwarding to the database. (6) **Document attributes** -- add `accessControlList` or `permissions` array to each document; the application filters based on the current user's permissions at query time.

## Q35: How do you implement authorization with HashiCorp Vault?
**A:** Vault authorization: (1) **Policies** -- HCL-based: `path "secret/data/projects/*" { capabilities = ["read", "list"] }`. (2) **Capabilities** -- create, read, update, delete, list, sudo, deny. (3) **Policy templates** -- parameterized with user/group/token metadata: `path "secret/data/{{identity.entity.name}}/*"`. (4) **Sentinel policies** -- for enterprise, fine-grained ABAC policies in Sentinel language. (5) **Identity system** -- entities (users) and groups with aliases across auth methods. (6) **Policy inheritance** -- policies attached to groups are inherited by members. (7) **Control groups** -- require authorization from N-of-M designated approvers before access. (8) **Endpoint restriction** -- policies can restrict which Vault endpoints a user can access, enabling fine-grained authorization within Vault's secret management.

## Q36: How do you implement authorization for a feature flag system?
**A:** Authorization for feature flags: (1) Flags are evaluated per-user, per-context. (2) The authorization system decides: can this user see/use this feature? (3) Use OPA or Rego for complex targeting rules: `allow_feature("dark-mode") { input.user.tier == "premium" }`. (4) For multi-tenant feature flags, combine tenant subscription level with user roles: `feature_enabled("analytics") { input.tenant.plan == "enterprise" AND has_role(input.user, "admin") }`. (5) Cache feature flag decisions aggressively (short TTL) since they're checked frequently. (6) Feature flags can reference authorization context -- user country, device type, beta group membership. (7) Use the authz system's audit logging to track feature flag access for compliance.

## Q37: How does the AWS IAM policy condition operator matrix work?
**A:** AWS IAM condition operators: (1) **String** -- `StringEquals`, `StringNotEquals`, `StringLike` (with `*` and `?` wildcard), `StringNotLike`. ## Q38: How do you implement authorization with a policy decision diagram?
**A:** A Policy Decision Diagram (PDD) is a tree representation for efficient policy evaluation: (1) Root node represents the first attribute to evaluate (e.g., user role). (2) Each branch represents a value or range (e.g., "admin", "user", "guest"). (3) Leaf nodes represent the decision (allow/deny) or a reference to another PDD. (4) PDDs are constructed from policy sets (XACML, ALFA) by combining rules and eliminating redundancy. (5) Evaluation: traverse the diagram from root, following branches matching the request attributes. (6) PDDs provide O(1) to O(log n) policy evaluation regardless of the number of policies. (7) Used in high-performance authorization systems where low latency is critical. PDDs can be updated by invalidating and rebuilding when policies change.

## Q56: How do you implement authorization with XACML (eXtensible Access Control Markup Language)?
**A:** XACML is an XML-based standard for attribute-based access control. Components: (1) **PAP (Policy Administration Point)** -- where policies are authored. (2) **PDP (Policy Decision Point)** -- evaluates policies against requests. (3) **PEP (Policy Enforcement Point)** -- intercepts requests and enforces decisions. (4) **PIP (Policy Information Point)** -- provides external attributes. A XACML policy has: `Target` (which subjects, resources, actions it applies to), `Effect` (Permit/Deny), `Condition` (boolean expression), and `Obligations` (actions to execute with the decision). XACML supports combining algorithms: permit-overrides, deny-overrides, first-applicable, and only-one-applicable. ALFA is a more developer-friendly alternative to raw XACML XML.
