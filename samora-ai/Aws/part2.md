# AWS Interview Questions and Answers - Part 2

## Q1: How would you design a VPC with public, private, and isolated subnets across multiple AZs?
**A:** Use 3 AZs minimum. In each AZ, create a public subnet (routes 0.0.0.0/0 to IGW), a private subnet (routes 0.0.0.0/0 to NAT Gateway), and an isolated subnet (no internet route at all). Public subnets host ALBs/NAT Gateways. Private subnets host application servers. Isolated subnets host databases (RDS, ElastiCache). Use separate route tables per subnet type. Each NAT Gateway goes in a public subnet and provides outbound internet for private subnets in the same AZ (for HA, deploy one NATGW per AZ). Use VPC Flow Logs to monitor traffic.

## Q2: Explain the difference between Security Groups and NACLs with a real-world example.
**A:** Security Groups are stateful, instance-level firewalls supporting only allow rules. If you allow inbound HTTPS, the outbound response is automatically allowed. NACLs are stateless, subnet-level firewalls supporting allow AND deny rules evaluated by rule number. Example: To block a specific IP (e.g., known attacker), you MUST use a NACL (deny rule). SG cannot deny specific IPs. Typical setup: use SGs for whitelist-based access control and NACLs as a stateless defense-in-depth layer with explicit deny rules for known bad IP ranges.

## Q3: How do you design IAM policies with permission boundaries and service control policies?
**A:** Layered approach: SCPs (in AWS Organizations) define the maximum allowed permissions for all accounts in an OU. Permission boundaries set the maximum permissions for a specific IAM user/role within the account. Identity-based policies grant actual permissions within those boundaries. Example: SCP denies all EC2 instances in non-us-east-1 regions. Permission boundary limits a developer role to only EC2 and S3. The identity policy further restricts to specific S3 buckets and EC2 instance types. Effective permissions = Intersection of SCP ∩ Boundary ∩ Identity.

## Q4: When should you use S3 bucket policies vs IAM policies for access control?
**A:** Use S3 bucket policies when: 1) Granting cross-account access (bucket policy specifies the external account/role). 2) Setting service-wide conditions (e.g., require MFA, enforce TLS, block public access). 3) Controlling access based on requester IP/VPC endpoint. Use IAM policies when: 1) Granting access to multiple AWS services from a single role. 2) Following least privilege per user/role. 3) Managing access centrally via IAM. Best practice: use IAM for user/role-level permissions and bucket policies for bucket-level restrictions and cross-account access.

## Q5: How do CloudFront origin groups work for failover?
**A:** Origin groups in CloudFront let you define a primary and secondary origin. If CloudFront gets a 5xx error (or timeout/connection failure) from the primary origin, it fails over to the secondary origin. Configure failover criteria (which HTTP status codes trigger failover). This is useful for multi-region active-passive setups: primary origin is an ALB in us-east-1, secondary is an ALB in eu-west-1. Combined with Route53 latency routing, this provides regional failover at the CDN level.

## Q6: What are the tradeoffs between RDS and DynamoDB for a social media application?
**A:** RDS (relational): Strong consistency, complex queries (JOINs), ACID transactions, fixed schema. Good for user profiles, relationships (friend graphs). DynamoDB: Single-digit ms latency at any scale, flexible schema, auto-scaling, serverless. Good for feed items, likes, comments (high-write, key-value access). Tradeoffs: RDS needs manual scaling, has write constraints. DynamoDB lacks complex queries, JOINs, has 400KB item limit. Typical social app uses both: RDS for user data, DynamoDB for activity feeds. Use DynamoDB for high-traffic, simple access patterns; RDS for complex relational queries.

## Q7: What techniques optimize Lambda cold starts?
**A:** 1) Provisioned Concurrency: pre-warms a set number of execution environments. 2) SnapStart (Java 11+): takes a snapshot of the initialized execution environment and resumes from it. 3) Use lighter runtimes (Node, Python) over Java/C#. 4) Minimize deployment package size (remove dev dependencies). 5) Use Layers for shared dependencies. 6) VPC cold starts are worse (ENI attachment) - use VPC endpoints, or avoid VPC if not needed. 7) Keep functions warm with scheduled CloudWatch Events (not reliable). 8) Use Lambda Response Streaming for first-byte latency. 9) Use SnapStart for Java functions (reduces cold start from ~5s to ~200ms).

## Q8: What criteria would you use to choose between ECS (with Fargate) and EKS?
**A:** Choose ECS+Fargate when: 1) You want minimal operational overhead (no control plane management). 2) Your team knows AWS-native tools. 3) You need simple service discovery, ALB integration. 4) You don't need Kubernetes-specific features. Choose EKS when: 1) You need Kubernetes portability (multi-cloud, hybrid). 2) You use Kubernetes-native tools (Helm, Kustomize, ArgoCD). 3) You need advanced orchestration (StatefulSets, CRDs, Operators). 4) Your team has K8s expertise. Tradeoffs: ECS is simpler, EKS is more flexible. EKS control plane costs $0.10/hr, Fargate costs more per pod than EC2 launch type.

## Q9: How do CloudWatch and X-Ray differ in observability?
**A:** CloudWatch: metrics, logs, alarms, dashboards. Best for: resource utilization (CPU, memory), application logs, operational health, threshold-based alerting. X-Ray: distributed tracing, service maps, trace analysis. Best for: understanding request flow across services, identifying latency bottlenecks, debugging errors in distributed systems. They complement each other: CloudWatch tells you WHAT is failing (high error rate), X-Ray tells you WHERE and WHY (which service, which call). Use CloudWatch Logs for log aggregation, X-Ray SDK for trace instrumentation, and CloudWatch ServiceLens to unify both views.

## Q10: How would you optimize costs for a production workload running on AWS?
**A:** 1) Reserved Instances/Savings Plans for steady-state compute (up to 72% discount). 2) Spot Instances for fault-tolerant, flexible workloads (up to 90% discount). 3) S3 Intelligent-Tiering for unknown access patterns. 4) S3 Lifecycle policies to move old data to Glacier. 5) Right-size instances with Compute Optimizer. 6) Use Auto Scaling to match demand. 7) Use Graviton instances (ARM-based, 20-40% better price/performance). 8) Use Fargate Spot for non-critical containers. 9) Delete unused EBS snapshots and old EBS volumes. 10) Use CloudWatch Budgets + Anomaly Detection alerts. 11) Use S3 Transfer Acceleration only when needed. 12) Monitor data transfer costs (often overlooked).

## Q11: Walk through the Well-Architected Framework review process for an existing workload.
**A:** Step 1: Identify workload and stakeholders. Step 2: Use the Well-Architected Tool in AWS Console. Step 3: Answer questions across all six pillars (Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability). Each pillar has design principles and questions. Step 4: Review answers and identify high-risk issues (HRIs). Step 5: Prioritize improvements based on business impact. Step 6: Create an action plan with milestones. Step 7: Implement improvements. Step 8: Schedule regular reviews (quarterly/semi-annual). Key: each pillar has specific lenses (Serverless, SaaS, HPC, etc.) for specialized review.

## Q12: How would you design a multi-region active-active application?
**A:** 1) Route53 with latency-based + geolocation routing to direct users to nearest region. 2) Application deployed identically in each region (CloudFormation/CDK stacks). 3) Global DynamoDB tables (multi-region, multi-master) for session/data that needs global access. 4) RDS cross-region read replicas for read-heavy workloads; Aurora Global Database for primary-region writes. 5) S3 Cross-Region Replication for static assets and user uploads. 6) CloudFront as global CDN with origin failover. 7) Global Accelerator for static IP and traffic optimization. 8) Separate CI/CD per region with gradual rollout. 9) Data consistency strategy: CRDTs or last-writer-wins for conflicts. 10) Monitoring with CloudWatch cross-region dashboards.

## Q13: How do VPC endpoints (Gateway vs Interface) differ and when do you use each?
**A:** Gateway Endpoints: free, use route table entries to reach S3 and DynamoDB privately. Interface Endpoints (AWS PrivateLink): charged by hour + data processed, use ENI with private IPs, support many services (SQS, SNS, Lambda, KMS, API Gateway, etc.). Choose Gateway for S3/DynamoDB (free, no IAM policy conditions needed). Choose Interface for all other AWS services (requires PrivateLink). Gateway endpoints work via prefix lists in route tables; Interface endpoints use DNS resolution. For S3 access from VPC, always prefer Gateway Endpoint unless you need PrivateLink features (IAM conditions with sourceVpce, on-premises access via DX/VPN).

## Q14: Explain S3 bucket policies vs IAM policies in detail with a cross-account example.
**A:** Cross-account S3 access requires BOTH: 1) Bucket policy granting access to the external account/role. 2) IAM policy in the external account allowing the user/role to use that access. Example: Account A's bucket policy: `{"Effect":"Allow","Principal":{"AWS":"arn:aws:iam::AccountB:root"},"Action":"s3:GetObject","Resource":"arn:aws:s3:::bucket-a/*"}`. Account B's IAM policy: `{"Effect":"Allow","Action":"s3:GetObject","Resource":"arn:aws:s3:::bucket-a/*"}`. Without BOTH, access fails. Bucket policy alone is insufficient if the external identity doesn't have IAM permissions. IAM policy alone is insufficient because the resource (bucket) hasn't granted access.

## Q15: How does CloudFront handle multiple behaviors and cache behaviors?
**A:** CloudFront behaviors map URL path patterns to origin configurations. Each behavior specifies: origin, viewer protocol policy, allowed HTTP methods, cache policy, origin request policy, response headers policy, Lambda@Edge associations, and TTL settings. Path patterns are evaluated from most specific to least (default `*` behavior). Example: `/api/*` -> ALB origin (no caching, forward all headers), `/images/*` -> S3 origin (cache with long TTL), `*` (default) -> custom origin. Behaviors enable serving different content from different origins with different caching rules within a single distribution.

## Q16: Compare Route53 routing policies: Geoproximity vs Geolocation vs Latency.
**A:** Geolocation: routes based on user's geographic location (country/state). Good for content localization, licensing restrictions. Latency: routes to region with lowest latency for that user. Good for performance optimization. Geoproximity: routes based on geographic distance with optional bias to shift traffic. Good for gradually shifting traffic between regions. Key difference: Latency routes to fastest region (may not be closest geographically due to network conditions). Geolocation routes based on boundaries. Geoproximity combines distance + bias for fine-grained control. Use Geoproximity with a bias to implement gradual traffic shifting during migrations.

## Q17: How do DynamoDB Transactions work and what are their limitations?
**A:** DynamoDB Transactions provide ACID across one or more tables within a single AWS account/region. `TransactWriteItems` supports up to 25 actions (Put, Update, Delete, ConditionCheck) atomically. `TransactGetItems` reads up to 25 items consistently. Limitations: 1) 25 actions max per transaction. 2) 4MB total data size. 3) Higher cost (2x RCU/WCU for transactional reads/writes). 4) Idempotency tokens (optional, prevent duplicate processing within 10 minutes). 5) No cross-region transactions. 6) Conditional writes within transaction support only simple conditions. Use transactions for critical financial operations, not for high-volume operations.

## Q18: What causes Lambda cold starts and how does SnapStart help?
**A:** Cold start happens when Lambda needs to create a new execution environment: download code, initialize runtime, run init code (constructor, static initializers), then run handler. It takes 100ms-10s depending on runtime and package size. SnapStart (Java 11+ only): takes a snapshot of the execution environment AFTER initialization but BEFORE handler runs. When invoked, Lambda resumes from the snapshot instead of re-initializing. This reduces cold start from ~5-10s to ~200ms for Java. SnapStart constraints: no network connections or unique runtime state during init, need to restore connections in handler. Not available for Python/Node/Go.

## Q19: How would you implement a VPC with shared services architecture (hub-and-spoke)?
**A:** Hub VPC contains shared services: NAT Gateways, Transit Gateway, Direct Connect/VPN, central egress, shared services (AD, DNS, monitoring). Spoke VPCs are attached to Transit Gateway. Each spoke uses Transit Gateway to reach hub for internet access (no direct IGW). Use RAM (Resource Access Manager) to share subnets for centralized NAT. Route tables: spoke routes 0.0.0.0/0 to TGW attachment; hub routes spoke CIDRs to appropriate TGW attachments. Security: Network Firewall in hub for centralized inspection, VPC Flow Logs on all attachments. This pattern centralizes egress, reduces NAT Gateway costs, and simplifies network security.

## Q20: How does AWS Reserved Instances work and what are Convertible vs Standard?
**A:** Reserved Instances offer significant discounts (up to 72%) for committing to 1 or 3 years. Standard RIs: lowest discount, fixed instance family, can sell on RI Marketplace, can modify AZ/scope/size. Convertible RIs: slightly lower discount, but can change instance family, OS, tenancy during the term. Payment options: All Upfront (highest discount), Partial Upfront, No Upfront. Scope: Regional (applies to any AZ in region) or Zonal (specific AZ, reservation capacity). RI Marketplace sells unused Standard RIs. Savings Plans are the newer, more flexible alternative that applies to compute usage across EC2, Lambda, Fargate.

## Q21: How does S3 Object Lambda work for transforming data at retrieval time?
**A:** S3 Object Lambda lets you add custom code to process data retrieved from S3 before returning it to the application. You create an S3 Object Lambda Access Point that points to a standard S3 Access Point and a Lambda function. When a GET request comes through the Access Point, S3 invokes your Lambda function with the original object. The function transforms the data (redact PII, resize images, convert formats, add watermarks) and returns the processed result. Benefits: no need to maintain multiple copies of data, transformation happens at access time, supports conditional transformations based on request context.

## Q22: How do CloudWatch Contributor Insights and CloudWatch Logs Insights differ?
**A:** CloudWatch Logs Insights: query language for analyzing log data ad-hoc. You write queries to filter, aggregate, and visualize log patterns. Good for debugging and operational investigation. CloudWatch Contributor Insights: continuously analyzes log data to identify top contributors (e.g., top error-generating endpoints, top talkers by IP). Automatically creates time-series rules to track contributors without writing queries. Use Contributor Insights for ongoing monitoring (e.g., "which API keys make the most 4xx errors") and Logs Insights for deep-dive troubleshooting.

## Q23: How would you design a cost-effective disaster recovery strategy (pilot light vs warm standby vs multi-region)?
**A:** Pilot Light: replicate data (RDS snapshots, S3 CRR), keep minimal services running (a single small EC2, a small RDS instance). On disaster: scale up full environment. RTO: minutes to hours, RPO: minutes. Cost: ~10-20% of production. Warm Standby: run a scaled-down copy of production (reduced instance sizes, fewer AZs). On disaster: scale up and shift traffic. RTO: minutes, RPO: seconds. Cost: ~30-40% of production. Multi-region Active-Active: full production in multiple regions. Traffic distributed via Route53. RTO: near-zero, RPO: seconds (synchronous replication). Cost: 2x+ production. Choose based on business RTO/RPO requirements.

## Q24: How does AWS CloudFormation StackSets work for multi-account deployments?
**A:** StackSets extends CloudFormation to deploy stacks across multiple accounts and regions from a single template. You define a template, specify target accounts (via OU or account list), and regions. StackSets creates stacks in each target account/region. You can deploy in order (serial) or parallel. Stack instances are managed centrally: updates propagate to all accounts. Self-service vs service-managed permissions: service-managed uses AWS Organizations to auto-deploy to new accounts. Use StackSets for account baselines (Config rules, CloudTrail, guardrails) and cross-account resources.

## Q25: How does Aurora Auto Scaling work and when would you use it?
**A:** Aurora Auto Scaling automatically adjusts the number of Aurora Replicas based on CPU utilization or connections. It adds replicas when load increases (scaling out) and removes idle replicas when load decreases (scaling in). Target tracking scaling policy (e.g., keep average CPU at 40%). Benefits: automatic read capacity scaling, improves read-heavy workload performance. Limits: doesn't scale writes (primary instance size is fixed), doesn't work with Aurora Serverless v1 (v2 supports it). Use for variable-traffic applications. For predictable traffic, manual replica management may be more cost-effective.

## Q26: How does ECS Service Auto Scaling differ from EC2 Auto Scaling?
**A:** ECS Service Auto Scaling: adjusts the desired count of tasks in an ECS service based on CloudWatch metrics (CPU, memory, ALB request count per target). Uses target tracking, step, or scheduled scaling. Operates at the service level. EC2 Auto Scaling: adjusts the number of EC2 instances in the cluster. ECS manages both: Scale service tasks based on load (application-level scaling), and scale cluster instances based on cluster resource reservation (infrastructure-level scaling). For Fargate, only service scaling is needed (no instances to manage). For EC2 launch type, you typically use both.

## Q27: How does DynamoDB Global Tables work and what are consistency tradeoffs?
**A:** DynamoDB Global Tables replicate data across multiple AWS regions automatically. Based on DynamoDB Streams: writes in any region are streamed and replayed in all other regions within seconds. Multi-master: all regions accept writes. Conflict resolution: last writer wins (based on timestamp). Tradeoffs: 1) Eventually consistent (cross-region replication lag of ~1 second). 2) No cross-region strong consistency. 3) Higher cost (replicated writes consume WCU in each region). 4) Conflict resolution is simple (LWW, may lose data). Best for: globally distributed applications that can tolerate eventual consistency. Use with DynamoDB Accelerator (DAX) for in-region read caching.

## Q28: How does Step Functions Express vs Standard workflow choice impact cost and durability?
**A:** Standard Workflows: exactly-once execution, up to 1 year duration, $0.025 per 1,000 state transitions. Best for long-running, auditable processes (order fulfillment, multi-step approval). Express Workflows: at-least-once execution, up to 5 minutes duration, $1.00 per 1 million executions. Best for high-volume, short-duration events (streaming data processing, request-response). Cost comparison: Express is dramatically cheaper for high-volume workloads. Durability: Standard guarantees exactly-once; Express may retry on failure. Choose Standard for business-critical processes, Express for event-driven high-throughput scenarios.

## Q29: How would you implement cross-account ECR access for container images?
**A:** Option 1 - Cross-account IAM: ECR repository policy grants `ecr:BatchGetImage`, `ecr:GetDownloadUrlForLayer` to the consuming account. Consuming account's task role needs `ecr:GetAuthorizationToken` + the above permissions. Option 2 - ECR pull-through cache (simpler): create a pull-through cache rule in the consuming account's ECR that caches images from the source account. Option 3 - Replication: configure ECR replication to push images to consumer accounts automatically. For production, prefer pull-through cache or replication over direct cross-account IAM. ECR repository policies follow the same resource-based policy model as S3.

## Q30: How does AWS Transit Gateway handle inter-region peering and bandwidth?
**A:** Transit Gateway inter-region peering connects TGWs in different regions via a peering attachment. Traffic stays on the AWS global network (no internet). Data transfer costs apply (standard inter-region rates, not internet rates). Bandwidth: up to 50 Gbps per peering attachment (burst). No transitive routing across peering: TGW-A can talk to TGW-B, but TGW-C connected to TGW-B cannot reach TGW-A unless direct peering exists. Supports static routes over peering attachments (no BGP). Use inter-region TGW peering for multi-region network connectivity without VPNs.

## Q31: How does AWS Compute Optimizer generate recommendations?
**A:** Compute Optimizer uses machine learning to analyze historical utilization metrics (CPU, memory, network I/O, EBS IOPS) over the past 14 days (with sufficient data). It generates recommendations for: EC2 instance types, Auto Scaling groups, EBS volumes, Lambda functions. Findings: Over-provisioned (recommend downsizing), Under-provisioned (recommend upsizing), Optimized (no change needed), Inactive (no data). Options: 1) Enhanced infrastructure metrics (CWA agent for memory/disk metrics). 2) Recommendation preferences (performance vs cost). 3) Savings estimates (On-Demand vs RI vs Spot pricing). Use to right-size resources quarterly.

## Q32: How does AWS Backup work for cross-region/account backups?
**A:** AWS Backup is a managed backup service with centralized policies. Backup plans define: frequency, retention, backup window, lifecycle rules. Vaults store backups with optional vault lock (WORM). Cross-region backup: copy backups to another region automatically. Cross-account backup: copy backups to another account using backup policies with AWS Organizations. Supports EC2, RDS, EFS, DynamoDB, EBS, Aurora, FSx, Storage Gateway. Features: point-in-time restoration, backup audits (compliance reports), cold storage tier (archive after X days). Use centralized backup policies via Organizations for enterprise compliance.

## Q33: How does RDS Performance Insights work and what can you diagnose?
**A:** RDS Performance Insights provides database performance analysis with a visual dashboard. It collects and displays: DB Load (average active sessions), waits (top SQL, top waits by wait event), and host/user metrics. Native mode: provides top SQL by load, wait events (CPU, IO, Lock, Latches). Enhanced mode (paid): retains performance data for 2 years. Use cases: 1) Identify inefficient SQL queries causing high load. 2) Detect lock contention. 3) Find throttled queries (IO bottlenecks). 4) Identify sudden load spikes. 5) Compare performance before/after schema changes. Performance Insights is critical for production database optimization.

## Q34: How would you design a serverless data pipeline with EventBridge Pipes?
**A:** EventBridge Pipes builds point-to-point integrations between event sources and targets with built-in filtering, enrichment, and transformation. Design: Source = DynamoDB Stream (or SQS, Kinesis); Filter = only events where `eventType = "ORDER_PLACED"`; Enrichment = Lambda function that looks up customer details; Target = Step Functions workflow for order fulfillment. Pipes handles source polling, filter application, enrichment execution, and target delivery. Benefits: no custom poller code, built-in error handling (DLQ), dead-letter destinations, built-in input transformers. Much simpler than building with EventBridge event buses for simple pipelines.

## Q35: How does AWS Shield Advanced DDoS mitigation work beyond basic Shield?
**A:** Shield Advanced provides: 1) Enhanced detection (layer 3/4 and layer 7 DDoS attacks). 2) Always-on flow-based monitoring with DDoS Response Team (DRT) access. 3) Proactive engagement (DRT helps during attacks). 4) Application Layer (L7) DDoS mitigation with WAF integration. 5) Cost protection (AWS credits for scaling charges during attacks). 6) Real-time attack visibility via Shield console and CloudWatch metrics. 7) Health-based detection (ALB/NLB health check degradation indicates attack). 8) 24/7 access to DRT. Costs $3,000/month for each consolidated account. Must have WAF for L7 protection.

## Q36: How does AWS Config Conformance Packs work for compliance?
**A:** Conformance Packs are collections of AWS Config Rules and remediation actions deployable as a single entity. They provide pre-packaged compliance frameworks (CIS, HIPAA, NIST, PCI DSS) with managed rules and parameters. Each conformance pack: contains rule definitions, remediation actions, and parameter values. You can create custom packs for organizational policies. Conformance packs evaluate resource compliance against the framework. Results are aggregated in the Config console showing compliant/non-compliant resources per framework. Use for continuous compliance monitoring across accounts.

## Q37: How do you implement least-privilege IAM roles for Lambda functions accessing DynamoDB?
**A:** 1) Create a Lambda execution role with a trust policy allowing `lambda.amazonaws.com` to assume the role. 2) Attach a policy that grants only the specific DynamoDB actions needed: `dynamodb:GetItem`, `dynamodb:Query`, `dynamodb:PutItem`, `dynamodb:UpdateItem` (no `DeleteItem`, no `Scan` unless needed). 3) Restrict to specific table ARN: `arn:aws:dynamodb:us-east-1:123456789012:table/orders`. 4) Use condition keys to restrict further: `dynamodb:LeadingKeys` to restrict partition key access. 5) Add condition `aws:SourceArn` to prevent confused deputy. 6) If Lambda is in a VPC, add `AWSLambdaVPCAccessExecutionRole` managed policy.

## Q38: How does S3 Intelligent-Tiering work and when is it cost-effective?
**A:** S3 Intelligent-Tiering automatically moves data between access tiers based on changing access patterns, without performance impact or operational overhead. Tiers: Frequent Access, Infrequent Access, Archive Instant Access, Archive Access (optional), Deep Archive Access (optional). It monitors access patterns and moves objects that haven't been accessed for 30 days to IA, then to Archive tiers. Monitoring charge: $0.0025 per 1,000 objects. Cost-effective when: 1) Access patterns are unknown or unpredictable. 2) Data has mixed access frequencies. 3) You want automatic cost optimization without lifecycle rules. Not cost-effective for: always-accessed data (Standard is cheaper) or data with very predictable patterns.

## Q39: How does DynamoDB Accelerator (DAX) improve read performance?
**A:** DAX is an in-memory cache for DynamoDB that delivers up to 10x read performance (microsecond latency). It sits between your application and DynamoDB. Write-through cache: writes go to DAX and DynamoDB simultaneously. Read-through cache: cache hits served from DAX, misses go to DynamoDB and populate cache. DAX handles cache invalidation automatically. Use DAX for: 1) Read-heavy workloads (social feeds, gaming leaderboards). 2) Applications requiring microsecond response times. 3) Reducing provisioned read capacity costs. DAX does not support: transactions, global tables, or item TTL beyond 5 minutes. DAX cluster nodes are single-AZ by default (multi-AZ supported).

## Q40: How would you implement cross-region RDS disaster recovery?
**A:** Options: 1) Cross-region read replicas: promote to standalone if primary fails. RPO: seconds (async replication), RTO: minutes (promotion + DNS change). 2) Cross-region automated backups: copy RDS snapshots to another region. RPO: hours (snapshot schedule), RTO: ~15 minutes (restore from snapshot). 3) Aurora Global Database: one primary region (read/write) + up to 5 secondary regions (read-only, <1s replication lag). RPO: <1s, RTO: <1 minute (promote secondary). For production: Aurora Global Database for lowest RPO/RTO; cross-region replicas for cost-sensitive; snapshot copying for compliance retention.

## Q41: How does Lambda@Edge work and what are its use cases?
**A:** Lambda@Edge runs Lambda functions at CloudFront edge locations, triggered by CloudFront events: 1) Viewer Request (before cache check). 2) Origin Request (after cache miss). 3) Origin Response (after origin returns). 4) Viewer Response (before returning to viewer). Use cases: A/B testing (serve different content variants), URL rewrites, authentication (JWT validation at edge), custom headers, bot detection, content personalization, response compression. Limitations: 5-second execution (viewer events) or 30 seconds (origin events), 128-3008 MB memory, no VPC access, no environment variables, 1MB response size. Code is replicated to all edge locations automatically.

## Q42: How would you design a VPC with a centralized egress pattern using AWS Network Firewall?
**A:** Create a dedicated Inspection VPC (or use Transit Gateway + Network Firewall). Architecture: 1) All spoke VPCs route internet-bound traffic (0.0.0.0/0) to Transit Gateway. 2) Transit Gateway routes to Inspection VPC's Network Firewall endpoint. 3) Network Firewall applies stateful rules (domain filtering, protocol detection, threat signature matching). 4) After inspection, traffic routes to NAT Gateway in Inspection VPC, then to IGW. 5) Return traffic reverses the path. Benefits: centralized egress inspection, consistent policy across VPCs, threat prevention (Suricata-based), domain filtering. Limitations: adds latency (extra hop), higher cost (per-GB inspection), single-point-of-entry complexity.

## Q43: How does EBS Multi-Attach work and when is it appropriate?
**A:** EBS Multi-Attach allows a single `io1`/`io2` EBS volume to be attached to up to 16 Nitro-based EC2 instances in the same AZ. All instances have read/write access. Requires a cluster-aware file system (like OCFS2, GPFS, or similar) that handles concurrent writes. Use cases: 1) Clustered databases (Oracle RAC, SAP HANA). 2) Shared storage for container orchestration. 3) High-availability clusters requiring shared block storage. NOT for: regular file sharing (use EFS), database replication (use database-native replication), multi-AZ (same AZ only). Each instance's I/O is limited to the volume's provisioned IOPS. Use with caution: concurrent writes without cluster FS can corrupt data.

## Q44: How does AWS Cost Anomaly Detection work?
**A:** Cost Anomaly Detection uses machine learning to continuously monitor cost and usage patterns and detect anomalous spends. It creates a baseline of normal spending per account/service/member. When it detects a meaningful deviation (anomaly), it creates an anomaly record with: 1) Impact magnitude. 2) Time period. 3) Root cause analysis (which service/region/resource caused the anomaly). 4) Anomaly score. It learns seasonal patterns (weekends, holidays). Alerts via SNS/email. Best practice: set up root cause monitoring with anomaly alerts per account and service. Review anomalies weekly to catch cost spikes from: misconfigurations, unexpected traffic, orphaned resources.

## Q45: How does S3 Batch Operations work for large-scale object management?
**A:** S3 Batch Operations processes billions of objects at scale with a single API request. You create a job specifying: 1) Object list (via S3 Inventory or CSV). 2) Operation (copy, invoke Lambda, restore from Glacier, set tags/ACL, delete, put object lock). 3) Completion report (success/failure per object). 4) Priority. S3 manages retries, progress tracking, and completion notifications. Example: Invoke Lambda function to transform every object in a 10PB dataset. Progress tracked in S3 console and via EventBridge. Use for: mass tagging, encryption updates, data format migration, GDPR deletion requests. Costs: per-object charge + Lambda/compute costs.

## Q46: How would you implement a blue-green deployment for Lambda functions?
**A:** Lambda supports traffic shifting via aliases and weighted routing. 1) Create an alias (e.g., `prod`). 2) Publish version 1 as the initial version for `prod`. 3) Publish version 2 (new code). 4) Update alias to route 90% to v1, 10% to v2. 5) Monitor errors/latency on v2. 6) Gradually shift 25%, 50%, 75%, 100%. 7) If errors: rollback by shifting 100% back to v1. AWS CodeDeploy automates this with Lambda deployment configurations: Canary10Percent5Minutes (10% for 5min, then 100%), Linear10PercentEvery1Minute, AllAtOnce. Combined with CloudWatch alarms, CodeDeploy auto-rolls back on alarm triggers.

## Q47: How does DynamoDB Streams integration with Lambda work for real-time processing?

## Q49: How does Service Control Policy (SCP) evaluation work with multiple OUs?
**A:** SCPs follow a deny-by-default model. If ANY SCP in the hierarchy denies an action, it's denied. Effective permissions = Intersection of ALL SCPs applied at root, OU, and account levels. Example: Root SCP allows EC2 and S3. OU-A SCP allows S3 only. Accounts in OU-A have S3 only (EC2 denied by OU-A SCP). Accounts in the root (outside OU-A) have EC2 and S3. SCPs do NOT grant permissions (no Allow in SCP is meaningful alone; they set boundaries). SCPs can be applied at root level (full org), OU level, or direct to account. Use SCP inheritance to layer restrictions: root-level baseline SCPs + OU-specific SCPs.

Choose based on time-sensitivity of your monitoring.

## Q57: How do DynamoDB Adaptive Capacity and Auto Scaling work together?
**A:** Adaptive Capacity automatically manages partition throughput: if one partition gets more traffic (hot partition), DynamoDB automatically allocates more WCU/RCU to that partition without increasing provisioned capacity. Auto Scaling adjusts the table's provisioned capacity based on usage. Together: Auto Scaling sets the base capacity; Adaptive Capacity handles temporary traffic imbalances across partitions. Adaptive Capacity doesn't help if total traffic exceeds provisioned capacity. Auto Scaling adjusts total capacity over minutes. For unpredictable workloads, use On-Demand capacity (auto-scales instantly but costs more). Adaptive Capacity is critical for uniform partition key design.

## Q62: How does DynamoDB Time-to-Live (TTL) work for cost optimization?
**A:** DynamoDB TTL automatically deletes expired items after a timestamp attribute passes the current time. It's free, doesn't consume WCU, and runs in the background. Expired items are deleted within 48 hours of expiration. Best for: session management, event data, temporary records, compliance data retention. Benefits: 1) No scan/delete code needed. 2) Automatic storage cost reduction. 3) Stream events for expired items (you can process deletions via DynamoDB Streams). Limitations: 1) Deletion timing is approximate (up to 48 hours). 2) Deleted items still appear in queries/scans until physically deleted. 3) TTL deletion doesn't reduce provisioned capacity cost, only storage cost.

## Q67: How does DynamoDB Transactions affect throughput and cost?
**A:** Transactions consume double capacity: each transactional read consumes 2 RCU instead of 1 (or 4 for strongly consistent). Each transactional write consumes 2 WCU instead of 1. This means: 1) Provisioned throughput needs 2x for transactions. 2) On-Demand costs 2x per request. 3) If mixing transactional and non-transactional operations, you still pay transactional pricing for operations within a transaction. Example: Transactional `GetItem` of 4KB = 2 RCU (vs 1 RCU for regular). Always consider if ACID is truly needed; many workloads can use conditional writes and optimistic locking instead.

Use for: standardizing infrastructure, cost governance, ensuring security compliance.

## Q92: How does RDS Performance Insights metric correlation help in diagnosing issues?
**A:** Performance Insights metric correlation identifies which database metrics are correlated with high DB load. It shows: 1) Wait events correlated with high load (IO:DataFileRead, CPU, Lock:RowLockWait). 2) SQL queries causing the load. 3) Hosts and users contributing. 4) Time-series correlation (when load spikes, which metrics spike simultaneously). Correlation analysis helps identify: 1) Whether the issue is IO-bound (correlated with ReadIOPS). 2) CPU-bound (correlated with CPU utilization). 3) Lock contention (correlated with active sessions). 4) Memory pressure (correlated with swap usage). This makes root cause diagnosis much faster.

## Q93: How does ECS Capacity Providers manage infrastructure scaling?
**A:** ECS Capacity Providers manage the EC2 instances (or Fargate) that back ECS tasks. They work with Auto Scaling groups to scale infrastructure based on task demand. Key concepts: 1) Capacity provider: manages the compute resource (EC2 ASG or Fargate). 2) Managed scaling: ECS automatically scales the ASG based on cluster resource reservation. 3) Managed termination protection: ECS protects instances with running tasks. 4) Capacity provider strategy: spread tasks across providers (e.g., 60% Fargate, 40% EC2 Spot). Benefits: automatic infrastructure scaling based on task demand, spot/fargate mix for cost optimization, simplified infrastructure management.

## Q94: How does DynamoDB Time-series pattern with partition key design avoid hot partitions?
**A:** For time-series data (IoT sensor data, logs), naively using timestamp as partition key creates hot partitions (all writes go to the latest timestamp partition). Solutions: 1) Sharding: combine sensor ID + timestamp: `sensor_123#2024-01-15`. 2) Write sharding: add a random/distributing suffix: `sensor_123#2024-01-15#shard1`. 3) Hierarchical partitioning: partition by `sensor_type`, sort by timestamp. 4) Adaptive capacity handles some imbalance but better design prevents it. For high-write-rate time series: use write sharding + DynamoDB Streams to aggregate. Avoid single partition key for time-series writes.

## Q95: How does AWS Cloud Map work for service discovery?

## Q96: How would you implement a cost-effective S3 lifecycle strategy for log data?
**A:** For log storage (CloudTrail, ALB, application logs): 1) S3 Standard for the first 30 days (frequent access). 2) Transition to S3 Standard-IA for 31-90 days (less frequent but may need quick access). 3) Transition to S3 Glacier Deep Archive for 91-365+ days (compliance retention). 4) Expire (delete) after 7 years (or as needed). Use S3 Lifecycle policy with prefix filters (e.g., `logs/cloudtrail/*`, `logs/app/*`). Use S3 Object Expiration to set hard delete date. Optional: use S3 Intelligent-Tiering if access patterns are unpredictable. Estimated cost for 1TB logs/year: Standard ~$23/month for 30 days, then Glacier Deep Archive ~$0.001/GB/month.

## Q97: How does Lambda SnapStart handle its initialization restrictions?
**A:** SnapStart only works with Java 11+ runtime. Restrictions: 1) Cannot establish persistent network connections during init (they break when snapshot is resumed in a different execution environment). 2) Cannot generate unique runtime identifiers (UUIDs, timestamps) during init. 3) Cannot load native libraries that don't support serialization. 4) Cannot leave files open (must close during init). Best practices: 1) Move connection creation to handler (lazy initialization). 2) Use SnapStart-specific hooks (`RuntimeHook`). 3) Implement `CrashHandler` to refresh connections on resume. 4) Test thoroughly: connections established during init are NOT preserved across snapshots.

## Q98: How does CloudFront Origin Shield work and when is it useful?
**A:** Origin Shield is an additional caching layer (regional cache) between CloudFront edge locations and your origin. Benefits: 1) Higher cache hit ratio: multiple edge locations share one Origin Shield cache, reducing origin requests. 2) Lower origin load: aggregated requests reduce load spikes on origin. 3) Reduced latency: SSL/TLS connection reuse. 4) Lower costs: fewer origin requests, less data transfer. Origin Shield sits in an AWS region (choose strategically). Costs: per-request charge for Origin Shield. Best for: 1) Origins with high request variability (many edge locations). 2) Origins that are expensive per-request (Lambda, DynamoDB). 3) Reducing origin infrastructure costs.

## Q99: How would you design a data encryption strategy across S3, EBS, and RDS?
**A:** Use KMS for centralized key management. S3: SSE-S3 (AES-256 default), SSE-KMS (with key rotation), or SSE-C (customer keys). Best practice: SSE-KMS with a CMK per application (enables key policies, audit, rotation). EBS: EBS encryption via KMS CMK, enforced via EC2 launch template or account-level setting. RDS: enable encryption at rest (enables during creation), KMS CMK per database. Cross-service: use same KMS key for S3/RDS/EBS to simplify permissions. Key rotation: enable automatic yearly rotation for all CMKs. Key policies: restrict KMS key usage to specific IAM roles/accounts. Bucket policies: enforce `aws:SecureTransport` and deny `s3:*` without encryption header.

## Q100: Walk through the complete AWS Organizations hierarchy for a typical enterprise.
**A:** Root (Management Account): consolidated billing, full admin access. OU Structure:
1) Security OU: Security account (GuardDuty, Security Hub, Config aggregator), Log Archive account (CloudTrail, Config snapshots, VPC Flow Logs), Audit account (audit tools).
2) Infrastructure OU: Shared Services account (Transit Gateway, DNS, AD), Network account (VPC, Firewalls), DevOps account (CI/CD tools, ECR, Code artifacts).
3) Workload OUs: Dev/Test OU (lower controls, higher dev flexibility), Pre-Prod OU (closer to prod controls), Production OU (strict SCPs, mandatory encryption, MFA, backup policies).
SCPs at root: baseline restrictions (deny non-approved regions, enforce instance types). Per-OU SCPs: Production OU denies public S3, denies IAM changes outside CloudFormation. Each account has service control + resource control + IAM for granularity. Use Control Tower to automate this landing zone.
