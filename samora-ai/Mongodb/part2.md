# MongoDB Interview Questions and Answers - Part 2

## Q1: How does the MongoDB query optimizer select between multiple competing indexes, and how can you influence its decision?
**A:** MongoDB's query optimizer evaluates candidate indexes by running a trial period where it executes the query plan with different indexes in parallel (typically 100-1000 trial rounds). It caches the winning plan for the same query shape. To influence the optimizer, use `$hint()` to force an index, or restructure queries to make better index selection obvious. The `planCacheClear()` command clears cached plans to force re-evaluation. You can examine optimizer decisions with `explain("allPlansExecution")`.

## Q2: What is the impact of a large number of collections in MongoDB on performance and resource usage?
**A:** Each collection consumes memory for metadata, indexes, and namespace entries. With thousands of collections, MongoDB must manage more namespace locks, file descriptors, and database files. WiredTiger allocates a cache page for each collection's index root. The `nsSizeMB` parameter (default 16MB, supporting ~24K namespaces) may need tuning. For time-series or multi-tenant designs, evaluate if sharded collections or a single collection with a tenant discriminator field would be more efficient than thousands of separate collections.

## Q3: How does the WiredTiger storage engine handle concurrent write operations at the document level, and what are the locking implications?
**A:** WiredTiger uses Multi-Version Concurrency Control (MVCC) with document-level optimistic concurrency control. Writers take intent locks at the database and collection levels but only conflict at the document level. Write operations are atomic at the document level — a single write to a document either succeeds completely or fails. For multi-document transactions, WiredTiger uses a read timestamp and write timestamp with conflict detection. Concurrent modifications to the same document cause a write conflict, and one operation retries.

## Q4: What happens to read preferences like `nearest` during a network partition in a replica set, and how can it cause stale reads?
**A:** With `nearest` read preference, drivers route reads to the lowest-latency member. During a network partition, a secondary that has fallen behind (or a primary that has stepped down but the driver hasn't detected) may have the lowest latency. This can result in reading stale data. Use `maxStalenessSeconds` (e.g., `maxStalenessSeconds=90`) to prevent routing reads to secondaries that are more than N seconds behind the primary. Pair with appropriate read concern (`majority`) for consistency guarantees.

## Q5: How do you design a shard key to avoid jumbo chunks, and what are the remediation strategies when they occur?
**A:** Jumbo chunks are chunks that exceed the max chunk size (default 64MB) and cannot be split because all documents share the same shard key value (monotonically increasing or low-cardinality key). Prevention: use hashed shard keys, compound shard keys with high-cardinality suffix, or range-based keys with even distribution. Remediation: temporarily increase chunk size with `splitVector` to allow splits, then reduce back; or manually split using `splitAt()`; or re-shard with a different key (requires dump/restore or a new collection).

## Q6: Explain the `$lookup` pipeline optimization — when does MongoDB push `$match` stages before `$lookup`, and when does it not?
**A:** MongoDB optimizes `$lookup` by pushing `$match` conditions before the `$lookup` when the `$match` references only fields from the source collection. When `$match` references fields from the joined collection, it cannot be pushed. Additionally, `$lookup` with `pipeline` syntax allows indexed joins: if the foreign collection has an index on the join field, joins are efficient. Foreign pipelines with `$match` + `$limit` can leverage indexes on the foreign collection. Without a pipeline, `$lookup` performs a full collection scan on the foreign collection by default.

## Q7: What is the difference between `writeConcern.majority` journaling behavior in WiredTiger vs In-Memory storage engine?
**A:** In WiredTiger, `writeConcern: { w: "majority", j: true }` waits for the journal (write-ahead log) to commit on a majority of replica set members. With the In-Memory storage engine, there is no journal — `j: true` is ignored and writes are acknowledged immediately. This means the In-Memory engine cannot provide the same durability guarantees. For testing or caching layers, this is acceptable; for production data, never use the In-Memory engine with `j: true` expecting crash recovery.

## Q8: How does MongoDB handle index intersection, and what are its limitations compared to compound indexes?
**A:** Index intersection uses multiple indexes to satisfy a query by intersecting document IDs from each index scan. MongoDB's query optimizer may use index intersection when no single index covers the query, but it requires scanning both indexes and merging results. Limitations: (1) cannot use `$sort` from intersected indexes, (2) only works with `$and` queries, (3) covered queries are not possible, (4) performance is typically worse than a well-designed compound index. Compound indexes are preferred; index intersection is a fallback.

## Q9: What are the implications of using `mongos` as a connection pool bottleneck in sharded clusters, and how do you scale it?
**A:** Each `mongos` instance is stateless but consumes memory for route caches, connection pools to shards, and configuration servers. With many client connections, a single `mongos` can become CPU-bound (parsing BSON, routing). Deploy multiple `mongos` instances behind a load balancer. Each `mongos` independently caches routing metadata. The `--port` option allows running multiple `mongos` on the same host. Connection pooling per `mongos` should be tuned: `maxPoolSize` in drivers should account for the number of `mongos` instances to avoid connection storms.

## Q10: How does the WiredTiger eviction algorithm work, and what happens when the cache is under pressure?
**A:** WiredTiger's eviction algorithm uses a combination of LRU and a "hazard pointer" mechanism to manage pages in cache. It maintains a "page table" with information about page usage and recency. Under cache pressure (cache usage exceeds `evictionTarget`), WiredTiger aggressively evicts clean pages first, then dirty pages (which must be written to disk first). If eviction cannot keep up, application threads are forced to do eviction work before proceeding, causing latency spikes. Monitor `wiredTiger.cache.tracked_dirty_bytes` and `wiredTiger.cache.pages_evicted` to detect cache pressure.

## Q11: What is the `$out` stage's behavior with sharded collections, and what are the constraints?
**A:** `$out` replaces an entire collection with the aggregation pipeline's results. When outputting to a sharded collection, the pipeline results are written to the shards, but the collection must have the same shard key. If the output collection is sharded with a different shard key, the pipeline fails. `$out` is not supported within multi-document transactions. For large aggregations, `$merge` is often preferred because it allows incremental updates and can respect existing sharding.

## Q12: How do you safely perform a primary node failover test in a replica set without causing data loss?
**A:** To safely test failover: (1) Force a stepDown: `rs.stepDown(60)` on the primary, (2) Monitor the election: check `rs.status()` to confirm a new primary is elected, (3) Verify writes resume on the new primary, (4) Reconnect the old primary — it rejoins as a secondary and catches up via oplog replay. Ensure `writeConcern: majority` is set to prevent rollback of acknowledged writes. After the test, verify rollback data (if any) using `mongodb-brain` or manual oplog inspection on the rejoining member.

## Q13: What causes rollbacks in MongoDB replica sets, and how can you minimize their occurrence and impact?
**A:** Rollbacks occur when a former primary had writes that were not replicated to a majority of the replica set before it stepped down. The new primary may have conflicting oplog entries. Minimization: (1) Use `writeConcern: majority` for all writes, (2) Set `settings.chainingAllowed: false` to force direct replication from primary, (3) Monitor replication lag and alert on high lag, (4) Configure `votes` on low-priority members to avoid undesirable elections, (5) Set `catchUpTimeoutMillis` to give primaries time to catch up before stepping down. Rollback data is written to rollback files — use `mongorestore` with caution.

## Q14: How does the `Change Streams` feature handle sharded cluster topology changes like chunk migrations?
**A:** Change streams in sharded clusters use global logical timestamps and the `resumeToken` to maintain cursor position across shards. When a chunk migration moves data from one shard to another, the change stream continues to watch on both shards and deduplicates events using the document's `_id`. The `resumeToken` is based on the cluster time, which is consistent across shards. If a shard is unavailable, the change stream may encounter an error — use `startAfter` instead of `resumeAfter` to skip lost events.

## Q15: What is the `defaultReadConcern` and `defaultWriteConcern` in MongoDB 5.0+, and how do they affect transaction defaults?
**A:** Since MongoDB 5.0, you can set default read and write concerns at the database or collection level: `db.setDefaultReadConcern("majority")`. These defaults are inherited by transactions started on that database. If no default is set, transactions inherit the session's read/write concern. This simplifies application code by centralizing consistency requirements. Changing defaults affects only new operations; existing operations continue with their specified concerns.

## Q16: How do you monitor and tune the WiredTiger read-cache hit ratio for read-heavy workloads?
**A:** Monitor `serverStatus().wiredTiger.cache` metrics: `bytes currently in cache`, `maximum bytes configured`, `pages read into cache`, and `pages requested from cache`. The cache hit ratio is `(pages requested - pages read) / pages requested`. A ratio below 95% indicates the cache is too small. Tuning: increase `storage.wiredTiger.engineConfig.cacheSizeGB` (up to 90% of RAM in dedicated deployments), or for read-heavy workloads, consider using the `storageEngine.wiredTiger.cacheSize` parameter. Also check `read into cache` vs `read from cache` in the `block-manager` section.

## Q17: What are the trade-offs between `hashed` and `ranged` sharding for time-series data with monotonically increasing shard keys?
**A:** Ranged sharding on time-based keys creates "hot spots" — all writes go to a single shard at any given time, and chunks migrate to other shards later. This causes uneven load. Hashed sharding distributes writes evenly across shards but makes range queries on time (e.g., "last 24 hours") hit all shards. Solution: use compound hashed shard keys — prefix with a high-cardinality field (e.g., `deviceId`) and suffix with time. Or use ranged sharding with zone sharding to pin time ranges to specific shards. For insert-heavy workloads, hashed sharding is preferred.

## Q18: How does the `$facet` stage handle memory limits, and what happens if a single facet exceeds 100MB?
**A:** Each `$facet` sub-pipeline is independently subject to the 100MB memory limit per stage. If a sub-pipeline exceeds the limit without `allowDiskUse: true`, the aggregation fails with a `QueryExceededMemoryLimitError`. With `allowDiskUse: true`, temporary files are written to the `dbPath` directory. Unlike regular aggregation stages, `$facet` sub-pipelines write intermediate state separately. For large `$facet` queries, enable `allowDiskUse` and monitor disk usage. Consider splitting facets into separate aggregation queries for very large datasets.

## Q19: How do you implement a "soft delete" pattern in MongoDB while preserving query performance?
**A:** Common pattern: add a `deletedAt` field (null by default, set to timestamp on deletion). Create a partial filter index: `db.collection.createIndex({ userId: 1, status: 1 }, { partialFilterExpression: { deletedAt: null } })`. Query with `{ deletedAt: null }` to exclude soft-deleted documents. For hard deletion after a retention period, use TTL index on `deletedAt` field to automatically remove expired soft-deleted documents. This avoids a collection scan for filtering out deleted records.

## Q20: What happens when a `mongod` instance runs out of disk space, and how does it recover?
**A:** When the disk is full, MongoDB enters a "read-only" mode — writes fail with `DiskSpaceExhausted` errors. Reads continue unless compaction or index creation is needed. Recovery: (1) Free space by removing unnecessary data or extending disk, (2) The server automatically resumes writes once space is available, (3) Run `db.repairDatabase()` if files are corrupted (but this requires significant free space), (4) Restart if the instance was force-killed while full. Preventive: configure `storage.wiredTiger.engineConfig.directoryForIndexes` to split indexes across disks, and set monitoring alerts at 80% disk usage.

## Q21: How does the aggregation pipeline's `$lookup` with correlated subqueries (`let` + `pipeline`) differ in performance from the original `$lookup` syntax?
**A:** The original `$lookup` syntax performs an equality join between a local field and a foreign field. The pipeline syntax with `let` allows correlated subqueries with `$match` stages. Performance differs: pipeline syntax can use indexes on the foreign collection (if the `$match` uses indexed fields), while the basic syntax always uses the foreign field index. However, pipeline syntax has more overhead per document because it creates a sub-query context. For simple equality lookups, the original syntax is faster. For filtered lookups, pipeline syntax is necessary and can be faster with proper indexing.

## Q22: What is a "sparse index" in MongoDB, and when would you use it over a partial index?
**A:** A sparse index only indexes documents that contain the indexed field(s). If a document lacks the indexed field, it is omitted from the index. Partial indexes (since 3.2) are more flexible — they index documents matching a filter expression. Sparse indexes are now considered a legacy feature; partial indexes are preferred because they support a wider range of queries (including `$exists: false` which sparse indexes cannot). Use sparse indexes only when retrofitting older deployments where partial indexes cannot be applied.

## Q23: How do you handle MongoDB transaction retry logic for transient errors like `WriteConflict` or `LockTimeout`?
**A:** MongoDB transactions can encounter transient errors that are safe to retry. The recommended pattern is a retry loop: (1) Start the transaction with `session.startTransaction()`, (2) Execute operations inside `try-catch`, (3) On `TransientTransactionError` (includes `WriteConflict`), abort and retry up to N times with exponential backoff, (4) On `UnknownTransactionCommitResult`, retry the commit (idempotent), (5) On `DuplicateKeyError` or validation errors, abort and don't retry. The `MongoTransactionError` class in drivers helps categorize errors. Set `maxTransactionLockRequestTimeoutMillis` (default 5ms) to control lock wait time.

## Q24: How does MongoDB's `Cache Pressure` manifest, and what metrics indicate it in production?
**A:** Cache pressure occurs when WiredTiger's internal cache is too small for the working set. Symptoms: increasing page evictions, high disk read rates, latency spikes, and query slowdowns. Key metrics in `serverStatus`: `wiredTiger.cache.pages_evicted` (high rate), `wiredTiger.cache.pages_read_into_cache` (high rate of reading from disk), `wiredTiger.concurrentTransactions.readThroughput` (dropping), `wiredTiger.cache.pages_queued_for_eviction` (non-zero). Also watch `globalLock.activeClients.readers` queuing behind cache access. Resolution: increase cache size, add indexes, or reduce working set.

## Q25: What are the risks of using `$lookup` inside a multi-document transaction with a sharded cluster?
**A:** In a sharded cluster, `$lookup` within a transaction only supports equality joins where the local collection is sharded and the foreign collection is either unsharded or shares the same shard key. Cross-shard `$lookup` in transactions is not supported and throws an error. Additionally, transactions acquire locks on all involved shards, and a `$lookup` accessing documents on another shard can cause distributed deadlocks. For cross-shard lookups, prefer doing the join at the application level or using aggregation pipelines outside of transactions.

## Q26: How does the `stable API` (API versioning) in MongoDB 5.0+ help with driver compatibility, and what are its limitations?
**A:** The Stable API (`apiVersion: "1"`) in MongoDB 5.0+ allows drivers to specify which API version to use. MongoDB guarantees backward compatibility within the same API version, preventing breaking changes from affecting applications. Commands not in the stable API are rejected. Limitations: (1) Not all commands are covered by the stable API (e.g., some admin commands), (2) It restricts the use of newer features not yet in the stable API, (3) It requires MongoDB 5.0+ on all nodes. It is most useful for enterprise applications that prioritize stability over new features.

## Q27: What is the `oplog` size's effect on replication, and how do you choose the right oplog size?
**A:** The oplog is a capped collection; default size is 5% of free disk (minimum 990MB, maximum 50GB). A small oplog causes secondary nodes to fall out of sync (become "too stale to catch up") if they lag behind the amount the oplog can hold. Choose size based on: (1) Peak write volume — the oplog should hold at least as many operations as produced during the longest planned maintenance window, (2) Worst-case replication lag allowance, (3) `mongodump` — if you dump from a secondary, the oplog must span the dump duration. For write-heavy workloads, set oplog size to at least 24 hours of writes. Monitor with `rs.printReplicationInfo()`.

## Q28: How do you migrate documents with different schemas during a gradual schema migration without downtime?
**A:** Gradual schema migration strategy: (1) Write new documents with the new schema, (2) For old documents, apply changes lazily — on read, check the schema version (stored as a field like `schemaVersion: 2`), transform to new schema, and write back, (3) Run a background batch migration script to update old documents in batches (using `bulkWrite` with `ordered: false`), (4) Apply a TTL or manual cleanup to remove old schema support code. This avoids locking the entire collection and allows zero-downtime migration.

## Q29: How does MongoDB handle index builds on a running primary, and what is the impact on read/write operations?
**A:** Index builds on a primary (since MongoDB 4.2) are "background" by default and do not block writes. However, they still consume significant resources: CPU for index sorting, disk I/O for writing index entries, and RAM for the index build buffer. On the primary, the index build replicates as an oplog entry, and secondaries rebuild the index locally (not via oplog replay). For large collections, index builds cause replication lag as secondaries cannot process other oplog entries while building the index. Use `createIndexes` with `commitQuorum` to control how many replica set members must finish before the build commits. In sharded clusters, index builds by default run in "rolling" mode across shards.

## Q30: What is the MongoDB `Catalog Cache`, and how does it affect query routing in sharded clusters?
**A:** Each `mongos` caches collection metadata (shard key distribution, chunk ranges) from the config server in the Catalog Cache. When chunk migrations happen, the cache becomes stale. `mongos` detects stale cache on query execution: the targeted shard returns a `ShardNotFound` or `StaleConfig` error, triggering a cache refresh. Frequent chunk migrations cause repeated cache invalidations, increasing query latency. Mitigation: (1) Use `mongos` with `--configdb` pointing to config server replicas, (2) Increase `cursorTimeoutMillis` to avoid frequent re-fetching, (3) Monitor `connPoolStats` for stale config errors.

## Q31: How do you implement a MongoDB replica set with arbiters correctly, and what are the failure scenarios?
**A:** An arbiter participates in elections but does not hold data. Use arbiters in odd-numbered replica sets where adding a data-bearing member is cost-prohibitive (e.g., 2 data members + 1 arbiter). Never run an arbiter on the same machine as a data member (no redundancy). Failure scenarios: (1) If the arbiter is lost and one data member fails, the remaining member cannot elect itself primary (no majority), (2) In a PSSA (Primary-Secondary-Secondary-Arbiter) setup, if the arbiter fails, the set still works with 3 data nodes, (3) Arbiters have no acknowledgment of write concern — `w:majority` with an arbiter requires only 2 out of 3 members. In MongoDB 7.0+, consider using a `voteLess` replica set member instead of an arbiter.

## Q32: What is the `$planCacheStats` aggregation stage, and how can you use it for query tuning?
**A:** `$planCacheStats` returns cached query plans for a collection. It shows for each query shape: (1) `cachedPlan` — the winning plan, (2) `creationExecStats` — stats when the plan was cached, (3) `candidatePlanScores` — scores of competing plans. Use it to: identify queries that have suboptimal cached plans, check if the optimizer chose the right index, and clear specific cached plans with `planCacheClear()` to trigger re-evaluation. This is a powerful debugging tool when queries suddenly become slow after a period of good performance.

## Q33: How does the `$search` aggregation stage (Atlas Search) integrate with the aggregation pipeline, and what are its performance implications?
**A:** `$search` (Atlas Search) runs on the Lucene-backed search index before the aggregation pipeline. It must be the first stage in the pipeline. Performance tips: (1) Use `$search` with `returnStoredSource` to avoid fetching full documents, (2) Combine with `$compound` queries for complex searches, (3) Use `index` parameter to specify the search index, (4) After `$search`, use `$limit` early to reduce pipeline input, (5) `$search` does not use MongoDB indexes — only Lucene indexes. It processes data on the same `mongod` node, consuming CPU and memory. For large result sets, use `$searchMeta` to get metadata without document retrieval.

## Q34: What is the `encryption` at rest architecture in MongoDB Enterprise, and how does key management work?
**A:** MongoDB Enterprise supports encryption at rest using WiredTiger's native encryption. It encrypts all database files (including oplog, journal, and indexes) using AES-256-CBC (or AES-256-GCM in 4.2+). Key management options: (1) Local Key Management — store the master key in a keyfile on each node (least secure), (2) KMIP (Key Management Interoperability Protocol) — integrate with enterprise KMS like Vault, Thales, or IBM Security Guardium, (3) MongoDB Atlas — uses Cloud KMS (AWS KMS, Azure Key Vault, GCP Cloud KMS). Key rotation can be performed online in MongoDB 4.2+ using `rotateMasterKey`. Performance impact: ~5-15% overhead on read/write operations.

## Q35: How does MongoDB handle `cursor exhaustion` and batch retrieval in the driver?
**A:** MongoDB cursors fetch results in batches (default 101 documents or 16MB, whichever is smaller). The driver's `cursor.hasNext()` triggers a `getMore` command when the local batch is exhausted. If the cursor is idle for 10 minutes (default `cursorTimeoutMillis`), MongoDB kills it — subsequent `getMore` calls fail with `CursorNotFound`. For large result iteration: (1) Set `noCursorTimeout(true)` on the cursor and manage cleanup manually, (2) Use aggregation with `$limit` to bound results, (3) Use tailable cursors for capped collections, (4) Implement pagination instead of large cursors.

## Q36: How do you configure MongoDB for PCI-DSS or HIPAA compliance, specifically around auditing and encryption?
**A:** For compliance: (1) Enable audit log: `--auditDestination syslog` with appropriate audit filters for DDL, DML, and authentication events, (2) Enable encryption at rest with KMIP integration and periodic key rotation, (3) Enable TLS 1.2+ for all connections (disable older versions), (4) Configure RBAC with separation of duties (separate DBAs from security admins), (5) Set `security.journal.enabled: true` and `writeConcern: majority`, (6) Enable `net.tls.allowConnectionsWithoutCertificates: false`, (7) Use SCRAM-SHA-256 or x.509 authentication, (8) Disable `$where` JavaScript execution in production (`security.javascriptEnabled: false`).

## Q37: What is the `mongotop` command's `rowcount` output, and how do you interpret it for identifying hot collections?
**A:** `mongotop` reports read/write activity per collection in milliseconds. The output shows total time spent per operation type per namespace. `mongotop 5` refreshes every 5 seconds. To identify hot collections: look for high `total` times consistently. Interpreting: (1) High reads on `system.profile` — profiling may be too verbose, (2) High writes on oplog — expected for primaries but check for replication churn, (3) High latency on collections with large documents — check for unfiltered scans, (4) Use `--locks` flag to see lock contention. Compare across primaries and secondaries to identify uneven load distribution.

## Q38: How does the `aggregation pipeline` handle `$group` with high-cardinality grouping keys, and what are the memory implications?
**A:** `$group` accumulates results in memory. For high-cardinality keys (millions of unique values), the `$group` stage can exhaust the 100MB memory limit quickly. Solutions: (1) Enable `allowDiskUse: true` to spill to disk, (2) Pre-filter with `$match` to reduce input, (3) Use `$bucket` or `$bucketAuto` for range-based grouping, (4) Consider `$sort` + `$group` with `$last` for deduplication, (5) For very high cardinality, use `$project` to reduce field sizes before `$group`. Monitor with `explain("executionStats")` to see `totalDocsExamined` and memory usage.

## Q39: What happens when you use `db.collection.drop()` on a sharded collection, and how do you cleanly drop a sharded collection?
**A:** Dropping a sharded collection removes all chunks and their data across all shards, deletes the collection metadata from the config server, and drops associated indexes. The operation is atomic but can take time for large collections. After dropping, the shard key space is freed. Cleanup steps: (1) Ensure no pending chunk migrations, (2) Drop the collection from `mongos` (not directly from a shard), (3) Verify with `sh.status()`, (4) Optionally `db.collection.dropIndexes()` first to speed up the drop, (5) If the balancer was running, it may need time to reconcile. Do not drop the config database's collections manually.

## Q40: How do you diagnose and resolve a "host not found" or "connection refused" error in a multi-datacenter replica set?
**A:** Multi-datacenter replica sets require proper DNS resolution, open firewall ports (default 27017), and consistent seed lists. Diagnostics: (1) Verify DNS resolution from each node for all other nodes' hostnames, (2) Check `iptables`/`security groups` for port 27017 access, (3) Verify `/etc/hosts` consistency, (4) Check `mongod.log` for `Failed to connect to` errors, (5) Verify each node's `net.bindIp` includes the IP used for replication. Resolution: use IP addresses in `replSetInitiate` to avoid DNS issues, configure `net.bindIp: 0.0.0.0` (with firewall protection), or set `net.ipv4` address specifically.

## Q41: How does the `$function` aggregation operator differ from `$accumulator` for custom JavaScript logic?
**A:** `$function` defines a stateless JavaScript function that operates on a single input and returns a transformed value — it cannot accumulate state across documents. `$accumulator` defines a custom accumulator with `init`, `accumulate`, `merge`, and `finalize` functions for use in `$group` — it maintains state across documents in a group. `$function` is simpler and supports any custom logic per document. `$accumulator` is powerful for complex group calculations like running totals, custom percentiles, or financial calculations. Both have performance overhead versus native operators.

## Q42: What is the `VM.overcommit_memory` kernel parameter, and why does MongoDB recommend a specific setting?
**A:** MongoDB recommends `vm.overcommit_memory = 1` (always overcommit) on Linux systems. When set to `0` (heuristic overcommit), the kernel may deny memory allocation requests even when enough is available, causing `mongod` to crash. Setting `1` allows the WiredTiger storage engine to allocate its internal cache without being denied by the kernel. The risk is that the OOM killer may terminate the process, but WiredTiger manages its own memory within the configured cache size. Also set `vm.overcommit_ratio` appropriately if using huge pages.

## Q43: How do you implement a CQRS (Command Query Responsibility Segregation) pattern with MongoDB?
**A:** CQRS with MongoDB: Use a dedicated primary (or replica set) for writes (commands) and a set of read replicas (secondaries with specific read preferences) for reads (queries). Denormalize data into read-optimized collections using change streams and materialized views. Write operations go to the command model (normalized/transactional). A subscriber listens to the oplog via change streams and updates read models (denormalized collections or Atlas Search indexes). Tools like `mongo-connector` or custom change stream consumers handle the synchronization. This pattern trades write latency for read performance.

## Q44: What are the differences between `SCRAM-SHA-1`, `SCRAM-SHA-256`, and `MONGODB-CR` authentication mechanisms?
**A:** `MONGODB-CR` (legacy, pre-3.0) is deprecated and insecure — it sends password hashes over the wire and uses weak MD5-based challenge-response. `SCRAM-SHA-1` (default since 3.0) uses salted challenge-response with SHA-1 hashing — secure but SHA-1 is considered weak. `SCRAM-SHA-256` (since 4.0) uses SHA-256 — the current recommendation. MongoDB 4.0+ defaults to SCRAM-SHA-256 with SCRAM-SHA-1 as fallback. For FIPS compliance, use SCRAM-SHA-256. Authentication mechanism is configured at user creation: `db.createUser({ mechanisms: ["SCRAM-SHA-256"] })`.

## Q45: How do you configure and use `mongosync` for live data migration between MongoDB clusters?
**A:** `mongosync` (part of MongoDB Atlas or MongoDB Enterprise) performs live, homogeneous data migration between clusters with minimal downtime. Process: (1) `start` command begins initial sync, (2) `commit` command finalizes migration after verifying lag is minimal, (3) The source cluster remains writable during sync. Requirements: both clusters must run compatible MongoDB versions, have network connectivity, and the target must have sufficient capacity. `mongosync` handles schema differences, indexes, and ongoing writes via oplog tailing. For heterogeneous migrations (e.g., on-prem to Atlas), use `mongomirror` (deprecated) or custom change stream applications.

## Q46: What is the `WiredTiger` "tiger" file format, and how does fragmentation affect performance?
**A:** WiredTiger stores data in `.wt` files using a B-tree (row-store) and LSM-tree (for log-structured merge). Fragmentation occurs as documents are inserted, updated, and deleted, causing page splits and internal fragmentation. High fragmentation increases disk I/O because more pages must be read for the same logical data. Symptoms: database file size significantly larger than actual data, and degraded read performance. Mitigation: (1) Use `compact` command (`db.runCommand({ compact: "collection" })`) to defragment, (2) Schedule periodic compaction for write-heavy collections, (3) Pre-allocate space by inserting placeholder documents and then updating them. Compaction blocks operations on the collection.

## Q47: How do you handle a `MongoError: E11000 duplicate key error` in a sharded cluster when the shard key constraint creates false duplicates?
**A:** In sharded clusters, the unique index must include the shard key. A unique compound index on `{ shardKey, otherField }` ensures uniqueness across shards while allowing per-shard optimization. False duplicates can occur if two documents with the same unique field value are routed to different shards (MongoDB only enforces uniqueness per shard unless the shard key is included). To avoid: (1) Always include the shard key in unique indexes, (2) Use a hashed shard key for even distribution if the unique field is not the shard key, (3) Consider using the unique field as the shard key itself.

## Q48: What are the trade-offs of using `WiredTiger`'s `log_compression` settings (snappy, zlib, zstd) for the journal?
**A:** WiredTiger supports snappy (default), zlib, and zstd compression for the journal (write-ahead log). Snappy: fastest compression/decompression, ~50% compression ratio, minimal CPU overhead — best for write-heavy workloads. Zlib: slower but better compression ratio (~60-70%) — saves disk space at the cost of latency. Zstd: balances speed and ratio, approaching zlib compression with snappy-like speed — good for mixed workloads. Journal compression is configured in `storage.wiredTiger.engineConfig.journalCompressor`. For SSDs with limited space, zstd or zlib may be worth the CPU cost. For write-heavy, latency-sensitive workloads, stick with snappy.

## Q49: How do you deploy MongoDB with `bindIp` set to multiple IP addresses, and what are the security implications?
**A:** `bindIp` in `mongod.conf` accepts a comma-separated list of IP addresses (e.g., `127.0.0.1,192.168.1.10,10.0.0.1`). Each IP corresponds to a network interface the server listens on. Security implications: (1) Binding to `0.0.0.0` exposes MongoDB to all interfaces — only do this behind a firewall, (2) Bind to specific internal IPs and never expose port 27017 to the internet, (3) Enable TLS if binding to any non-localhost IP, (4) Use multi-homing for replica sets: one IP for client traffic, one for replication traffic (via different ports or services), (5) IP whitelisting with `net.ssl.allowConnectionsWithoutCertificates` disabled with TLS.

## Q50: How does the aggregation pipeline `$unionWith` stage work, and what are its use cases vs `$lookup`?
**A:** `$unionWith` combines documents from two collections (like SQL UNION ALL) without requiring a join condition. It appends documents from the specified collection to the pipeline. Use cases: (1) Combining current month data from one collection with historical data from another, (2) Searching across multiple collections with same schema, (3) Unifying logs from different sources. Unlike `$lookup` (which adds nested arrays to each document), `$unionWith` creates a flat result set. Performance: `$unionWith` may perform a collection scan on the second collection — ensure indexes exist on filter fields used after `$unionWith`.

## Q51: What is the `defaultDbRouting` in a sharded cluster, and how does it affect unsharded collections on shards?
**A:** In a sharded cluster, unsharded collections reside on the primary shard (the shard configured as the database's primary). All queries to unsharded collections are routed to that single shard. The primary shard of a database is set when `enableSharding(database)` is called. If an unsharded collection grows large, it becomes a bottleneck. Mitigation: (1) Shard collections that are large or have high throughput, (2) Manually move the database primary shard with `movePrimary` to distribute unsharded collections, (3) Monitor the primary shard's load with `db.stats()` and `sh.status()`.

## Q52: How do you implement field-level encryption (FLE) using MongoDB's Client-Side Field Level Encryption (CSFLE)?
**A:** CSFLE encrypts fields on the client before sending data to MongoDB. Setup: (1) Generate a local master key or use a KMS (AWS KMS, Azure Key Vault, GCP Cloud KMS), (2) Create a data encryption key stored in a `keyVault` collection, (3) Configure the `MongoClient` with automatic field encryption: specify which fields to encrypt, the encryption algorithm (`AEAD_AES_256_CBC_HMAC_SHA_512` or `AEAD_AES_256_CBC_HMAC_SHA_512_Random`), and key ID, (4) Encrypted fields cannot be queried directly (unless using `queryType: "equality"` which supports direct equality matching), (5) Indexes on encrypted fields use randomized encryption by default, preventing range queries. Supports `deterministic` encryption for equality queries on indexed fields.

## Q53: What is the difference between `updateOne` with `upsert: true` and `replaceOne` with `upsert: true` in terms of behavior and potential pitfalls?
**A:** `updateOne` with `upsert: true` applies update operators (`$set`, `$inc`) to the matched document or creates a new document by combining the query filter with update operators. `replaceOne` with `upsert: true` replaces the entire matched document or inserts the provided replacement document. Pitfalls: (1) `replaceOne` removes fields not included in the replacement document — use with caution, (2) `updateOne` with `upsert: true` and a query using `$or` or `$and` can create unexpected documents with only the filter fields, (3) `upsert` on a sharded collection requires the filter to include the shard key — otherwise it errors. Always test `upsert` behavior with your specific query shape.

## Q54: How does MongoDB handle time-series collections internally in 5.0+, and what are the limitations you must plan for?
**A:** Time-series collections automatically organize data into buckets (based on `timeField`), compressing and storing them for efficient queries. The `bucketMaxSpanSeconds` (default 3600) and `bucketRoundingSeconds` control bucketing. Benefits: reduced storage (up to 90% compression), automatic indexing on time, and columnar compression. Limitations: (1) Only INSERT and query operations — no UPDATE or DELETE (except `db.drop()` or `db.collection.deleteMany()`), (2) `secondary` granularity minimum for time fields, (3) Limited secondary indexes (only `metaField` and `timeField` can be indexed), (4) Document size restrictions per time series event, (5) Sharding a time-series collection is available only in MongoDB 6.0+.

## Q55: How do you detect and handle `long-running operations` in MongoDB beyond using `currentOp`?
**A:** Beyond `db.currentOp()`: (1) Use `$currentOp` aggregation stage to filter: `db.aggregate([{ $currentOp: {} }, { $match: { secs_running: { $gt: 5 } } }])`, (2) Set up `maxTimeMS` on all queries to prevent runaway operations, (3) Configure `operationProfiling.mode: slowOp` with `slowOpThresholdMs` (default 100ms), (4) Use the `Database Profiler` to capture slow operations in `system.profile`, (5) Monitor `serverStatus().opLatencies` for operation distribution, (6) Use Atlas monitoring or Ops Manager for historical trend analysis. Kill operations with `db.killOp(opid)` when needed.

## Q56: What are the considerations for using MongoDB with GraphQL (e.g., through `mongodb-memory-server` or `Mongoose` resolvers)?
**A:** GraphQL + MongoDB requires careful N+1 prevention. Each GraphQL resolver can trigger separate MongoDB queries. Solutions: (1) Use DataLoader for batching and caching queries, (2) Design resolvers to use aggregation pipelines that return denormalized data, (3) Implement query complexity limiting to prevent abusive queries, (4) Use `$project` to limit fields returned matching GraphQL selections, (5) Consider Atlas Search for full-text or faceted GraphQL queries, (6) Use `mongoose` lean() in resolvers (skips hydrating Mongoose documents) for performance, (7) Be aware that GraphQL's recursive nested queries can produce extremely expensive MongoDB queries.

## Q57: How do you set up MongoDB for ephemeral or testing environments where data persistence is not required?
**A:** Options: (1) Always use `--fork` or `systemd` for clean daemonization, (2) Use `--dbpath /tmp/mongodb-{port}` and `--logpath /dev/null` for disposable instances, (3) Use `mongod --replSet` with `--nojournal` (MongoDB 4.0+ — journaling cannot be disabled) or `storage.journal.enabled: false` (deprecated), (4) Use `inMemory` storage engine for speed: `--storageEngine inMemory --inMemorySizeGB 1`, (5) Use `docker run --rm` for containerized testing, (6) Use `mongodb-memory-server` npm package for unit tests (spins in-process `mongod`), (7) Remember: disable auth and TLS for local dev, enable them for shared dev environments.

## Q58: How does MongoDB handle DNS Seed List Connection Format (`mongodb+srv://`), and what are the benefits over standard connection strings?
**A:** The `mongodb+srv://` protocol uses DNS SRV records to discover all nodes in a replica set or sharded cluster. Format: `mongodb+srv://server.example.com/db`. Benefits: (1) Automatic topology discovery — no need to list all nodes in the connection string, (2) Automatic load balancing — the driver connects to nodes based on SRV record priority, (3) Simplified configuration — change DNS records instead of updating connection strings across all applications, (4) TLS is automatically enabled. Requirements: (1) The DNS server must support SRV records on port 27017 (`_mongodb._tcp.server.example.com`), (2) A TXT record provides connection options (e.g., `authSource=admin&replicaSet=rs0`). Limitations: (1) Custom ports are not directly supported, (2) Some DNS providers have latency.

## Q59: What is the MongoDB `Router` (mongos) connection pool behavior under high load, and how do you tune it?
**A:** Each `mongos` maintains connection pools to each shard and the config server. Under high load, connection pools expand (`maxPoolSize` per shard). If `maxPoolSize` is too low, connections queue and requests timeout. If too high, `mongos` and shards consume excessive memory for connections. Tuning: (1) `maxPoolSize` should balance concurrency and resource limits — typical range 10-100 per `mongos`, (2) `waitQueueTimeoutMS` controls how long operations wait for a connection from the pool, (3) Monitor `connPoolStats` in `mongos` for `poolHits`, `poolMisses`, and `connectionsInUse`, (4) Use connection pooling at the application level too. For high-throughput workloads, add more `mongos` instances behind a load balancer.

## Q60: How do you implement multi-tenancy in MongoDB at scale, and what are the trade-offs of different isolation models?
**A:** Three multi-tenancy models: (1) **Database per tenant** — each tenant gets its own database. Best isolation, easiest backup/restore per tenant, but hits MongoDB's namespace limit with thousands of tenants. (2) **Collection per tenant** — each tenant has a dedicated collection. Good for sharding per tenant but same namespace scaling issue. (3) **Document field per tenant** (single collection with `tenantId` field) — best for scale (billions of documents), requires careful indexing (compound index on `{tenantId, ...}`). Trade-offs: model (3) requires all queries to include `tenantId` for performance; a slow query from one tenant can affect others. Model (1) is simplest for <1000 tenants; model (3) is necessary for millions of tenants.

## Q61: What are the common causes of `SecondaryCatchUpTooSlow` errors, and how do you resolve them?
**A:** This error occurs when a newly elected primary cannot catch up with the latest oplog entries from the old primary within `catchUpTimeoutMillis`. Causes: (1) Network latency between nodes, (2) Under-provisioned secondaries (slower disks, less RAM), (3) Write-heavy workload with large oplog entries, (4) Long-running index builds blocking replication. Resolutions: (1) Increase `catchUpTimeoutMillis` (default -1, infinite), (2) Improve secondary resource allocation, (3) Increase oplog size, (4) Set `catchUpTakeoverDelayMillis` for delayed members, (5) For urgent recovery, manually step down the stuck primary.

## Q62: How do you use the `hierarchical` shard key to support multiple access patterns efficiently?
**A:** A hierarchical shard key compounds fields matching common query patterns. Example: `{ region: 1, date: 1, userId: 1 }`. Queries on `region` alone target a specific shard (data locality). Queries on `region + date` also target a single shard. Queries on `date` alone broadcast to all shards but this may be acceptable. Design considerations: (1) The prefix field should have high cardinality for even distribution, (2) Avoid "hot spots" by ensuring the prefix has enough values to distribute writes, (3) If one prefix value dominates (e.g., 80% of data is `region=US`), that shard becomes a bottleneck, (4) Use hashed sharding on the prefix with range on a secondary field for some queries.

## Q63: What happens internally when MongoDB splits a chunk in a sharded cluster?
**A:** Chunk split process: (1) The `mongos` or balancer detects a chunk exceeds `chunkSize` (default 64MB), (2) MongoDB calculates split point(s) using `splitVector` command (samples ~100K keys or reads chunk metadata), (3) The chunk is split at median shard key values into roughly equal pieces, (4) The config server records the new chunk boundaries, (5) Data is not immediately migrated — the balancer may move the new chunks later. Splits do not cause data movement but update metadata. If the estimated split points are wrong (uneven distribution), subsequent chunk migrations may be needed.

## Q64: How does MongoDB's `transaction-lifetime` limit work, and what is the maximum transaction runtime?
**A:** Transactions have a default lifetime of 60 seconds (`transactionLifetimeLimitSeconds`). If a transaction exceeds this, it is automatically aborted. Additionally, each transaction has `maxTransactionLockRequestTimeoutMillis` (default 5ms) for lock acquisition and the `maxTimeMS` limit on individual operations within the transaction. For long-running transactions: (1) Increase `transactionLifetimeLimitSeconds` on all nodes (must be consistent), (2) Keep transactions as short as possible — only perform necessary operations, (3) Avoid holding transactions open across HTTP requests, (4) Use retry logic for `TransientTransactionError` which indicates `WriteConflict` — a sign the transaction held locks too long.

## Q65: How do you create a custom aggregation function using `$accumulator` for advanced statistical calculations?
**A:** `$accumulator` allows custom JavaScript-based accumulators. Example for calculating a running median:
```javascript
{
  $accumulator: {
    init: function() { return { values: [] }; },
    accumulate: function(state, value) { state.values.push(value); return state; },
    accumulateArgs: ["$score"],
    merge: function(s1, s2) { return { values: s1.values.concat(s2.values) }; },
    finalize: function(state) {
      state.values.sort((a,b) => a-b);
      const mid = Math.floor(state.values.length/2);
      return state.values.length % 2 ? state.values[mid] : (state.values[mid-1] + state.values[mid]) / 2;
    },
    lang: "js"
  }
}
```
Limitations: (1) JavaScript execution is slower than native operators, (2) Cannot be sharded efficiently (`merge` must combine all partial results), (3) Only available in `$group` stage.

## Q66: What is the difference between `mongodump` with `--oplog` and a file-system snapshot backup?
**A:** `mongodump --oplog` captures a point-in-time snapshot by recording oplog entries during the dump. It runs as a `mongod` process, causing load on the server. It does not capture all data atomically (dumps collection by collection) but uses oplog to reconstruct consistency. File-system snapshots (LVM, EBS snapshots, `fsyncLock` + copy) capture an atomic point-in-time image but require filesystem support and briefly pause writes. Snapshots are faster for large datasets and impose less load on MongoDB. Recovery from snapshots is simpler (just copy files). `mongodump` is portable across MongoDB versions and platforms; snapshots are not.

## Q67: How do you configure MongoDB for a multi-datacenter deployment with high write availability?
**A:** Multi-datacenter MongoDB: (1) Distribute replica set members across datacenters (e.g., 2 in DC1, 2 in DC2, 1 arbiter in a tie-breaking DC3), (2) Set `writeConcern: majority` — requires acknowledgment from 3 out of 5 nodes, surviving one datacenter failure, (3) Set `readPreference: nearest` for low-latency reads, (4) Configure `settings.chainingAllowed: false` to force direct replication across DCs (reduces lag), (5) Set `electionTimeoutMillis: 10000` (default 10000ms) to avoid false elections during network jitter, (6) Use `settings.catchUpTimeoutMillis` to configure write propagation, (7) Monitor cross-DC network latency — >20ms RTT increases replication lag.

## Q68: How does `$sample` work internally — when does it use random cursor vs reservoir sampling?
**A:** `$sample` with `size <= N` (where N was 100 before 4.4, 8192 in 4.4+) uses a pseudo-random cursor that skips documents with a random offset. This is fast but produces non-uniform random samples if documents are not uniformly distributed on disk. For `size > N` or when non-deterministic results are required, `$sample` uses reservoir sampling — scanning all matching documents and randomly selecting N. The reservoir algorithm ensures statistical uniformity regardless of document distribution but scans the entire input. For large collections, prefer the random cursor approach by keeping sample sizes small.

## Q69: What are the implications of using `network message compression` (snappy, zstd, zlib) between clients and MongoDB?
**A:** MongoDB supports compression (`snappy`, `zstd`, `zlib`, or `disabled`) for wire protocol messages. Compression reduces network bandwidth but increases CPU usage on both client and server. Configuration: `net.compression.compressors: ["snappy", "zstd"]` on the server; drivers enable compression in connection string: `?compressors=snappy`. Snappy is the default — good balance of speed and compression (~50% reduction). Zstd offers better compression ratio (~60% reduction) with moderate CPU. Zlib offers best compression (~70% reduction) but highest CPU. For local networks, compression may not be beneficial (CPU cost > bandwidth savings). For WAN connections or cloud-to-cloud, compression is highly recommended.

## Q70: How do you handle a scenario where a MongoDB shard's data becomes inconsistent with the config server's metadata?
**A:** Inconsistent metadata can cause queries to return incomplete results or errors. Diagnosis: (1) Check `sh.status()` for orphaned chunks or missing chunks, (2) Compare `db.collection.getShardDistribution()` against expected distribution, (3) Run `cleanupOrphaned` on each shard to remove documents that belong to other shards, (4) Use `checkShardingIndex` to verify index consistency. Resolution: (1) Run `sh.mergeChunks()` if chunks overlap, (2) Manually `sh.splitAt()` and `sh.moveChunk()` to redistribute, (3) As a last resort, dump and restore the collection. Prevention: the balancer and splitter handle most cases automatically; disable manual chunk operations in production.

## Q71: How does the `$merge` stage differ from `$out` in handling sharded output collections and incremental updates?
**A:** `$merge` is more flexible than `$out`: (1) `$merge` can output to a sharded collection (including when the pipeline runs on a sharded source), `$out` has restrictions, (2) `$merge` supports incremental updates with `on` fields to match existing documents, (3) `$merge` offers multiple `whenMatched` actions (replace, merge, keepExisting, pipeline, fail) and `whenNotMatched` actions (insert, discard, fail), (4) `$merge` can use custom fields (not just `_id`) for matching. `$out` replaces the entire target collection atomically — simpler but less flexible. Use `$merge` for materialized views and incremental ETL; use `$out` for full refreshes.

## Q72: What are the root causes of `Tailable Cursor` timeouts in capped collections, and how do you handle them?
**A:** Tailable cursors (used for tailing capped collections like the oplog) hang on `getMore` waiting for new data. Timeouts occur when: (1) `cursorTimeoutMillis` (10 minutes) expires with no activity, (2) The query exceeds `maxTimeMS`, (3) The capped collection reaches max size and the cursor's position is overwritten. Handling: (1) Set `noCursorTimeout(true)` and manage timeout in application, (2) Use `awaitData: true` with `maxAwaitTimeMS` to poll efficiently, (3) For oplog tailing, use Change Streams instead (they handle resume and failures better), (4) Catch `CursorNotFound` error and recreate the cursor from the last known position.

## Q73: How do you implement a MongoDB migration strategy for rolling upgrades across major versions?
**A:** Rolling upgrade process: (1) Check version compatibility — MongoDB supports upgrades from one major version to the next (e.g., 4.4 -> 5.0, not 4.4 -> 6.0 directly), (2) Set `featureCompatibilityVersion` (`setFeatureCompatibilityVersion`) after all nodes are upgraded, (3) Upgrade secondaries one at a time (shut down, upgrade binary, restart), (4) Step down the primary with `rs.stepDown()`, upgrade the former primary, (5) After all nodes are upgraded, set `featureCompatibilityVersion` to the new version, (6) For sharded clusters, upgrade config servers first, then `mongos`, then shards. Always test upgrades in a staging environment first. Check for deprecated features and breaking changes in the release notes.

## Q74: How does MongoDB handle `snapshot` read concern with transactions, and what consistency guarantees does it provide?
**A:** Snapshot read concern in transactions provides a point-in-time consistent view of data across multiple documents, collections, and (in 4.2+) shards. The transaction reads from a snapshot of the data at the transaction's `readTimestamp` (typically the start time). Guarantees: (1) No dirty reads, (2) No non-repeatable reads, (3) No phantom reads — within the transaction, repeated reads return the same data, (4) All reads see a consistent cross-collection state. In sharded clusters, the snapshot is coordinated across shards using the cluster-wide time. Snapshot isolation prevents write skew in some cases but not all (depending on the operation).

## Q75: What is the `exportCollection` and `importCollection` in MongoDB Shell, and how do they differ from `mongoexport`/`mongoimport`?
**A:** Starting MongoDB 5.0, `db.collection.exportCollection()` and `db.collection.importCollection()` are shell utilities that use the `mongod` server's internal mechanisms for faster data transfer compared to `mongoexport`/`mongoimport`. They work over the wire protocol, support BSON format (preserving all types), and are integrated with the shell. They are generally faster than the standalone tools but require the shell to maintain a connection. They support progress reporting and can be resumed. However, they do not support CSV/JSON output like `mongoexport`. For large datasets, they are preferred over the standalone tools.

## Q76: How do you handle MongoDB user-defined roles with privileges spanning multiple databases?

Cross-database roles simplify management but increase the blast radius if a user is compromised.

## Q77: What is the `wiredTigerConcurrentReadTransactions` and `wiredTigerConcurrentWriteTransactions` configuration, and when should you tune them?
**A:** These settings (renamed from `concurrentReadTransactions` and `concurrentWriteTransactions`) control the number of reader/writer slots in WiredTiger's internal concurrency model. Defaults: 128 for reads, 128 for writes (MongoDB 4.0+). When all slots are used, operations queue. Tuning: (1) For high-concurrency workloads on multi-core machines (32+ cores), increase both, (2) For write-heavy workloads with many small documents, increase write slots, (3) For read-heavy workloads with large documents, increase read slots, (4) If `serverStatus().wiredTiger.concurrentTransactions.write.available` is frequently 0, consider increasing write slots. However, increasing too high causes context switching overhead. Monitor throughput vs latency to find the sweet spot.

## Q78: How do you implement a custom conflict resolution strategy for multi-master MongoDB deployments?
**A:** MongoDB natively supports only single-master (primary-secondary) replication. Multi-master requires custom solutions: (1) Use Change Streams to detect conflicts — compare `oplog` entries from different clusters, (2) Implement CRDTs (Conflict-free Replicated Data Types) in application logic (e.g., use `$max` for timestamps, `$addToSet` for sets), (3) Use "last-writer-wins" (LWW) with logical clocks or wall-clock timestamps, (4) Assign write zones — different regions write to different document fields, (5) For bidirectional sync, use `mongosync` (Atlas) or third-party tools. MongoDB Bi-Temporal features can also help track conflicting writes. Remember: custom conflict resolution is complex and error-prone.

## Q79: How does the `$vectorSearch` aggregation stage work in MongoDB Atlas for vector similarity search?

    path: "embedding",

Cannot be used with sharded collections (vector search indexes are not yet supported on sharded collections).

## Q80: What are the edge cases with `updateMany` and the `$` positional operator in nested arrays?
**A:** The positional operator `$` identifies the first matching element in an array. For nested arrays, `$[identifier]` with `arrayFilters` is needed. Edge cases: (1) The `$` operator matches only the first array element that satisfies the query condition — for multiple matches, use `$[elem]` with `arrayFilters`, (2) ArrayFilters with multiple levels require positional all `$[]` or filtered positional `$[identifier]`, (3) Updating deeply nested arrays requires specific patterns — `arrayFilters` only go one level deep relative to the array being filtered, (4) The `$` operator does not work in upsert operations (throws error), (5) When using `$`, the array field must appear in the query filter.

## Q81: How do you monitor and tune the `flow control` mechanism introduced in MongoDB 4.2 for replica sets?
**A:** Flow control (enabled by default in 4.2+) limits the rate at which the primary applies writes to prevent secondaries from falling too far behind. It uses a "majority point" — if a secondary's lag exceeds `flowControlTargetLagSeconds` (default 10 seconds), the primary's write throughput is throttled. Monitoring: `serverStatus().flowControl` shows `isLaggedCount`, `lags` metrics, and `numRequestsThrottled`. Tuning: (1) If secondaries cannot keep up despite throttling, increase `flowControlTargetLagSeconds` or investigate why secondaries are slow (under-provisioned, network issues), (2) Disable flow control (`enableFlowControl: false`) only if you accept the risk of secondaries falling behind. High throttling indicates a systemic replication bottleneck that should be addressed.

## Q82: How do you handle MongoDB `connection storms` when many application instances start simultaneously?
**A:** Connection storms occur when many application instances connect simultaneously, overwhelming `mongos` or `mongod` with connection setup handshakes and auth operations. Mitigation: (1) Implement connection pooling with `maxPoolSize` that considers total expected connections, (2) Use connection pool warm-up: pre-create connections before handling traffic, (3) Configure `net.maxIncomingConnections` (default 1000000) to a reasonable limit, (4) Use `net.serviceExecutor: adaptive` for better connection handling, (5) Stagger application instance startup with readiness probes, (6) Use a connection proxy (like HAProxy or Envoy) in front of MongoDB, (7) For Kubernetes, use `container startupProbe` with initial delay. Monitor `connections.current` and `connections.available` in `serverStatus()`.

## Q83: What is the `logicalSessionCache` in MongoDB and how does it affect connection management?
**A:** MongoDB drivers create logical sessions (LSIDs) for each client for features like retryable writes and transactions. These sessions are tracked on the server in the `config.system.sessions` collection. The `logicalSessionRefreshMillis` (default 300 seconds) refreshes sessions; `localLogicalSessionTimeoutMinutes` (default 30 minutes) cleans up stale sessions. Issues: (1) Many concurrent sessions consume memory on `mongos`, (2) Stale sessions accumulate in `system.sessions` — monitor with `serverStatus().logicalSessionRecordCache`, (3) If you see `too many logical sessions` errors, the session cache may need clearing: `db.adminCommand({ refreshSessions: [] })`, (4) In sharded clusters, session information is distributed, adding overhead. Drivers manage sessions automatically; typically no manual intervention is needed.

## Q84: How does the aggregation pipeline `$replaceRoot` stage work with `$mergeObjects` for dynamic field renaming?
**A:** `$replaceRoot` replaces the entire document with a specified sub-document. Combined with `$mergeObjects`, it can dynamically restructure documents. Example — prefix all fields with a namespace:
```javascript
{
  $replaceRoot: {
    newRoot: {
      $mergeObjects: [
        { namespace: "users" },
        { $arrayToObject: { $map: {
          input: { $objectToArray: "$$ROOT" },

        }}}
      ]
    }
  }

```
This pattern is powerful for schema reshaping but is complex and has performance implications due to `$objectToArray` and `$arrayToObject` conversions. Use sparingly and only when necessary.

## Q85: How do you implement MongoDB auditing without enabling the built-in audit log (for performance reasons)?
**A:** If built-in auditing (Enterprise) has too much performance overhead, alternatives: (1) Use Change Streams on critical collections — capture document-level changes with minimal overhead, (2) Implement application-level auditing — log all writes through a middleware layer that records before/after states in an audit collection, (3) Use the oplog — tail the `local.oplog.rs` collection for write operations (requires oplog size to retain sufficient history), (4) Use database profiler on specific collections — `db.setProfilingLevel(1, { slowms: 0 })` on specific collections, (5) Use `mongod` logs — set `logLevel: 1` to capture DDL operations. Each has trade-offs in completeness, performance, and complexity.

## Q86: What are the considerations for storing and querying geospatial data with MongoDB's 2dsphere index and GeoJSON format?
**A:** MongoDB stores geospatial data in GeoJSON format: `{ type: "Point", coordinates: [lng, lat] }`. 2dsphere indexes support `$near`, `$geoWithin`, `$geoIntersects`, and `$geoNear` (aggregation). Considerations: (1) Always use GeoJSON (not legacy coordinate pairs) for 2dsphere, (2) Ensure coordinates are in [longitude, latitude] order (not lat/lng), (3) `$near` automatically sorts by distance — use `$geoWithin` if sorting is not needed (faster with index), (4) 2dsphere index supports `$minDistance` and `$maxDistance` in meters, (5) For complex shapes (polygons, multi-polygons), ensure the polygon is [right-hand rule](https://docs.mongodb.com/manual/reference/geojson/) compliant, (6) For planet-wide queries, use `{ spherical: true }` option with legacy coordinates. The `$geoNear` aggregation stage can include distance calculations and location data in the output.

## Q87: How does `transactionRetryWrites` behavior differ between sharded clusters and replica sets?
**A:** Retryable writes automatically retry write operations if a network error or replica set election occurs. In replica sets, retryable writes store the write operation's `txnNumber` and `lsid` in the oplog, so retries are idempotent. In sharded clusters, retryable writes are more complex: (1) A single write may target multiple shards if the operation modifies documents on different shards, (2) Each shard independently tracks retry state, (3) If one shard succeeds and another fails, retry may succeed on one but fail on another (inconsistent state), (4) Multi-document transactions across shards are the solution for cross-shard atomicity. Enable `enableTestCommands` for testing retry behavior in development.

## Q88: How do you configure MongoDB to handle `ingestion` of millions of small documents per second (IoT use case)?
**A:** For high-throughput IoT ingestion: (1) Use time-series collections (MongoDB 5.0+) — they batch and compress data automatically, (2) If not using time-series, batch inserts with `insertMany` (100-1000 documents per batch), (3) Use `writeConcern: { w: 1 }` (ack from primary only) instead of `majority`, (4) Use hashed shard key to distribute writes, (5) For raw `mongod`, set `net.maxIncomingConnections` appropriately, (6) Use `BulkWrite` operations in `ordered: false` mode for higher throughput, (7) Consider separate `mongos` for writes vs reads in sharded clusters, (8) Monitor `serverStatus().opcounters.insert` to track ingestion rate, (9) Use SSDs with high IOPS for the `dbPath`, (10) Pre-allocate documents (insert with placeholder data, then update) if schema is known in advance.

## Q89: How does the `$convert` operator with `onError` handle type conversion failures, and what are the use cases?
**A:** `$convert` converts a value to a specified type with graceful error handling:
```javascript
{
  $convert: {
    input: "$age",
    to: "int",
    onError: 0,       // default value if conversion fails
    onNull: null      // default value if input is null
  }
}
```
Use cases: (1) Safely converting string dates to Date objects without crashing the pipeline, (2) Handling mixed-type fields in collections with evolving schema, (3) Defaulting numeric fields that may contain non-numeric data to a sensible default, (4) ETL pipelines where source data quality is unknown. Without `onError`, `$toInt`, `$toString`, etc., throw errors on invalid input, aborting the pipeline.

## Q90: How do you migrate from a standalone `mongod` to a replica set without downtime?
**A:** Steps for zero-downtime migration: (1) Stop the standalone `mongod` and restart with `--replSet rs0`, (2) Use `rs.initiate()` on the standalone (now a single-node replica set), (3) The application connection string must be updated to use the replica set name (e.g., `mongodb://host:27017/?replicaSet=rs0`), (4) Add a new secondary node: start a new `mongod` with `--replSet rs0`, (5) Use `rs.add("new-host:27017")` — initial sync begins, (6) Wait for the secondary to reach `SECONDARY` state, (7) Optionally add more secondaries. Application impact: during step 3, briefly stop writes (only for the connection string change) or use a rolling connection migration with a proxy. Once the replica set is active, you have automatic failover.

## Q91: How does MongoDB handle `$lookup` with `foreignField` that is an array, and what are the performance implications?
**A:** When `foreignField` is an array, MongoDB performs a multikey index scan on the foreign collection — each element in the array creates a join condition. If the foreign collection has a multikey index on the array field, joins can use the index. However, if the foreign collection has no matching index, `$lookup` does a full collection scan for each source document. Performance traps: (1) A single source document with a large array causes N lookups (one per array element), (2) The foreign collection's multikey index size grows with array cardinality, (3) Avoid `foreignField` as an array in high-throughput systems — consider a separate join collection instead.

## Q92: What happens when MongoDB's `balancer` is running during high-write periods, and how do you schedule it?
**A:** The balancer constantly evaluates chunk distribution and migrates chunks to balance shards. Balancer activity consumes: (1) Network bandwidth for chunk data transfer, (2) CPU and I/O on source and target shards, (3) Config server metadata updates. During high-write periods, migrations can cause: (1) Increased query latency, (2) Replication lag, (3) `StaleConfig` errors. Scheduling: (1) Set `sh.setBalancerState(false)` to disable, then `sh.setBalancerState(true)` during low-traffic windows, (2) Use the `balancerWindow` settings: `sh.startBalancerTime` and `sh.stopBalancerTime` in `sh._updateBalancerSettings()`, (3) For MongoDB 6.0+, use `sh.setBalancerWindow()` for time-based scheduling, (4) Monitor migration success with `sh.status()` and `sh._lastMigration`.

## Q93: How does the aggregation pipeline handle timezone conversions with `$dateToString` and `$dateFromParts`?
**A:** MongoDB stores all Date values in UTC. Timezone conversion happens at query time: (1) `$dateToString: { format: "%Y-%m-%d %H:%M:%S", date: "$timestamp", timezone: "America/New_York" }` converts UTC to specified timezone, (2) `$dateFromParts: { year: 2023, month: 1, day: 15, timezone: "Asia/Tokyo" }` interprets components in given timezone and converts to UTC, (3) `$dateFromString: { dateString: "2023-01-15", timezone: "UTC" }` with `$dateToString` for timezone-aware queries. Performance: timezone conversions are CPU-bound. For large aggregations, convert once and store in a UTC-calculated field. Timezone names (IANA/Olson) like "America/Chicago" are supported; offset strings like "+05:30" also work.

## Q94: What is `refineCollectionShardKey`, and how do you change a shard key without downtime?
**A:** `refineCollectionShardKey` (MongoDB 5.0+) allows adding a suffix field to an existing shard key without re-sharding or downtime. Example: from `{ userId: 1 }` to `{ userId: 1, date: 1 }`. Process: (1) `db.adminCommand({ refineCollectionShardKey: "db.collection", key: { userId: 1, date: 1 } })`, (2) Existing chunks remain with the old key but future splits consider the new key, (3) New documents are distributed based on the compound key, (4) The change propagates to config servers. Limitations: (1) Cannot change the existing prefix fields — only add suffix fields, (2) Old documents are not re-distributed automatically, (3) The balancer may need time to rebalance. For complete shard key replacement (change prefix), a dump/restore is still required.

## Q95: How does MongoDB's `all` and `batchSize` affect `getMore` operations for large aggregation results?
**A:** Aggregation pipeline results are returned in batches (default `batchSize`: 101 for `find`, varies for aggregation). The `getMore` command retrieves subsequent batches. For large result sets: (1) If the aggregation can produce the entire result in memory, `allowDiskUse` is not needed, (2) The cursor stays alive until all batches are consumed or `cursorTimeoutMillis` (10 min) expires, (3) If results are consumed slowly, cached cursor state holds memory on the server, (4) For streaming large results, request a smaller `batchSize` to reduce per-iteration memory, (5) Use `$limit` before cursor traversal to reduce total data, (6) The `$out` and `$merge` stages avoid cursor issues by writing results directly to a collection.

## Q96: How do you implement a write-heavy leaderboard system using MongoDB's atomic operators and sorted data structures?
**A:** Leaderboard implementation: (1) Use a single collection with `{ playerId, score, updatedAt }`, (2) Create a compound index on `{ score: -1, updatedAt: 1 }` for sorted retrieval, (3) Use `updateOne` with `$inc: { score: points }` for atomic score updates, (4) For real-time rank retrieval, use `find({ score: { $gt: currentScore } }).count() + 1` (or `$rank` aggregation), (5) For top-N, use `find().sort({ score: -1 }).limit(N)` with the compound index, (6) For large player bases (millions), shard on `playerId` (hash) but this makes global ranking expensive — consider a dedicated leaderboard service or Redis for top-N caching, (7) For tied scores, use `updatedAt` for tie-breaking, (8) Materialize top-N views using `$merge` with periodic updates for performance.

## Q97: How does `PartialFilterExpression` indexing interact with the query optimizer, and what are the limitations?
**A:** Partial indexes (with `partialFilterExpression`) only index documents matching the filter. The query optimizer considers them when the query predicate is a subset of the filter expression. Limitations: (1) If the query does not include the partial filter condition, MongoDB will not use the partial index — it may do a collection scan or use a different index, (2) Compound partial indexes require all indexed fields that are not in the filter to be present in the query, (3) `$exists`, `$type`, and `$regex` in `partialFilterExpression` may not work as expected for index selection, (4) Partial indexes cannot have a filter expression on a field that is part of a compound index's prefix. Always use `explain()` to verify the partial index is being used.

## Q98: What is the `vectorClock` in sharded clusters, and how does it coordinate distributed transactions?
**A:** The vector clock (cluster-wide logical time) in MongoDB sharded clusters is maintained by the config server's `topologyTime` and the cluster-wide `logicalTime`. For distributed transactions: (1) Each shard has a local `logicalTime`, (2) The config server coordinates the global `clusterTime`, (3) Transactions use `clusterTime` for snapshot isolation across shards, (4) The vector clock ensures causal consistency — if operation A happens before B, the clock reflects this ordering, (5) Drivers include `$clusterTime` in requests for causal consistency, (6) The clock enables exact point-in-time reads across shards. Monitoring: `serverStatus().shardingStatistics` shows cluster time state.

## Q100: How does MongoDB handle `UUID` as `_id` compared to `ObjectId`, and what are the performance trade-offs?
**A:** UUIDs (stored as BinData) offer uniqueness without coordination but have significant performance trade-offs vs ObjectId: (1) UUIDs are 16 bytes vs ObjectId's 12 bytes — larger index entries and more memory, (2) UUIDs are random — inserting into a B-tree index causes page splits and index fragmentation (ObjectId is monotonic, enabling sequential index growth), (3) UUID _id insertion is ~20-40% slower than ObjectId due to random B-tree insert patterns, (4) UUIDs cannot be sorted chronologically (no embedded timestamp). Mitigations: (1) Use `Binary subtype 4` (UUID) for storage efficiency, (2) Consider a monotonic UUID variant (version 7 UUID), (3) Pre-allocate documents or use `unordered` bulk inserts, (4) Use `mmapv1` (legacy) storage engine for better random insert performance (not recommended). For most use cases, ObjectId is preferred — use UUID only when uniqueness must be guaranteed without database coordination.
