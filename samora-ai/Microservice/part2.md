# Microservice Interview Questions and Answers - Part 2

## Q1: How does the Saga pattern handle rollback with a long chain of 15 services?
**A:** Orchestration-based Saga is strongly preferred for long chains. A central coordinator (Step Functions, Temporal, Axon) tracks each step's state and compensations. If step 12 fails, the coordinator sequentially invokes compensating actions for steps 11, 10...1 in reverse order. The coordinator persists state in a database so it survives crashes and resumes compensation on restart. Choreography-based Saga for 15 services is highly complex and not recommended.

## Q2: What are the consistency tradeoffs between CQRS and Event Sourcing?
**A:** CQRS separates commands (writes) and queries (reads). The write side is immediately consistent (returns after commit). The read side is eventually consistent (lags behind writes). Event Sourcing stores all state changes as events; the read model is built by replaying events. Combined CQRS+ES: reads are eventually consistent (projection processing lag). For strong consistency, query the write model directly. Eventual consistency lag varies from milliseconds (in-memory projections) to seconds (batch-processed projections).

## Q3: How does Istio handle mTLS and what are the performance implications?
**A:** Istio uses Envoy sidecar proxies to intercept all traffic. mTLS uses SPIFFE identities (K8s service accounts) with certificates rotated every 24 hours by Citadel agent. Performance overhead: 1-3ms latency per request (TLS handshake + proxy processing), 3-10% CPU increase per sidecar, ~50MB memory per proxy, ~1-2KB per connection for headers. Use PERMISSIVE mode during migration (accepts both mTLS and plaintext), then switch to STRICT. Mitigations: connection pooling, tuned Envoy buffer sizes, exclude health check ports from proxy.

## Q4: How would you implement the Bulkhead pattern in a Java microservice?
**A:** Use Resilience4j Bulkhead. Two implementations: 1) SemaphoreBulkhead (lightweight, non-blocking, config: `maxConcurrentCalls=10, maxWaitDuration=500ms`). 2) FixedThreadPoolBulkhead (bounded thread pool). Each downstream dependency gets its own bulkhead. Example: Order service has separate bulkheads for Payment, Inventory, and Shipping. If Payment is slow, it exhausts only Payment's pool. In Spring Boot: `@Bulkhead(name="paymentService", type=THREADPOOL)`. Combine with CircuitBreaker: bulkhead limits concurrent calls, circuit breaker prevents calls when failure rate is high.

## Q5: What is the exact role boundary between API Gateway and Service Mesh?
**A:** API Gateway: edge-level north-south traffic (external client to service). Handles auth, rate limiting, request transformation, API versioning, caching, CORS, DDoS protection. Service Mesh: infrastructure-level east-west traffic (service to service). Handles mTLS, traffic routing (canary, mirroring), observability (metrics, traces), circuit breaking, retries. Gateway operates at L7, Mesh operates via sidecar proxies. They overlap in routing/observability but serve different traffic directions. Production systems typically use both: Gateway at edge, Mesh inside.

## Q6: How does the Outbox pattern solve the dual-write problem?
**A:** The dual-write problem: writing to DB and publishing a message must be atomic. Outbox pattern: 1) In the same DB transaction, insert business data AND an outbox event record. 2) A background process polls the outbox table for unsent events. 3) Publishes each event to the message broker. 4) Marks as sent after successful publish. CDC variant (Debezium): captures DB binlog changes from outbox table and streams to Kafka - eliminates polling, reduces latency, more reliable. The outbox table lives in the same DB as business data, so the transaction ensures atomicity of business data and event metadata.

## Q7: How does gRPC streaming outperform REST for inter-service communication?
**A:** gRPC streaming types: unary, server-streaming, client-streaming, bidirectional. Benefits: 1) Binary protobuf (4-5x smaller payload than JSON). 2) HTTP/2 multiplexing (multiple streams on one connection, no HoLB). 3) Native streaming without WebSocket. 4) Built-in deadlines/cancellation. 5) Strong typing via .proto. Downsides: harder debugging, limited browser support, not human-readable. Best for: high-throughput internal services, real-time streaming, mobile backends. REST with HTTP/1.1 requires multiple connections; HTTP/2 helps but gRPC's protobuf serialization and streaming primitives give significant performance advantage.

## Q8: How would you implement distributed tracing across 50 services with OpenTelemetry?
**A:** 1) Instrument all services with OTel SDK (auto-instrumentation for Spring Boot, Express, Flask). 2) Use W3C Trace Context propagation (`traceparent` header) across HTTP/gRPC/messaging. 3) Configure OTel Collector as sidecar or DaemonSet to batch, sample, and export spans. 4) Backend: Jaeger/Tempo for tracing, with head-based sampling (decide at request entry) for production. 5) Trace across async boundaries: inject trace context into message headers (Kafka, SQS). 6) Use `Baggage` for custom propagation (tenant ID, user ID). 7) Verify coverage with trace heatmaps. 8) Sampling strategies: probabilistic (1% for production), rate-limited (fixed spans/second), tail-based (keep traces with errors).

## Q9: How do error budgets drive operational decisions in microservices?
**A:** Error budget = 100% - SLO target. For 99.9% SLO over 30 days: ~43 minutes downtime budget. If budget is healthy: deploy safely, do canaries, experiment. If budget is nearly exhausted: halt releases, focus on reliability, rollback risky features, add redundancy. Developers have autonomy within the budget (no approvals needed). On-call alerted only when error budget burn rate is high. Multi-window multi-burn-rate alerting detects fast and slow budget consumption. SLOs are user-facing (availability, latency). SLIs are technical metrics (p99 latency, error rate). Error budget policy must be enforced by the platform team.

## Q10: How does the Strangler Fig pattern work for monolith decomposition?
**A:** 1) Identify bounded contexts (order management, payments). 2) Build a new microservice for one boundary while monolith runs. 3) Deploy new service alongside monolith. 4) Routing layer (API Gateway, reverse proxy) intercepts calls for the extracted functionality and routes to new service. 5) Synchronization between monolith and new service (dual-write or event-based) for data consistency. 6) Gradually increase traffic to new service. 7) Once stable, remove old code from monolith. 8) Repeat for each bounded context. The routing layer dynamically decides monolith vs new service using feature flags or path-based routing, enabling incremental migration with zero downtime.

## Q11: How does Canary deployment work at the infrastructure level?
**A:** 1) Deploy new version to a subset of instances (1 of 10 K8s pods). 2) Route small traffic percentage to canary (5% via Istio VirtualService weights or service mesh traffic splitting). 3) Monitor metrics: error rate, latency p99, request volume, business metrics (conversion rate). 4) Compare with baseline (old version). 5) If healthy after X minutes, increase (25%, 50%, 75%, 100%). 6) If degraded, rollback to 100% old version. Tools: Istio (VirtualService), AWS CodeDeploy (Lambda traffic shifting), Nginx/HAProxy (weighted upstreams). Canary must receive production traffic to be meaningful. Use observability tools to detect subtle p99 latency or error rate changes.

## Q12: How do you design an event-driven architecture to avoid event storms?
**A:** Event storm: one event triggers cascade overwhelming consumers. Mitigations: 1) Throttling: rate-limited event buses, concurrency limits. 2) Deduplication: idempotent consumers with dedup keys. 3) Bulkheads: separate event streams per domain. 4) Prioritization: critical events on separate streams from non-critical. 5) Filtering: emit only significant events, not every state change. 6) Backpressure: reactive streams, monitor consumer lag. 7) Circuit breakers: if consumer falls behind, stop emitting downstream events. 8) Dead letter queues: isolate failed events after retries exhausted. 9) Limit event fan-out depth (max 3 levels deep).

## Q13: How does the BFF pattern solve API design problems for different clients?
**A:** BFF creates a separate backend per client type (mobile, web, IoT, smart TV). Benefits: 1) Each BFF optimized for its client's data needs (mobile needs less data, web needs full data with SSR). 2) Client-specific logic stays out of shared services. 3) BFF aggregates upstream microservice calls into client-optimized responses. 4) Mobile BFF handles offline sync, push registration; Web BFF handles SSR, SEO metadata. 5) Each BFF team owns its client experience end-to-end. 6) Eliminates "one-size-fits-all" API bloat. Downsides: code duplication across BFFs, more services to maintain. Recommended when client requirements diverge significantly.

## Q14: How do you handle zero-downtime database migrations in microservices?
**A:** Expand-Migrate-Contract (parallel change) pattern: Phase 1 (Expand): Add new schema elements (columns, tables), keep old ones. Deploy code writing to both old and new. Phase 2 (Migrate): Deploy code reading from new schema only. Stop writing to old fields. Phase 3 (Contract): Remove old schema elements. Each phase is a deployable version. DB changes must be backward-compatible at each step. Use versioned migrations (Flyway, Liquibase). Never break existing queries. Use feature flags for risky schema changes. Prefer additive changes (new table) over altering existing tables.

## Q15: How do Circuit Breaker, Retry, and Timeout interact?
**A:** Proper ordering: Timeout -> Retry -> Circuit Breaker. 1) Timeout is innermost: if response exceeds deadline (2s), fail fast. 2) Retry wraps timeout: on failure, retry with exponential backoff + jitter (max 3 retries). 3) Circuit Breaker wraps retry: track failure rate in sliding window (last 100 calls). If failure rate > 50%, open circuit. While open: fail immediately (skip timeout and retry). Half-open: allow probe request. Config: `failureRateThreshold=50, waitDurationInOpenState=10s, slidingWindowSize=100`. This prevents retries overwhelming struggling service (circuit opens first) and cascading failures.

## Q16: How would you implement idempotency without a centralized store?
**A:** 1) Client generates idempotency key (UUID v4) per request, sends in header. 2) Service stores key + response in database with unique constraint on key. 3) On duplicate key: return cached response. 4) Without centralized store: use deterministic request hashing (hash request body + timestamp). 5) Use database unique constraints: `INSERT ON CONFLICT DO NOTHING`. 6) For events: use event ID as dedup key. Database-driven idempotency is more reliable than cache-based but higher latency. Use Redis for speed with DB fallback for reliability.

## Q17: How does Kubernetes NetworkPolicy isolate microservices?
**A:** NetworkPolicy defines allowed ingress/egress traffic at pod level using label selectors. Default: pods are non-isolated (allow all traffic). Once a NetworkPolicy selects a pod, unallowed traffic is denied. Example: `podSelector: app: orders`, `ingress: from: podSelector: app: payment, ports: 8080`. Only `payment` pods can reach `orders:8080`. Key concepts: `podSelector`, `policyTypes` (Ingress, Egress), `from/to` (`namespaceSelector`, `podSelector`, `ipBlock`). Requires CNI plugin (Calico, Cilium). Cilium supports L7 HTTP-aware policies. Use for zero-trust: default deny all, explicitly allow only necessary communication.

## Q18: How does Kafka guarantee message order and what are the tradeoffs?
**A:** Kafka guarantees order within a partition, not across partitions. Use the entity ID as partition key: `producer.send(new ProducerRecord("orders", orderId, event))`. Same key = same partition = ordered delivery. Consumer processes sequentially within partition. For ordered processing across entities, use one partition (limits parallelism). Partition count = max consumer parallelism. More partitions = more throughput but no cross-partition order. Best practice: partition by entity ID for ordered entity processing. For high throughput with partial order: partition by entity group and accept per-group ordering.

## Q19: How would you implement a feature flag system for microservices?
**A:** 1) Centralized flag service (LaunchDarkly, Split, or custom with Redis/etcd). 2) Each service has a client library caching flags locally, subscribing to updates (SSE or polling). 3) Flag targeting: user-based, percentage-based, environment-based, custom attributes (region, plan tier). 4) Use for: gradual rollout, kill switches, A/B testing, trunk-based development. 5) Flags must be short-lived (remove after stabilization). 6) Use typed flags (boolean, percentage, string). 7) Audit log flag changes. 8) Avoid nested conditionals (strategy pattern instead). 9) Evaluate flags at edge (API Gateway) for early routing decisions.

## Q20: How does distributed caching affect consistency in microservices?
**A:** Caching introduces stale data. Patterns: 1) Cache-aside: read from cache, miss -> read DB -> populate cache. TTL-based invalidation. 2) Write-through: write to DB and cache atomically. 3) Write-behind: write to cache, async write to DB (data loss risk). 4) Cache invalidation: publish event on DB update, all instances invalidate cache. Tradeoffs: cache-aside has best read perf but stale data within TTL. Write-through stronger consistency but higher write latency. Event-based invalidation is eventually consistent but low latency. Best practices: short TTLs (1-5 min), cache invalidation on writes, use read-through for high-read workloads, avoid caching write-hot data.

## Q21: How does the Ambassador pattern differ from Sidecar?
**A:** Both use proxy containers but differ in scope. Ambassador: purpose-built proxy for a specific external dependency (Redis, specific API). Handles connection pooling, retry, circuit breaking for that dependency. Sidecar: general-purpose proxy for ALL network traffic (Envoy in Istio). Ambassador may be lighter (single purpose), Sidecar is configurable but heavier. Ambassador is often at the edge (API Gateway). Sidecar is internal (service mesh). In practice, Ambassador pattern is used for dependency-specific concerns; Sidecar is used for universal traffic management.

## Q22: How would you implement backpressure in a REST-based system?
**A:** REST is synchronous with no native backpressure. Strategies: 1) Client-side: if 429 received, slow down (exponential backoff). 2) Service-side: rate limiting (token bucket) -> signal via 429 + Retry-After header. 3) Load shedding: reject requests when queue depth exceeds threshold. 4) Reactive Streams (RSocket): protocol with built-in backpressure. 5) Circuit breaker: open when downstream is overwhelmed. 6) Bulkhead: separate thread pools, isolate overwhelmed services. 7) Message queue: decouple with queue backpressure (Kafka consumer lag indicates backpressure). Most practical: rate limiting at API Gateway + circuit breakers + bulkheads.

## Q23: How do you implement contract testing for async messaging with Pact?
**A:** Pact supports Message Pact for async. 1) Consumer defines expected message structure (provider, topic, body, metadata). 2) Pact generates a contract file. 3) Provider-side Pact test verifies it can produce messages matching the contract. 4) Unlike HTTP Pact (replays requests), Message Pact verifies provider's message producer code creates the correct format. 5) Consumer-side handlers tested with Pact-generated messages. Key: tests each side independently using contract verification, not integration tests. For Kafka: Pact verifies Avro/JSON message schemas between producer and consumer. Detects breaking schema changes without running full system.

## Q24: How does choreography-based Saga handle partial failures in a 5-service flow?
**A:** Each service emits events and listens for compensation events. Example flow: OrderService creates order (emits `OrderCreated`), InventoryService reserves (emits `InventoryReserved`), PaymentService processes (emits `PaymentProcessed`), ShippingService ships (emits `OrderShipped`). If Payment fails: emits `PaymentFailed`. InventoryService listens and compensates (release inventory, emits `InventoryReleased`). OrderService listens and compensates (mark order failed). Challenges: complex event flows, no central visibility, circular compensation risk, silent failure handling. Best practice: use orchestration for 3+ service chains. Choreography is simple for 2-3 services but becomes unmanageable beyond that.

## Q25: How would you handle schema evolution with Avro and Schema Registry?
**A:** 1) Define Avro schema per event type. 2) Store schemas in Confluent Schema Registry (or AWS Glue Schema Registry). 3) Producers validate writes against schema, consumers validate reads. 4) Schema evolution rules: Backward compatible (new schema can read old) - add optional fields, remove fields with defaults. Forward compatible (old schema can read new) - delete fields, add optional fields. Full compatible (both). 5) Schema IDs embedded in each message (4-byte magic + 4-byte schema ID). 6) Producer: serialize with Avro, include schema ID. Consumer: deserialize using schema ID to fetch schema. 7) Breaking changes require new topic or subject. Without Schema Registry, schema changes cause deserialization errors across services.

## Q26: How does HPA work with custom metrics for microservice scaling?
**A:** 1) Deploy metrics-server for resource metrics (CPU, memory). 2) Deploy Prometheus Adapter for custom metrics (queries Prometheus, exposes to HPA). 3) HPA config: `type: Pods` (pods metric - requests/sec per pod) or `type: Object` (object metric - queue depth per service). Example: scale orders-service when `kafka_consumer_lag > 100`. HPA calculates: `desired = current_replicas * (current_metric / desired_metric)`. Best for: scaling on business metrics (queue depth, request latency, error rate) rather than just CPU/memory. Supports multiple metrics (max across all).

## Q27: How does Process Manager (Saga Orchestrator) handle timeouts and retries?
**A:** Process Manager is a stateful service orchestrating multi-step sagas. It handles: 1) Timeouts: if step doesn't complete within expected time, emits timeout event. 2) Retries: configurable per step (max attempts, backoff). 3) Compensation: on failure/retries exhausted, calls compensating actions in reverse order. 4) Persistence: PM stores state (current step, context) in DB (PostgreSQL, DynamoDB). 5) Recovery: if PM crashes, resumes from persisted state. Steps after timeout: log, notify monitoring, execute compensation, optionally escalate to human. Implementation: AWS Step Functions, Temporal, Axon, or custom state machine + event bus.

## Q28: How do you design idempotent event consumers?
**A:** 1) Event carries unique event ID (UUID). 2) Consumer maintains dedup store (Redis with TTL, or DB table with unique constraint on event ID). 3) On receive: check dedup store. If exists, skip and return success. If not, process and store event ID. 4) Processing must be idempotent: `UPDATE account SET balance = 100` is idempotent; `UPDATE account SET balance = balance + 10` is not. 5) Use DB constraints: `INSERT INTO payment (id, amount) VALUES (eventId, amount) ON CONFLICT (id) DO NOTHING`. 6) For Kafka: save offset + process result atomically using Kafka transactional API (process-then-ack). 7) Process-then-ack prevents duplicate processing but requires manual offset management.

## Q29: How would you implement a service mesh without sidecars using eBPF?
**A:** Cilium Service Mesh uses eBPF instead of sidecars. Architecture: 1) Cilium agent runs on each node (DaemonSet). 2) eBPF programs attach to network interfaces at kernel level. 3) Service identity via K8s pod labels (not sidecar certs). 4) mTLS: WireGuard encryption (Linux kernel 5.6+) or TLS via eBPF + key management. 5) L3/L4/L7 policies enforced via eBPF. 6) Observability: Hubble (eBPF-based flow visibility). Benefits: no sidecar overhead (CPU/memory), lower latency (kernel-level, no extra hop), simpler deployment, compatible with all apps. Downsides: requires Linux 5.10+, less mature, limited traffic management vs Istio.

## Q30: How does retry with exponential backoff + jitter prevent thundering herd?
**A:** Without jitter: all clients retry at same time = thundering herd. With jitter: random variation spreads retries. Full jitter: `sleep = random(0, min(cap, base * 2^attempt))`. Equal jitter: `sleep = min(cap, base * 2^attempt) / 2 + random(0, ... / 2)`. De-correlated jitter: `sleep = min(cap, random(base, sleep_prev * 3))`. Example: base=100ms, cap=30s. Without jitter: 100 clients retry at t=200ms, 400ms, 800ms (coordinated spikes). With jitter: retries spread across window. Max attempts: 5-10. Full jitter is most common and simplest.

## Q31: How do you handle cross-service queries with database-per-service?
**A:** Since you can't JOIN across services: 1) API Composition: Service A calls Service B's API and joins in app code (works for small datasets). 2) CQRS with Read Model: pre-join data into read-optimized view (Elasticsearch, Materialized View). 3) Event-based data duplication: services emit events, consumers store local copies for querying. 4) GraphQL Federation: unified graph across services. 5) Aggregator service: dedicated service for cross-service queries. Example: Order (PostgreSQL) + User (MongoDB). For "orders with user names": create OrderView in PostgreSQL that includes user name (via UserUpdated events). Avoids cross-service JOINs.

## Q32: How would you implement distributed rate limiting across instances?
**A:** Centralized rate limiter (Redis): 1) Sliding window: sorted sets per user, trim timestamps, count. 2) Token bucket: Redis counters per user (`rate_limit:user123`, tokens, TTL). Use Lua scripts for atomic operations. 3) Sliding window counter: track request count per minute. For high-scale: use local rate limiter with periodic sync (eventually consistent approximations). Use Redis Cluster for distributed limit storage. Inform clients via headers: `X-RateLimit-Remaining`, `X-RateLimit-Reset`. API Gateway (Kong, Envoy) for external APIs. Service mesh rate limiting (Istio/Envoy with Redis) for internal.

## Q33: How do Liveness vs Readiness probes affect availability?
**A:** Liveness: determines if pod should be restarted. If fails, kubelet restarts container. Use for: deadlock detection, process freeze. Readiness: determines if pod receives traffic. If fails, removed from Service endpoints (not restarted). Use for: waiting for DB connection, cache warmup. Startup: delays liveness/readiness checks for slow-starting apps. Incorrect: liveness tied to external dependency (if DB slow, all pods restart = cascading failure). Best practice: liveness = app-internal (process alive); readiness = external dependencies (DB, cache). Use `initialDelaySeconds` and `failureThreshold` to handle startup.

## Q34: How would you implement graceful shutdown in Spring Boot?
**A:** 1) K8s sends SIGTERM to pod. 2) Spring Boot graceful shutdown (`server.shutdown=graceful`, `spring.lifecycle.timeout-per-shutdown-phase=30s`) stops accepting new requests (returns 503). 3) Active requests complete within timeout. 4) Release resources: close DB connections, drain thread pools, close message consumers. 5) Deregister from service discovery (Eureka, K8s preStop hook). 6) If not finished within `terminationGracePeriodSeconds` (default 30s), K8s sends SIGKILL. Config: `terminationGracePeriodSeconds=60`, preStop sleep 15s (allows LB to de-register). Readiness probe returns false before shutdown.

## Q35: How does Transactional Outbox achieve atomicity between DB writes and message publication?
**A:** Problem: writing to DB and publishing a message must be atomic (both succeed or both fail). With two operations, system can crash between them. Transactional Outbox: 1) In same DB transaction, insert business record AND outbox event record. 2) Background process polls outbox for events that haven't been published. 3) Publishes each event to message broker. 4) Marks as published after success. CDC variant (Debezium): captures DB binlog entries from outbox table and streams to Kafka - eliminates polling, reduces latency. The outbox table lives in the same DB as business data, so the transaction ensures atomicity.

## Q36: How would you implement a chaos engineering experiment for microservices?
**A:** 1) Define steady state: normal metrics (p99 latency, error rate, throughput). 2) Hypothesis: "If Payment goes down, Order degrades gracefully." 3) Blast radius: start small (1% traffic, non-prod). 4) Run experiment (Chaos Mesh, Litmus, AWS FIS): kill pods, inject latency, block traffic. 5) Observe: metrics change vs steady state. 6) Check: circuit breakers open? bulkheads isolate? retries degrade gracefully? 7) Automate rollback: if SLO violation, halt experiment. 8) Game day: simulate region failure, DB failover, cert expiry. 9) Document findings: add missing circuit breakers, update timeouts. 10) Add experiments to CI/CD.

## Q37: How does API versioning interact with consumer-driven contracts?
**A:** Consumer-driven contracts (Pact): 1) Consumer creates Pact file with expected API behavior. 2) Provider verifies it satisfies contract. 3) For v2 endpoints: create new contract. 4) Provider runs both v1 and v2 contract verification in CI. 5) Deprecation: inform consumers of v1 end date, stop verifying v1 contracts after migration. Use URL versioning (`/v1/orders`, `/v2/orders`) so both versions coexist with separate contracts. Consumer migration: consumer updates to v2, creates v2 contract, verifies against provider. CDC catches breaking changes before deployment.

## Q38: How does BFF pattern affect microservice team structure?
**A:** BFF aligns with team boundaries. Teams own end-to-end: 1) Web Team owns Web BFF + web frontend. 2) Mobile Team owns Mobile BFF + mobile apps. 3) Each BFF owned by client team (not separate team). 4) BFF aggregates upstream services + client-specific logic. 5) Teams negotiate APIs with upstream services. Tradeoffs: BFF team needs both frontend + backend skills. If BFF becomes complex, split by domain. Cross-cutting (auth, logging) handled by API Gateway before BFF. Conway's Law: BFF structure reflects team structure. A frontend team owning its BFF moves faster with no dependency on separate backend team.

## Q39: How would you handle distributed transactions without Saga?
**A:** Options (preference order): 1) Redesign boundaries (move conflicting operations into one service). 2) Try-Confirm/Cancel (TCC): reserve resources (try), then confirm or cancel. 3) Two-Phase Commit (2PC): coordinator asks all to prepare, then commit. NOT recommended (blocking, long locks, poor scalability). 4) Best-effort delivery with retries + idempotency. 5) Compensating transactions (Saga). 2PC is the only option guaranteeing atomicity, but doesn't scale across heterogeneous databases and holds locks long-term. Saga with eventual consistency is the practical approach.

## Q40: How does gRPC load balancing differ from REST?
**A:** REST (HTTP/1.1): client connects to LB (ALB/NGINX). Each request may use new connection or pool. Load balancing is per-request. gRPC (HTTP/2): client maintains one long-lived connection to a server. HTTP/2 multiplexes many RPCs over one TCP connection. Load balancing is per-connection. Problem: if gRPC client opens one connection to one backend, that instance gets all traffic (imbalanced). Solutions: 1) Client-side LB: client receives backend list, picks per RPC (round_robin, weighted_target). 2) Look-aside LB: gRPC uses external LB (Envoy) for per-request balancing. 3) K8s headless service: client discovers all pod IPs, uses client-side LB. Best practice: service mesh (Envoy sidecar) for per-request gRPC balancing.

## Q41: How would you design a cross-service notification system?
**A:** 1) Any service emits `SendNotification` event (Kafka/SNS) with recipient, template, params. 2) Notification Service subscribes. 3) Deduplicates (event ID), templates message, sends via channel (SES for email, SNS for SMS, FCM for push). 4) On failure: retry (exponential backoff) then DLQ. 5) User preferences service: check opt-in/out before sending. 6) Rate limiting: per user, per channel. 7) Delivery tracking: bounce/complaint handling. 8) Notification status service: track delivery state. 9) Eventual consistency: notification may take seconds to send, user may see confirmation before receiving email. Compensating: cancelled order can't recall sent email.

## Q42: How do you implement tracing context propagation across message queues?
**A:** 1) Create trace context before sending to queue. 2) Serialize context (trace_id, span_id, baggage) into message headers. 3) Producer injects context: `producer.send(new ProducerRecord<>("orders", orderId, event).withHeaders(traceparent: "00-0af7...-b9c7...-01"))`. 4) Consumer extracts from headers, creates child span or continues trace. 5) For batch consumers (Kafka poll): each message may have different trace context. Create separate span per message while using batch context for poll. 6) With OpenTelemetry: `TextMapPropagator` to inject/extract. 7) Challenges: batch processing (one poll returns many trace IDs), redelivery (maintain trace ID). Proper context propagation is essential for end-to-end visibility across async boundaries.

## Q43: How would you gradually adopt a service mesh (Istio) without breaking apps?
**A:** Gradual Istio adoption: 1) Start with observability (no traffic management, no mTLS). Install with `meshConfig.accessLogFile`. Existing apps continue without sidecars. 2) Enable sidecar injection for few namespaces (non-prod first). Monitor overhead. 3) Enable PERMISSIVE mTLS (accept both mTLS and plaintext). All services work. 4) Once all services in namespace have sidecars, switch to STRICT mTLS. 5) Add traffic management (VirtualService, DestinationRule) for canary deployments. 6) Add authorization policies. Risk mitigation: use `includeInboundPorts` to control proxy ports, add `sidecar.istio.io/inject: "false"` for incompatible services. Test with mirrored traffic. Rollback: disable injection, traffic falls back to direct.

## Q44: How does Event Sourcing projection rebuilding work?
**A:** Projections transform event streams into read models. To rebuild: 1) Stop the running projection. 2) Drop the read model database. 3) Replay all events from the event store from the beginning. 4) Projection re-processes every event and rebuilds the read model. Events are immutable and contain all state changes. Optimizations: 1) Snapshotting: periodically save aggregate state so replay starts from latest snapshot. 2) Parallel projections: different projections rebuild independently. 3) Chunked replay: replay in batches. 4) Versioned projections: if schema changes, rebuild from events (event store doesn't change). This is a key advantage: you can always reconstruct any view from original events.

## Q45: How do you handle schema conflicts when merging microservice databases?
**A:** Schema conflicts when services merge/synchronize. Strategies: 1) Anti-corruption layer: translate between bounded context schemas. 2) Shared kernel: share only truly common concepts (customer ID, product ID). 3) Event-based sync: each service emits events with its schema; downstream transforms to own schema. 4) Schema-on-read: store as JSON, interpret at read time. 5) Canonical data model: standard intermediate format (Avro, Protobuf). 6) DB views: map to shared schema. Best practice: avoid merging entirely. Each service owns its schema, translates at boundaries. If required (data warehouse), use ETL pipeline from each service's event stream.

## Q46: How does Health Check API affect K8s rolling updates?
**A:** During rolling update: 1) New pod starts with startup probe (checks if initialized). 2) Readiness probe must return success before K8s adds pod to Service. 3) Old pod readiness probe returns false (traffic drained). 4) Old pod receives SIGTERM (graceful shutdown). If readiness probe is too aggressive: new pod never becomes ready, update stalls. Too lenient: new pod receives traffic before ready (503 errors). Best practices: readiness checks DB connectivity and cache warmup; liveness is simple (process alive); startup probe for slow-starting apps. Set `minReadySeconds` to wait for app stability.

## Q47: How would you design cross-service search with Elasticsearch?

## Q50: How would you implement distributed counters (views/likes)?

## Q51: How does Saga compensation handle cascading failures in compensations?
**A:** 5-step Saga (A->B->C->D->E), step D fails: 1) Orchestrator invokes compensations in reverse: D_compensate, C, B, A. 2) Each compensation must be idempotent. 3) If B's compensation fails: Saga enters "compensation failed" state. Options: retry failed compensation with backoff, log and alert for manual intervention, continue compensating remaining steps. 4) Saga must be durable: store state in DB so orchestrator resumes compensation on restart. 5) Compensations must eventually succeed (retries, DLQs, manual escalation). Make compensations as simple as possible. Monitor compensation failures as high-priority alerts.

## Q57: How do you handle the dual-write problem during monolith-to-microservice migration?
**A:** During migration, data in both monolith DB and new service DB. Dual write: both must be updated atomically. Solutions: 1) Event-based sync: monolith writes to its DB, emits event. ## Q60: How does BFF pattern differ from GraphQL Federation?

## Q61: How would you implement distributed locks for microservices?
**A:** Options: 1) Redis (Redlock): `SET resource_key uuid NX PX 10000` (acquire with 10s TTL). Release with Lua script verifying ownership. Redlock uses N/2+1 Redis nodes. 2) ZooKeeper: ephemeral sequential znodes, lowest seq holds lock, watcher for notification. 3) etcd: `Txn` API with lease. 4) Database: `SELECT FOR UPDATE` or advisory locks. ## Q63: How do you handle database indexing strategies in CQRS?
**A:** Command side (write model): minimal indexes (PK only, few secondary). Avoid over-indexing (impacts write performance). Query side (read model): heavily indexed for read patterns. Each query type gets its own index or materialized view. Tradeoff: CQRS allows different indexing on each side. Write model optimized for fast inserts (few indexes). Read model optimized for fast queries (many indexes, denormalized). ES projections often use specialized indexes (Elasticsearch inverted index, Redis sorted sets for ranking). The separation is a key benefit: write indexing doesn't affect read performance and vice versa.

## Q64: How does the Gateway Aggregation pattern differ from BFF?

Reduces client round-trips (client makes 1 call instead of N). BFF: separate backend per client type. Gateway Aggregation is generic (one gateway for all clients). BFF is client-specific. Gateway Aggregation is simpler (single aggregation layer) but lacks client customization. BFF provides client-optimized responses but more code. ## Q65: How would you implement cross-service transactions using the TCC pattern?
**A:** Try-Confirm/Cancel (TCC) for distributed transactions: 1) Try phase: each participant reserves resources (not executes). Payment service reserves $100 (pending hold), Inventory service reserves item. 2) If all Try succeed: Confirm phase. Payment captures $100, Inventory decrements item. 3) If any Try fails: Cancel phase. Payment releases hold, Inventory unreserves. TCC differs from Saga: Try + Confirm is the success path (not just "do then compensate"). Benefits: shorter lock duration (Try holds, Confirm releases quickly). Challenges: idempotency required for Confirm/Cancel, requires each service to implement Try/Confirm/Cancel endpoints. TCC is practical for payment scenarios (credit card authorization + capture).

## Q66: How does the Event Carried State Transfer pattern work?
**A:** Instead of querying another service for related data, each service includes necessary state in its events. Example: Order Service emits `OrderPlaced` event with order details + customer name + shipping address (duplicated from Customer service). Benefits: 1) Consumer services don't need to call Customer service to get name/address. 2) Reduces synchronous coupling. 3) Events are self-contained for processing. 4) Consumers can work offline. Downsides: 1) Data duplication across events. 2) If customer updates address, old events have stale data. 3) Schema evolution is more complex. Best for: data that doesn't change often or where staleness is acceptable. Use with event versioning to handle field changes.

## Q67: How would you handle database query routing (read/write splitting) in microservices?
**A:** Each microservice manages its own read/write splitting. Patterns: 1) Application-level: service has two data sources (write master, read replicas). Use `@Transactional(readOnly=true)` to route to replica. 2) Database proxy: proxy (Pgpool, ProxySQL, RDS Proxy) handles routing transparently. 3) CQRS: naturally separates read and write stores. Challenges: 1) Replica lag (read replica may be behind master). 2) Read-your-writes consistency: after write, need to read from master (session consistency). 3) Statement-based vs row-based replication differences. Best practice: use CQRS if read/write models differ significantly. Use proxy layer for simple read/write splitting with configurable consistency.

## Q68: How does the API Composition pattern handle partial failures?
**A:** API Composition: service calls multiple downstream services and aggregates results. Partial failure handling: 1) Timeout per downstream call (don't wait indefinitely). 2) Circuit breaker per downstream. 3) Partial response: if one service fails, return results from others (with error information). 4) Fallback: if non-critical service fails, return cached/default data. 5) Retry with backoff for transient failures. 6) Parallel vs sequential calls: parallel reduces latency but may increase error rate. 7) Error aggregation: collect all errors, return partial success + error details. 8) Idempotency: if composition fails mid-way, retry must handle partially successful state.

## Q69: How does the Backend for Frontend pattern handle authentication across clients?
**A:** BFF handles client-specific authentication: 1) Web BFF: session-based auth (cookie + server-side session). CSRF protection. 2) Mobile BFF: token-based auth (JWT, OAuth2 refresh tokens). ## Q70: How does the Saga pattern interact with database transaction isolation levels?
**A:** Saga spans multiple services/databases, so ACID isolation doesn't apply. Issues: 1) Dirty reads: Saga step A writes, step B reads and makes decision based on uncommitted (in-saga) data. 2) Non-repeatable reads: between Saga steps, another process modifies data. 3) Lost updates: concurrent Sagas interleave steps. Mitigations: 1) Semantic locking: Saga step marks record as "in-progress" to prevent other operations. 2) Commutative updates: operations can be applied in any order (counters). 3) Pessimistic view: Saga reads current state, verifies no changes before committing. 4) Reread values: before compensating, verify current state matches expected. Saga sacrifices isolation for performance and availability.

## Q71: How would you implement a distributed scheduler for cron jobs across microservices?

## Q72: How does mTLS work in a service mesh for inter-service authentication?

## Q74: How do you handle multi-tenant data isolation in microservices?
**A:** Patterns: 1) Database per tenant: each tenant gets its own DB. Strong isolation, easier backup/restore per tenant. Higher operational complexity. 2) Schema per tenant: same DB, different schemas per tenant. Good for medium isolation. 3) Row-level tenant isolation: same table, `tenant_id` column. Simpler, but weaker isolation (one bad query can affect all). Row-level requires every query to filter by `tenant_id`. Use connection pooling per tenant or `SET session tenant_id`. Ensure tenant context propagation across service calls.

## Q75: How does the Cryptographic Shredding pattern work for data deletion?
**A:** Instead of deleting data from all databases/replicas, delete the encryption key. Data becomes permanently inaccessible (undecipherable). Implementation: 1) Each tenant/user has a unique data encryption key (DEK). 2) DEK is encrypted by a key encryption key (KEK) stored in a key management system (AWS KMS, Vault). 3) To delete: delete the KEK from the KMS. The DEK is permanently unrecoverable. 4) Data can remain in databases/backups but is unreadable. Benefits: 1) Instant deletion (no need to scan all data). 2) Backups automatically become inaccessible. 3) GDPR/CCPA compliance (right to deletion). 4) No need to track all copies of data. Drawback: you lose the data too. Use for: PII, regulatory compliance data.

## Q76: How would you implement rate limiting with the Token Bucket algorithm across services?
**A:** 1) Each user/API key gets a token bucket (Redis key: `rate_limit:{user_id}`). 2) Bucket has capacity (max tokens) and refill rate (tokens per second). 3) On request: check if tokens available. If yes, decrement and allow. If no, reject with 429. 4) Redis Lua script for atomic check-and-decrement: `local tokens = redis.call("GET", key); if tokens and tonumber(tokens) > 0 then redis.call("DECR", key); return 1; else return 0 end`. 5) Refill: use sliding window or separate timer. 6) Distributed: Redis Cluster for multi-region. 7) Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`. 8) Enforcement at API Gateway for external APIs, service mesh (Envoy rate limit filter) for internal.

## Q77: How do you implement caching strategies for read-heavy microservices?
**A:** 1) Cache-aside (lazy loading): on read, check cache; if miss, read DB, populate cache, return. TTL-based expiry. Simple but cache stampede risk. 2) Read-through: cache layer handles DB reads transparently. 3) Write-through: write to cache + DB synchronously. Strong consistency, higher write latency. 4) Write-behind: write to cache, async write to DB. Lower write latency, data loss risk. 5) Cache invalidation: on update, invalidate cache entry (delete or update). Use events (CDC from DB) to invalidate all cache instances. 6) Cache warming: pre-populate cache for expected hot data (product catalog). 7) Multi-tier: local (in-process) + distributed (Redis) cache. Local is faster but limited to instance; distributed is shared but higher latency.

## Q78: How does the Strangler Fig pattern handle database splitting?
**A:** During monolith decomposition, the database must also be split. Approach: 1) Start with shared database (monolith). 2) New service writes to its own DB + synchronizes back to monolith DB. 3) Gradually migrate read/write operations from monolith DB to service DB. 4) Use database triggers or CDC (Debezium) to keep monolith and service DBs in sync during migration. 5) Monolith reads: first from service DB, fallback to monolith DB. 6) Once no operations read from monolith tables, migrate those tables to service's new DB. 7) Remove synchronization once fully migrated. Tools: Debezium, AWS DMS (continuous replication). This is the riskiest part of strangling - incorrect sync leads to data inconsistency. Plan rollback at each step.

## Q79: How would you implement health check endpoints for microservice dependencies?
**A:** 1) Each service exposes `/health` endpoint returning overall status (UP/DOWN). 2) `/health/ready` returns readiness for traffic (check DB, cache, message queue, downstream services). 3) `/health/live` returns liveness (process alive). 4) Aggregate health of critical dependencies: DB connection pool non-empty, cache reachable, queue connected. 5) Circuit breaker status: include downstream service health indicator (if circuit breaker is open for Payment, mark as DEGRADED). 6) Custom indicators: disk space, memory usage, thread pool utilization. 7) Response format (RFC): `{"status": "UP", "components": {"db": {"status": "UP"}, "redis": {"status": "DOWN"}}}`. 8) Use Spring Boot Actuator, Quarkus Health, or Flask Healthcheck.

## Q80: How does the Event Sourcing event store handle concurrent writes?
**A:** Event Sourcing appends events, never updates. Concurrent writes: 1) Optimistic concurrency: each aggregate has a version number. Write includes expected version. If version doesn't match (concurrent write already incremented), reject with conflict. 2) Pessimistic locking: lock aggregate before appending (slower, less common). 3) Event store (EventStoreDB, PostgreSQL with append-only tables) handles concurrency natively. 4) Expected version check prevents lost updates: if two writers try to append at version 5, one succeeds (version becomes 6), the other fails (version is 6, not 5). 5) Read-side projections don't block on writes (append-only is fast). Concurrency conflicts are less common in Event Sourcing (no UPDATEs) but still matter for command validation that reads current state before appending events.

## Q81: How would you implement a service mesh with mutual TLS for legacy applications?
**A:** 1) Deploy sidecar proxy (Envoy) alongside legacy app without modifying app code. 2) Sidecar intercepts all TCP traffic via iptables rules (transparent proxy). 3) mTLS: sidecar handles TLS handshake with peer sidecar. 4) Legacy app communicates over plaintext localhost to sidecar (no TLS in app). 5) PERMISSIVE mode: sidecar accepts both mTLS and plaintext during migration. 6) Once all services have sidecars, switch to STRICT mode. 7) Certificate management: Istio Citadel issues SPIFFE certificates to sidecars. 8) Authorization: sidecar validates peer identity, enforces policies. Legacy app doesn't need to know about TLS, certificates, or service identity. This is a key benefit of service mesh: add mTLS to any app without code changes.

## Q82: How does the Offset Tracking pattern work for Kafka consumers?
**A:** 1) Consumer reads messages from a partition. 2) After processing each message (or batch), consumer commits the offset (the position in the partition). 3) If consumer crashes, it resumes from the last committed offset. 4) Auto-commit (default): commit periodically (5s). Risk: crash between processing and commit = duplicate processing. 5) Manual commit: at-least-once (commit after processing) - may reprocess if crash before commit. 6) Exactly-once: Kafka transactions - producer sends to output topic + consumer offset commits atomically (read-process-write cycle). 7) Manual partition assignment: consumer controls which partitions it reads, useful for stateful processing. 8) Seek: consumer can seek to specific offset (reprocess from beginning, skip, or go to latest). Best practice: at-least-once with idempotent processing (simplest).

## Q83: How would you implement a dead letter queue strategy for message processing?
**A:** 1) After max retries (configurable, typically 3-5 with exponential backoff), failed message goes to DLQ. 2) DLQ is a separate queue/topic. 3) Messages include: original payload, error details, retry count, stack trace, original timestamp. 4) DLQ monitoring: alert when DLQ depth > threshold. 5) DLQ processing: manual or automatic replay after fixing the issue. 6) DLQ partitioning: separate DLQs by error type (transient vs permanent), priority, or service. 7) TTL for DLQ messages: messages expire and are deleted after retention period. 8) DLQ analysis: periodically analyze DLQ messages for patterns (systematic vs one-off failures). 9) In Kafka: use dead letter topic (DLT) with the same key as original for ordered replay. 10) DLQ should not cause alert fatigue - alert only on persistent patterns, not transient blips.

## Q84: How does the Sidecar pattern handle configuration reloading?
**A:** 1) Sidecar proxies (Envoy) support hot reload: configuration changes via xDS API (discovery service) without restart. 2) Envoy watches for config changes from control plane (Istio, Consul). 3) On change: Envoy gracefully updates listeners, clusters, routes without dropping connections. 4) Rolling update: gradually drain old connections while accepting new ones with updated config. 5) Rate limit config: sidecar picks up new rate limit rules without restart. 6) TLS certificate rotation: Envoy detects new certs on disk, reloads them gracefully. 7) If config is invalid: Envoy keeps last valid config (failsafe). 8) Fallback: if control plane is unreachable, sidecar continues with last known good config. Benefits: config changes propagate within seconds without pod restarts.

## Q85: How do you implement structured error responses across all microservices?

## Q86: How would you implement cross-service feature flags with gradual rollout?
**A:** 1) Feature flag service (LaunchDarkly, Split, or custom) stores flag configurations. 2) Each service's flag client caches flags locally (polling or streaming). 3) Flag targeting: user ID, email domain, plan tier, region, random percentage. 4) Gradual rollout: start at 1% of users, monitor metrics, increase to 5%, 25%, 50%, 100%. 5) Kill switch: flag set to false instantly disables feature across all services. 6) A/B testing: different user groups see different behavior, results measured in analytics. 7) Flag lifecycle: create -> rollout -> stabilize -> remove (delete after full rollout). 8) Technical flags: for operations (enable debug logging, bypass cache). 9) Canary flags: new service is flagged on for 1% of traffic. 10) No nested flags: each flag independently toggleable.

## Q87: How does the API Gateway handle WebSocket connections for microservices?
**A:** 1) API Gateway (AWS API Gateway, Kong, Envoy) accepts WebSocket upgrade request. 2) Gateway routes WebSocket connection to appropriate service based on path (e.g., `/ws/chat/:roomId` -> Chat Service). 3) Connection state: Gateway maintains mapping of connection ID to backend service. 4) Scalability: Gateway handles millions of concurrent connections, forwards messages to backend without holding state. 5) Backend services broadcast messages via pub/sub (Redis Pub/Sub, Kafka). 6) Connection lifecycle: connect (auth at Gateway), message (forwarded), disconnect (cleanup). 7) Sticky sessions: if using stateful backends, Gateway pins connection to specific instance. 8) Graceful close: if backend goes down, Gateway closes connections with error. 9) Backpressure: if backend is slow, Gateway buffers or drops messages. 10) Metrics track: active connections, messages/second, message latency.

## Q88: How do you handle schema evolution for events stored in Kafka (long-term retention)?
**A:** 1) Use Schema Registry with Avro/Protobuf. 2) Schema evolution rules: BACKWARD (new schema reads old), FORWARD (old schema reads new), FULL (both). 3) Never delete/rename fields: make optional with defaults instead. 4) Add fields: optional with default value (backward compatible). 5) Remove fields: can only be removed if they had defaults (forward compatible). 6) Rename: create new field, deprecate old, migrate consumers, remove old. 7) Schema versioning: each change creates new version, registry tracks history. 8) Compatibility checking: schema registry rejects incompatible changes at produce time. 9) For incompatible changes: create new topic, migrate consumers gradually. 10) No schema registry: use JSON with `$schema` field, validate at consumer. Risk: silent data corruption if schemas drift. Schema Registry is strongly recommended for Kafka with long retention.

## Q89: How would you implement cross-service leader election for a singleton service?
**A:** 1) Use distributed lock: Redis (Redlock), ZooKeeper (ephemeral sequential znodes), etcd (leases). 2) Multiple instances attempt to acquire a lock with TTL (heartbeat). 3) The instance holding the lock is the leader. 4) Leader performs the singleton task (e.g., periodic cleanup, batch processing). 5) Non-leaders wait or poll for lock. 6) If leader crashes: lock expires (TTL), another instance acquires it. 7) If leader's heartbeat fails: lock released, new leader elected. 8) Implementation: K8s leader election using ConfigMap (built-in), or library (Apache Curator for ZooKeeper, Redisson for Redis). 9) Graceful handoff: leader releases lock on shutdown, successor can immediately take over. 10) Split-brain prevention: lock with fencing token (epoch) to reject stale leaders.

## Q90: How does the Backend for Frontend pattern handle mobile offline scenarios?
**A:** 1) Mobile BFF implements offline-first patterns: 2) Sync API: BFF provides `/sync` endpoint returning all data changes since last sync timestamp. 3) Conflict detection: server-side timestamps per entity, BFF returns conflicts when mobile submits stale data. 4) Queue writes: mobile sends mutations to BFF message queue when offline. 5) BFF processes queue on mobile coming online, handles conflicts. 6) Data schema: BFF uses `updated_at` + `version` fields for conflict detection. 7) Compression: BFF compresses sync responses (gzip) for mobile bandwidth. 8) Pagination: BFF supports cursor-based pagination for large datasets. 9) Push notifications: BFF sends notification when sync data is ready (via WebSocket/APNs/FCM). 10) BFF manages device registration, push tokens, and sync state per device.

## Q91: How would you implement a distributed cache invalidation strategy?
**A:** 1) Event-based invalidation: when data changes, service emits `EntityUpdated` event with entity ID. 2) All cache instances subscribe to invalidation events and evict the key. 3) Redis Pub/Sub or Kafka for event distribution. 4) TTL-based invalidation (fallback): even if event missed, cache eventually expires. 5) Write-through: cache is updated synchronously when data changes (strong consistency but higher latency). 6) Cache-aside with invalidation: on write, delete cache key, next read fetches fresh data. 7) Stale-while-revalidate: serve stale data while asynchronously refreshing cache. 8) Cache tagging: tag related cache entries (e.g., `user:123` tags all user's cached data), invalidate all by tag. 9) Gradual invalidation: not all instances invalidate simultaneously (avoid cache stampede). 10) Monitor cache hit rate: drop indicates invalidation issue.

## Q92: How does the Gateway Routing pattern handle multi-region deployments?
**A:** 1) Each region has its own API Gateway instance (regional endpoint). 2) Global load balancer (Route53, Global Accelerator) routes users to nearest region's gateway. 3) Gateway routes requests to services within the region (intra-region). 4) If region has an outage: global LB routes traffic to another region. 5) Gateway checks regional health before routing. 6) Cross-region requests: if a service isn't in the region, gateway can forward to another region (with latency penalty). 7) Regional gateway configuration is identical (config as code, synced across regions). 8) Data gravity: prefer routing to region where data resides. 9) Sticky routing: if user session created in US-East, route subsequent requests to US-East (unless failover). 10) Gateway handles regional telemetry: request count, latency, error rate per region.

## Q93: How do you implement the Circuit Breaker pattern with a half-open state probe interval?
**A:** Circuit breaker states: CLOSED -> OPEN -> HALF_OPEN -> CLOSED or OPEN. Configuration: 1) `failureRateThreshold`: 50% (open if > 50% of last 100 calls fail). 2) `waitDurationInOpenState`: 10 seconds (stay open for 10s before trying half-open). 3) `permittedNumberOfCallsInHalfOpenState`: 3 (if all 3 succeed, close; if any fails, reopen). 4) `slidingWindowType`: COUNT_BASED or TIME_BASED. 5) `minimumNumberOfCalls`: 10 (don't evaluate until at least 10 calls recorded). Half-open probe: after `waitDurationInOpenState`, circuit breaker allows a limited number of probe requests. If probe requests succeed, service is considered recovered. If probe fails, circuit reopens and waits again. Probe requests carry real user traffic (not health checks), so they accurately test service health. Set `waitDurationInOpenState` to balance quick recovery vs protecting the downstream.

## Q94: How would you design a service for graceful degradation under load?
**A:** 1) Identify critical vs non-critical features. Under load: disable non-critical. 2) Circuit breakers for downstream dependencies: if Payment is slow, offer "pay later" fallback. 3) Resource limits per feature: separate thread pools per feature (bulkhead). 4) Response degradation: under high load, return cached or simplified responses. 5) Request prioritization: premium users get priority over free users (priority queues). 6) Load shedding: reject requests when queue depth exceeds threshold (return 503 with Retry-After). 7) Feature flags: toggle non-critical features off under load. 8) Data freshness degradation: serve stale cache instead of hitting overloaded DB. 9) Timeout reduction: reduce timeouts under load (fail fast instead of waiting). 10) Monitor: track degradation status and expose via health endpoint (degraded vs healthy). 11) Recovery: when load normalizes, gradually restore features.

## Q95: How does the Process Manager pattern handle idempotency for compensating actions?
**A:** 1) Each compensating action receives a unique compensation ID. 2) Process Manager stores which compensations have been executed. 3) If PM crashes mid-compensation: on restart, it skips already-executed compensations and continues from where it stopped. 4) Compensating actions must be idempotent: `cancelOrder(id)` returns success even if order already cancelled. 5) PM uses idempotency key for each compensation call. 6) If compensation call times out: PM retries with same ID (idempotent ensures no double-effect). 7) Compensation store: PM saves executed compensation IDs in its database. 8) Recovery: PM checks compensation store before executing. 9) If compensation permanently fails: PM marks saga as "compensation failed", alerts operators, and will retry periodically. 10) Manual override: operators can force-compensate via admin API (also idempotent).

## Q96: How do you implement a distributed database migration across microservices?
**A:** 1) Each service manages its own schema independently (Flyway/Liquibase). 2) For changes affecting multiple services: a) Add new field/message format. b) Both producer and consumer deploy updated versions (consumer first, handles both old and new format). c) Once all consumers support new format, producer switches to new format exclusively. d) Remove old format support from consumers. 3) Coordination: shared calendar for migrations, feature flags to control migration timing. 4) Zero-downtime: always backward-compatible schema changes. 5) Data backfill: if adding new required field, backfill existing data with default values before deployment. 6) Migration ordering: service dependency graph - migrate leaf services first, upstream last. 7) Rollback: each migration has a rollback script. 8) Test: run migrations against staging with production-like data volume.

## Q97: How would you implement cross-service correlation for async sagas?
**A:** 1) Each saga has a unique saga ID (UUID). 2) Saga ID is propagated in events, messages, and service calls throughout the saga flow. 3) All logs include `saga_id` field. 4) All metrics include `saga_type` and `saga_id` tags (for Prometheus/Datadog). 5) Tracing: saga orchestrator creates a trace with saga ID as root span. 6) Each step creates child spans with saga context. 7) Dashboard: track active sagas by type, duration histograms, step-level latency. 8) Alert: saga duration > p99 threshold, compensation triggered, step failure rate > threshold. 9) Audit: store all saga events in an audit log (saga ID, step, timestamp, action, result). 10) Debug: search by saga ID to see all events, logs, and traces across services. This provides end-to-end visibility into distributed sagas.

## Q98: How does the Sidecar pattern handle network failures with the control plane?
**A:** 1) Sidecar proxies (Envoy) cache last known good configuration locally. 2) If control plane is unreachable (Istiod, Consul server down): sidecar continues operating with cached config. 3) Dynamic updates stop: new services added during control plane outage won't be discovered. 4) Existing connections continue uninterrupted. 5) Envoy health checking continues (local decision, not dependent on control plane). 6) Certificates (mTLS): sidecar caches certificates (24h TTL). If control plane is down at cert expiry, sidecar continues with current cert until control plane recovers (within cert grace period). 7) Rate limiting: if rate limit service is down, Envoy can fail open (allow all) or fail closed (deny all), configurable. 8) Fallback: sidecar uses DNS fallback for service discovery if xDS fails. 9) Reconnection: sidecar keeps retrying control plane with exponential backoff.

## Q99: How would you implement a cross-service data export/import feature?
**A:** 1) Export: a) User initiates export from any service. b) Export service creates a `DataExportRequested` event with user ID. c) Each service listens for export events, collects user data from its domain. d) Services upload data to shared storage (S3) with user ID prefix. e) Export service monitors progress, zips all files, notifies user when ready. f) Data format: JSON or CSV with schema per service. g) TTL: export files expire after 7 days. 2) Import: a) User uploads file (e.g., from another platform). b) Import service validates format, splits by service domain. c) Each service receives its portion via event/API. d) Services validate and import data (with conflict resolution). e) Results aggregated back to user. 3) GDPR: export includes all PII across services. Import must handle data ownership boundaries.

## Q100: Design a comprehensive observability strategy for a 50-microservice platform.
**A:** Three pillars: 1) Logging: structured JSON logs with standardized fields (`trace_id`, `span_id`, `service`, `level`, `message`, `correlation_id`, `duration_ms`). Centralized in Elasticsearch (Loki alternative). Retention: 7 days hot, 30 days warm, 1 year cold. Log levels: ERROR (alert), WARN (dashboard), INFO (debug), DEBUG (disabled production). 2) Metrics: RED metrics (Rate, Errors, Duration) per endpoint per service. RED metrics per dependency (DB, queue, downstream service). Saturation metrics (CPU, memory, connections, queue depth). Business metrics (orders placed, revenue). Prometheus + Grafana dashboards per service and cross-service. 3) Tracing: 100% tracing for errors, 1% sampling for successful requests. OpenTelemetry SDK in all services. Jaeger/Tempo backend. Traces correlated with logs (trace_id in log entries). 4) Alerting: SLO-based (99.9% availability, p99 latency < 500ms). Multi-window burn rate alerts. Alert on error budget consumption rate. 5) Dashboards: Service overview (RED metrics), Dependency graph (service map), Business metrics, SLO tracking, Cost per service. 6) SLOs: define SLIs per service (latency p99, error rate, throughput). Error budget policy: deployment freeze when budget exhausted. 7) On-call: PagerDuty integration, runbooks for common alerts, incident management process.
