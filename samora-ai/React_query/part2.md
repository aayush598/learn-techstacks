# React Query Interview Questions and Answers - Part 2

## Q1: How do you design a query key hierarchy for a large-scale application with hundreds of endpoints?
**A:** Use a structured factory pattern: Define a queryKeys object with methods returning structured keys: queryKeys.users.all() -> ['users'], queryKeys.users.detail(id) -> ['users', id], queryKeys.users.posts(id) -> ['users', id, 'posts']. Consistency: domain first, IDs, sub-resource. Invalidation: queryClient.invalidateQueries({ queryKey: ['users'] }) invalidates ALL user queries. Use scoped keys for non-hierarchical: ['feature', 'component', ...params]. For TypeScript, use @tanstack/query-key-factory. The pattern enables broad invalidation (all users) and targeted invalidation (specific user's posts).

## Q2: How do dependent queries with multiple levels of nesting affect React Query's caching behavior?
**A:** With deep nesting (3+ levels), each query's cache is independent. Example: useQuery(['user', id]) -> useQuery(['projects', userId], { enabled: !!user }) -> useQuery(['tasks', projectId], { enabled: !!project }). Level 1 data may be stale while level 2 and 3 are fresh. Strategies: set staleTime shortest at the top level (1 min for user, 5 min for projects, 30 min for tasks). Prefetch all levels upfront after first query succeeds via prefetchQuery in onSuccess. Use keepPreviousData to avoid cascading loading states when any dependency changes.

## Q3: How do you implement parallel queries with dynamic query keys that depend on an array of IDs?
**A:** Use useQueries for dynamic parallel queries: const users = useQueries({ queries: userIds.map(id => ({ queryKey: ['users', id], queryFn: () => fetchUser(id) })) }). React Query deduplicates identical IDs. For streaming arrival of IDs, wrap in a hook maintaining the ID list. Cache semantics: when IDs change, stale queries drop, new ones fetch, unchanged persist. For large arrays (>100), batch into smaller groups with throttling. Error handling: useQueries returns individual query states per item - handle each independently.

## Q4: How does AbortSignal integration work with React Query for race condition prevention?
**A:** React Query passes an AbortSignal as the second argument to queryFn: queryFn: ({ signal }) => fetch(url, { signal }). When a component unmounts, query key changes, cancelQueries is called, or the query is garbage collected, React Query calls abort() on the AbortController. The fetch API natively supports signal - the network request is cancelled. This prevents stale updates from previous requests overwriting fresh data, wasted bandwidth, and setState on unmounted component warnings. Always handle AbortError in error boundaries.

## Q5: How do you implement optimistic updates for a list where items need to be inserted at a specific position?
**A:** For position-aware optimistic updates: In onMutate of useMutation, cancel all list queries, snapshot the current cache via getQueryData, compute the new list with the item inserted at the correct sorted position using setQueryData with a comparator. For paginated lists, track the target page for insertion. Use a unique client-generated ID for the optimistic item (temp-${nanoid()}) so it can be matched when the server returns the real ID. In onSuccess, replace the temp ID with the real server ID. In onError, rollback to the snapshot. Show a subtle loading indicator on the optimistic item.

## Q6: How does cursor-based pagination with useInfiniteQuery handle cache invalidation when a new item is inserted into the first page?
**A:** When a new item is inserted at the beginning of a cursor-based list, all existing page caches are offset by one. Four approaches: (A) Invalidate all pages - discards cached pages, re-fetches from scratch. (B) Manually prepend to page 1 via setQueryData - works if subsequent pages' cursors are still valid. (C) Re-fetch only page 1, invalidate remaining pages. (D) Use getPreviousPageParam to prepend - if API supports backward pagination. For real-time apps, approach B is most efficient. The safest: invalidate all pages and use fetchNextPage lazily as user scrolls.

## Q7: How do you implement a query invalidation strategy that avoids the waterfall problem when multiple mutations invalidate overlapping queries?
**A:** The waterfall problem: mutation A invalidates ['data', 1], mutation B invalidates ['data', 2], causing sequential re-fetches. Solutions: (1) Batch invalidations - wrap in unstable_batchedUpdates or invalidateQueries with a predicate: predicate: query => query.queryKey[0] === 'data'. (2) Debounced invalidation - debounce with 100ms window, fires once after batch completes. (3) Global mutation observer - subscribe to getMutationCache() events, invalidate based on meta tags: meta: { invalidates: ['list1', 'list2'] }. (4) Async batch invalidation - use Promise.all to invalidate simultaneously.

## Q8: How does React Query's hydration work with Next.js's App Router (server components)?
**A:** Next.js App Router: Create a QueryClient in the server component, prefetch data, dehydrate: const queryClient = new QueryClient(); await queryClient.prefetchQuery(['data'], fetcher); const dehydratedState = dehydrate(queryClient). Pass dehydratedState to a client component wrapped with HydrationBoundary. Inside the client component, use useQuery normally - it sees pre-filled cache. Create a new QueryClient per request (not singleton) to avoid cross-request leakage. For streaming SSR, use prefetchQuery without await. React Query v5 integrates with React's cache() for memoizing QueryClient creation per request.

## Q9: How do you implement React Query with TypeScript using discriminated union types for query results?

Create a typed hook per domain: useUserQuery = (id: string) => useQuery({ queryKey: ['users', id], queryFn: fetchUser(id) }). Type mutations: useMutation<ResponseType, ErrorType, VariablesType>. Use the satisfies operator in TS 4.9+: if (status === 'success') { data satisfies User; }. Create typed query key factories for consistent type inference across the app.

## Q12: How do you implement a retry with exponential backoff and jitter strategy for mutations that handles transient vs permanent failures?
**A:** For mutations, retry is not automatic. Implement a wrapper: async function withRetry(fn, opts) { for (let i = 0; i <= opts.retries; i++) { try { return await fn() } catch (e) { if (i === opts.retries) throw e; if (!isRetriable(e)) throw e; const delay = Math.min(opts.baseDelay * 2**i + Math.random() * 1000, opts.maxDelay); await new Promise(r => setTimeout(r, delay)) } } }. isRetriable: retry 5xx and network errors, not 4xx except 429. For 429, use Retry-After header. Jitter prevents thundering herd. Track retry count in mutation meta for UI display.

## Q15: How do you implement staleTime and gcTime strategies for an e-commerce product listing with real-time inventory?

## Q18: How does React Query's resetQueries differ from removeQueries and when should each be used?
**A:** resetQueries: resets to initial state (data: undefined, error: null, status: 'loading'). Query survives, refetches on next access. Use after logout, clearing filters, or undo operations. removeQueries: completely removes from cache - no trace remains. Use for permanent cleanup, freeing memory, clearing sensitive data on logout. resetQueries preserves query key and observer; removeQueries destroys them. After removeQueries, next useQuery recreates from scratch. After resetQueries, next access triggers immediate refetch with loading state.

## Q19: How do you implement optimistic updates with rollback for a mutation that can fail after multiple side effects?
**A:** Take cache snapshot before mutation: const snapshot = queryClient.getQueriesData({ queryKey: ['related-data'] }). On onSettled, invalidate affected queries. Track mutation state via isPending. For concurrent mutations, use mutation keys for correct rollback. Use mutateAsync in try/catch pattern. Snapshot capture must be inside onMutate.

## Q22: How does React Query handle focus refetching in a multi-tab application and how can conflicts be resolved?

## Q23: How do you use React Query's select option to derive data without losing referential stability?
**A:** The select function MUST be memoized - wrap in useCallback or define outside component. React Query compares selected values using referential equality (===). If select creates a new array/object each render, component re-renders even if data hasn't changed. Structural sharing applies inside the query but NOT inside select. Solution 1: notifyOnChangeProps: ['data'] + memoized selector. Solution 2: useMemo on the query result: const names = useMemo(() => data?.map(u => u.name), [data]). Solution 3: Stable reference ensures selector runs only when dataUpdatedAt changes.

## Q24: How do you implement pessimistic updates with React Query and when should you prefer them over optimistic?
**A:** Pessimistic: call mutation -> show loading -> server responds -> update cache -> show success. Implementation: onSuccess updates setQueryData, onError shows error. Prefer pessimistic for: financial transactions (incorrect balance is confusing), collaborative apps (concurrent conflicts), complex validation (unpredictable server rules), low mutation failure rate, regulatory compliance. Optimistic is better for: frequent mutations, low risk, instant feedback (likes, toggles). Hybrid: show optimistic UI with subtle 'saving...' indicator and rollback on failure.

## Q26: How do you implement query cancellation for cleanup when React components unmount?

## Q28: How does React Query's structural sharing work and what are its implications for immutable data patterns?
**A:** Structural sharing deep-compares new response with cached data. If a sub-object is deeply equal to cached version, React Query reuses the cached reference. Example: response { users: [...], meta: { page: 1 } } - if meta hasn't changed, the meta reference is reused. Implications: components get new references only for changed sub-trees, optimizing re-renders. Expects IMMUTABLE data - mutating cache directly breaks structural sharing. Only works with JSON-serializable data. Clones internally - has overhead for large payloads. For >1MB, consider disabling with structuralSharing: false.

## Q33: How does React Query handle query retry when the server returns specific HTTP status codes?
**A:** Default retry: 3 times on ANY error.
