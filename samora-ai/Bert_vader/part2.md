# BERT and VADER Interview Questions and Answers - Part 2

## Q1: How does BERT's multi-head attention mechanism work in detail?
**A:** Multi-head attention runs multiple scaled dot-product attention operations in parallel. Each head: 1) Projects input into Query (Q), Key (K), Value (V) subspaces via learned weight matrices. 2) Computes attention scores: `softmax(Q * K^T / sqrt(d_k)) * V`. 3) Different heads learn different relationships (syntax, semantics, position). 4) BERT-base has 12 heads, BERT-large has 16. Concatenates all heads' outputs and projects through a final linear layer. This allows the model to attend to information from different representation subspaces at different positions simultaneously.

## Q2: Explain BERT's transformer encoder layer architecture (layer norm placement, feed-forward, residual connections).
**A:** Each encoder layer: 1) Multi-head self-attention (with residual connection). 2) Add & LayerNorm: `LayerNorm(x + Attention(x))`. 3) Feed-forward network: two linear layers with GeLU activation (typically 4x hidden size: 768 -> 3072 -> 768 in BERT-base). 4) Add & LayerNorm: `LayerNorm(x + FFN(x))`. BERT uses pre-layer norm (LayerNorm before sub-layer) in original implementation. Dropout (0.1) on attention weights, FFN outputs, and embeddings. Residual connections allow gradients to flow through 12/24 layers without vanishing.

## Q3: What is the difference between WordPiece (BERT) and BPE (GPT/RoBERTa) tokenization?
**A:** WordPiece (BERT): builds vocabulary by maximizing the likelihood of training data using a language model. Starts with characters, iteratively merges pairs that maximize the likelihood increase. Uses `##` prefix for subword continuations. BPE (Byte-Pair Encoding, GPT/RoBERTa): merges the most frequent pair of bytes/tokens iteratively. No special continuation markers. WordPiece is more linguistically motivated (likelihood-based). BPE is simpler (frequency-based). BPE may produce more aggressive subword splits. Both handle OOV words by breaking into subwords. BERT's WordPiece vocabulary = ~30K tokens. RoBERTa's BPE = ~50K.

## Q4: How does learned positional encoding in BERT differ from sinusoidal positional encoding in the original Transformer?
**A:** BERT uses learned positional embeddings (randomly initialized, trained during pre-training). Original Transformer (Vaswani et al.) uses fixed sinusoidal positional encodings: `PE(pos, 2i) = sin(pos / 10000^(2i/d_model))`, `PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))`. Advantages of sinusoidal: 1) Can extrapolate to longer sequences than seen during training. 2) No additional parameters. Advantages of learned: 1) More flexible (can learn position-specific patterns). 2) Better performance in practice. BERT's learned positions are limited to 512 tokens (can't extrapolate). Learned embeddings outperform sinusoidal but limit maximum sequence length.

## Q5: How do you fine-tune BERT with a gradual unfreezing strategy?
**A:** Instead of fine-tuning all layers at once: 1) Freeze all BERT layers. Train only the classification head for 1 epoch. 2) Unfreeze last transformer layer. Train for 1 epoch. 3) Gradually unfreeze more layers from top to bottom each epoch. 4) Use discriminative learning rates: lower layers get lower lr (2e-6), higher layers get higher lr (2e-5). Rationale: lower layers capture general language features (should not change much), higher layers are more task-specific (need more adaptation). Prevents catastrophic forgetting. Implemented via: `param_group['lr'] = base_lr * (layer_discount ^ (num_layers - layer_idx))`. Typically 3-4 epochs suffice for full gradual unfreezing.

## Q6: How does RoBERTa's dynamic masking improve over BERT's static masking?
**A:** BERT pre-processes data once: 15% of tokens masked statically. The model sees the same masking pattern every epoch. RoBERTa generates a new masking pattern each time a sequence is fed to the model (dynamic masking). Benefits: 1) Model sees more diverse masking patterns across 500K steps. 2) Reduces risk of overfitting to specific mask positions. 3) More robust contextual representations. 4) Effectively more training data diversity. Implementation: replicate the dataset 10x, mask each copy differently (RoBERTa approach), or mask on-the-fly during data loading. Dynamic masking contributed ~0.5% to RoBERTa's improvements over BERT.

## Q7: How does ALBERT's cross-layer parameter sharing reduce memory while maintaining performance?
**A:** ALBERT shares all parameters across transformer layers (self-attention + FFN weights are identical in every layer). Benefits: 1) Dramatically fewer parameters: BERT-large has 340M, ALBERT-xxlarge has 235M (but 12M unique + 223M shared). 2) Stabilizes training (shared layers converge more consistently). 3) Enables deeper networks (ALBERT-xxlarge has 24 layers). Despite parameter sharing, ALBERT maintains competitive performance because: 1) Layer normalization differs per layer (different scaling after shared weights). 2) Each layer still has unique intermediate activations (shared weights process different inputs). 3) Shared layers act as repeated refinement, like an iterative refinement process.

## Q8: How does ELECTRA's replaced token detection pre-training differ from BERT's MLM?
**A:** ELECTRA uses a generator-discriminator framework: 1) Generator (small MLM BERT) replaces masked tokens with plausible alternatives. 2) Discriminator (ELECTRA model) learns to detect which tokens were replaced. 3) Pre-training objective: binary classification (replaced vs original) for EVERY token, not just masked ones. Benefits: 1) Model learns from all tokens (not just 15% masked). 2) Computationally more efficient (learns more per training step). 3) Outperforms BERT and RoBERTa on GLUE with same compute. 4) Discriminator can be directly fine-tuned for downstream tasks. ELECTRA achieves BERT-level performance with 1/4 of the training compute.

## Q9: How does VADER's lexicon construction work and what types of features does it include?
**A:** VADER's lexicon was constructed using a combination of: 1) Initial seed words from existing sentiment lexicons (LIWC, ANEW, GI). 2) Human raters (10+ independent raters) scored each word on a -4 (extremely negative) to +4 (extremely positive) scale. 3) Only words with non-zero mean and low standard deviation (rater agreement) were included. 4) ~7,500 features including: common words (good, bad, terrible), slang (sucks, blows), emoticons (:-), :-(), :D), emoji (❤️, 😞), acronyms (LOL, WTF, SMH), intensifiers (very, extremely, somewhat), negations (not, never, no), idioms (cutting edge). Each feature has a valence score (mean rating). The lexicon is manually curated for social media relevance.

## Q10: How do you build a hybrid BERT + VADER sentiment system?
**A:** Hybrid approaches combine speed (VADER) with accuracy (BERT). 1) Tiered: VADER processes all text (fast, ~1μs). If VADER confidence > threshold (e.g., |compound| > 0.8), use VADER result. If uncertain, pass to BERT for deeper analysis. 2) Ensemble: average VADER compound score + BERT probability (weighted). 3) Feature augmentation: use VADER scores (compound, pos, neg, neu) as additional features fed into BERT's classification head. 4) Weak supervision: use VADER to generate noisy labels for unlabeled data, then fine-tune BERT. 5) Cascading: VADER = first pass (99% of requests), BERT = second pass for flagged/challenging content. In production: tiered approach processes 90%+ of traffic with VADER, BERT only for ambiguous cases, reducing total compute cost by 10x.

## Q11: How do you deploy BERT with ONNX Runtime optimization?
**A:** 1) Export fine-tuned BERT to ONNX: `torch.onnx.export(model, dummy_input, "bert.onnx", opset_version=14)`. 2) Optimize with ONNX Runtime: session options (graph optimization level, execution provider). 3) Use ORT's transformers optimizer: `python -m onnxruntime.transformers.optimizer --model bert.onnx --output bert_opt.onnx --model_type bert`. 4) optimizations: layer fusion, constant folding, attention fusion, Gelu fusion. 5) Execution providers: CUDA (GPU), OpenVINO (Intel), DirectML (Windows). 6) Quantize INT8: `quantize_dynamic(bert_opt.onnx, bert_int8.onnx)`. Results: 2-4x inference speedup vs PyTorch eager mode, 4x smaller with INT8 quantization. ONNX Runtime can serve BERT-base in ~5ms on CPU (vs ~50ms PyTorch).

## Q12: What is the impact of model quantization on BERT accuracy?
**A:** Quantization reduces precision (FP32 -> INT8) to reduce model size and speed inference. Post-training quantization (PTQ): minimal accuracy loss (0.5-1% F1) for classification tasks. Quantization-aware training (QAT): trains with simulated quantization, almost no accuracy loss (<0.3%). Impact varies by task: 1) Text classification: very robust (0-1% drop). 2) NER: slightly more sensitive (1-2% drop). 3) QA (SQuAD): moderate sensitivity (1-3% EM/F1 drop). 4) Sentiment analysis: very robust (<0.5% drop). INT8 quantization = 4x smaller model (400MB -> 100MB), 2-4x inference speedup on CPU. Use QAT for production systems where accuracy matters. Use PTQ for rapid deployment. Mixed-precision (FP16) has negligible accuracy loss for most tasks and gives 2x speedup on GPU.

## Q13: How does BERT's attention visualization reveal model behavior?
**A:** Attention weights show which tokens the model focuses on when encoding each token. Visualization tools (BertViz, exBERT, attention heads viewer): 1) Head view: shows attention patterns for a single attention head. 2) Layer view: shows attention across all layers. 3) Neuron view: shows which neurons activate for specific inputs. Patterns: lower layers learn surface/syntactic patterns (adjacent words, same POS), middle layers learn semantic relationships (pronoun-antecedent, subject-verb), higher layers learn task-specific patterns. Attention can reveal: 1) How the model resolves coreference. 2) Which words influence the [CLS] token. 3) If attention is focused on [SEP] tokens (model not learning properly). Attention maps help debug model predictions and identify spurious correlations.

## Q14: How do you adapt BERT for a domain with very different vocabulary (e.g., legal, medical)?
**A:** Domain adaptation strategies: 1) Continue pre-training (domain-adaptive pre-training): continue MLM training on domain-specific unlabeled data before fine-tuning. Most effective for vocabulary shift. 2) Vocabulary extension: add domain-specific tokens to BERT's vocabulary (medical terms, legal jargon), initialize embeddings randomly, continue pre-training. 3) Use domain-specific BERT: BioBERT (biomedical), FinBERT (financial), Legal-BERT, SciBERT (scientific). 4) Adapter-based fine-tuning: train small adapter layers per domain while keeping BERT frozen. 5) For VADER: extend the lexicon with domain-specific terms and their sentiment scores. Continue pre-training for 50K-100K steps on domain data typically recovers most of the domain adaptation gap.

## Q15: How do you evaluate BERT interpretability using SHAP or Integrated Gradients?
**A:** Interpretability methods attribute predictions to input tokens: 1) SHAP (SHapley Additive exPlanations): computes each token's contribution using game-theoretic Shapley values. Partition SHAP handles transformer attention. Output: per-token attribution scores. 2) Integrated Gradients: integrates gradients along the path from baseline (e.g., zero embedding) to input. Provides token-level importance. 3) Layer-wise Relevance Propagation (LRP): backpropagates relevance from output to input. 4) Attention attribution: weight * gradient product. Use to: 1) Verify model focuses on sentiment-bearing words. 2) Detect spurious correlations. 3) Debug misclassifications (which tokens drove the wrong prediction). 4) Build trust with stakeholders. SHAP is more theoretically grounded but slower; Integrated Gradients is faster and commonly used.

## Q16: How does knowledge distillation (DistilBERT, TinyBERT, MobileBERT) work for BERT?
**A:** Knowledge distillation trains a smaller "student" model to mimic a larger "teacher" BERT. Loss function combines: 1) Soft target loss: student's softmax probabilities match teacher's (temperature-scaled). 2) Hidden state loss: student's intermediate representations match teacher's (MSE). 3) Attention loss: student's attention matrices match teacher's. 4) Hard target loss: standard cross-entropy with true labels. DistilBERT: 40% smaller, 60% faster, retains 97% performance. Loss: soft target + masked LM loss. TinyBERT: 7.5x faster, 9.4x smaller. Uses intermediate layer distillation + prediction layer distillation. MobileBERT: bottleneck architecture for mobile devices. Distillation typically trains on the teacher's pre-training data (not task-specific data) for best results.

## Q17: How does BERT's attention handle long-range dependencies beyond 512 tokens?
**A:** BERT's self-attention is O(n^2) in sequence length, limiting it to 512 tokens. Alternatives for longer texts: 1) Sliding window: process 512-token chunks with overlap, aggregate predictions (voting/average). 2) Hierarchical: encode sentences independently, then encode sentence representations with another transformer. 3) Longformer: replaces O(n^2) attention with sliding window attention (local) + global attention on specific tokens. Handles up to 4096 tokens. 4) BigBird: sparse attention (random + window + global), O(n) complexity. 5) Reformer: LSH (locality-sensitive hashing) attention, O(n log n). 6) Performer: Fast attention via kernel approximation (FAVOR+), O(n). For production, Longformer and BigBird are common choices for long-document tasks.

## Q18: How do you handle the [CLS] token pooling strategy for sentence embeddings?
**A:** BERT's [CLS] token is the default sentence representation. Issues: [CLS] embeddings are not sentence-level optimized (pre-trained on MLM+NSP, not similarity). Better pooling strategies: 1) Mean pooling: average all token embeddings (last hidden layer). Works well for sentence similarity. 2) Max pooling: element-wise max across tokens. 3) Weighted pooling: attention-weighted average (learnable weights). 4) Concatenate last 4 layers: sum/concat hidden states from last 4 layers (captures different abstraction levels). 5) CLS + mean: concatenate CLS and mean-pooled embeddings. Sentence-BERT uses mean pooling on top of siamese BERT with contrastive loss, significantly outperforming raw [CLS] for semantic similarity. For sentiment, [CLS] or mean pooling both work well.

## Q19: How does VADER's sentiment scoring handle punctuation and emphasis?
**A:** VADER has specific heuristics for punctuation: 1) Exclamation marks: increase sentiment intensity. If text ends with `!`, sentiment score is boosted (multiplier = 0.292). Multiple `!!` increase boost proportionally. 2) Question marks: reduce sentiment intensity. If text ends with `?`, reduce. 3) CAPITALIZATION: ALL CAPS words get sentiment intensity increased (multiplier = 0.733). 4) Combined emphasis: "BEAUTIFUL!!!" gets both CAPS boost + punctuation boost. 5) Degree modifiers: "sort of good" < "good" < "very good" < "extremely good". VADER's booster dictionary assigns different multipliers to different intensifiers. These heuristics are based on empirical observations of social media text patterns.

## Q20: How do you implement BERT-based aspect-based sentiment analysis (ABSA)?
**A:** ABSA identifies sentiment towards specific aspects. BERT approaches: 1) BERT for aspect extraction (sequence labeling: B-ASPECT, I-ASPECT, O) + BERT for sentiment classification per aspect. 2) BERT-ADA (Aspect-Dependent Attention): incorporate aspect embedding into BERT's attention. 3) BERT + CRF for joint extraction + classification. 4) Span-based: predict aspect spans and sentiment labels simultaneously. 5) Prompt-based: "The sentiment of [ASPECT] in this review is [MASK]". Architecture: 1) Tokenize "The food was great but service was slow". 2) Identify aspect tokens "food" and "service". 3) For each aspect, classify sentiment (positive/negative). BERT-ADA passes aspect embedding through an attention layer that focuses on context relevant to that aspect. State-of-the-art on SemEval ABSA benchmarks.

## Q21: How does BERT's layer-wise learning rate decay work in practice?
**A:** Lower BERT layers capture universal language features (should change little during fine-tuning). Higher layers are more task-specific (should change more). Layer-wise decay: 1) Set base learning rate for the classification head (e.g., 5e-5). 2) Apply decay factor (e.g., 0.95) per layer from top to bottom. 3) Layer 12 (top): lr = base * 1.0 = 5e-5. Layer 11: lr = 5e-5 * 0.95 = 4.75e-5. Layer 10: lr = 5e-5 * 0.95^2 = 4.51e-5. Layer 1 (bottom): lr = 5e-5 * 0.95^11 = 2.87e-5. 4) Embedding layer: same as bottom layer. Implementation in Hugging Face: group parameters by layer, assign different learning rates. Benefits: prevents catastrophic forgetting of general language knowledge, better on small datasets.

## Q22: How do you handle multi-label sentiment classification with BERT?
**A:** Multi-label (e.g., "bittersweet" = both positive and negative): 1) Replace softmax with sigmoid activation on output layer. 2) Use binary cross-entropy loss (BCEWithLogitsLoss) instead of cross-entropy. 3) Each label is an independent binary classifier. 4) Threshold (typically 0.5) per label determines if that label is active. 5) Example: output = [0.8, 0.1, 0.6] for [positive, neutral, negative]. Positive and negative both active (bittersweet). 6) Evaluation: label-based F1 (macro/micro) or exact match accuracy. 7) Data: each sample has multiple binary labels. BERT's [CLS] -> linear layer with `num_labels` outputs -> sigmoid. For multi-label with hierarchical labels, use cascading classifiers or graph neural networks.

## Q23: How do you perform model pruning on BERT for deployment?
**A:** Pruning removes unimportant weights to reduce model size and speed inference. Techniques: 1) Magnitude pruning: remove weights with smallest absolute values (0% to 90% sparsity). 2) Structured pruning: remove entire attention heads or FFN neurons (hardware-friendly). 3) Movement pruning: remove weights based on their movement during training (better than magnitude). 4) Lottery Ticket Hypothesis: find sparse subnetwork that can train to similar accuracy. Results: 40-60% of BERT weights can be pruned with <1% accuracy loss. Structured pruning of entire attention heads: BERT can lose up to 40% of heads with minimal impact. After pruning, fine-tune remaining weights to recover accuracy. Pruning + quantization = 8-10x compression.

## Q24: How does VADER handle sarcasm and why does it fail?
**A:** VADER fails at sarcasm because it relies on word-level sentiment without understanding context or contradiction. Example: "Great, another meeting that could have been an email." VADER sees "great" (positive) and "meeting" (neutral) -> positive score. But the sentence is sarcastic (negative). Why VADER fails: 1) No contextual understanding (can't detect positive words used in negative contexts). 2) No discourse analysis (can't detect contradiction between words and situation). 3) Fixed lexicon (doesn't learn that "great" is negative when paired with complaints). 4) No pragmatic understanding (can't detect that "I love waiting" is sarcastic). BERT captures these patterns: it learns from data that positive words in certain contexts indicate negative sentiment.

## Q25: How do you train BERT from scratch for a specific domain/language?
**A:** Training BERT from scratch: 1) Data: 3-10B+ tokens of domain text (e.g., all PubMed articles for biomedical). 2) Tokenizer: train WordPiece tokenizer on domain data (vocab size: ~30K). 3) Pre-training: MLM (15% masking) + NSP (optional) for 500K-1M steps. 4) Hardware: 16-64 TPUs/GPUs (TPUv3-8 for BERT-base takes ~4 days). 5) Optimization: AdamW, linear warmup (10K steps), linear decay, batch size 256-4096. 6) Sequence length: 128 for 90% of steps, 512 for last 10%. 7) Mixed precision (FP16) for speed. Alternatives instead of training from scratch: 1) Domain-adaptive pre-training (DAPT): continue pre-training from existing BERT. 2) Vocabulary extension: add domain tokens to existing BERT. DAPT with 100K steps typically recovers most of the benefit of training from scratch.

## Q26: How does Sentence-BERT (SBERT) differ from BERT for semantic similarity?
**A:** SBERT modifies BERT for efficient semantic similarity: 1) Siamese/contrastive architecture: two BERTs share weights (or same BERT encodes two sentences). 2) Pooling: mean pooling on token embeddings (not [CLS]). 3) Loss functions: triplet loss (anchor, positive, negative), cosine similarity loss, MSE for STS. 4) Inference: encode sentence once -> embedding vector. Similarity = cosine(embedding1, embedding2). Benefits: 1) Speed: 10,000+ sentences/second on GPU (vs BERT's 1 comparison at a time). 2) Semantic search: use FAISS for nearest neighbor search over embeddings. 3) Clustering: group similar sentences. 4) Superior to raw BERT [CLS] embeddings. SBERT achieves best results on STS benchmarks. Use SBERT for semantic search, clustering, paraphrase detection.

## Q27: How do you implement cross-lingual sentiment analysis with mBERT?
**A:** Multilingual BERT (mBERT) trained on 104 languages. For cross-lingual sentiment: 1) Fine-tune mBERT on sentiment data in one language (e.g., English). 2) Zero-shot transfer: evaluate on target language (e.g., Spanish) without any target-language training data. 3) Performance: typically 70-85% of in-language fine-tuning performance. 4) For better results: add target-language unlabeled data during fine-tuning (unsupervised adaptation). 5) Language-specific tokens: mBERT handles all languages with shared vocabulary (WordPiece on 104 languages). 6) XLM-RoBERTa is better for cross-lingual transfer (larger vocab, trained on 2.5TB data across 100 languages). 7) Translate-train: translate English training data to target language, fine-tune. Sometimes beats zero-shot. Best: XLM-R with translate-train augmentation.

## Q28: How does BERT handle negation in sentiment analysis compared to VADER?
**A:** BERT captures negation through its bidirectional attention mechanism. For "The movie was not good": 1) BERT's self-attention connects "not" with "good" directly (attends to both). 2) BERT learns from data that "not good" often co-occurs with negative sentiment labels. 3) No explicit rule needed: the attention weights automatically learn the scope of negation. 4) Can handle complex negation: "I wouldn't say it's bad" (not + bad = positive, but double negative). 5) Implicit negation: "The movie failed to impress" (failed negates impress). VADER: uses a rule-based tri-gram approach: if negation word within 3 tokens before sentiment word, flip polarity. Simpler but doesn't handle: 1) Long-distance negation. 2) Double negatives. 3) Implicit negation. 4) Context-dependent negation. BERT's learned approach is more robust.

## Q29: How do you implement active learning for BERT fine-tuning with minimal labeled data?
**A:** Active learning reduces labeling effort by selecting the most informative examples. Strategies: 1) Uncertainty sampling: pick examples where model is most uncertain (highest entropy, lowest max probability). 2) Margin sampling: pick examples with smallest difference between top 2 predicted probabilities. 3) Diversity sampling: pick examples that cover diverse regions of embedding space (k-means centers). 4) Query-by-committee: train multiple models, pick examples with highest disagreement. Workflow: 1) Start with small labeled set (100 examples). 2) Train BERT. 3) Predict on large unlabeled pool. 4) Select N most informative examples (e.g., 50). 5) Human labels them. 6) Add to labeled set, retrain. 7) Repeat until performance plateaus. Results: can achieve 90%+ of full-data accuracy with only 10-20% of training data.

## Q30: How does BERTScore compute similarity between generated and reference text?
**A:** BERTScore evaluates text generation by computing token-level similarity using BERT embeddings. Steps: 1) Encode reference and candidate texts with BERT. 2) Compute pairwise cosine similarity between all token embeddings of reference (r_i) and candidate (c_j). 3) Precision: max similarity for each candidate token over reference tokens. 4) Recall: max similarity for each reference token over candidate tokens. 5) F1: harmonic mean. 6) Optional: IDF weighting (rare words matter more). Benefits over BLEU/ROUGE: 1) Captures semantic similarity (not just n-gram overlap). 2) Handles paraphrasing. 3) Correlates better with human judgment. BERTScore with RoBERTa-large achieves the highest correlation with human evaluation for text generation tasks. Used in: machine translation, summarization, image captioning evaluation.

## Q31: How do you implement a BERT sentiment classifier that provides confidence-calibrated probabilities?
**A:** BERT's softmax outputs are often overconfident. Calibration techniques: 1) Temperature scaling: divide logits by temperature T > 1 (learned on validation set). `softmax(logits / T)`. Higher T = more uniform (less confident). 2) Label smoothing: replace hard labels (0/1) with smoothed (e.g., 0.1/0.9). Reduces overconfidence. 3) MC Dropout: enable dropout at inference, run forward passes N times (e.g., 10), average predictions. Variance = uncertainty. 4) Deep Ensemble: train N BERT models with different seeds, average predictions. 5) Dirichlet calibration: learn Dirichlet distribution parameters for better uncertainty. In production: temperature scaling is simplest and effective (adds no inference overhead). MC Dropout provides uncertainty estimates (for OOD detection). Use expected calibration error (ECE) to measure calibration quality.

## Q32: How does VADER's compound score formula work and how can you modify it?
**A:** VADER compound score = `sum / sqrt(sum^2 + alpha)`, where `sum` is the sum of all adjusted valence scores, alpha = 15 (normalization constant). This compresses the raw sum into [-1, +1]. Why 15: empirical tuning on social media text - produces good separation between positive/negative/neutral. To modify: 1) Change alpha: lower alpha = more binary (push scores toward -1/1). Higher alpha = more neutral (compress toward 0). 2) Change thresholds: standard > 0.05 = positive, < -0.05 = negative, in between = neutral. For stricter sentiment: > 0.2 = positive, < -0.2 = negative. 3) Modify lexicon: update valence scores for specific words. 4) Add custom boosters: add new intensifiers with custom multipliers. Customizing thresholds is the most common modification for domain adaptation.

## Q33: How do you implement a BERT service for real-time sentiment inference with high throughput?
**A:** Production BERT serving: 1) Model optimization: ONNX Runtime (CPU) or TensorRT (GPU), FP16/INT8 quantization. 2) Batching: dynamic batching (collect requests within 10ms window, process as batch). GPU: batch size 32-64 for optimal throughput. 3) Model server: Triton Inference Server (NVIDIA) or TorchServe. Supports model pipelining, concurrent model execution. 4) Horizontal scaling: multiple model replicas behind load balancer. 5) Caching: cache frequent inference results (Redis). 6) Request queuing: buffer requests during traffic spikes. 7) Cold start: keep models warm with keep-alive requests. 8) Graceful degradation: fallback to VADER if BERT unavailable. Throughput: BERT-base FP16 on A10G = ~1000 inferences/sec (batch=32). DistilBERT = ~3000 inferences/sec. BERT INT8 on CPU = ~200 inferences/sec.

## Q34: How does BERT's embedding visualization (t-SNE/UMAP) help understand model behavior?
**A:** Dimensionality reduction (t-SNE, UMAP) of BERT embeddings reveals: 1) [CLS] token embeddings of sentiment classes should form distinct clusters (positive vs negative separated). 2) Toxic vs non-toxic separation. 3) Aspect categories: embeddings of different aspects (food, service, ambiance) cluster separately. 4) Out-of-distribution detection: OOD inputs embed far from training data clusters. 5) Language patterns: similar sentiment phrases (e.g., "very good" and "excellent") close in embedding space. 6) Ambiguity: ambiguous inputs embed between clusters. Use for: 1) Qualitative evaluation (are embeddings as expected?). 2) Debugging (why did this misclassify?). 3) Hard negative mining (find examples near decision boundary). 4) Data leakage detection. UMAP is preferred over t-SNE for production (faster, preserves global structure better).

## Q35: How do you implement contrastive learning for BERT sentiment analysis?
**A:** Contrastive learning improves BERT's representation quality by pulling similar pairs together and pushing dissimilar pairs apart: 1) Siamese BERT: encode two sentences with same BERT (shared weights). 2) Positive pairs: same-sentiment examples, paraphrases, augmented versions (back-translation, synonym replacement). 3) Negative pairs: different-sentiment examples, hard negatives (same topic, opposite sentiment). 4) Loss: NT-Xent (Normalized Temperature-scaled Cross-Entropy), triplet loss, or supervised contrastive loss (`SupConLoss`). 5) Temperature: controls separation (lower = harder separation). 6) Training: first pre-train with contrastive loss, then fine-tune with classification loss. 7) Benefits: more robust embeddings, better performance with limited data, improved OOD detection. SimCLR, SupCon, and ConSERT are common contrastive frameworks for BERT.

## Q36: How does VADER handle the word "but" as a contrastive conjunction?
**A:** VADER's "but" heuristic: the clause after "but" gets higher weight in the overall sentiment score. Example: "The food was good but the service was terrible." Without "but" handling: the sum would roughly cancel out (good + terrible ~ 0). With "but": the sentiment after "but" (terrible) is weighted more heavily, making the compound score negative. Implementation: VADER detects "but" in the text, then applies a multiplier (>1.0) to words after "but" and/or a discount to words before. This captures the linguistic pattern that "but" signals the main sentiment of the sentence. The exact multiplier was empirically tuned on social media data. This is one of VADER's most valuable heuristics - many sentiment analysis systems have a similar "but reversal" rule.

## Q37: How do you implement a BERT regression model for continuous sentiment scores?
**A:** Instead of classification (positive/negative/neutral), regression predicts a continuous sentiment score (e.g., -1 to +1, or 1-5 stars). Implementation: 1) Replace classification head with a single linear layer outputting a scalar. 2) Loss: MSE (Mean Squared Error) or MAE (Mean Absolute Error). 3) Evaluation: Pearson correlation, Spearman correlation, MAE. 4) Output range: optional tanh/sigmoid to constrain output (tanh -> -1 to 1). 5) Data: use VADER compound scores or star ratings as regression targets. Challenges: 1) Regression is more sensitive to label noise. 2) Outliers have high impact. 3) Calibration: predicted score distribution should match target distribution. BERT regression typically outperforms classification + mapping to scores because it treats the sentiment scale as continuous. Use MSE loss with robust scaling of targets.

## Q38: How does the attention mechanism in BERT handle polysemy (words with multiple meanings)?
**A:** BERT's contextualized embeddings inherently handle polysemy because the same word's embedding changes based on context. "Bank" in "river bank" vs "money bank": 1) Both tokenized as "bank" (same token). 2) Self-attention: in "river bank", "bank" attends to "river" (and vice versa). In "money bank", "bank" attends to "money". 3) The hidden state of "bank" in layer 1 is the same. By layers 8-12, the representation diverges based on attended context. 4) Attention heads learn different patterns: some heads focus on syntactic dependencies, others on semantic relatedness. 5) This allows BERT to assign different sentiment to "sick" (ill = negative, cool = positive) based on context. VADER has a single score for "sick" (can't handle polysemy), which is a key limitation.

## Q39: How do you implement zero-shot sentiment analysis with BART or T5?
**A:** Zero-shot sentiment uses natural language inference (NLI) or text generation models. BART-based: 1) Hugging Face pipeline: `classifier("Great movie!", candidate_labels=["positive", "negative"])`. 2) Converts to NLI: premise = "Great movie!", hypothesis = "This text expresses positive sentiment." 3) BART MNLI model computes entailment score for each hypothesis. 4) Softmax over entailment scores gives sentiment probabilities. T5-based: 1) Prompt: "sentiment: Great movie!" -> generates "positive". 2) No task-specific fine-tuning needed. Performance: BART zero-shot is ~85-90% of fine-tuned BERT on standard benchmarks. Best for: 1) Rapid prototyping without labeled data. 2) New sentiment categories not seen during training. 3) Multi-lingual (using multilingual NLI models). Limitations: 1) Slower (NLI requires 2 forward passes per label). 2) Sensitive to prompt wording. 3) Struggles with domain-specific sentiment.

## Q40: How do you implement a BERT model that handles both sentiment and emotion detection?
**A:** Multi-task learning for sentiment + emotion: 1) Shared BERT encoder (all layers). 2) Two task-specific heads: sentiment head (3 classes: pos/neg/neu) and emotion head (6+ classes: joy, anger, sadness, fear, surprise, disgust). 3) Joint loss: `L_total = alpha * L_sentiment + (1-alpha) * L_emotion`. 4) Alpha controls importance weighting (typically 0.5-0.7 for sentiment). 5) Training: alternately sample batches from sentiment and emotion datasets. 6) Alternative: hierarchical output - predict sentiment first (positive/negative), then emotion within that sentiment (joy within positive, anger within negative). 7) Multi-label: some emotions co-occur (joy + surprise). Benefits: 1) Shared representation improves both tasks. 2) Emotion detection provides finer-grained analysis. 3) Single model reduces deployment complexity. Use GoEmotions dataset for emotion (27 fine-grained categories).

## Q41: How does BERT's hidden state dimensionality affect task performance?
**A:** BERT-base: 768 hidden dim, 12 layers. BERT-large: 1024 dim, 24 layers. Impact: 1) Larger hidden dim = more capacity = better on complex tasks (QA, NLI) but more parameters and compute. 2) DistilBERT: 768 dim, 6 layers - similar performance on simple tasks (sentiment classification) but drops on complex tasks. 3) TinyBERT: 312 dim, 4 layers - significant drop on reasoning tasks. For sentiment analysis: BERT-base (768 dim) is sufficient for most cases. Dim >= 512 is needed for reasonable sentiment accuracy. Tradeoffs: 1) Inference time scales linearly with dim * layers. 2) Memory: BERT-base (110M params) vs BERT-large (340M). For production sentiment: DistilBERT (768 dim, 6 layers) offers the best accuracy/speed tradeoff. For aspect-based: BERT-base or RoBERTa-base typically needed.

## Q42: How do you implement a custom VADER lexicon for a specialized domain?
**A:** Customizing VADER for a domain (e.g., finance, medical): 1) Collect domain-specific sentiment words from domain text. 2) Rate each word on -4 to +4 scale by multiple domain experts. 3) Add words to VADER lexicon: `analyzer.lexicon.update({"bullish": 2.5, "downgrade": -2.0})`. 4) Add custom boosters/negations if needed. 5) For financial: add "beat earnings" (positive), "missed estimates" (negative), "volatile" (negative). 6) For medical: "efficacious" (positive), "adverse" (negative), "prognosis" + "positive" -> compound phrase handling. 7) Test against domain-specific labeled data. 8) Adjust thresholds: financial text may need stricter thresholds (subtle sentiment). Custom VADER can approach the accuracy of simple ML models for domain-specific tasks while maintaining speed and interpretability.

## Q43: How does BERT's subword tokenization affect sentiment of compound words?
**A:** WordPiece breaks words into subwords. Impact on sentiment: 1) "unhappy" -> ["un", "happy"]. "happy" has positive sentiment, "un" is a negation prefix. BERT learns this compositionally. 2) "heartbroken" -> ["heart", "break", "##en"]. The combined meaning is learned, not just individual subwords. 3) "not_good" - if tokenized as ["not", "good"], BERT learns from data that this pair signals negative sentiment. 4) Emoji: some tokenizers split emoji into bytes (loss of semantic meaning). Better to use emoji-aware tokenizers or keep as special tokens. 5) Misspellings: "gooood" -> ["go", "##oo", "##od"]. BERT may not recognize the emphasized sentiment. Preprocessing misspellings before BERT can help. Subword tokenization is generally beneficial: it handles OOV words by decomposing into known subwords with sentiment-bearing components.

## Q44: How do you implement data augmentation for BERT sentiment using masked language modeling?
**A:** BERT-based augmentation uses MLM to generate diverse training samples: 1) Select sentences from training data. 2) Mask random tokens (not sentiment-bearing words). 3) Use BERT to predict replacements (top-k candidates). 4) Generate augmented sentences with different word choices but preserved sentiment. 5) Filter: keep augmentations where BERT's sentiment prediction matches the original label. 6) Mix: combine original + augmented data for training. Benefits: 1) Preserves sentiment (unlike back-translation which may change nuance). 2) Increases lexical diversity. 3) Improves robustness. Example: "The movie was great" -> mask "great" -> BERT predicts "amazing", "fantastic", "wonderful" -> "The movie was amazing". Implementation: `transformers.BertForMaskedLM` to predict masked tokens with temperature sampling. A typical augmentation rate is 10-30% of training data.

## Q45: How does VADER handle emoji sentiment compared to BERT?
**A:** VADER: includes emoji in its lexicon with pre-assigned sentiment scores (❤️ = +2.5, 😞 = -2.0). Emoticons also included (:) = +1.5, :( = -1.5). This makes VADER effective for social media where emoji convey significant sentiment. BERT: if the tokenizer was trained on text with emoji, BERT learns emoji embeddings from context. However: 1) Many BERT tokenizers split emoji into byte pairs (no semantic meaning). 2) BERT may not have seen enough emoji training data. 3) For BERT to handle emoji well, fine-tune on data containing emoji (tweets, comments). BERTweet (trained on 850M tweets) handles emoji naturally. Best practice: use a dedicated emoji-to-text converter (e.g., emoji -> ":heart:" description) or keep emoji as special tokens.

## Q46: How do you implement a BERT-based system for target-dependent sentiment analysis?
**A:** Target-dependent sentiment: sentiment towards a specific entity/target in the text. Approaches: 1) Target injection: prepend target to input: `[CLS] target [SEP] sentence [SEP]`. BERT learns to focus on target-relevant context. 2) Target-specific attention: incorporate target embedding into attention computation. 3) End-to-end: use BERT to extract target spans + classify sentiment per span. 4) Prompting: "sentiment of [TARGET] in '[text]' is [MASK]". Example: "Apple's new iPhone is amazing but customer service is terrible." Target iPhone -> positive. Target customer service -> negative. BERT with target injection achieves >90% F1 on targeted sentiment benchmarks like SemEval 2014 Task 4 (Restaurant domain). The target injection approach is simplest and effective.

## Q47: How does BERT's attention head redundancy affect model efficiency?
**A:** Research shows that many BERT attention heads are redundant and can be pruned without significant accuracy loss. Findings: 1) 30-50% of heads in BERT-base can be removed with <1% accuracy drop on GLUE. 2) Some heads are task-specific (prune for one task, keep for another). 3) Pruning strategy: score heads by attention confidence (or gradient-based importance), remove lowest-scoring heads. 4) 12 heads -> 6 heads: 2x speedup, minimal accuracy loss on sentiment. 5) 12 heads -> 3 heads: 4x speedup, ~2-3% accuracy drop. Head Pruning: 1) Train with head importance masking (learn which heads to keep). 2) Iteratively prune and fine-tune. 3) Result: smaller, faster model without full retraining. Structured pruning (remove entire heads) is hardware-friendly and reduces memory.

## Q48: How do you implement a BERT ensemble for robust sentiment predictions?
**A:** Ensemble combines multiple BERT models for better accuracy and uncertainty estimation. Approaches: 1) Model diversity: different seeds (random initialization), different training data subsets (bagging), different architectures (BERT-base + RoBERTa-base), different pre-trained checkpoints. 2) Voting: majority vote (hard) or average probabilities (soft). 3) Stacking: meta-classifier trained on individual model outputs. 4) Bagging: train each model on bootstrap sample of training data. 5) Snapshot ensemble: single training run, save checkpoints at different epochs (cyclic LR). Benefits: 1) 1-2% accuracy improvement over single model. 2) Better calibration (ensemble uncertainty). 3) More robust to domain shift. Tradeoffs: 5x compute at inference (5 models). Use smaller models (DistilBERT) in ensemble to reduce cost. DistilBERT ensemble of 5 = 1 BERT-base cost, potentially better accuracy.

## Q49: How does VADER's handling of ALL CAPS compare to modern transformer models?
**A:** VADER has an explicit ALL CAPS rule: words in all capital letters get sentiment intensity multiplied by 0.733. This captures emphasis in social media ("AMAZING" > "amazing"). Transformers (BERT): don't have explicit casing rules. Cased models distinguish "AMAZING" vs "amazing" as different tokens (different embeddings). Uncased models lowercase everything, losing emphasis information. BERTweet (tweet-specific) preserves casing and learns emphasis patterns from data. Comparison: 1) VADER: rule-based, consistent but simplistic. 2) BERT cased: learns from data but may not generalize emphasis. 3) BERT uncased: loses emphasis entirely. Best practice: for social media sentiment, use BERTweet (cased, trained on tweets) which naturally learns capitalization patterns. For VADER: ALL CAPS handling is a well-tuned heuristic that works well in practice.

## Q50: How do you implement few-shot sentiment analysis using SetFit?
**A:** SetFit (Sentence Transformer Fine-tuning) for few-shot sentiment: 1) Load a sentence transformer (all-MiniLM-L6-v2). 2) Generate contrastive pairs from labeled data: same-label pairs (positive), different-label pairs (negative). 3) Fine-tune sentence transformer with contrastive loss (cosine similarity). 4) Train a classification head (logistic regression) on the embeddings. 5) Evaluate. Benefits: 1) Works with 8-64 examples per class. 2) No prompts or verbalizers needed. 3) Fast training (minutes). 4) Outperforms GPT-3 few-shot on many sentiment tasks. Process: with 16 examples per class (32 total for binary), SetFit achieves ~90% of full-data BERT performance. The key insight: contrastive fine-tuning on few examples creates task-specific embeddings that separate sentiment classes well. SetFit is the best choice for few-shot sentiment: simple, fast, effective.

## Q51: How does BERT handle emoji in sentiment analysis and what preprocessing is needed?
**A:** BERT's handling of emoji depends on the tokenizer and pre-training data. Issues: 1) Original BERT tokenizer (WordPiece) may split emoji into byte pairs (e.g., 😊 -> multiple byte tokens), losing semantic meaning. 2) Emoji are out-of-vocabulary for many BERT variants. 3) Solutions: a) Convert emoji to text descriptions: 😊 -> `:smiling_face:` (emoji library). b) Add emoji as special tokens in vocabulary. c) Use BERTweet (trained on tweets with emoji) - handles emoji naturally. d) Use Unicode normalization. For production: best to convert emoji to text descriptions or use a model pre-trained on emoji-rich data (BERTweet, RoBERTa-twitter). Emoji carry significant sentiment (especially in social media), so handling them properly is important for sentiment accuracy.

## Q52: How do you implement a BERT model that handles multiple languages simultaneously?
**A:** Multilingual BERT (mBERT) and XLM-RoBERTa support 100+ languages. Implementation: 1) Tokenize input with multilingual tokenizer (language-agnostic WordPiece). 2) Fine-tune on sentiment data in multiple languages simultaneously (joint training). 3) For language-specific logic: add language ID token or use language embeddings. 4) Mixed-language batches: each batch contains samples from different languages. 5) Performance: XLM-R is better than mBERT for cross-lingual transfer. Training data mix: balance languages (up-sample low-resource languages). 6) If sentiment categories differ by language (some languages have more nuanced sentiment vocabulary), use per-language classification heads sharing the BERT encoder. 7) Joint multilingual training often improves performance on each language (positive transfer). XLM-R large achieves 95%+ of language-specific BERT performance on each language.

## Q53: How does the VADER normalization constant alpha affect sentiment distributions?
**A:** The compound score uses: `compound = sum / sqrt(sum^2 + alpha)`. Alpha = 15 was empirically chosen for social media text. Effect: 1) Low alpha (e.g., 5): more binary scores (tends toward -1 or +1). Less neutral zone. 2) High alpha (e.g., 30): more compressed toward 0. More neutral scores. 3) Alpha = 0: compound = sign(sum) (pure binary). 4) Alpha = infinity: compound = 0 (all neutral). Visualization: with sum = 10, alpha = 15, compound = 10/sqrt(100+15) = 10/sqrt(115) = 0.93. With alpha = 5: 10/sqrt(100+5) = 10/sqrt(105) = 0.98. With alpha = 50: 10/sqrt(150) = 0.82. The choice of alpha affects how many texts fall into neutral vs strong sentiment. For domains with subtle sentiment (financial), higher alpha may be better. For social media, alpha = 15 works well.

## Q54: How do you implement BERT inference optimization using TensorRT?
**A:** TensorRT optimizes BERT for NVIDIA GPUs: 1) Convert fine-tuned BERT to ONNX. 2) Use TensorRT's ONNX parser to import. 3) Apply FP16 or INT8 precision. 4) Apply layer fusion: fuse GeLU, layer norm, attention components into single kernels. 5) Apply QKV fusion: combine Query, Key, Value projections into single GEMM. 6) Apply attention fusion: fused multi-head attention implementation. 7) Apply plugin: BERT-specific plugins (fused softmax, masked scaling). 8) Build TensorRT engine (serialized, loaded at runtime). Results on T4 GPU: BERT-base FP32: 8ms, FP16: 3ms, INT8: 2ms. TensorRT INT8 BERT is ~4x faster than PyTorch FP32. TensorRT also supports dynamic shapes (variable batch sizes/sequence lengths) with shape optimization profiles.

## Q55: How does BERT handle sentiment in code-mixed text (e.g., Hinglish)?
**A:** Code-mixed text (Hinglish: Hindi + English) challenges: 1) Models must handle mixed vocabulary. 2) Sentiment expression differs across languages. Approaches: 1) mBERT: multilingual embeddings handle both Hindi and English tokens. 2) XLM-RoBERTa: better for code-mixed due to larger vocabulary and cross-lingual training. 3) HingBERT: specifically pre-trained on Hinglish text. 4) Transliteration: normalize Romanized Hindi to Devanagari for consistency. 5) Character-level: handle OOV words at character level. 6) Data augmentation: create synthetic code-mixed data from parallel corpora. Performance: XLM-R achieves ~75% F1 on Hinglish sentiment benchmarks. Language-specific models (HingBERT) achieve ~80%. VADER is English-only and fails on Hinglish. Best practice: use XLM-R with code-mixed fine-tuning data.

## Q56: How do you measure and optimize BERT inference latency in production?
**A:** Latency optimization: 1) Quantize: FP16 (GPU) gives 2x speedup with <0.5% accuracy loss. INT8 (CPU/GPU) gives 3-4x speedup with ~1% loss. 2) Distill: DistilBERT is 2x faster than BERT-base with 97% performance. 3) Prune: remove 30-50% attention heads. 4) Batch: process requests in batches (higher throughput, not latency). 5) ONNX Runtime: 2x speedup over PyTorch eager mode. 6) TensorRT: 4x speedup on GPU. 7) Hardware: use inference-optimized instances (AWS Inferentia, NVIDIA T4/A10G). 8) Caching: cache frequent results. 9) Speculative execution: predict sentiment for common patterns. Typical latency targets: <50ms for real-time, <200ms for interactive. BERT-base INT8 on CPU ~10ms. DistilBERT INT8 ~5ms. TinyBERT INT8 ~2ms.

## Q57: How does BERT's tokenization handle hashtags and mentions in social media sentiment?
**A:** Social media preprocessing for BERT: 1) Hashtags (#HappyBirthday): segment into words (word segmentation: "Happy Birthday") or keep as single token. Word segmentation improves BERT understanding. 2) @mentions: replace with `@USER` token (reduces vocabulary, prevents overfitting to specific users). BERTweet uses this approach. 3) URLs: replace with `HTTPURL` token. 4) Emoji: convert to text or keep as special tokens. 5) Repeated punctuation ("!!!", "???") -> normalize. Tokenization impact: 1) BERT's WordPiece splits `#HappyBirthday` as `["#", "Happy", "Birthday"]`. `#` may not have been seen in pre-training. 2) Better: preprocess hashtags to `["happy", "birthday"]` (remove #, segment). BERTweet handles social media formatting natively (pre-trained on raw tweets). For general BERT: segment hashtags, replace mentions/URLs.

## Q58: How do you implement a BERT-based sentiment system with calibrated uncertainty for OOD detection?
**A:** Out-of-distribution (OOD) detection identifies inputs the model hasn't seen during training. Methods: 1) Softmax confidence: max softmax probability. OOD samples often have lower confidence. 2) MC Dropout: run forward pass N times (dropout enabled), variance across passes indicates uncertainty. 3) Distance-based: compute distance to training data in embedding space ([CLS] embeddings via Mahalanobis distance). 4) Energy-based: energy score = -log(sum(exp(logits_i))). Lower energy = more in-distribution. 5) ODIN: temperature scaling + small input perturbation for better OOD separation. Implementation: 1) Compute OOD score per input. 2) Threshold: reject inputs with score < threshold (send to human). 3) Calibrate threshold on validation OOD dataset. 4) In production: reject ~5% of inputs (most uncertain) to human review for safety-critical applications.

## Q59: How does VADER handle emoticons compared to modern approaches?
**A:** VADER includes emoticons in its lexicon (:) = +1.5, :( = -1.5, :D = +2.5, etc.). This was innovative for 2014 and makes VADER effective for social media where emoticons are common. Modern approaches: 1) Keep emoticons as special tokens (let BERT learn embeddings from context). 2) Map emoticons to sentiment scores (feature engineering). 3) Use emoji2vec (pre-trained emoji embeddings). 4) Replace emoticons with text descriptions. VADER's approach is transparent and deterministic. BERT: emoticons like ":)" may be split into [":", ")"] (losing meaning). Better to replace with `EMOTICON_POSITIVE` token or preprocess to keep as single token. For modern models: preserving emoticon information is important for social media sentiment, and preprocessing to keep them as meaningful tokens improves accuracy.

## Q60: How do you implement an active learning pipeline for BERT sentiment with a human-in-the-loop?
**A:** Production active learning pipeline: 1) Deploy BERT sentiment model. 2) For each prediction, compute uncertainty (max softmax, margin, or entropy). 3) If uncertainty > threshold T1 (high confidence): auto-approve, use prediction. 4) If T1 > uncertainty > T2 (medium): send to human for review. 5) If uncertainty < T2 (very low): auto-reject, use fallback (rule-based or generic response). 6) Human-reviewed samples are added to labeled dataset. 7) Periodically (daily/weekly) retrain model with new labeled data. 8) Track: review rate (% of inputs sent to human), human agreement rate, model accuracy over time. 9) Drift detection: if review rate increases significantly, trigger model retraining. Benefits: 1) Improves model over time. 2) Reduces human labeling effort (only uncertain samples). 3) Handles edge cases gracefully. 4) Detects domain shift early.

## Q61: How does BERT's self-attention compare to LSTM for long-range sentiment dependencies?
**A:** BERT's self-attention: O(n^2) complexity, direct connections between any two tokens (long-range with non-position bias). LSTM: O(n) complexity, but information must flow through hidden state sequentially (bottleneck for long range). For sentiment: 1) "The movie, despite its slow start and confusing middle act, which made me want to leave, actually had a brilliant ending." BERT attends directly from "brilliant" to "movie" (long distance). LSTM must carry information through 20+ time steps (difficult). 2) BERT captures bidirectional context simultaneously (both "despite" before and "actually" after the long clause). LSTM is left-to-right only. 3) BERT significantly outperforms LSTM on sentiment tasks requiring long-range dependencies (>50 words). For short text (tweets, reviews <50 words), LSTM and BERT are more comparable. BERT's architecture fundamentally solves the long-range dependency problem that plagues RNNs/LSTMs.

## Q62: How do you implement a BERT model that provides explainable sentiment with attention weights?
**A:** Using BERT's attention weights for explainability: 1) Extract attention weights from the last layer (or aggregate across layers). 2) Attention to [CLS] token indicates which words contributed to the classification. 3) Attention rollout: multiply attention matrices across layers to get input-to-output attention. 4) Attention + gradient: multiply attention weights by gradient of output w.r.t. attention (more faithful explanations). 5) Visualization: highlight words by attention weight (red = high attention, blue = low). 6) Aggregation: average attention over all heads (or select most interpretable head). 7) Implementation: `model.bert.encoder.layer[l].attention.self.attn_output_weights` from Hugging Face BERT outputs. 8) Caveat: attention weights may not represent "importance" perfectly (debated in literature). For production, use Integrated Gradients or SHAP over raw attention for more faithful explanations.

## Q63: How does VADER handle domain-specific sentiment without modification?
**A:** VADER's general-purpose lexicon works reasonably well on informal text but shows limitations on domain-specific text: 1) Financial: "volatile" (negative to most, but day-traders see opportunity). 2) Medical: "acute" (negative for a condition, neutral/technical in clinical notes). 3) Legal: "settlement" (neutral legal term but may imply guilt). 4) Gaming: "nerfed" (negative to players but not in VADER). Without modification: VADER scores these based on general sentiment (may be wrong). Mitigations: 1) Domain-specific preprocessing: normalize domain terms before VADER. 2) Post-processing: adjust compound thresholds per domain. 3) Weighted VADER: combine VADER score with domain-specific rule scores. 4) Best practice: customize the lexicon for the domain (add terms, adjust scores). For unmodified VADER, expect 5-15% lower accuracy on domain-specific text vs general text.

## Q64: How do you implement BERT with a custom classification head for hierarchical sentiment?
**A:** Hierarchical sentiment: coarse (positive/negative/neutral) -> fine-grained (joy/sadness/anger within negative). Architecture: 1) Shared BERT encoder. 2) Coarse head: [CLS] -> linear(768, 3) -> softmax (pos/neg/neu). 3) Fine-grained head: for each coarse class, separate linear layer: neg_head: linear(768, 3) -> softmax (sadness/anger/fear). pos_head: linear(768, 2) (joy/surprise). 4) Cascaded training: first train coarse, then fine-grained (with coarse labels as input). 5) Joint training: loss = L_coarse + L_fine. 6) Inference: coarse prediction selects which fine-grained head to use. Benefits: 1) Hierarchy improves both levels (coarse features help fine). 2) More interpretable. 3) Handles class imbalance (coarse classes have more data). Alternative: multi-label flat classification (output 6 probabilities, threshold each). Hierarchical is better when sentiment has a clear taxonomy.

## Q65: How do you handle BERT's maximum sequence length for long document sentiment?
**A:** For documents >512 tokens: 1) Sliding window: split into 512-token chunks with overlap (50 tokens). Predict per chunk, aggregate: majority vote (classification) or average (regression). 2) Hierarchical: encode each sentence with BERT, aggregate sentence embeddings with another model (LSTM, transformer, attention pooling). 3) Long-context transformer: Longformer (4096 tokens), BigBird (4096), Reformer (up to 64K). Fine-tune on sentiment task. 4) Truncation: keep first 512 tokens (works well for reviews where key info is at start). 5) Hybrid: truncate to 512, plus use VADER on the full text for additional signal. 6) Section-based: analyze intro, body, conclusion separately (good for structured documents). For most product reviews: truncation works fine (key sentiment in first 512 tokens). For academic/legal: hierarchical or Longformer is better.

## Q66: How do you implement prompt-based sentiment analysis with fine-tuned BERT?
**A:** Prompt-based sentiment: reframe classification as MLM prediction. 1) Template: "[TEXT] This text expresses [MASK] sentiment." 2) Target labels mapped to vocabulary: "positive" -> token IDs of "positive", "negative" -> "negative". 3) Fine-tune BERT on this prompt format (instead of standard [CLS] classification). 4) Inference: predict [MASK] token, map back to label. Benefits: 1) Better few-shot performance. 2) Uses BERT's MLM head (pre-trained knowledge). 3) More robust to domain shift. Implementation with Hugging Face: 1) Create prompt function. 2) Use `BertForMaskedLM` for training. 3) Train on prompt-formatted data. Results: prompt-based fine-tuning with 100 examples per class matches standard fine-tuning with 500 examples. PET (Pattern-Exploiting Training) extends this with multiple prompts and knowledge distillation. Best for: low-resource sentiment, domain adaptation.

## Q67: How does VADER handle comparative sentences?
**A:** VADER struggles with comparative sentences because it processes each word independently. Examples: 1) "Product A is better than Product B." VADER sees "better" (+1.5) -> positive. But the sentence compares two products, and the sentiment depends on which product is being evaluated. 2) "This phone is worse than my old one." VADER sees "worse" (-2.0) -> negative. Accurate here but only because the comparative word itself is negative. 3) "This restaurant is more expensive than the one next door." "Expensive" is negative to VADER, but the comparison may be neutral. VADER's limitation: no understanding of comparison structure. For comparative sentiment: use dependency parsing to identify comparative structure and target. BERT can learn comparative patterns from data. Best: use a model specifically trained for comparative sentiment (Google CompareNet).

## Q68: How do you implement a BERT model trained on noisy labels from VADER (weak supervision)?
**A:** Weak supervision with VADER: 1) Apply VADER to large unlabeled dataset (~1M texts). 2) Use VADER's compound score to generate noisy labels: > 0.05 = positive, < -0.05 = negative. 3) Keep only high-confidence VADER predictions (|compound| > 0.7) for cleaner labels. 4) Train BERT on this noisy labeled data. 5) Noise-robust training: use bootstrapping (model generates soft labels, combine with VADER labels). 6) Self-training: train BERT on VADER labels, then use BERT to label more data iteratively. 7) Label smoothing: reduce confidence in noisy labels (use 0.8 weight for VADER label, 0.2 uniform). Results: BERT trained on VADER-labeled data achieves 80-85% of the accuracy of BERT trained on human-labeled data (vs 90-95% for human data). This is a cost-effective way to create sentiment models without manual labeling. Best for: rapid prototyping, large-scale deployment with limited budget.

## Q69: How does BERT's performance on sentiment analysis vary with training set size?
**A:** BERT's data efficiency: 1) 100 examples: ~80% accuracy (few-shot, strong base performance). 2) 1K examples: ~88% accuracy. 3) 10K examples: ~92% accuracy. 4) 100K examples: ~94% accuracy (diminishing returns). 5) 1M+ examples: ~95% accuracy (plateau). Compare to traditional ML (TF-IDF + SVM): 1) 100 examples: ~65%. 2) 10K examples: ~82%. 3) 100K examples: ~88%. BERT's advantage is most pronounced in low-data regimes (pre-training knowledge). For sentiment: 1K-10K examples is typically sufficient for production-level accuracy (~90-92%). Increasing beyond 100K yields marginal gains (<2%). VADER (zero-shot): ~70-75% accuracy on general text, varies significantly by domain. This data efficiency is why BERT dominates sentiment analysis: even with moderate labeled data, it outperforms traditional approaches with 10x more data.

## Q70: How do you implement a BERT model that handles temporal sentiment drift?
**A:** Sentiment expressions change over time (e.g., "sick" = positive slang). Handling drift: 1) Temporal fine-tuning: periodically fine-tune on recent data (monthly). 2) Online learning: update BERT continuously with streaming data (small batches, gradient updates). 3) Domain adaptation: treat time periods as domains, use domain-adversarial training. 4) Ensemble of time-specific models: train separate BERT for each year/quarter, ensemble predictions. 5) Active learning: send uncertain predictions (due to drift) to human labelers, add to training set. 6) Monitoring: track accuracy on held-out recent data. If accuracy drops >2%, trigger retraining. 7) Data weighting: weight recent training samples higher than old ones. Implementation: maintain rolling 6-month training window, retrain monthly. BERT adapts relatively quickly (2-3 epochs on 10K recent samples is enough to capture drift).

## Q71: How does cross-attention in BERT-based ABSA models differ from self-attention?
**A:** In ABSA, cross-attention allows the model to attend to aspect-relevant context. Self-attention (standard BERT): all tokens attend to all tokens. Cross-attention (ABSA-specific): aspect representation attends to the text representation. Architecture: 1) Encode text with BERT (self-attention). 2) Encode aspect separately (or use aspect embedding). 3) Cross-attention layer: aspect queries attend to text keys (weighted sum of text tokens based on aspect relevance). 4) Output: aspect-aware text representation for sentiment classification. Benefits: 1) Model focuses on text regions relevant to the aspect. 2) Different aspects of the same text get different representations. 3) Interpretable: cross-attention weights show which text words are important for each aspect. BERT-ADA and AEN (Attentive Encoder Network) use cross-attention for state-of-the-art ABSA.

## Q72: How do you implement a hybrid lexicon-neural sentiment model that combines VADER and BERT?

Benefits: 1) VADER provides strong prior (especially for social media text). 2) BERT learns from context where VADER fails (sarcasm, complex negation). 3) Hybrid outperforms either alone (1-2% improvement). 4) VADER acts as inductive bias, improving few-shot performance. ## Q73: How do you handle domain adaptation for VADER without labeled data?
**A:** Unsupervised VADER adaptation: 1) Collect domain-specific text (e.g., 10K financial news articles). 2) Compute VADER scores for each text. 3) Identify words with high variance in scores (domain words with inconsistent VADER sentiment). 4) Use word co-occurrence with known sentiment words to estimate domain-specific valence. 5) Alternatively: use seed words with known domain sentiment, expand lexicon using word embeddings (Word2Vec trained on domain text). 6) Adjust VADER thresholds based on domain distribution (financial texts may have weaker sentiment signals). 7) Add domain-specific emoticons/acronyms. 8) Manual: domain experts review 100-500 domain words, adjust scores. This semi-automated approach can improve domain VADER accuracy by 5-10% without any labeled data.

## Q74: How does BERT's layer representation evolve from general to specific across layers?
**A:** BERT's layers capture different abstraction levels: 1) Layer 1-2: surface features (position information, part-of-speech). 2) Layer 3-4: syntactic features (dependency relations, phrase boundaries). 3) Layer 5-8: semantic features (entity types, roles, coreference). 4) Layer 9-12: task-specific features (closest to output, most specialized). For sentiment: layer 8-12 capture sentiment-related features. [CLS] token from layer 12 is used for classification, but concatenating layers 10+12 often improves performance. Evidence: 1) Probing tasks: lower layers predict POS tags better, higher layers predict sentiment better. 2) Fine-tuning: lower layers change less, higher layers change more. 3) Attention: lower layers attend to local context, higher layers attend more broadly. 4) Representation similarity: layer representations diverge more in higher layers. This hierarchy justifies layer-wise learning rate decay and gradual unfreezing.

## Q75: How do you implement semi-supervised learning for BERT sentiment with self-training?
**A:** Self-training for BERT: 1) Train initial BERT on small labeled set (e.g., 500 examples). 2) Predict on large unlabeled dataset (e.g., 100K texts). 3) Select high-confidence predictions (max probability > 0.95) as pseudo-labels. 4) Add pseudo-labeled data to training set. 5) Retrain BERT. 6) Repeat for 3-5 iterations. 7) Filter: only keep pseudo-labels where confidence exceeds threshold (increase threshold each iteration). Benefits: 1) Can leverage large unlabeled datasets. 2) Improves accuracy by 3-5% over supervised-only baseline. 3) Simple to implement. Risks: 1) Confirmation bias: model reinforces its own mistakes. 2) Noise accumulation: early mistakes compound. Mitigations: 1) Use confidence threshold. 2) Use data augmentation on pseudo-labeled samples. 3) Mix pseudo-labeled with labeled data (weighted). 4) Use ensemble (self-training with multiple models). Self-training with BERT is effective for sentiment when labeled data is scarce.

## Q76: How does VADER compare to lexicon-based approaches like TextBlob and SentiWordNet?
**A:** VADER vs TextBlob: 1) VADER is specifically tuned for social media (emoticons, slang, CAPS). TextBlob uses the Pattern lexicon (general). 2) VADER provides compound score (normalized -1 to +1). TextBlob provides polarity (-1 to +1) and subjectivity (0 to 1). 3) VADER handles intensifiers ("very"), negation ("not good"), and contrastive "but". TextBlob has minimal heuristic support. 4) On social media: VADER significantly outperforms TextBlob (85% vs 75% accuracy). VADER vs SentiWordNet: 1) SentiWordNet covers 117K synsets (much larger). 2) But SentiWordNet is not domain-tuned. 3) VADER's heuristics give it better practical performance. 4) For formal text (news, reports): SentiWordNet can match VADER accuracy. VADER is preferred for general-purpose social media sentiment. For specific domains, custom lexicon + heuristics can outperform VADER.

## Q77: How do you implement aspect extraction for ABSA using BERT-CRF?
**A:** BERT-CRF for aspect extraction (sequence labeling): 1) Tokenize input text. 2) Pass through BERT: get token-level hidden states (768-dim per token). 3) Linear layer: map BERT outputs to tag scores (B-ASPECT, I-ASPECT, O). 4) CRF layer: learn transition constraints between tags (B must follow O, I must follow B or I, O cannot follow I without B). 5) Loss: CRF negative log-likelihood. 6) Decoding: Viterbi algorithm for best tag sequence. 7) Training: manually labeled aspect boundaries. Benefits: 1) CRF ensures valid label sequences (fixes invalid transitions). 2) BERT provides rich contextual features. 3) Joint CRF learns tag dependencies (e.g., "customer service" -> [I-ASPECT, I-ASPECT] both tagged, which is invalid unless preceded by B-ASPECT. CRF prevents this). 4) State-of-the-art on aspect extraction benchmarks (90%+ F1). Alternative: use BERT + softmax per token (independent predictions) - simpler but slightly less accurate.

## Q78: How do you implement a BERT model that handles sentiment across different data modalities (text + emoji + hashtags)?

Concatenate patterns: `[CLS] text tokens [SEP] emoji tokens [SEP]`. 2) Text + hashtags: segment hashtags into words, add as separate segment. 3) Text + images: use vision-language model (CLIP, ViLT). Encode image with ViT, text with BERT, fuse embeddings. 4) Multi-task: predict text sentiment + emoji sentiment + image sentiment with shared representation. 5) Fusion: early fusion (concatenate before BERT), late fusion (separate encoders + attention fusion), or intermediate fusion (cross-attention between modalities). BERT-base can handle text + emoji + hashtags natively (just convert to text). For image + text: use CLIP (Contrastive Language-Image Pre-training) which aligns text and image embeddings, then add sentiment head.

## Q79: How does BERT's gradient flow differ between fine-tuning and training from scratch?
**A:** Fine-tuning: 1) Start from pre-trained weights (good local optimum). 2) Low learning rates (2e-5) to avoid catastrophic forgetting. 3) Gradients are small (close to minima). 4) Few epochs (2-4). 5) Most gradient updates affect higher layers (lower layers change little). Training from scratch: 1) Random initialization. 2) High learning rates (1e-4), warmup schedule. 3) Large gradients initially (far from optimum). 4) Many epochs (500K-1M steps). 5) All layers updated significantly. 6) Requires much more data and compute. Gradient implications: 1) Fine-tuning: stable, predictable, low variance. 2) From scratch: unstable, high variance, sensitive to initialization and LR. 3) Fine-tuning with discriminative LR: lower layers < 2e-6, higher layers < 2e-5. 4) Gradient clipping: fine-tuning rarely needs it (small gradients). From scratch: essential (norm clip = 1.0). This is why fine-tuning is so successful: it leverages pre-trained features and requires minimal data.

## Q80: How do you implement data balancing for imbalanced sentiment datasets with BERT?
**A:** Imbalanced sentiment (e.g., 90% positive, 5% negative, 5% neutral). Strategies: 1) Weighted loss: `weighted_cross_entropy` with class weights inversely proportional to frequency. `pos_weight = 0.1 / 0.9 = 0.11, neg_weight = 0.1 / 0.05 = 2.0`. 2) Oversampling: duplicate minority class samples. Risk: overfitting. 3) Undersampling: remove majority class samples. Risk: losing information. 4) Focal loss: `FL(p_t) = -alpha_t * (1-p_t)^gamma * log(p_t)`. Gamma = 2 focuses on hard examples. Alpha balances classes. 5) Data augmentation: back-translate minority class to create synthetic samples. 6) Threshold moving: adjust decision threshold (default 0.5) to optimize F1. 7) Ensemble: train on balanced subsets, ensemble. For BERT: weighted loss + oversampling of minority class is most effective. Focal loss with gamma=2 gives 1-2% F1 improvement on highly imbalanced datasets. Monitor macro-F1, not accuracy.

## Q81: How does BERT's training with mixed precision (FP16) affect sentiment model accuracy?
**A:** Mixed precision training (FP16 + FP32) for BERT: 1) Forward/backward pass in FP16 (faster, less memory). 2) Master weights in FP32 (for accurate gradient updates). 3) Loss scaling to prevent underflow. Impact on sentiment: 1) No significant accuracy difference (within 0.1% of FP32). 2) Training speed: 2-3x faster on V100/A100 GPUs. 3) Memory: ~40% less (can double batch size). 4) Inference: BERT FP16 is 2x faster than FP32 on GPU. Hugging Face Trainer: `fp16=True`. For fine-tuning: FP16 is safe and recommended. For quantization-aware training: use FP16 + QAT for INT8. For inference: FP16 is the standard for GPU deployment. For CPU: INT8 quantization gives more speedup (4x) than FP16 (which requires GPU). Mixed precision doesn't degrade sentiment accuracy in practice.

## Q82: How do you implement cross-validation for BERT sentiment on small datasets?
**A:** K-fold cross-validation for BERT (N <= 1000): 1) Stratified K-fold (preserve class distribution). 2) For each fold: train on K-1 folds, validate on 1 fold. 3) Use early stopping on validation loss. 4) Average metrics across folds. 5) Final model: train on full data (or ensemble of fold models). Challenges: 1) Training BERT K times is expensive (K=5 means 5x training time). 2) Small dataset: high variance across folds. Alternatives: 1) Repeated holdout: split 80/20 multiple times with different random seeds. 2) Leave-one-out: for very small datasets (<100 examples). 3) Bootstrap: sample with replacement N times, train on bootstrap sample, evaluate on OOB samples. For efficiency: use DistilBERT for cross-validation (faster, similar trends). Report mean ± std across folds. If std > 3%, dataset may be too small or noisy.

## Q84: How do you implement a BERT-based model that provides both sentiment score and explanation?

## Q85: How do you implement a BERT model for sentiment in conversational text (chat, dialogue)?
**A:** Dialogue sentiment has unique challenges: 1) Context across turns. 2) Speaker-specific sentiment. 3) Sentiment evolution. Approaches: 1) Contextual BERT: concatenate previous N turns as context: `[CLS] turn-1 [SEP] turn-2 [SEP] current [SEP]`. 2) Speaker-aware: add speaker tokens `[SPEAKER_A]`, `[SPEAKER_B]` before each turn. 3) Hierarchical: encode each turn with BERT, encode sequence of turns with another transformer. 4) Position encoding: add turn position embeddings (layer that knows turn order). 5) Dialogue history: use all previous turns in the same session as context. 6) Emotion dynamics: predict sentiment trajectory (improving vs deteriorating conversation). 7) Datasets: DailyDialog, MELD, EmoryNLP for training. For production: contextual BERT with 3-5 previous turns works well. The [SEP] tokens help BERT distinguish turn boundaries.

## Q86: How do you implement a VADER-based system with automatic threshold tuning?
**A:** Automatic threshold optimization for VADER: 1) Collect labeled data (even small set: 100 examples). 2) Compute VADER compound scores. 3) Search for optimal positive/negative thresholds that maximize F1-score: grid search over thresholds from -1 to 1 (step 0.01). 4) Default: pos >= 0.05, neg <= -0.05. Optimal may differ by domain. 5) Calibration: use Platt scaling on VADER compound scores (fit logistic regression on compound -> label) to get calibrated probabilities. 6) For multi-class: use one-vs-rest with different thresholds per class. 7) Uncertainty: if compound score is near threshold (e.g., 0.03 to 0.07), mark as uncertain instead of neutral. Benefits: threshold tuning can improve VADER accuracy by 2-5% on domain-specific data without modifying the lexicon. Simple to implement, significant impact.

## Q87: How does BERT's positional encoding limit affect sentiment performance on long texts?
**A:** BERT's 512-token limit: 1) Sentiment-bearing content beyond token 512 is ignored. 2) For Amazon reviews (avg ~100 tokens) - no problem. 3) For movie reviews (avg ~250 tokens) - most fit. 4) For news articles (500-2000 tokens) - truncation loses information. 5) For legal/financial documents (5000+ tokens) - severe truncation. Impact: 1) If key sentiment is in the truncated tail, BERT misses it. 2) BERT accuracy drops by 5-15% on long documents vs full-text models. 3) Hierarchical BERT or Longformer recovers this gap. 4) Empirical: for reviews, first 512 tokens contain >90% of sentiment signal (most reviews open with overall opinion). Mitigations: 1) Hierarchical BERT (document ~99% recovery). 2) Longformer (document ~99% recovery). 3) Truncation + heuristic (pick first + last 256 tokens). 4) For production reviews: standard BERT with 512 truncation is usually sufficient.

## Q88: How do you implement a BERT model that handles multiple sentiment dimensions (valence, arousal, dominance)?

## Q91: How does BERT's attention pattern differ for sentiment analysis across languages?
**A:** Cross-lingual attention patterns: 1) All languages show similar layer-wise progression (surface -> syntax -> semantics). 2) Attention distances: English sentiment-focused attention tends to be local (near the sentiment word), while German/Dutch show longer-distance attention (due to verb-final structure in subordinate clauses). 3) Free word order languages (Russian, Czech): sentiment-bearing words may be far apart, requiring longer attention spans. 4) Character-based languages (Chinese, Japanese): token boundaries are different, attention shows more focus on individual characters. 5) Code-switching: attention tends to focus within each language span. These differences affect cross-lingual transfer: a model fine-tuned on English sentiment may attend differently when applied to German. XLM-R handles these differences better than mBERT.

## Q92: How do you implement BERT with a mixup augmentation for sentiment?
**A:** Mixup for BERT: interpolate between two input-label pairs. Implementation: 1) Take two training examples (x1, y1) and (x2, y2). 2) Sample lambda ~ Beta(alpha, alpha) (typically alpha=0.2). 3) Interpolate embeddings: mixed_emb = lambda * emb1 + (1-lambda) * emb2. 4) Interpolate labels: mixed_label = lambda * y1 + (1-lambda) * y2. 5) Forward pass through BERT with mixed embeddings. 6) Loss: lambda * CE(pred, y1) + (1-lambda) * CE(pred, y2). Benefits: 1) Regularizes model (linear behavior between training examples). 2) Improves robustness. 3) Better calibration (reduced overconfidence). 4) 1-2% accuracy improvement on sentiment tasks. Implementation nuance: mix at the embedding layer (not token level) to avoid creating nonsensical token sequences. Sequence-level mixup (mix [CLS] embeddings) is simpler and also effective.

## Q93: How does VADER handle question sentences?
**A:** VADER has a special rule for question marks: text ending with `?` has its sentiment intensity reduced. Rationale: questions are often neutral in sentiment. Examples: 1) "Is this a good product?" VADER sees "good" (+1.5) but the `?` reduces intensity. Compound score typically ~0.2-0.3 (neutral to slightly positive). 2) "Don't you love this?" VADER sees "love" (+3.0) and "don't" (negation), plus question effect. Score may be near neutral. Limitations of this rule: 1) Rhetorical questions ("How can you be so stupid?") are clearly negative, but VADER may give neutral. 2) Enthusiastic questions ("Isn't this amazing?!") combine CAPS, punctuation, and intensifiers. VADER underdetects rhetorical question sentiment. BERT handles questions better by learning from context whether a question is rhetorical, enthusiastic, or genuinely neutral.

## Q94: How do you implement a BERT model that provides confidence intervals for sentiment predictions?
**A:** Confidence intervals for BERT sentiment: 1) MC Dropout: enable dropout at inference, run 10-50 forward passes, compute mean and std of predictions. 95% CI = mean ± 1.96 * std. 2) Deep Ensemble: train 5 BERT models with different seeds. Predictions = mean ± std across ensemble. 3) Quantile regression: BERT outputs predicted quantiles (10th, 50th, 90th percentiles) for regression tasks. 4) Bayesian approximation: use Bayesian neural network techniques (Bayes by Backprop) - computationally expensive. MC Dropout is most practical: 1) No model changes needed. 2) Single model (no ensemble storage). 3) 10 forward passes adds 10x inference cost but provides uncertainty estimates. 4) Use for: flag low-confidence predictions for human review. In production: run 10 passes, use mean prediction, flag if std > 0.2 (sigmoid output). This enables risk-aware sentiment decisions.

## Q95: How do you implement a BERT-based sentiment system that handles spelling errors and informal language?

## Q96: How does BERT's token-level vs sentence-level sentiment analysis differ?
**A:** Token-level sentiment: each token gets a sentiment score (aspect extraction, opinion target detection). Sentence-level: single sentiment for the entire sentence. BERT supports both: 1) Token-level: use `BertForTokenClassification` (linear layer per token). 2) Sentence-level: use `BertForSequenceClassification` (linear layer on [CLS]). For aspect extraction: token-level (BIO tagging). For overall sentiment: sentence-level. Can combine: 1) Multi-task: predict token-level aspects + sentence-level sentiment. 2) Hierarchical: sentence-level sentiment weighted by token-level aspect predictions. ## Q97: How do you implement a BERT model that is robust to adversarial sentiment attacks?
**A:** Adversarial attacks: small input perturbations that change sentiment prediction. Attack types: 1) Word substitution ("good" -> "great"). 2) Character perturbation ("good" -> "g0od"). 3) Insertion of irrelevant tokens. Defenses: 1) Adversarial training: generate adversarial examples during training (FGSM, PGD, FreeLB). BERT sees perturbed inputs and learns to be robust. 2) FreeLB (Free Large-Batch): state-of-the-art adversarial training for BERT. Adds adversarial perturbations to embedding layer during training. 3) Certified robustness: randomized smoothing (add Gaussian noise to embeddings, predict by majority). 4) Input sanitization: spell-check, normalize. 5) Ensemble: multiple models reduce attack success rate. Adversarial training with FreeLB: 3-6x training time, but reduces attack success rate from 80% to 20%. For sentiment: adversarial training is recommended for production systems where malicious users may try to game the system (review filtering, content moderation).
