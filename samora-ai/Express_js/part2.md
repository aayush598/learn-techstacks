# Express.js Interview Questions and Answers - Part 2

## Q1: How do you implement HTTP/2 support in Express.js?
**A:** Express.js does not natively support HTTP/2 because it relies on Node's `http` module. To use HTTP/2, create a server with Node's `http2` module and pass the Express app as a handler: `const http2 = require('http2'); const app = express(); http2.createSecureServer({ key, cert }, app).listen(443);`. Note that Express middleware works, but some HTTP/2 features like server push and multiplexing require manual implementation through the raw `http2` stream API.

## Q2: How do you use Express with TypeScript?
**A:** Install `typescript`, `@types/node`, `@types/express`. Create a `tsconfig.json` with `"outDir": "./dist"`. Write server code in `.ts` files using ES module imports. Use `Request`, `Response`, `NextFunction` types from Express. For custom request properties, extend the `Express.Request` interface via declaration merging. Compile with `tsc` and run with `node dist/server.js`, or use `ts-node`/`tsx` for development.

## Q3: How do you implement request correlation IDs in Express.js?
**A:** A correlation ID (or request ID) traces a request across services. Generate a UUID at the start of the request lifecycle via middleware: `app.use((req, res, next) => { req.id = uuidv4(); res.setHeader('X-Request-ID', req.id); next(); })`. Pass the ID to downstream services via headers, and include it in all log entries using a structured logger like Winston or Pino. This enables debugging distributed request flows.

## Q4: How do you set up Prometheus metrics monitoring for Express.js?
**A:** Use the `prom-client` library. Create a Registry, define metrics (Counter for request count, Histogram for request duration, Gauge for active connections). Add middleware that measures response time and increments counters. Expose a `/metrics` endpoint (often guarded) for Prometheus scraping. Example: `const httpRequestDurationMicroseconds = new client.Histogram({ name: 'http_request_duration_ms', help: 'Duration of HTTP requests in ms', labelNames: ['method', 'route', 'code'], buckets: [50, 100, 200, 300, 500, 1000, 2000] })`.

## Q5: How do you implement structured logging with Winston in Express.js?
**A:** Create a Winston logger with transports (console, file). Use a middleware that captures request details and creates a child logger per request: `app.use((req, res, next) => { req.logger = logger.child({ requestId: req.id, method: req.method, url: req.url }); next(); })`. Log at different levels (`req.logger.info`, `req.logger.error`). Include response status and duration via `res.on('finish')`. Structured logs (JSON format) enable log aggregation tools (ELK, Datadog) to parse and search effectively.

## Q6: How do you implement a rate limiter backed by Redis in Express.js?
**A:** Use `rate-limit-redis` package with `express-rate-limit`. Configure a Redis store: `const RedisStore = require('rate-limit-redis').default; const limiter = rateLimit({ store: new RedisStore({ sendCommand: (...args) => client.sendCommand(args) }), windowMs: 15 * 60 * 1000, max: 100 });`. Redis-based rate limiting persists across server restarts and works in multi-instance deployments, unlike the default in-memory store which is per-process.

## Q7: How do you implement API key authentication in Express.js?
**A:** Define a middleware that checks the `x-api-key` header (or query param) against a stored set of valid keys: `function apiKeyAuth(req, res, next) { const key = req.headers['x-api-key']; if (!key || !validKeys.includes(key)) return res.status(401).json({ error: 'Invalid API key' }); next(); }`. For production, store keys hashed (bcrypt) in a database, not in memory. Apply the middleware to protected routes: `app.use('/api', apiKeyAuth)`.

## Q8: How do you implement JWT authentication with refresh tokens in Express.js?
**A:** On login, issue an access token (short-lived, e.g., 15 min) and a refresh token (long-lived, e.g., 7 days). Store the refresh token hash in a database. The access token authenticates API calls via middleware: `jwt.verify(token, secret)`. When the access token expires, the client sends the refresh token to a `/refresh` endpoint. If valid and not revoked, issue a new access token. Store refresh tokens in an HTTP-only cookie for web apps. Blacklist revoked tokens server-side.

## Q9: How do you implement role-based access control (RBAC) in Express.js?
**A:** Create a middleware factory: `function authorize(...allowedRoles) { return (req, res, next) => { if (!req.user || !allowedRoles.includes(req.user.role)) return res.status(403).json({ error: 'Forbidden' }); next(); }; }`. Apply after authentication: `app.delete('/users/:id', authenticate, authorize('admin'), deleteUser)`. Roles (admin, moderator, user) are typically stored in the JWT payload or database. For fine-grained control, consider attribute-based access control (ABAC) with permissions rather than broad roles.

## Q10: How do you prevent SQL injection in Express.js?
**A:** Never concatenate user input into SQL strings. Always use parameterized queries (prepared statements). With `pg` (node-postgres): `client.query('SELECT * FROM users WHERE email = $1', [email])`. With Sequelize: `User.findOne({ where: { email } })`. With Knex: `knex('users').where('email', email)`. Validate and sanitize input using libraries like `express-validator` or `zod` as an additional layer. Use an ORM/query builder that handles escaping automatically.

## Q11: How do you implement a health check endpoint with database connectivity verification?
**A:** A comprehensive health check endpoint verifies the application's critical dependencies: `app.get('/health', async (req, res) => { const health = { status: 'ok', timestamp: new Date(), uptime: process.uptime() }; try { await db.raw('SELECT 1'); health.database = 'connected'; } catch (e) { health.database = 'error'; health.status = 'error'; } res.status(health.status === 'ok' ? 200 : 503).json(health); })`. Some implementations support detailed checks via `/health/ready` (readiness) and `/health/live` (liveness) for Kubernetes.

## Q12: How do you implement request body validation with Zod in Express.js?
**A:** Define a Zod schema and create a validation middleware: `const userSchema = z.object({ name: z.string().min(2), email: z.string().email(), age: z.number().positive().optional() }); function validate(schema) { return (req, res, next) => { const result = schema.safeParse(req.body); if (!result.success) return res.status(400).json({ errors: result.error.flatten().fieldErrors }); req.validatedBody = result.data; next(); }; }`. Apply: `app.post('/users', validate(userSchema), handler)`. Zod provides type inference for TypeScript and detailed error messages.

## Q13: How do you document an Express.js API with Swagger/OpenAPI?
**A:** Use `swagger-jsdoc` to generate an OpenAPI spec from JSDoc comments, and `swagger-ui-express` to serve the UI. Add route annotations: `/** @swagger \n * /users: \n *   get: \n *     responses: \n *       200: \n *         description: List of users \n */`. Mount the UI: `app.use('/api-docs', swaggerUi.serve, swaggerUi.setup(swaggerSpec))`. Alternative: use `express-oas-generator` for auto-generation, or write the spec manually as YAML/JSON.

## Q14: How do you implement CORS with specific origin allowlists in Express.js?
**A:** Configure the `cors` middleware with a dynamic origin callback: `const allowedOrigins = ['https://app.example.com', 'https://admin.example.com']; app.use(cors({ origin: (origin, callback) => { if (!origin || allowedOrigins.includes(origin)) callback(null, true); else callback(new Error('Not allowed by CORS')); }, credentials: true, methods: ['GET', 'POST', 'PUT', 'DELETE'], allowedHeaders: ['Content-Type', 'Authorization'] }))`. The `!origin` check allows server-to-server requests (no Origin header).

## Q15: How do you implement WebSocket authentication in Express.js with Socket.io?
**A:** Socket.io can use the same JWT-based auth as Express. On the client, pass the token in the handshake: `io('wss://example.com', { auth: { token } })`. On the server, use a middleware: `io.use((socket, next) => { const token = socket.handshake.auth.token; jwt.verify(token, secret, (err, user) => { if (err) return next(new Error('Authentication failed')); socket.user = user; next(); }); })`. This runs before any connection event, ensuring only authenticated users can connect.

## Q16: How do you implement server-sent events (SSE) with authentication in Express.js?
**A:** SSE in Express uses `res.writeHead(200, { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache', 'Connection': 'keep-alive', 'X-Accel-Buffering': 'no' })`. Authenticate before establishing the SSE connection, either via middleware or by checking a token in the query string. Send events: `res.write(`data: ${JSON.stringify(data)}\n\n`)`. Handle cleanup: `req.on('close', () => { /* remove client from active connections set */ })`. Keep a set of response objects to broadcast events to all connected clients.

## Q17: How do you implement a job queue with Bull and Redis in Express.js?
**A:** Create a Bull queue: `const emailQueue = new Bull('email', redisConfig)`. Define a processor: `emailQueue.process(async (job) => { await sendEmail(job.data) })`. In route handlers, add jobs: `await emailQueue.add({ to: user.email, template: 'welcome' }, { attempts: 3, backoff: { type: 'exponential', delay: 2000 } })`. Expose endpoints to check job status: `app.get('/jobs/:id', async (req, res) => { const job = await emailQueue.getJob(req.params.id); res.json({ state: await job.getState(), progress: job.progress }) })`.

## Q18: How do you implement database transactions across multiple repositories in Express.js?
**A:** With Sequelize: pass the transaction object through a transaction middleware or request context. `app.use(async (req, res, next) => { const t = await sequelize.transaction(); req.transaction = t; res.on('finish', () => t.commit()); res.on('close', () => { if (res.statusCode >= 400) t.rollback(); }); next(); })`. With Knex: use `knex.transaction(trx => { ... })`. With TypeORM: use `getManager().transaction(entityManager => { ... })`. The key is ensuring rollback on errors and commit on success.

## Q19: How do you implement Express.js with Prisma ORM?
**A:** Install `@prisma/client`, define a schema in `prisma/schema.prisma`. Create a single Prisma client instance and attach it to the request or app: `const prisma = new PrismaClient(); app.use((req, res, next) => { req.prisma = prisma; next(); })`. In routes: `const users = await req.prisma.user.findMany({ where: { email: { contains: query } } })`. Prisma provides type safety, auto-completion, and handles connection pooling. On shutdown: `process.on('SIGTERM', async () => { await prisma.$disconnect(); process.exit(0); })`.

## Q20: How do you implement pagination with cursor-based (keyset) pagination in Express.js?
**A:** Cursor-based pagination is more efficient than offset for large datasets. The client sends a `cursor` (the last seen ID) and `limit`. Query: `SELECT * FROM users WHERE id > $1 ORDER BY id ASC LIMIT $2`. Return `{ data, nextCursor: data.length === limit ? data[data.length-1].id : null }`. In Express: `app.get('/users', async (req, res) => { const cursor = req.query.cursor ? parseInt(req.query.cursor) : 0; const limit = parseInt(req.query.limit) || 20; const users = await db.query('SELECT * FROM users WHERE id > $1 ORDER BY id LIMIT $2', [cursor, limit]); ... })`. This avoids the performance degradation of `OFFSET` on large tables.

## Q21: How do you implement background tasks with worker threads in Express.js?
**A:** For CPU-intensive tasks (image processing, PDF generation), offload work to worker threads to avoid blocking the event loop. Create a worker file: `const { parentPort } = require('worker_threads'); parentPort.postMessage(processImage(data))`. In Express: `app.post('/process', (req, res) => { const worker = new Worker('./workers/image.js', { workerData: req.body.image }); worker.on('message', result => res.json(result)); worker.on('error', err => res.status(500).json({ error: err.message })); })`. Consider using `piscina` for a worker thread pool.

## Q22: How do you implement request logging with Pino (fast structured logger) in Express.js?
**A:** Pino is the fastest Node.js logger. Create a logger: `const pino = require('pino'); const logger = pino({ level: process.env.LOG_LEVEL || 'info' })`. Use `pino-http` middleware for automatic HTTP request logging: `app.use(require('pino-http')({ logger }))`. Access per-request logger via `req.log`. Pino outputs JSON logs by default, optimized for machine consumption. For local development, pipe through `pino-pretty`: `node server.js | pino-pretty`.

## Q23: How do you implement Helmet with custom Content Security Policy in Express.js?
**A:** Helmet sets various HTTP security headers. For a custom CSP: `app.use(helmet({ contentSecurityPolicy: { directives: { defaultSrc: ["'self'"], scriptSrc: ["'self'", "https://cdn.example.com"], styleSrc: ["'self'", "'unsafe-inline'"], imgSrc: ["'self'", "data:", "https://images.example.com"], connectSrc: ["'self'", "https://api.example.com"], fontSrc: ["'self'"], objectSrc: ["'none'"], frameAncestors: ["'none'"], upgradeInsecureRequests: [] } } }))`. CSP prevents XSS by controlling which resources can be loaded. Send violations to a reporting endpoint via `reportUri`.

## Q24: How do you implement graceful shutdown with connection draining in Express.js?
**A:** Listen for termination signals, stop accepting new connections, drain active requests, then close: `const server = app.listen(port); process.on('SIGTERM', async () => { console.log('SIGTERM received. Shutting down...'); server.close(async () => { await db.disconnect(); await redis.disconnect(); process.exit(0); }); setTimeout(() => { console.error('Forced shutdown'); process.exit(1); }, 30000); })`. In Kubernetes, use `preStop` lifecycle hook with a sleep to allow the service mesh to deregister the pod before connections are drained.

## Q25: How do you implement cluster mode with PM2 in Express.js?
**A:** PM2's cluster mode spawns multiple worker processes across CPU cores. Ecosystem file: `module.exports = { apps: [{ name: 'api', script: 'server.js', instances: 'max', exec_mode: 'cluster', env: { NODE_ENV: 'production' } }] }`. Start: `pm2 start ecosystem.config.js`. PM2 handles zero-downtime reloads: `pm2 reload all`. Each worker listens on the same port (PM2 uses a built-in load balancer). Ensure session stores and rate limiters use a shared backend (Redis) rather than in-memory.

## Q26: How do you implement database connection pooling with node-postgres in Express.js?
**A:** Create a pool once at startup: `const { Pool } = require('pg'); const pool = new Pool({ max: 20, idleTimeoutMillis: 30000, connectionTimeoutMillis: 2000 })`. In route handlers, acquire a client: `app.get('/users', async (req, res) => { const result = await pool.query('SELECT * FROM users'); res.json(result.rows) })`. For transactions, use `const client = await pool.connect()` then `client.query('BEGIN')`. Release the client with `client.release()` after the transaction. The pool reuses connections efficiently and handles timeouts.

## Q27: How do you implement retry logic with exponential backoff in Express.js?
**A:** When calling external APIs or databases that may fail transiently, implement retry: `async function withRetry(fn, { maxRetries = 3, baseDelay = 100 } = {}) { for (let i = 0; i <= maxRetries; i++) { try { return await fn(); } catch (err) { if (i === maxRetries) throw err; const delay = baseDelay * Math.pow(2, i) + Math.random() * 100; await new Promise(r => setTimeout(r, delay)); } } }`. Use libraries like `p-retry` or `async-retry`. Add jitter to prevent thundering herd. Only retry idempotent requests (GET, PUT) — not non-idempotent POSTs.

## Q28: How do you implement a circuit breaker pattern in Express.js?
**A:** A circuit breaker prevents cascading failures by stopping requests to a failing service. Use `opossum` library: `const circuitBreaker = new CircuitBreaker(externalServiceCall, { timeout: 5000, errorThresholdPercentage: 50, resetTimeout: 30000 })`. In Express: `app.get('/data', async (req, res) => { try { const result = await circuitBreaker.fire(req.params.id); res.json(result); } catch (err) { if (err.name === 'CircuitBreakerOpenException') res.status(503).json({ error: 'Service temporarily unavailable' }); else res.status(500).json({ error: err.message }) } })`. The circuit trips to open after 50% failures, then half-opens after 30s to test recovery.

## Q29: How do you implement file upload validation with Multer in Express.js?

Handle Multer errors: `app.use((err, req, res, next) => { if (err instanceof multer.MulterError) return res.status(400).json({ error: err.message }); next(err) })`.

## Q32: How do you implement rate limiting by user ID instead of IP in Express.js?
**A:** Create a custom key generator for `express-rate-limit`: `const limiter = rateLimit({ keyGenerator: (req) => req.user?.id || req.ip, windowMs: 60 * 1000, max: 30, handler: (req, res) => res.status(429).json({ error: 'Too many requests', retryAfter: Math.ceil(req.rateLimit.resetTime / 1000 - Date.now() / 1000) }) })`. Rate limiting by user ID prevents one user from exhausting another user's quota when behind a shared IP (NAT). Use a distributed store like Redis for accuracy.

## Q33: How do you implement Express.js with Socket.io for real-time bidirectional communication?

## Q34: How do you implement API versioning using the Accept header in Express.js?
**A:** Rather than URL-based versioning (`/v1/`), use content negotiation: check the `Accept` header. Middleware: `app.use((req, res, next) => { const accept = req.headers['accept']; if (accept?.includes('application/vnd.api.v2+json')) req.apiVersion = 2; else req.apiVersion = 1; next(); })`. In routes, branch logic: `app.get('/users', (req, res) => { if (req.apiVersion === 2) return res.json(usersV2); res.json(usersV1) })`. This keeps URLs clean but can make code harder to maintain compared to URL versioning.

## Q35: How do you implement request throttling with token bucket algorithm in Express.js?
**A:** Implement a token bucket manually or use a library. Algorithm: a bucket holds `capacity` tokens, refilled at `rate` per second. Each request consumes one token. If no tokens remain, reject: `async function tokenBucket(req, res, next) { const key = req.ip; const bucket = await redis.get(key); if (!bucket) { await redis.set(key, capacity - 1, 'EX', 1); return next(); } if (parseInt(bucket) <= 0) return res.status(429).json({ error: 'Rate exceeded' }); await redis.decr(key); next(); }`. Token bucket allows short bursts while enforcing a long-term average rate.

## Q36: How do you implement Express with gRPC for microservice communication?
**A:** Express handles external HTTP/REST requests, while internal microservice communication uses gRPC. Define `.proto` files, generate client/server code with `grpc-tools`. In an Express route handler, call the gRPC service: `const client = new services.GreeterClient('localhost:50051', grpc.credentials.createInsecure()); app.post('/greet', (req, res) => { client.sayHello({ name: req.body.name }, (err, response) => { if (err) return res.status(500).json({ error: err.message }); res.json({ message: response.message }) }) })`. gRPC provides strong typing, streaming, and better performance than REST for internal services.

## Q37: How do you implement caching with ETags in Express.js?
**A:** Express automatically generates ETags for JSON and sendFile responses: `app.set('etag', 'strong')` (default). For fine-grained control, set custom ETags: `app.get('/users', async (req, res) => { const users = await db.query('SELECT * FROM users'); const etag = crypto.createHash('md5').update(JSON.stringify(users)).digest('hex'); if (req.headers['if-none-match'] === etag) return res.status(304).end(); res.set('ETag', etag); res.json(users) })`. Clients send `If-None-Match` and get 304 (Not Modified) when data hasn't changed.

## Q38: How do you implement Express.js with MongoDB and Mongoose?

Mongoose provides validation, middleware (pre/post hooks), population (joins), and schema methods.

This approach gives full control over SQL and performance.

## Q41: How do you implement error classification and custom error classes in Express.js?
**A:** Create custom error classes extending Error: `class AppError extends Error { constructor(message, statusCode, code) { super(message); this.statusCode = statusCode; this.code = code; this.isOperational = true; Error.captureStackTrace(this, this.constructor) } }; class NotFoundError extends AppError { constructor(resource = 'Resource') { super(`${resource} not found`, 404, 'NOT_FOUND') } }`. In route handlers: `throw new NotFoundError('User')`. Centralized error handler: `app.use((err, req, res, next) => { const status = err.isOperational ? err.statusCode : 500; res.status(status).json({ error: err.code || 'INTERNAL_ERROR', message: err.isOperational ? err.message : 'Something went wrong' }) })`.

## Q42: How do you implement request logging with correlation IDs across microservices in Express.js?
**A:** Generate a unique correlation ID at the API gateway or first service. Pass it via HTTP headers (`X-Correlation-ID`). In each Express service, extract and propagate: `app.use((req, res, next) => { req.correlationId = req.headers['x-correlation-id'] || uuidv4(); res.setHeader('X-Correlation-ID', req.correlationId); req.logger = logger.child({ correlationId: req.correlationId }); next() })`. When calling downstream services, include the header: `fetch(url, { headers: { 'X-Correlation-ID': req.correlationId } })`. All logs across services share the same correlation ID for tracing.

## Q43: How do you implement content negotiation to serve both JSON and HTML in Express.js?
**A:** Use `res.format()`: `app.get('/users', async (req, res) => { const users = await getUsers(); res.format({ 'text/html': () => res.render('users', { users }), 'application/json': () => res.json(users), 'default': () => res.status(406).send('Not Acceptable') }) })`. Express checks the `Accept` header to determine which handler to invoke. Alternatively, use a query parameter (`?format=json`) to override: `const format = req.query.format || req.accepts(['json', 'html']) || 'json'`.

## Q44: How do you implement server-side caching with Redis in Express.js?
**A:** Create a caching middleware: `async function cache(req, res, next) { const key = `cache:${req.originalUrl}`; const cached = await redis.get(key); if (cached) return res.json(JSON.parse(cached)); const originalJson = res.json.bind(res); res.json = (body) => { redis.setex(key, 300, JSON.stringify(body)); originalJson(body); }; next(); }`. Apply to specific routes: `app.get('/expensive-data', cache, handler)`. Adjust TTL based on data volatility. Invalidate cache on mutations: `await redis.del('cache:/expensive-data')`. Use cache tags for group invalidation.

## Q45: How do you implement webhook signature verification in Express.js?
**A:** Webhook providers (Stripe, GitHub, Slack) sign payloads with a secret. Verify the signature in middleware to ensure authenticity: `function verifyWebhook(secret) { return (req, res, next) => { const signature = req.headers['x-hub-signature-256'] || req.headers['stripe-signature']; const computed = 'sha256=' + crypto.createHmac('sha256', secret).update(JSON.stringify(req.body)).digest('hex'); if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(computed))) return res.status(401).send('Invalid signature'); next(); } }`. Always use `crypto.timingSafeEqual` to prevent timing attacks.

## Q92: How do you implement Express.js error tracking with Sentry?
**A:** Install `@sentry/node` and `@sentry/tracing`. Initialize at the top of your app: `Sentry.init({ dsn: process.env.SENTRY_DSN, environment: process.env.NODE_ENV, tracesSampleRate: 1.0 }); app.use(Sentry.Handlers.requestHandler()); app.use(Sentry.Handlers.tracingHandler())`. Use Sentry's error handler before your custom one: `app.use(Sentry.Handlers.errorHandler())`. Capture manual errors: `Sentry.captureException(err)`. Sentry provides error grouping, stack traces, user context, and performance monitoring via transaction tracing.

## Q93: How do you implement Express.js subdomain routing?
**A:** Use the `vhost` middleware or check `req.subdomains`: `app.use((req, res, next) => { const host = req.hostname; if (host.startsWith('api.')) req.apiVersion = 1; else if (host.startsWith('admin.')) req.isAdmin = true; next() })`. For separate subdomain applications: `const apiApp = express(); apiApp.get('/users', (req, res) => res.json({ service: 'api' })); app.use(require('vhost')('api.example.com', apiApp))`. This allows deploying multiple Express apps on different subdomains through a single port.

## Q94: How do you implement Express.js with serverless (AWS Lambda) using the serverless-http package?

## Q95: How do you implement Express.js 2xx and 3xx response logging selectively?
**A:** Log only non-success responses to reduce noise: `app.use((req, res, next) => { const start = Date.now(); res.on('finish', () => { const duration = Date.now() - start; if (res.statusCode >= 400) { logger.warn({ method: req.method, url: req.originalUrl, status: res.statusCode, duration, body: req.body }, 'Request failed'); } else if (process.env.LOG_LEVEL === 'debug') { logger.debug({ method: req.method, url: req.originalUrl, status: res.statusCode, duration }, 'Request completed'); } }); next() })`. In production, avoid logging all 2xx responses — they're too noisy and increase costs in log aggregation systems.

## Q96: How do you prevent HTTP parameter pollution in Express.js?
**A:** HTTP Parameter Pollution occurs when an attacker sends multiple parameters with the same name. Express's `qs` library (extended query parser) parses duplicate params as arrays: `?id=1&id=2` becomes `{ id: ['1', '2'] }`. Prevent by using the simple query parser: `app.set('query parser', 'simple')`, or validate: `const id = req.query.id; if (Array.isArray(id)) return res.status(400).json({ error: 'Duplicate parameter' })`. For body parameters, Zod schemas can reject unexpected structures.

## Q97: How do you implement Express.js middleware for JSON schema validation with AJV?
**A:** AJV (Another JSON Validator) is the fastest JSON Schema validator. Create a validation middleware: `const Ajv = require('ajv'); const ajv = new Ajv({ allErrors: true }); function validateSchema(schema) { const validate = ajv.compile(schema); return (req, res, next) => { const valid = validate(req.body); if (!valid) return res.status(400).json({ errors: validate.errors.map(e => ({ path: e.instancePath, message: e.message })) }); next(); } }`. Define schemas: `const userSchema = { type: 'object', properties: { name: { type: 'string', minLength: 2 }, email: { type: 'string', format: 'email' } }, required: ['name', 'email'], additionalProperties: false }`. AJV is highly performant for production use.

## Q98: How do you implement Express.js with graceful degradation when external services are down?
**A:** Implement fallback responses when external services fail: `async function getUsers(req, res) { try { const users = await fetchExternalAPI('/users'); res.json(users); } catch (err) { if (req.accepts('html')) { res.render('users-fallback', { message: 'Service temporarily unavailable — showing cached data', cachedUsers: await getCachedUsers() }); } else { res.status(503).json({ error: 'Service unavailable', data: await getCachedUsers(), cached: true }); } } }`. Cache previous responses in Redis to serve as fallbacks. Set `Cache-Control: stale-if-error=86400` to allow CDNs to serve stale content when origin is down.

## Q99: How do you implement Express.js middleware to add security headers for preventing MIME sniffing?
**A:** MIME sniffing can lead to security vulnerabilities. Use Helmet or set headers manually: `app.use((req, res, next) => { res.setHeader('X-Content-Type-Options', 'nosniff'); // Prevents MIME sniffing res.setHeader('X-Frame-Options', 'DENY'); // Prevents clickjacking res.setHeader('X-XSS-Protection', '0'); // Disables legacy XSS filter (can cause issues) res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin'); res.setHeader('Permissions-Policy', 'camera=(), microphone=(), geolocation=()'); // Disables unnecessary APIs res.removeHeader('X-Powered-By'); // Hides Express info next(); })`. These headers provide defense-in-depth against common browser-based attacks.

## Q100: How do you implement Express.js with Knex.js query builder and transactions?

Transactions: `app.post('/transfer', async (req, res) => { await knex.transaction(async (trx) => { await trx('accounts').where('id', fromId).decrement('balance', amount); await trx('accounts').where('id', toId).increment('balance', amount); }); res.json({ success: true }) })`. Knex provides migration and seed tools as well.
