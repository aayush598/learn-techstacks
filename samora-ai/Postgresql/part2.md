# PostgreSQL Interview Questions and Answers - Part 2

## Q1: What is TOAST in PostgreSQL and how does it work?
**A:** TOAST (The Oversized-Attribute Storage Technique) handles large field values that don't fit in a standard page (8KB). When a row exceeds ~2KB, PostgreSQL compresses and/or moves large values into a separate TOAST table. Three strategies: `EXTENDED` (compress then move to TOAST), `EXTERNAL` (move to TOAST without compression), `MAIN` (compress but keep inline if possible), `PLAIN` (prevent TOAST). The TOAST table is transparent to queries.

## Q2: What is a GiST index and when would you use it?
**A:** GiST (Generalized Search Tree) is a balanced tree structure supporting operator classes for complex data types. Use it for: geometric data (points, polygons) with distance/containment operators, full-text search (ranking), range types (overlap checks), and custom data types. Unlike B-tree, GiST supports similarity queries (ORDER BY distance) and exclusion constraints. GiST indexes are larger and slower to build than B-tree but support more complex query patterns.

## Q3: What is a BRIN index and when is it beneficial?
**A:** BRIN (Block Range INdex) stores summary information about contiguous physical blocks. It's very efficient for large tables where data is naturally ordered (time-series, log data). A BRIN index is orders of magnitude smaller than a B-tree index (e.g., 100KB vs 100MB) because it stores min/max values per block range rather than per row. BRIN is best for columns with high correlation to physical storage order. It performs poorly on random data.

## Q4: How do you implement a unique constraint across multiple columns with NULL handling in PostgreSQL?
**A:** PostgreSQL treats NULLs as not equal to each other for unique constraints, so `UNIQUE (col1, col2)` allows multiple rows where both columns are NULL. To enforce uniqueness including NULLs, use a partial unique index: `CREATE UNIQUE INDEX idx_unique_null ON users (email) WHERE email IS NOT NULL` for single-column. For multi-column with NULL treatment: `CREATE UNIQUE INDEX idx_unique_nullable ON users (COALESCE(col1, ''), COALESCE(col2, ''))` though this has limitations.

## Q5: How do you implement table partitioning for time-series data in PostgreSQL?
**A:** Declarative partitioning (PostgreSQL 10+): `CREATE TABLE measurements (id BIGSERIAL, ts TIMESTAMPTZ NOT NULL, value FLOAT) PARTITION BY RANGE (ts)`. Create monthly partitions: `CREATE TABLE measurements_2024_01 PARTITION OF measurements FOR VALUES FROM ('2024-01-01') TO ('2024-02-01'); CREATE TABLE measurements_2024_02 PARTITION OF measurements FOR VALUES FROM ('2024-02-01') TO ('2024-03-01')`. For efficient management, use `pg_partman` extension for automatic partition creation. Partition pruning eliminates scanning irrelevant partitions based on WHERE conditions.

## Q6: What is the difference between pg_stat_statements and pg_stat_activity?
**A:** `pg_stat_statements` tracks cumulative statistics (total time, calls, rows, block hit/read ratios) for normalized queries across all sessions — it identifies the slowest queries over time. `pg_stat_activity` shows currently running queries/connections with state, wait events, and duration — it identifies what's happening right now. Use `pg_stat_statements` for performance tuning and `pg_stat_activity` for real-time troubleshooting of blocking and long-running queries.

## Q7: How do you implement connection pooling with PgBouncer?
**A:** PgBouncer is a lightweight connection pooler that sits between the application and PostgreSQL. Configure `pgbouncer.ini`: `[databases]; mydb = host=localhost port=5432 dbname=mydb; [pgbouncer]; listen_addr = 0.0.0.0; listen_port = 6432; pool_mode = transaction; max_client_conn = 1000; default_pool_size = 25`. Pool modes: `session` (connection held for entire session), `transaction` (returned to pool after each transaction), `statement` (returned after each statement). Transaction pooling is most common for web applications.

## Q8: How do you perform online schema migration without downtime in PostgreSQL?
**A:** Use tools like `pgroll` or `liquibase` with safe migration patterns: 1) Add columns with NULL default (no table rewrite). 2) For adding NOT NULL columns, first add as nullable, backfill data, then add NOT NULL. 3) Create indexes with `CONCURRENTLY` to avoid locks: `CREATE INDEX CONCURRENTLY idx ON table (col)`. 4) For column type changes, add a new column, migrate data with triggers, then swap. 5) Use `CHECK CONSTRAINTS` with `NOT VALID` then validate later to avoid long locks.

## Q9: What is transaction ID wraparound and how do you prevent it?
**A:** PostgreSQL uses 32-bit transaction IDs (XID), allowing ~4 billion transactions. When the limit approaches, old XIDs "wrap around" to zero, making old rows appear as future transactions. Symptoms: forced shutdown, emergency vacuum. Prevention: monitor `SELECT datname, age(datfrozenxid) FROM pg_database` and ensure `autovacuum` runs frequently enough. Age should stay below `autovacuum_freeze_max_age` (default 200M). VACUUM freezes tuples by marking their XID as "frozen" (transaction 2).

## Q10: How do you implement logical replication with conflict resolution?
**A:** Logical replication (publication/subscription model) can face conflicts with bi-directional setups. PostgreSQL doesn't have built-in conflict resolution beyond "latest timestamp wins". For custom resolution: 1) Use `pglogical` extension for last-update-wins or first-update-wins strategies. 2) Implement conflict handlers via triggers on the subscriber that check `pg_last_xact_replay_timestamp()` to skip replayed changes. 3) Use application-level conflict resolution by tracking version numbers. In PostgreSQL 16+, improved conflict detection with `pg_stat_subscription_stats`.

## Q11: How do you implement zero-downtime failover with PostgreSQL?
**A:** Set up streaming replication with a hot standby. For failover: 1) Use Patroni (most popular HA solution) which manages automatic leader election via etcd/Consul/ZooKeeper. 2) Configure connection pooling with PgBouncer and handle failover via DNS updates or a load balancer. 3) Use `pg_auto_failover` for simpler setups. 4) For manual failover: promote standby with `pg_ctl promote` or `SELECT pg_promote()` in PostgreSQL 12+. Ensure application retry logic handles connection drops during failover.

## Q12: How do you implement cursor-based pagination (keyset pagination) in PostgreSQL?
**A:** Cursor-based pagination is more efficient than OFFSET for large datasets. Use WHERE on indexed columns: `SELECT * FROM users WHERE id > $1 ORDER BY id ASC LIMIT 20`. Return the last ID as the cursor for the next page. For multi-column sort: `SELECT * FROM users WHERE (created_at, id) > ($1, $2) ORDER BY created_at ASC, id ASC LIMIT 20`. This avoids scanning skipped rows that OFFSET requires. Always use a unique column (like id) as tiebreaker for stable ordering.

## Q13: How do you use `pg_stat_statements` to identify and fix slow queries?
**A:** Query `pg_stat_statements` to find performance bottlenecks: `SELECT queryid, query, calls, total_exec_time / calls AS avg_time, rows, shared_blks_hit / (shared_blks_hit + shared_blks_read) * 100 AS hit_ratio FROM pg_stat_statements WHERE query NOT LIKE '%pg_%' ORDER BY total_exec_time DESC LIMIT 10`. Low hit ratio (<99%) indicates inefficient index usage. High avg_time with high calls indicates a hot query needing optimization. Use `EXPLAIN ANALYZE` on identified queries to find missing indexes or bad join strategies.

## Q14: How do you implement full-text search with ranking and highlighting in PostgreSQL?
**A:** Full-text search with ranking: `SELECT title, ts_rank(to_tsvector('english', title || ' ' || body), to_tsquery('english', 'postgresql & performance')) AS rank, ts_headline('english', body, to_tsquery('english', 'postgresql & performance'), 'MaxWords=50, MinWords=30') AS headline FROM articles WHERE to_tsvector('english', title || ' ' || body) @@ to_tsquery('english', 'postgresql & performance') ORDER BY rank DESC LIMIT 20`. Create a GIN index: `CREATE INDEX idx_fts ON articles USING GIN(to_tsvector('english', title || ' ' || body))`. Use `ts_headline()` for snippet generation with highlighted search terms.

## Q15: How do you implement recursive CTEs for hierarchical data traversal?
**A:** Recursive CTEs are essential for tree structures: `WITH RECURSIVE org_tree AS (SELECT id, name, manager_id, 1 AS depth, ARRAY[id] AS path FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.id, e.name, e.manager_id, ot.depth + 1, ot.path || e.id FROM employees e JOIN org_tree ot ON e.manager_id = ot.id) SELECT id, name, depth, repeat('--', depth - 1) || name AS indented_name FROM org_tree ORDER BY path`. The base case selects roots, the recursive step joins children. The depth and path track hierarchy. Prevent infinite recursion by limiting depth or detecting cycles.

## Q16: How do you implement advanced JSONB querying with path operations in PostgreSQL?
**A:** JSONB provides powerful path queries: `-- Check if key exists: SELECT * FROM docs WHERE data ? 'metadata'; -- Check if any of keys exist: SELECT * FROM docs WHERE data ?| ARRAY['name', 'title']; -- Check if all keys exist: SELECT * FROM docs WHERE data ?& ARRAY['name', 'email']; -- Path access: SELECT data #>> '{metadata, author}' FROM docs; -- JSON path (PG12+): SELECT * FROM docs WHERE data @? '$.features[*] ? (@.enabled == true)'; -- Containment: SELECT * FROM docs WHERE data @> '{"status": "active"}'; -- Nested containment: SELECT * FROM docs WHERE data @> '{"tags": ["postgresql"]}'`. GIN indexes support all these operators.

## Q17: How do you implement custom aggregate functions in PostgreSQL?
**A:** Create custom aggregates using state functions: `CREATE FUNCTION sum_sfunc(INTEGER, INTEGER) RETURNS INTEGER AS 'SELECT $1 + $2' LANGUAGE SQL IMMUTABLE; CREATE FUNCTION sum_ffunc(INTEGER) RETURNS INTEGER AS 'SELECT $1' LANGUAGE SQL IMMUTABLE; CREATE AGGREGATE my_sum(INTEGER) (SFUNC = sum_sfunc, STYPE = INTEGER, INITCOND = '0')`. For complex aggregates: `CREATE AGGREGATE array_agg_distinct(anyelement) (SFUNC = array_append, STYPE = anyarray, INITCOND = '{}', FINALFUNC = array_distinct)`. PostgreSQL 14+ supports `AGGREGATE` with `PARALLEL = SAFE` for parallel execution.

## Q18: How do you implement LISTEN/NOTIFY for real-time application features?
**A:** LISTEN/NOTIFY provides asynchronous event notification: `-- Session 1 (listener): LISTEN channel_name; -- Session 2 (notifier): NOTIFY channel_name, '{"event": "user_created", "id": 42}'; -- In application (Python): import asyncio, asyncpg; async def listen(): conn = await asyncpg.connect(DSN); await conn.add_listener('channel_name', callback); -- The callback receives the payload string. -- Listen for multiple channels: LISTEN user_events; LISTEN order_events`. NOTIFY delivers events to all connected sessions listening on that channel. Payload is limited to 8000 bytes. Use for cache invalidation, real-time dashboards, or triggering background workers.

## Q19: How do you implement advanced table inheritance strategies in PostgreSQL?
**A:** Table inheritance: `CREATE TABLE products (id SERIAL, name TEXT, price NUMERIC, CHECK(price > 0)); CREATE TABLE books (pages INT, author TEXT) INHERITS (products); CREATE TABLE electronics (warranty_months INT) INHERITS (products)`. Constraints and indexes on parent are not inherited. Query parent: `SELECT * FROM ONLY products WHERE ...` (excludes children). Use `CHECK NO INHERIT` constraints for parent-only rules. Inheritance is useful for partitioning-like patterns before PostgreSQL 10, but declarative partitioning is now preferred.

## Q20: How do you implement exclusion constraints for scheduling/booking systems?
**A:** Exclusion constraints prevent overlapping ranges: `CREATE EXTENSION btree_gist; CREATE TABLE bookings (room_id INT, period TSRANGE, employee_id INT, EXCLUDE USING gist (room_id WITH =, period WITH &&))`. This prevents booking the same room with overlapping time ranges. The `&&` operator checks range overlap. GiST index required. Also works with other operators: `-- Prevent scheduling the same employee for overlapping meetings: EXCLUDE USING gist (employee_id WITH =, period WITH &&)`. On conflict, the transaction aborts — catch the error in application code.

## Q21: How do you implement row-level security (RLS) in PostgreSQL?
**A:** RLS restricts which rows a user can access: `CREATE TABLE documents (id SERIAL, owner_id INT, content TEXT); ALTER TABLE documents ENABLE ROW LEVEL SECURITY; CREATE POLICY user_policy ON documents FOR SELECT USING (owner_id = current_setting('app.current_user_id')::int); CREATE POLICY insert_policy ON documents FOR INSERT WITH CHECK (owner_id = current_setting('app.current_user_id')::int)`. Set the app parameter: `SET app.current_user_id = 42`. RLS applies to all queries (SELECT, INSERT, UPDATE, DELETE) on the table. For admin bypass: `CREATE POLICY admin_all ON documents TO admin USING (true) WITH CHECK (true)`. Force RLS for table owners: `ALTER TABLE documents FORCE ROW LEVEL SECURITY`.

## Q22: How do you implement replication lag monitoring in PostgreSQL?
**A:** Monitor streaming replication lag: `SELECT application_name, state, sync_state, pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes, (EXTRACT(EPOCH FROM (NOW() - pg_last_xact_replay_timestamp())))::INT AS lag_seconds FROM pg_stat_replication`. For physical replication, check `pg_stat_replication` and `pg_stat_wal_receiver`. Alerts: lag > 10MB or > 30 seconds. Use `pg_current_wal_lsn()` for publisher and `pg_last_wal_receive_lsn()` / `pg_last_wal_replay_lsn()` for subscriber. For logical replication, check `pg_stat_subscription` and `pg_stat_replication_slots`.

## Q23: How do you implement a queue-like table with SKIP LOCKED in PostgreSQL?
**A:** Use `SKIP LOCKED` for a job queue where multiple workers consume tasks: `CREATE TABLE job_queue (id SERIAL PRIMARY KEY, status TEXT DEFAULT 'pending', payload JSONB, created_at TIMESTAMPTZ DEFAULT NOW()); CREATE INDEX idx_queue_status ON job_queue (status) WHERE status = 'pending'`. Worker query: `UPDATE job_queue SET status = 'processing', started_at = NOW() WHERE id = (SELECT id FROM job_queue WHERE status = 'pending' ORDER BY created_at ASC LIMIT 1 FOR UPDATE SKIP LOCKED) RETURNING *`. `SKIP LOCKED` skips rows already locked by other workers, preventing contention. This is the recommended approach for PostgreSQL-based job queues.

## Q24: How do you implement advanced indexing with INCLUDE columns (covering indexes)?
**A:** Covering indexes allow index-only scans by including non-key columns: `CREATE INDEX idx_users_covering ON users (email) INCLUDE (name, avatar_url, created_at)`. The index can answer `SELECT email, name, avatar_url FROM users WHERE email = 'test@example.com'` without touching the table. Key columns go in the index tree for searching; INCLUDE columns are stored only at the leaf level. Benefits: faster reads since fewer heap lookups. Costs: larger index, slower writes. Use for frequently accessed columns in read-heavy workloads.

## Q25: How do you implement temporal tables (system-versioned) in PostgreSQL?
**A:** PostgreSQL doesn't have built-in temporal tables, but you can implement them with triggers: `CREATE TABLE employees (id SERIAL PRIMARY KEY, name TEXT, salary NUMERIC, valid_from TIMESTAMPTZ NOT NULL DEFAULT NOW(), valid_to TIMESTAMPTZ NOT NULL DEFAULT 'infinity'); CREATE TABLE employees_history (LIKE employees INCLUDING ALL, EXCLUDING IDENTITY)`. Create a trigger: `CREATE FUNCTION save_history() RETURNS TRIGGER AS $$ BEGIN INSERT INTO employees_history SELECT (OLD).*, NOW(); RETURN NEW; END $$ LANGUAGE plpgsql; CREATE TRIGGER trg_save_history BEFORE UPDATE OR DELETE ON employees FOR EACH ROW EXECUTE FUNCTION save_history()`. Query historical state: `SELECT * FROM employees_history WHERE id = 1 AND '2024-01-15' BETWEEN valid_from AND valid_to`.

## Q26: How do you automate PostgreSQL backups with WAL archiving and PITR?
**A:** Configure continuous WAL archiving: `postgresql.conf`: `wal_level = replica; archive_mode = on; archive_command = 'cp %p /backups/wal/%f'`. Perform a base backup: `pg_basebackup -D /backups/base -Ft -z -P`. For PITR, restore base backup and create `recovery.conf` (PG12-) or `postgresql.auto.conf` (PG12+): `restore_command = 'cp /backups/wal/%f %p'; recovery_target_time = '2024-06-15 14:30:00 UTC'`. Use `pg_verifybackup` to validate backups. For automation, use `pgBackRest` or `barman` which manage base backups, WAL archiving, retention, and validation.

## Q27: How do you implement query optimization using pg_hint_plan?
**A:** Sometimes the PostgreSQL planner makes suboptimal choices. `pg_hint_plan` (extension) overrides planner decisions: `CREATE EXTENSION pg_hint_plan; SELECT /*+ SeqScan(t) */ * FROM users t WHERE id = 5; SELECT /*+ HashJoin(a b) */ * FROM orders a JOIN customers b ON a.customer_id = b.id; SELECT /*+ IndexScan(t idx_users_email) */ * FROM users t WHERE email = 'test@test.com'`. Hints include: scan methods (SeqScan, IndexScan), join methods (HashJoin, MergeJoin, NestLoop), join order (Leading), and parallelism (Parallel). Use hints as a last resort after optimizing schema and queries.

## Q28: How do you implement cross-database queries with foreign data wrappers (FDW)?
**A:** Use `postgres_fdw` to query other databases: `CREATE EXTENSION postgres_fdw; CREATE SERVER foreign_server FOREIGN DATA WRAPPER postgres_fdw OPTIONS (host 'remote_host', port '5432', dbname 'remote_db'); CREATE USER MAPPING FOR local_user SERVER foreign_server OPTIONS (user 'remote_user', password 'remote_pass'); CREATE FOREIGN TABLE remote_users (id INT, name TEXT) SERVER foreign_server OPTIONS (schema_name 'public', table_name 'users')`. Query: `SELECT * FROM remote_users WHERE id = 1`. PostgreSQL pushes WHERE conditions to the remote server. FDW supports INSERT/UPDATE/DELETE on foreign tables. Use for data integration without ETL.

## Q29: How do you implement advanced array operations in PostgreSQL?
**A:** PostgreSQL has powerful array support: `-- Create: SELECT ARRAY[1, 2, 3]; SELECT ARRAY(SELECT id FROM users); -- Access: SELECT arr[1:2]; SELECT arr[array_upper(arr, 1)]; -- Contains: SELECT ARRAY[1,2,3] @> ARRAY[1,2]; -- Overlap: SELECT ARRAY[1,2,3] && ARRAY[3,4,5]; -- Append: SELECT array_append(ARRAY[1,2], 3); -- Remove: SELECT array_remove(ARRAY[1,2,3], 2); -- Replace: SELECT array_replace(ARRAY[1,2,3], 2, 99); -- Unnest: SELECT unnest(ARRAY[1,2,3]); -- Array of distinct: SELECT ARRAY(SELECT DISTINCT unnest(col) FROM table ORDER BY 1)`. Array functions are optimized with GIN indexes using `@>` and `&&` operators.

## Q30: How do you implement PostgreSQL security hardening?
**A:** Hardening steps: 1) Use `scram-sha-256` authentication (not `md5` or `trust`). 2) Configure `pg_hba.conf` with host-based restrictions: `hostssl all all 10.0.0.0/8 scram-sha-256`. 3) Enable SSL: `ssl = on; ssl_cert_file = 'server.crt'; ssl_key_file = 'server.key'`. 4) Use row-level security for multi-tenancy. 5) Apply least privilege: create roles with minimal grants. 6) Enable `log_statement = 'ddl'` to track schema changes. 7) Set `password_encryption = 'scram-sha-256'`. 8) Regular `pg_verifybackup` checks. 9) Keep PostgreSQL updated for security patches. 10) Use `pgaudit` extension for audit logging.

## Q31: How do you implement PostgreSQL connection limits and resource management?
**A:** Configure `max_connections` in `postgresql.conf` (default 100). Each connection consumes ~10MB memory. Use connection pooling (PgBouncer) to handle thousands of clients with a small pool. Per-user limits: `ALTER ROLE app_user CONNECTION LIMIT 25`. Resource management via: `statement_timeout` (query timeout), `idle_in_transaction_session_timeout`, `lock_timeout`. Use `pg_stat_activity` to monitor: `SELECT state, wait_event, query_start, query FROM pg_stat_activity WHERE state = 'active'`. Terminate runaway queries: `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE query ILIKE '%bad_query%'`.

## Q32: How do you implement full outer join and its use cases in PostgreSQL?
**A:** Full outer join returns all rows from both tables, matching where possible: `SELECT e.name, d.name FROM employees e FULL OUTER JOIN departments d ON e.dept_id = d.id`. Results include employees without departments (dept name NULL) and departments without employees (employee name NULL). Use cases: data reconciliation (find records in one table but not the other), merging two data sources, and comparing schema changes. Identify unmatched rows: `WHERE e.id IS NULL OR d.id IS NULL`.

## Q33: How do you implement lateral joins in PostgreSQL?
**A:** LATERAL joins allow the right side to reference columns from the left side: `SELECT u.name, recent_orders.total FROM users u, LATERAL (SELECT SUM(amount) AS total FROM orders WHERE user_id = u.id AND created_at > NOW() - INTERVAL '30 days') recent_orders WHERE recent_orders.total IS NOT NULL`. LATERAL is powerful for: top-N per group: `SELECT d.name, e.name, e.salary FROM departments d, LATERAL (SELECT * FROM employees WHERE dept_id = d.id ORDER BY salary DESC LIMIT 3) e`. Without LATERAL, subqueries can't reference outer FROM items.

## Q34: How do you implement PostgreSQL data masking for sensitive columns?
**A:** Dynamic data masking hides sensitive data from unauthorized users. Approaches: 1) View-based masking: `CREATE VIEW users_public AS SELECT id, name, regexp_replace(email, '(.)(.*)(@.*)', '\\1***\\3') AS email_masked FROM users`. 2) Column-level security (grant column access selectively). 3) Custom masking function: `CREATE FUNCTION mask_ssn(ssn TEXT) RETURNS TEXT AS $$ SELECT CASE WHEN current_setting('app.role') = 'admin' THEN ssn ELSE overlay(ssn placing 'XXX-' from 1 for 4) END $$ LANGUAGE SQL STABLE`. 4) Use `anon` extension for GDPR-compliant anonymization with declarative masking rules.

## Q35: How do you implement MongoDB-style document aggregation in PostgreSQL using JSONB?
**A:** PostgreSQL can perform document-style aggregation: `SELECT doc ->> 'category' AS category, COUNT(*) AS count, AVG((doc ->> 'price')::NUMERIC) AS avg_price, JSONB_AGG(doc ->> 'name') AS names, JSONB_OBJECT_AGG(doc ->> 'name', doc -> 'metadata') AS metadata_map FROM products GROUP BY doc ->> 'category' HAVING COUNT(*) > 5 ORDER BY count DESC`. Use `JSONB_AGG` for collecting documents into arrays, `JSONB_OBJECT_AGG` for key-value maps. For nested grouping: `GROUP BY (doc #>> '{address, country}')`. GIN indexes on JSONB support efficient querying.

## Q36: How do you implement the "merge upsert" pattern in PostgreSQL?
**A:** Use `INSERT ... ON CONFLICT DO UPDATE`: `INSERT INTO users (email, name, updated_at) VALUES ('test@example.com', 'Test User', NOW()) ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name, updated_at = EXCLUDED.updated_at, version = users.version + 1 RETURNING *`. The `EXCLUDED` pseudo-table references the proposed values. For partial conflict targets: `ON CONFLICT WHERE active = true DO UPDATE SET ...`. For `DO NOTHING`: `INSERT INTO logs (event, timestamp) VALUES ('login', NOW()) ON CONFLICT DO NOTHING`. This is more efficient than a separate SELECT + INSERT/UPDATE.

## Q37: How do you implement PostgreSQL query optimization with materialized CTEs?
**A:** In PostgreSQL, CTEs act as optimization fences — they are materialized by default (executed once, results stored). This prevents pushing down WHERE conditions: `WITH regional_users AS (SELECT * FROM users WHERE region = 'US') SELECT * FROM regional_users WHERE active = true`. The entire US user set is materialized even if only active ones are needed. Since PostgreSQL 12, you can use `WITH [MATERIALIZED | NOT MATERIALIZED]` to control behavior: `WITH NOT MATERIALIZED regional_users AS (...)`. Use `NOT MATERIALIZED` when the CTE is referenced once to allow predicate pushdown. Use `MATERIALIZED` for expensive CTEs referenced multiple times.

## Q38: How do you implement a type 2 slowly changing dimension (SCD) in PostgreSQL?
**A:** SCD Type 2 tracks historical changes: `CREATE TABLE customer_dim (id SERIAL, customer_id INT, name TEXT, address TEXT, valid_from DATE NOT NULL DEFAULT CURRENT_DATE, valid_to DATE NOT NULL DEFAULT '9999-12-31', is_current BOOLEAN NOT NULL DEFAULT true)`. On change: `UPDATE customer_dim SET valid_to = CURRENT_DATE, is_current = false WHERE customer_id = 123 AND is_current = true; INSERT INTO customer_dim (customer_id, name, address) VALUES (123, 'New Name', 'New Address')`. Query current state: `WHERE is_current = true`. Query as-of: `WHERE '2024-01-15' BETWEEN valid_from AND valid_to`. Add a `version` column for ordering.

## Q39: How do you implement geospatial queries with PostGIS in PostgreSQL?
**A:** PostGIS extends PostgreSQL with spatial types and functions: `CREATE EXTENSION postgis; CREATE TABLE locations (id SERIAL, name TEXT, geom GEOMETRY(POINT, 4326)); INSERT INTO locations VALUES (1, 'Office', ST_SetSRID(ST_MakePoint(-73.9857, 40.7484), 4326))`. Spatial queries: `-- Find nearby: SELECT name, ST_Distance(geom, ST_SetSRID(ST_MakePoint(-73.9857, 40.7484), 4326)::GEOGRAPHY) AS dist_m FROM locations ORDER BY dist_m LIMIT 5; -- Spatial join: SELECT a.name, b.name FROM locations a JOIN locations b ON ST_DWithin(a.geom::GEOGRAPHY, b.geom::GEOGRAPHY, 1000) WHERE a.id != b.id; -- Bounding box: SELECT * FROM locations WHERE geom && ST_MakeEnvelope(-74, 40, -73, 41, 4326)`. Create GiST index: `CREATE INDEX idx_locations_geom ON locations USING GIST (geom)`.

## Q40: How do you implement query tuning with EXPLAIN (BUFFERS, ANALYZE)?
**A:** Detailed explain: `EXPLAIN (ANALYZE, BUFFERS, TIMING, COSTS, FORMAT JSON) SELECT * FROM users JOIN orders ON users.id = orders.user_id WHERE users.email = 'test@example.com'`. Key metrics: `actual time` (actual execution duration), `rows` vs `rows removed by filter` (accuracy of estimates), `shared hit/read/dirtied` (buffer usage: hit=cached, read=disk read), `I/O timings` (if track_io_timing enabled). Red flags: sequential scans with high row estimates, large row count mismatches (>10x), nested loops with inner tables scanned repeatedly, excessive buffers read.

## Q41: How do you implement PostgreSQL with Python using asyncpg?
**A:** asyncpg is the fastest PostgreSQL driver for Python asyncio: `import asyncio, asyncpg; async def main(): conn = await asyncpg.connect(user='user', password='pass', database='db', host='localhost'); await conn.execute('''CREATE TABLE IF NOT EXISTS users (id SERIAL, name TEXT, email TEXT UNIQUE)'''); await conn.execute('INSERT INTO users (name, email) VALUES ($1, $2)', 'Alice', 'alice@example.com'); row = await conn.fetchrow('SELECT * FROM users WHERE email = $1', 'alice@example.com'); print(row['name']); await conn.close()`. Use connection pools: `pool = await asyncpg.create_pool(dsn, min_size=5, max_size=20); async with pool.acquire() as conn: await conn.fetch('SELECT * FROM users')`. asyncpg supports prepared statements, COPY, and LISTEN/NOTIFY.

## Q42: How do you implement PostgreSQL with Node.js using node-postgres?
**A:** `pg` (node-postgres) is the most popular Node.js driver: `const { Pool } = require('pg'); const pool = new Pool({ connectionString: 'postgres://user:pass@localhost:5432/db', max: 20, idleTimeoutMillis: 30000 }); module.exports = { query: (text, params) => pool.query(text, params) }`. Usage: `const { rows } = await db.query('SELECT * FROM users WHERE id = $1', [userId])`. For transactions: `const client = await pool.connect(); try { await client.query('BEGIN'); await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [100, 1]); await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [100, 2]); await client.query('COMMIT') } catch (e) { await client.query('ROLLBACK'); throw e } finally { client.release() }`.

## Q43: How do you implement auto-incrementing custom sequences with skipping?
**A:** Create sequences with custom increment: `CREATE SEQUENCE order_number_seq START 1000 INCREMENT 1 CACHE 100; -- Assign in INSERT: INSERT INTO orders (order_no, ...) VALUES (nextval('order_number_seq'), ...)`. For gapless sequences (e.g., invoice numbers), don't use CACHE (defaults to 1, but slower). For gap-friendly IDs, use CACHE 100 for better performance. Use sequences across tables: `-- Shared sequence: CREATE SEQUENCE global_id_seq; ALTER TABLE users ALTER COLUMN id SET DEFAULT nextval('global_id_seq'); ALTER TABLE posts ALTER COLUMN id SET DEFAULT nextval('global_id_seq')`. Reset: `ALTER SEQUENCE seq_name RESTART WITH 1000`.

## Q44: How do you implement constraint deferring in PostgreSQL?
**A:** Deferrable constraints are checked at transaction commit rather than statement end: `CREATE TABLE orders (id SERIAL PRIMARY KEY, user_id INT REFERENCES users(id) DEFERRABLE INITIALLY DEFERRED); CREATE TABLE payments (id SERIAL PRIMARY KEY, order_id INT NOT NULL, CONSTRAINT fk_order FOREIGN KEY (order_id) REFERENCES orders(id) DEFERRABLE INITIALLY IMMEDIATE)`. `INITIALLY DEFERRED` means always deferred; `INITIALLY IMMEDIATE` can be deferred with `SET CONSTRAINTS ALL DEFERRED`. Use case: circular references where both tables reference each other. At `COMMIT`, constraints are checked. Performance: deferring unique/primary key constraints reduces work for multi-row operations.

## Q45: How do you implement joins with multiple conditions and OR clauses efficiently?
**A:** Multi-condition joins can degrade performance. Instead of: `SELECT * FROM a JOIN b ON a.id = b.ref_id OR a.email = b.email`, restructure: `SELECT * FROM a JOIN b ON a.id = b.ref_id UNION SELECT * FROM a JOIN b ON a.email = b.email AND a.id != b.ref_id`. Use a UNION when OR prevents hash/merge joins. For IN clauses: `SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE total > 100)` is often optimized, but: `SELECT * FROM users WHERE id = ANY(ARRAY[1,2,3])` is better for small lists. Use EXISTS for correlated conditions: `SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.total > 100)`.

## Q92: How do you implement PostgreSQL with read-after-write consistency in distributed systems?
**A:** In distributed setups with replication, read-after-write consistency ensures the client reads what it just wrote. Approaches: 1) Session-based: issue reads to the primary for a short period after writes (write-your-own-reads). 2) Application tracking: after a write, store the write LSN: `SELECT pg_current_wal_lsn()`. On reads from replicas: wait until `pg_last_wal_replay_lsn() >= stored_lsn`. 3) Use `target_session_attrs=read-write` in connection strings to always hit primary for critical reads. 4) CockroachDB-style: use synchronous replication for critical data. 5) Use `SELECT ... FOR UPDATE` on replicas (waits for replication). The simplest approach: route user-specific reads to primary for a few seconds after mutations.
