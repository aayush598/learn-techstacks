# Clerk Auth Interview Questions and Answers - Part 2

## Q1: How do you implement webhook signature verification in a Next.js API route for Clerk events?
**A:** Use the `svix` library. In your API route, extract the `svix-id`, `svix-timestamp`, and `svix-signature` headers. Pass the raw body (as a string, not parsed JSON) and headers to `Webhook.verify()`. Use `webhook.verify(rawBody, headers)` which returns the verified payload. Always verify the timestamp is within 5 minutes of your server time to prevent replay attacks. NEVER parse the body with `JSON.parse()` before verification since it changes the payload string.

## Q2: How do you configure Clerk for a B2B multi-tenant SaaS with per-organization SAML SSO?
**A:** Enable Organizations in the Clerk Dashboard. For each enterprise tenant, create a SAML connection under that organization's settings with their IdP metadata. Use `auth().orgId` to identify the tenant. Configure JWT templates to include `org_id` and `org_role` in claims. When a user authenticates via their organization's SSO, Clerk automatically scopes the session to that org. Use `<OrganizationSwitcher>` for users belonging to multiple orgs. The organization slug can be used in subdomain routing (e.g., `tenant.app.com`).

## Q3: How do you create a custom JWT template that includes data from your external database?
**A:** In the Clerk Dashboard under JWT Templates, create a new template. Use the shortcode syntax to reference Clerk data: `{{user.public_metadata}}`, `{{org.private_metadata}}`, etc. For external DB data, use Clerk's Backend API webhooks to sync external data into `public_metadata` or `private_metadata` on user create/update events. Then reference those metadata fields in your JWT template. You CANNOT make direct database calls from JWT templates — the shortcode system only has access to Clerk's stored data.

## Q4: How do you implement session-based rate limiting per user with Clerk?
**A:** Track session activity using `auth().sessionId` as a unique key in your rate limiter (e.g., Redis). For auth endpoints (sign-in, sign-up), Clerk handles rate limiting internally. For your own API, implement middleware that extracts `userId` from `auth()` and applies your own rate limit strategy. Use the `clerkClient.sessions.verifySession()` to check if a session is still valid before accepting rate-limited requests. For brute-force protection, monitor webhook events like `session.created` and `sign_in.attempt` to detect abnormal patterns.

## Q5: How do you use Clerk with tRPC for end-to-end type safety in your auth flow?
**A:** Create a tRPC middleware that uses `auth()` from `@clerk/nextjs/server` to inject user context. Define your context creator as `async ({ req }) => { const { userId, sessionId } = auth(); return { userId, sessionId }; }`. Create a `protectedProcedure` using `t.procedure.use(({ ctx, next }) => { if (!ctx.userId) throw new TRPCError({ code: 'UNAUTHORIZED' }); return next(); })`. For role-based access, check `auth().orgRole` or custom claims. Type the context so all protected procedures have typed access to user info without repeated auth checks.

## Q6: How do you handle Clerk OAuth token exchange for accessing third-party APIs on behalf of users?
**A:** When a user authenticates via OAuth (e.g., Google), Clerk stores the OAuth access and refresh tokens. Use `clerkClient.users.getUserOauthAccessToken(userId, 'oauth_google')` to retrieve the token server-side. The returned token object includes `token`, `provider`, and `scopes`. Use this token to call the third-party API (e.g., Google Calendar). Handle token refresh by checking for 401 responses and using Clerk's stored refresh token. Note that the access token is only available via the Backend API, not client-side for security reasons.

## Q7: How do you implement Clerk with Express.js middleware for a traditional Node.js backend?
**A:** Install `@clerk/backend` and initialize `createClerkClient({ secretKey: process.env.CLERK_SECRET_KEY })`. Create middleware that extracts the session token from the `Authorization: Bearer <token>` header (or `__session` cookie), then calls `clerkClient.verifyToken(sessionToken)`. The verified token contains `sub` (userId), `sid` (sessionId), and custom claims. Attach the payload to `req.auth`. For routes that need full user data, call `clerkClient.users.getUser(payload.sub)`. Use CORS configuration to allow the Clerk frontend API to communicate with your origin.

## Q8: How do you synchronize Clerk user data to an external PostgreSQL database using webhooks?
**A:** Create a webhook endpoint that listens for `user.created`, `user.updated`, and `user.deleted` events. Verify the webhook signature using `svix`. For `user.created`, upsert a row in your `users` table mapping Clerk `id` to your internal ID and storing relevant fields (email, name, metadata). For `user.updated`, update the corresponding row. For `user.deleted`, soft-delete or cascade clean up related data. Handle idempotency by checking the `svix-id` header to avoid duplicate processing. Use database transactions to ensure consistency between the webhook payload and any side effects.

## Q9: How do you implement a custom sign-in flow with Clerk that supports step-up authentication for sensitive operations?
**A:** Use the `useSignIn()` hook for the base sign-in. For step-up auth on sensitive actions (e.g., viewing billing info), check `auth().sessionClaims.iat`. If the session is older than your threshold (e.g., 5 minutes), trigger re-authentication using `signIn.create({ strategy: 'password', identifier: email })` then `signIn.attemptFirstFactor({ strategy: 'password', password })`. Alternatively, use Clerk's second factor strategies like `phone_code` or `totp`. Manage the step-up state with a React context that tracks which operations require fresh auth. The new session claim `fva` (factor verification age) can be used to verify recent auth.

## Q10: How do you configure Clerk for a Next.js middleware-based route protection with role-based redirects?
**A:** In `middleware.ts`, export `clerkMiddleware((auth, req) => { const { userId, sessionClaims } = auth(); if (!userId && !req.nextUrl.pathname.startsWith('/public')) { return redirect(new URL('/sign-in', req.url)); } const role = sessionClaims?.metadata?.role; const isAdminRoute = req.nextUrl.pathname.startsWith('/admin'); if (isAdminRoute && role !== 'admin') { return redirect(new URL('/dashboard', req.url)); } })`. Use `publicRoutes` and `ignoredRoutes` options to optimize middleware performance. Structure routes hierarchically so the middleware can pattern-match efficiently. Always redirect to a meaningful fallback that guides the user.

## Q19: How do you handle Clerk webhook idempotency and event ordering?
**A:** Each Svix webhook message includes a unique `id` field. Store processed webhook IDs in a database with a unique constraint or use Redis with a TTL. Before processing, check if the ID exists. If already processed, return 200 without processing. Handle event ordering by checking event timestamps. For example, if you receive `user.updated` with an older timestamp than the last processed `user.updated`, skip it. Use the `data.updated_at` field from the webhook payload to compare. For critical data, implement last-write-wins or use event sourcing with a monotonically increasing sequence.

## Q37: How do you implement Clerk's webhook-based user data synchronization with a Redis cache?
