# Server-Sent Events Interview Questions and Answers - Part 2

## Q1: What is the exact byte-level format of an SSE event stream and how does the parser handle field boundaries?
**A:** The SSE format uses UTF-8 encoding. Each event consists of one or more lines ending with LF (\n). Lines are formatted as 'field: value\n' where the colon separates field name from value. The space after colon is optional but conventional. An event is terminated by a double newline (\n\n). The parser: reads lines sequentially, accumulates 'data' fields by concatenating them with newlines, sets 'event', 'id', and 'retry' from their respective fields. On encountering an empty line (\n\n), it fires the complete event. Lines starting with ':' are comments (ignored). Lines without a colon are treated as field with empty value (ignored). Fields can be split across lines only for 'data' - other fields are single-line.

## Q2: How does the Last-Event-ID mechanism work at the TCP level for ensuring at-least-once delivery?
**A:** The browser stores the last received 'id' field value in memory. On disconnection detection (TCP RST, timeout, or HTTP error), the browser initiates a new HTTP connection. The new request includes the HTTP header 'Last-Event-ID: <value>'. The server reads this header and replays events from that ID onwards from a buffer. The TCP-level guarantee: because the browser tracks the last successfully received event ID (not the last sent), it ensures missed events between the last received and the reconnection are replayed. However, this is at-least-once, not exactly-once - the last event before disconnection may be replayed (duplicate). Clients must deduplicate using event IDs.

## Q3: How do SSE event stream formats handle dispatch timeout when using load balancers that enforce connection timeouts?
**A:** Load balancers (AWS ALB, nginx, HAProxy) have idle timeout defaults (60s for ALB, 60s for nginx). When no data flows for the timeout duration, they close the connection. SSE must send keepalive heartbeats before the timeout. Strategy: send a comment line (: keepalive\n\n) every 30s (half the typical LB timeout). For ALB, the idle timeout is configurable up to 4000s. For nginx, proxy_read_timeout must be set to a high value (e.g., 3600s). HTTP/2 connections have different timeout behavior - they're less likely to be closed by intermediaries. Response buffering must be disabled: proxy_buffering off (nginx), chunked transfer encoding, and explicit flush after each event.

## Q4: How do you implement SSE with token-based authentication when EventSource doesn't support custom headers?
**A:** EventSource API limitation: no custom headers (Authorization). Solutions: (1) URL query parameter: new EventSource('/events?token=' + encodeURIComponent(token)). Server validates token from query param. Security risk: token in URL may be logged by servers/proxies. (2) Cookie-based: set httpOnly secure cookie with session token. EventSource with withCredentials: true includes cookies automatically. (3) Short-lived token redirect: initial endpoint validates token (from body/header via fetch), returns 302 with short-lived token in URL to the SSE endpoint. (4) fetch-based SSE: use fetch() with custom headers, parse ReadableStream manually. Libraries like @microsoft/fetch-event-source provide this with reconnection support.

## Q5: How does SSE behavior differ across HTTP/1.1 and HTTP/2 in terms of multiplexing and connection limits?
**A:** HTTP/1.1: browsers limit 6-8 concurrent connections per domain. Each SSE stream uses one connection. Opening multiple SSE streams exhausts the pool, blocking other requests (AJAX, images). HTTP/2: multiplexed streams over a single TCP connection. Multiple SSE connections use separate streams within the same connection, not consuming the connection limit. HTTP/2's binary framing allows interleaving SSE events with other HTTP requests without head-of-line blocking. However, SSE over HTTP/2 has a caveat: some proxies and servers may downgrade SSE to HTTP/1.1. EventSource API behavior is identical regardless of HTTP version - the transport is transparent.

## Q6: How do you implement SSE reconnection with exponential backoff when the server sends a retry field?
**A:** The server sends 'retry: <ms>' field to set the browser's reconnection delay. Default: browser-specific (2-3s). For exponential backoff: the server can update the retry value based on failure count. However, EventSource resets retry to the last received value on each successful connection. Custom implementation: use fetch-based SSE instead of EventSource to control reconnection. Algorithm: initial delay 1s, multiply by 2 after each failure (1, 2, 4, 8, 16s), cap at 30s, add random jitter (0-1000ms). On successful connection, reset to initial delay. Track consecutive failures to detect permanent connection issues. Implement in a wrapper: function createReconnectingSSE(url, options) { /* custom backoff logic */ }.

## Q7: How does SSE handle HTTP response buffering by proxies and how do you configure common proxies to avoid it?
**A:** Proxies buffer HTTP responses by default, which breaks SSE streaming. Solutions: (1) nginx: proxy_buffering off; proxy_cache off; chunked_transfer_encoding on; proxy_http_version 1.1; proxy_set_header Connection '';. (2) Apache: SetEnv no-gzip for SSE endpoint, disable mod_deflate output buffering. (3) HAProxy: option http-server-close; no option http-keep-alive. (4) Cloudflare: automatically handles text/event-stream without buffering. (5) CDNs (Akamai, Fastly): disable caching for SSE path, increase origin timeout. (6) Python/Flask: use Response with direct stream (disable WSGI buffering) or use async framework. (7) Node.js: res.flush() after each write if using compression. (8) General: set Cache-Control: no-cache, no-store, must-revalidate.

## Q8: How do you implement SSE with Redis Pub/Sub for horizontal scaling across multiple server instances?
**A:** Architecture: (1) All SSE server instances subscribe to Redis Pub/Sub channels. (2) When a message needs to be sent to clients, publish to Redis: redis.publish('user:123', JSON.stringify(event)). (3) Each server instance has a Redis subscriber that receives the message and forwards to its locally-connected SSE clients for that user. (4) Connection pool: maintain Map<userId, Set<Response>> per server instance. (5) On receiving Redis message: iterate the user's connections, write event to each, handle write errors (remove disconnected clients). (6) Pattern matching: Redis supports PSUBSCRIBE for channel patterns: PSUBSCRIBE 'user:*' broadcasts to all instances, each filters for its connected users. (7) Cleanup: on server shutdown, unsubscribe from Redis channels. (8) Reliability: Redis Pub/Sub is fire-and-forget - if a server disconnects, messages are lost. Use Redis Streams or a message queue for guaranteed delivery.

## Q9: How does the EventSource stream format differ when handling binary data and what encoding strategies work?
**A:** SSE natively supports only UTF-8 text. Binary data must be encoded: (1) Base64 encoding: data: <base64>\n\n. Increases payload by ~33%. The client decodes: Uint8Array.from(atob(base64), c => c.charCodeAt(0)). (2) Base64URL (URL-safe) for when data may be in URLs. (3) Multipart: send binary in chunks with metadata in separate events. (4) Data URLs with Base64: data:image/png;base64,... (for images). (5) For streaming binary (audio, video), SSE is not appropriate - use WebSocket or raw HTTP streaming. (6) Binary framing: send binary length in a data event, then follow with raw bytes (requires parsing outside EventSource). (7) Recommendation: for binary payloads > 1KB, use WebSocket. For small binary metadata, Base64 encoding is acceptable.

## Q10: How do you implement SSE connection pooling and cleanup in Node.js/Express to prevent memory leaks?
**A:** Connection pool management: (1) Maintain a Map or Set of response objects: const clients = new Map<string, Response>(). (2) On new SSE connection: clients.set(clientId, res); req.on('close', () => clients.delete(clientId)). (3) Periodic cleanup: setInterval to check stale connections: for each client, try writing a heartbeat, if error occurs (socket destroyed), remove from pool. (4) Memory leak prevention: ensure all event listeners are removed on close. Use res.on('close') not req.on('end'). (5) Backpressure detection: check res.writableEnded and res.destroyed before writing. (6) Max connections limit: if clients.size > MAX_CONNECTIONS, reject new connections with 503. (7) Graceful shutdown: on server close, iterate all clients, send goodbye event, call res.end(). (8) Use WeakRef for client tracking if available, or implement manual cleanup intervals.

## Q11: How does SSE handle the WebSocket-like scenario where the client needs to send data back to the server?
**A:** SSE is unidirectional (server -> client). For bidirectional communication without upgrading to WebSocket: (1) HTTP POST for client-to-server messages: the client uses fetch() or XMLHttpRequest to send data to a separate endpoint. The SSE connection handles server-to-client streaming. (2) Multipart approach: use fetch-based SSE with ReadableStream, client can send data via HTTP headers or query params piggybacked on reconnection requests. (3) Long-polling fallback: client sends data via POST, server response may include events (poll-like). (4) Combined SSE + fetch: SSE for push, fetch for pull/commands. (5) Multiplexing over a single connection: not natively possible with EventSource. For bidirectional streaming, WebSocket is the correct choice. SSE + HTTP POST is the pragmatic pattern for adding client interactivity to SSE.

## Q12: How do you implement SSE with gRPC-web streaming as an alternative transport?
**A:** gRPC-web supports server-side streaming. The browser uses grpc-web client to establish a streaming RPC. The server sends protobuf-encoded messages. Comparison to SSE: (1) Binary protocol - no text/event-stream format, uses Protobuf for serialization. (2) Bidirectional support - gRPC-web supports both server-streaming and bidirectional streaming. (3) Requires Envoy proxy or gRPC-web proxy for browser compatibility. (4) Type safety - strongly typed via .proto files. (5) Performance - binary encoding is more efficient than text-based SSE. (6) Complexity - higher setup overhead than SSE. (7) Use case: if your stack already uses gRPC, gRPC-web streaming is a natural fit. Otherwise, SSE is simpler for server-to-client streaming.

## Q13: How does SSE behave when the browser tab is backgrounded and what strategies prevent connection loss?
**A:** Background tab behavior: (1) Browsers throttle JavaScript timers in background tabs (Chrome: 1s, Safari: variable). EventSource onmessage callbacks may be delayed. (2) Network throttling: Chrome may deprioritize SSE connections for background tabs, causing delayed delivery. (3) Connection may stay alive but with reduced throughput. (4) Strategies: (a) Web Worker: run EventSource in a DedicatedWorker - workers are not throttled as aggressively for network connections. (b) Service Worker: intercept SSE connection in service worker for reliable delivery even when tab is backgrounded. (c) Page Visibility API: detect tab hidden -> pause non-critical SSE consumption. (d) Audio wake-lock: if SSE is critical, request a wake lock (navigator.wakeLock.request('screen')) to prevent backgrounding. (e) Heartbeat: ensure server sends heartbeats to keep connection alive through browser throttling.

## Q14: How do you implement SSE with HTTP/2 server push for initial data + streaming updates?
**A:** Combine HTTP/2 Server Push (initial data) with SSE (streaming updates): (1) Client requests SSE endpoint. (2) Server sends HTTP/2 push promise for the initial data resource (e.g., /api/initial-data). (3) Client receives push and caches the initial snapshot. (4) SSE connection delivers incremental updates on top of the initial snapshot. (5) Benefits: initial data loads even faster (no round trip for the initial fetch). (6) Caveats: HTTP/2 Server Push is deprecated in Chrome and being removed - use 103 Early Hints instead. (7) Alternative: SSE sends initial snapshot as first event, then incremental updates. Simpler and more reliable than push. (8) For HTTP/2 multiplexing benefit alone: SSE over HTTP/2 provides multiplexing without Server Push - multiple SSE streams share one TCP connection.

## Q15: How do you implement SSE authentication with JWT tokens and handle token expiration during long-lived connections?
**A:** JWT with SSE: (1) Initial authentication: client gets JWT via login endpoint. (2) SSE connection: new EventSource('/events?token=' + jwt) or use fetch-based SSE with Authorization header. (3) Token expiration handling: JWTs have short expiry (15-60 min). SSE connections may last hours. Token expires -> connection drops. (4) Refresh strategy: (a) Short-lived token: before token expires, client establishes a NEW SSE connection with a fresh token, old connection closes. (b) Long-lived session token: use a session token (server-generated UUID) instead of JWT for SSE. The session token maps to the user's authenticated session server-side. (c) Silent refresh: client periodically calls /refresh endpoint (via fetch) to get a new token, then reconnects SSE with new token. (d) Server-side token validation: server validates token on connection and periodically (every 5 min) re-validates - if expired, sends a 'token_expired' event and closes connection. (5) Reconnection with new token: on token refresh, close old EventSource, create new one with new token. Maintain event buffer for continuity.

## Q17: How do you design an SSE system that handles 100K+ concurrent connections on a single machine?
**A:** Scaling to 100K+ connections: (1) Event-driven architecture: Node.js, Python asyncio, or Go (goroutines) for non-blocking I/O. Each connection should use < 1MB memory. (2) OS tuning: fs.file-max to 1M+; ulimit -n 1048576; net.ipv4.tcp_tw_reuse=1; net.core.somaxconn=65535; increase ephemeral port range. (3) Connection pooling: maintain Map<userId, Set<Response>>. Use efficient data structures (Map with WeakRef for cleanup). (4) Backpressure: monitor socket buffer size. If client is slow (TCP window full), drop non-critical events or disconnect with 'slow client' error. (5) Batching: coalesce events over small time window (50-100ms) before writing to reduce system calls. (6) Memory: each connection uses ~200KB for kernel buffers + ~10KB for app state. 100K connections = ~20GB. (7) Clustering: use Node.js cluster or run multiple processes sharing Redis for event distribution. (8) Monitoring: track active connections, write errors, event throughput. Use Prometheus metrics. (9) Graceful degradation: implement connection priority - premium users get guaranteed delivery, free users may be dropped under load.

## Q19: How do you implement SSE event deduplication at the client level using event IDs?
**A:** Client-side deduplication: (1) Track processed event IDs: const processedIds = new Set(). (2) On receiving event with 'id' field: if (processedIds.has(event.lastEventId)) { return } // skip duplicate. (3) Add to set: processedIds.add(event.lastEventId). (4) Size management: processedIds can grow unbounded. Use a bounded LRU cache: const processedIds = new LRUCache({ max: 10000 }). (5) On reconnection: the Last-Event-ID header may cause the server to replay the last event. Deduplication catches this. (6) Window-based dedup: only track IDs within a time window (e.g., last 5 minutes). Events older than that are safe to process. (7) ID format: use monotonically increasing IDs (timestamp + sequence) so the client can also detect gaps. (8) Gap detection: if event IDs are sequential and the client detects a gap, it knows events were missed. Request a catch-up from the server.

## Q21: How does SSE behave with HTTP/2 flow control and how do you prevent connection stalls?
**A:** HTTP/2 flow control: (1) Each stream has a flow control window (initial: 65,535 bytes for both connection and stream). (2) As the server sends SSE data, the window depletes. (3) When window is empty, the server must wait for the client to send WINDOW_UPDATE frames. (4) If client is slow reading events (e.g., JS processing takes time), the TCP socket buffers fill, the client stops sending WINDOW_UPDATE, and the SSE stream stalls. (5) Prevention: (a) Keep SSE events small (< 64KB) to avoid exhausting the initial window. (b) Monitor write completion callbacks to detect backpressure. (c) If using Node.js, check socket.writableLength for buffered data amount. (d) Implement application-level flow control: if buffer exceeds threshold, drop non-critical events. (e) Use HTTP/2's stream priority to deprioritize event streams when congested. (f) For large payloads, chunk events into smaller pieces. HTTP/2 flow control is transparent to EventSource API but affects throughput.

(6) Stateful token: generate a short-lived SSE token tied to the session, include as query param: /events?token=<uuid>. Validate on connection. (7) Short expiry: SSE token expires after 5 minutes. Client reconnects with fresh token. (8) HTTPS required: all cookies and tokens must be transmitted over HTTPS to prevent interception. (9) CORS: for cross-origin SSE, set Access-Control-Allow-Origin and withCredentials: true. Specific origin, not wildcard.

## Q23: How do you implement SSE with backpressure handling using Node.js streams?
**A:** Node.js backpressure for SSE: (1) Check writableHighWaterMark (default 16KB) and writableLength (buffered bytes). If writableLength > writableHighWaterMark, the stream is under backpressure. (2) Implement drain event: if res.write() returns false, pause event production, wait for 'drain' event, then resume. (3) Backpressure strategy: const canWrite = res.write(data); if (!canWrite) { isBackpressured = true; resumeProducer = pausedResume; }. On 'drain': isBackpressured = false; resumeProducer(). (4) Drop strategy: if backpressure persists beyond timeout, drop non-critical events: if (backpressured && event.priority < critical) { return }. (5) Buffer management: set highWaterMark: 65536 (64KB) for SSE to allow some buffering. (6) Slow consumer detection: measure write latency. If each write takes > 100ms, the client is slow. Disconnect with 'too slow' or downgrade to polling. (7) Graceful degradation: on persistent backpressure, send 'slow_connection' event and close the SSE connection, instructing client to switch to polling.

## Q24: How do you implement SSE with message queuing (RabbitMQ) for reliable delivery across restarts?
**A:** RabbitMQ + SSE: (1) Each client connection declares a unique queue (per connection or per user). (2) Server publishes events to an exchange, routed to the appropriate queue. (3) SSE generator consumes from the queue: channel.consume(queueName, (msg) => { res.write(formatSSE(msg)); channel.ack(msg) }). (4) Queue durability: durable queues survive broker restarts. Events are persisted to disk. (5) On reconnection: client sends Last-Event-ID. Server replays unacknowledged messages from the queue (checking message headers for event IDs). (6) Queue TTL: set message TTL (e.g., 1 hour) to auto-purge old events. (7) Consumer acknowledgements: ack only after event is successfully written to SSE socket. If write fails, nack (requeue) for retry. (8) Dead letter: messages that repeatedly fail delivery go to a dead-letter queue for analysis. (9) Performance: RabbitMQ handles 10K+ queues easily. Each queue is lightweight. Use topic exchanges for routing flexibility.

## Q27: How do you implement SSE with automatic reconnection and event catch-up using a server-side event buffer?
**A:** Server-side event buffer: (1) Maintain a circular buffer (Redis sorted set or in-memory deque) for recent events per user/channel. (2) On connection, client sends Last-Event-ID. Server queries buffer for events after that ID. (3) Catch-up: for (const event of buffer.getAfter(lastEventId)) { res.write(formatSSE(event)) }. (4) Buffer size: fixed capacity (e.g., 1000 events or 10MB) to limit memory. (5) Buffer TTL: events older than 5 minutes are purged. (6) On reconnection: after catch-up, the client is synchronized and receives live events. (7) Gap detection: if the requested event ID is no longer in buffer, client needs a full snapshot. Server sends a 'snapshot' event with complete state, followed by incremental updates. (8) Implementation: Redis sorted set with event timestamp as score. ZREVRANGEBYSCORE to get events after a timestamp. (9) For high-volume systems, buffer per channel rather than per user.

(7) Server setup: nginx 1.25+ with HTTP/3 module, Caddy (automatic), Cloudflare/Fastly (enabled by default). (8) Fallback: if client/network lacks HTTP/3, browser auto-downgrades to HTTP/2 or HTTP/1.1. (9) Performance: on lossy networks (>1% packet loss), HTTP/3 SSE delivers 15-40% lower latency than HTTP/2 SSE due to QUIC's independent stream handling and faster loss recovery. (10) Impact: connection migration alone eliminates most SSE reconnections on mobile, massively reducing event replay complexity.

## Q84: How do you implement SSE reconnection with advanced jitter strategies (full jitter, equal jitter, decorrelated jitter) to prevent thundering herds?
**A:** Advanced jitter strategies: (1) Full jitter (AWS recommended): `delay = random(0, min(cap, base * 2^attempt))`. Maximal spread for thundering herd prevention. (2) Equal jitter: `delay = exp/2 + random(0, exp/2)`. Half deterministic, half random. More predictable than full jitter. (3) Decorrelated jitter: `delay = min(cap, random(base, prevDelay * 3))`. Self-correcting, no attempt counter. Best for intermittent failures. (4) Implementation: `class ReconnectingSSE { reconnect() { const delay = this.jitterFn(this.attempt++, base, cap); setTimeout(() => this.connect(), delay) } }`. (5) Native EventSource limitation: no custom jitter. Use fetch-based SSE (e.g., @microsoft/fetch-event-source) with custom `onreconnect`. (6) Server-side retry jitter: send randomized `retry` values. Each client gets different retry. But browser applies it after disconnect — not ideal. (7) Success reset: reset attempt=0 on successful connection. (8) Failure cap: after 10 consecutive failures, cap at maxDelay, show user-visible indicator. (9) Network-aware: effectiveType 'slow-2g' gets 10s base; '4g' gets 1s base. (10) Thundering herd mitigation: after server restart, full jitter spreads reconnections across [0, cap] window, preventing overload.

## Q98: How do you implement SSE with BroadcastChannel API for cross-tab synchronization and shared single-connection ownership?
**A:** BroadcastChannel for cross-tab SSE: (1) Problem: each tab opens its own SSE connection, wasting server resources. (2) Solution: one tab owns the connection, shares events via BroadcastChannel: `const channel = new BroadcastChannel('sse-events')`. (3) Tab election: first tab creates SSE: `if (!sharedSSE) { sharedSSE = new EventSource(url); sharedSSE.onmessage = (e) => channel.postMessage(e.data) }`. (4) Late tabs: receive events via BroadcastChannel. Do not create their own SSE connection. Read last event from localStorage for initialization. (5) Tab crash recovery: if owning tab crashes, remaining tabs detect via channel 'messageerror' or absence. A remaining tab takes over SSE. (6) SharedWorker approach: SharedWorker opens ONE SSE connection, posts to all connected tabs via `self.onconnect`. Survives tab closure as long as 1+ tabs remain. (7) Leader election: BroadcastChannel with 'leader' role. Leader sends heartbeat. If no heartbeat for 5s, election held, new leader opens SSE. (8) Deduplication: all tabs receive same events via channel. Use event IDs for deduplication if side effects matter. (9) Backpressure: BroadcastChannel buffers for backgrounded tabs. Tab catches up when foregrounded. (10) Benefit: 1 SSE connection for N tabs — reduces server load by 50-80% in multi-tab apps (email, dashboards).

## Q99: How do you implement SSE with WebAssembly for client-side event processing and transformation?
**A:** Wasm for SSE event processing: (1) Use case: client-side event transformation, filtering, decompression, or decryption at near-native speed. (2) Load Wasm: `const wasm = await WebAssembly.instantiateStreaming(fetch('event-processor.wasm'))`. (3) Pipeline: EventSource receives raw event -> postMessage to Wasm worker -> Wasm processes (filter, transform, decode) -> returns to main thread. (4) Decompression: Wasm zlib/brotli 2-3x faster than JS pako for large payloads: `wasm.exports.decompress(ptr, len)`. (5) Filtering: Wasm evaluates complex rules faster than JS: `wasm.exports.evaluate(eventPtr, len, rulePtr, ruleLen)`. (6) Binary parsing: Base64-decoded binary events parsed efficiently in Wasm: `wasm.exports.parseBinary(base64Ptr, len)`. (7) Web Worker: run Wasm in Worker to avoid blocking main thread: `worker.postMessage(event); worker.onmessage = (e) => updateUI(e.data)`. (8) Security: Wasm is sandboxed. Safely execute user-provided filter rules without XSS risk. (9) Performance: 30-50% faster than JS for CPU-intensive event processing. For I/O-bound consumption, JS is sufficient. (10) Trade-off: Wasm adds complexity (compilation, memory management, marshaling). Use for >100 events/sec requiring complex processing. Future Wasm GC will improve interop.

## Q100: What are the proposed future developments for the SSE specification (WHATWG) and what new features can we expect?
**A:** SSE future developments: (1) Current spec: SSE is part of WHATWG HTML Standard (EventSource interface). Last major update was 2015. (2) Binary support: proposal for native binary event data (EventSource with ArrayBuffer), eliminating Base64 overhead. (3) Custom HTTP methods and headers: proposal for EventSource constructor to accept Request objects: `new EventSource(new Request(url, { method: 'POST', headers: {...} }))`. Enables flexible auth. (4) Bidirectional: proposal for `EventSource.send()` to send data back over the same connection. Merges SSE with simple bidirectional needs. (5) Compression negotiation: built-in per-event compression via `Accept-Encoding: sse-gzip`. Server compresses, client decompresses transparently. (6) Stream priorities: `EventSource.setPriority('high')` to influence browser's connection scheduling. (7) CORS relaxation: proposal to allow custom headers in EventSource without preflight for GET requests with Authorization. (8) Connection health API: `navigator.connection.ondowngrade` and `EventSource.connectionState === 'degraded'` for better network awareness. (9) Last-Event-ID in response headers: server can send Last-Event-ID as HTTP header instead of event field. (10) WebTransport integration: EventSource over WebTransport (QUIC) for 0-RTT reconnection and connection migration. (11) Retry policy in client: standardized retry config in constructor options (maxRetries, backoff, jitter) instead of relying on server retry field. (12) Service Worker integration: native EventSource support in Service Workers (currently SW fetch can intercept but not directly create EventSource). (13) Streams API integration: ReadableStream-based SSE replacing callback-based EventSource for better backpressure handling.
