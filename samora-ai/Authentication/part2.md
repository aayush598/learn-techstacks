# Authentication Interview Questions and Answers - Part 2

## Q1: How does the OAuth 2.0 Backend for Frontend (BFF) pattern improve security for SPAs?
**A:** The BFF pattern introduces a lightweight backend proxy between the SPA and the authorization server. The SPA never holds tokens directly — instead, the BFF manages the OAuth flow, stores tokens in HTTP-only secure session cookies, and proxies API requests. This eliminates token exposure to browser storage (XSS vulnerable) and prevents attackers from extracting tokens even if JavaScript is compromised. The BFF also handles refresh token rotation transparently.

## Q2: What is the OAuth 2.0 Mutual TLS (mTLS) Client Authentication and how does it work?
**A:** Defined in RFC 8705, mTLS client authentication allows confidential clients to authenticate to the authorization server using X.509 certificates during the TLS handshake. The authorization server validates the client certificate against registered certificates. This provides stronger security than client_secret_basic or client_secret_post, especially in high-security environments. It supports two modes: self-signed certificates (bound to a specific client) and PKI-issued certificates (validated against a trusted CA).

## Q3: How do you implement refresh token rotation with automatic reuse detection?
**A:** Each time a refresh token is used to obtain a new access token, the old refresh token is invalidated and a new refresh token is returned. Reuse detection works by monitoring if a revoked refresh token is ever presented again — this signals token theft. Upon detection, all refresh tokens for that user-client combination are immediately revoked, forcing re-authentication. Implementation requires storing a token family identifier with a generation counter; when the presented generation doesn't match the stored one, theft is assumed.

## Q4: What is the OAuth 2.0 Pushed Authorization Request (PAR) and why is it important?
**A:** PAR (RFC 9126) allows clients to push the authorization request payload directly to the authorization server via a POST request before redirecting the user. The server returns a request URI (`urn:ietf:params:oauth:request_uri:...`) which replaces the large query string in the browser redirect. PAR prevents manipulation of authorization parameters via the browser, eliminates URL length limits, and hides sensitive parameters (like `scope` or custom claims) from the user's browser history.

## Q5: What is a JWT Secured Authorization Request (JAR) and how does it differ from PAR?
**A:** JAR (RFC 9101) signs and/or encrypts the entire authorization request as a JWT. The client creates a Request Object containing all OAuth parameters, signed with its private key or encrypted with the authorization server's public key. This ensures integrity, authenticity, and confidentiality of the request. PAR and JAR are complementary — PAR pushes the request to the server; JAR secures the content. They are often combined: the Request Object is pushed via PAR.

## Q6: How does the OAuth 2.0 Rich Authorization Request (RAR) extend authorization granularity?
**A:** RAR (RFC 9396) introduces the `authorization_details` parameter, replacing or supplementing the traditional `scope` parameter. It allows structured, JSON-format authorization requests with typed details. For example: requesting access to specific bank accounts with transaction limits, or specific API endpoints with particular methods. The authorization server can present these granular details to the user for informed consent. RAR enables fine-grained delegation that scopes alone cannot express.

## Q7: What is DPoP (Demonstration of Proof of Possession) in OAuth 2.0?
**A:** DPoP (RFC 9449) binds access tokens to a specific client by requiring proof of possession of a private key. The client generates a DPoP key pair, sends the public key to the authorization server via a `cnf` (confirmation) claim in the JWT, and includes a DPoP header with each API request proving possession of the private key. This prevents token theft and replay: even if an access token is intercepted, the attacker cannot use it without the corresponding private key.

## Q8: How does OAuth 2.0 Token Binding prevent token theft?
**A:** Token Binding (RFC 8471) cryptographically binds tokens to the TLS connection between the client and the authorization server. The client proves possession of a private key that is tied to the TLS session. A bound token includes the `tbh` (token binding hash) claim. If an attacker steals the token, they cannot use it because their TLS connection won't have the same token binding. DPoP has largely superseded Token Binding due to simpler implementation requirements.

## Q9: What is the OAuth 2.0 Token Exchange grant type and when should you use it?
**A:** The Token Exchange grant type (RFC 8692) allows a client to exchange one token for another with transformed attributes or a different target service. Common uses: (1) Downstream service delegation — exchanging an access token for one scoped to a downstream API. (2) Impersonation — exchanging a token for one representing a different user. (3) Token chaining in microservice architectures. The `resource` parameter specifies the target service; `actor_token` specifies who is acting.

## Q10: How do you implement OAuth 2.0 authorization for native mobile apps using the AppAuth pattern?
**A:** The recommended pattern uses the system browser (Chrome Custom Tabs on Android, ASWebAuthenticationSession on iOS) for the authorization request. This ensures the user sees the real authorization server URL. The app registers a custom URL scheme or universal link for the redirect URI. After authorization, the code is caught by the OS and delivered to the app. PKCE is mandatory. The app should never use embedded WebViews for OAuth flows, as the user cannot trust the URL shown in an embedded WebView.

## Q11: What is the Identity, Security, and Risk (ISR) profile in OpenID Connect?
**A:** The ISR profile extends OIDC with enhanced identity assurance, security mechanisms, and risk signaling. It includes: verified claims (identity proofing level), fraud indicators, session management with logout tokens, and risk event sharing. ISR defines how Identity Providers share assurance levels, verification methods, and fraud signals with Relying Parties. This enables informed authorization decisions based on identity confidence.

## Q12: How does OpenID Connect for Identity Assurance (OIDC4IDA) work?
**A:** OIDC4IDA extends OIDC with verified claims about a user's identity. It defines new claims like `verified_claims` which include: (1) `verification` object — who performed verification, when, and to what assurance level. (2) `claims` — the actual verified attributes (name, birthdate, address). The verification method (e.g., document check, biometric verification) is communicated. Relying parties can request specific assurance levels via `acr` values.

## Q13: What is Self-Issued OpenID Provider (SIOP) and how does it enable decentralized identity?
**A:** SIOP allows users to act as their own OpenID Provider using a wallet or identity application on their device. The user's wallet generates keys, issues ID tokens, and presents them to Relying Parties. No centralized IdP is required. SIOP v2, combined with OpenID Connect for Verifiable Presentations (OIDC4VP), enables self-sovereign identity flows where users control their identity data and selectively disclose claims.

## Q14: How does the OpenID Connect Back-Channel Logout mechanism work?
**A:** Back-Channel Logout (OIDC Back-Channel) allows the OP to directly notify RPs when a user logs out, without relying on browser redirects. The OP constructs a Logout Token (a JWT with `events` claim containing `http://schemas.openid.net/event/backchannel-logout`) and sends it as an HTTP POST to the RP's back-channel logout endpoint. This ensures logout across all sessions even if the user has closed the RP's browser tab.

## Q15: How does OpenID Connect Front-Channel Logout work and what are its vulnerabilities?
**A:** Front-Channel Logout uses iframes on the OP's logout page that load logout URLs from each RP. The OP renders an iframe for each RP's logout endpoint, and the RP clears the session via Set-Cookie clearing. Vulnerabilities: (1) Third-party cookie blocking in modern browsers prevents iframe-based logout from working. (2) It relies on browser redirects — if the user navigates away, logout is incomplete. (3) Timing attacks — RPs must process logout within iframe timeout. Back-Channel Logout is more reliable.

## Q16: What is the difference between RP-Initiated Logout and Session Management in OIDC?
**A:** RP-Initiated Logout (defined in OIDC RP-Initiated Logout 1.0) is a request from the RP to log the user out of the OP. The RP redirects the user to the OP's `end_session_endpoint` with an `id_token_hint`. The OP logs the user out of the OP session and optionally logs the user out of all RPs. Session Management (OIDC Session Management 1.0) uses an iframe and the `postMessage` API for the RP to monitor the OP session state, enabling the RP to detect when the user logs out at the OP level.

## Q17: What is the SCIM (System for Cross-domain Identity Management) protocol and how does it integrate with authentication?
**A:** SCIM (RFC 7642-7644) is a RESTful protocol for automating identity lifecycle management across domains. It defines schemas for users, groups, and enterprises, and CRUD operations for provisioning identities. Integration with authentication: (1) When a user is created in the IdP (e.g., Okta), SCIM automatically provisions the account in downstream SaaS apps. (2) When a user is deactivated, SCIM removes access across all systems. (3) Group membership changes propagate automatically, ensuring authorization stays current.

## Q18: How do you implement a distributed session management system for authentication?
**A:** A distributed session store uses Redis or a database cluster to share session data across application instances. Key considerations: (1) Use sticky sessions (session affinity) or a centralized store. (2) Centralized store requires high availability — Redis Sentinel or Cluster. (3) Session data should be minimal (just a user ID + last activity) with full data fetched from the database per request. (4) Implement session regeneration on privilege escalation. (5) Handle split-brain scenarios with conflict resolution. (6) Use encrypted session cookies as fallback.

## Q19: How does the CAEP (Continuous Access Evaluation Protocol) work?
**A:** CAEP (Continuous Access Evaluation Protocol) enables real-time security event sharing between IdPs, RPs, and policy engines. When a risk event occurs (password change, device compromise, admin revokes session), the IdP emits a CAEP event via a shared stream or direct push. RPs subscribe to events and enforce immediate re-evaluation of access decisions. CAEP events include: session revoked, credential change, device compliance change, and assurance level change. This moves from point-in-time to continuous authorization.

## Q20: What is the Shared Signals Framework (SSF) and how does it relate to authentication?
**A:** SSF (RFC 9493-9495) is a framework for transmitting security events between parties. It defines a standardized event distribution mechanism using streams. In authentication, SSF enables: (1) Risk Incident Sharing and Coordination (RISC) — sharing security events between IdPs and RPs. (2) CAEP — continuous access evaluation. (3) Security event tokens (SET) carry event payloads. SSF replaces polling-based approaches with push-based event notification, enabling real-time response to security incidents across a federation.

## Q21: How do you implement phishing-resistant authentication using FIDO2/WebAuthn?
**A:** Phishing-resistant authentication with WebAuthn involves: (1) Server generates a registration challenge. (2) Client creates a key pair using `navigator.credentials.create()`, with `rpId` set to the application's origin. (3) The private key never leaves the authenticator. (4) Authentication uses `navigator.credentials.get()` — the authenticator signs a challenge that includes the `rpId`. (5) The signature is bound to the origin — if a user visits a phishing site, `rpId` won't match and authentication fails. This prevents credential harvesting and relay attacks.

## Q22: What is the difference between platform authenticators, roaming authenticators, and cross-platform authenticators in WebAuthn?
**A:** Platform authenticators are built into the device (Touch ID, Windows Hello, Android fingerprint). They cannot be removed. Roaming authenticators (cross-platform) are external devices like YubiKeys that can be used across devices — they connect via USB, NFC, or BLE. Cross-platform authenticators implement CTAP (Client to Authenticator Protocol) over these transports. Password managers increasingly act as cross-platform authenticators using FIDO2. Platform authenticators offer convenience; roaming authenticators provide portability and stronger security.

## Q23: How do passkeys sync across devices while maintaining security?
**A:** Passkeys use cloud-based key escrow managed by the platform vendor (Apple iCloud Keychain, Google Password Manager, Microsoft Account). The user's private key is encrypted with a device-specific key and uploaded to the vendor's cloud. When the user authenticates on a new device, the encrypted private key is downloaded and decrypted using the user's cloud account credentials (or a recovery key). The platform ensures end-to-end encryption — even the vendor cannot access the private key. This balances security with usability, though some argue it centralizes trust in platform vendors.

## Q24: How do you implement step-up authentication for high-risk operations?
**A:** Step-up authentication requires additional verification when the user performs sensitive actions. Implementation: (1) The application checks the current authentication assurance level (AAL) against the operation's required level. (2) If insufficient, the application initiates a step-up flow via OIDC `acr_values` parameter requesting a higher AAL. (3) The IdP prompts for additional factors (TOTP, biometric, hardware key). (4) The IdP returns an ID token with the achieved ACR. (5) The application validates and grants access. Common step-up triggers: financial transactions, password changes, accessing PII, admin actions.

## Q25: How does the OAuth 2.0 Device Authorization Grant work for IoT devices?
**A:** In the Device Authorization Grant (RFC 8628), the device: (1) Requests a device code and user code from the authorization server. (2) Displays the user code and a verification URL. (3) The user visits the URL on another device, enters the code, and authenticates. (4) The original device polls the token endpoint until the user completes authentication. (5) The device receives an access token. This is ideal for devices with limited input (smart TVs, IoT, CLI tools). The device must poll at appropriate intervals to avoid overwhelming the authorization server.

## Q26: How do you implement OAuth 2.0 for service-to-service (M2M) communication at scale?
**A:** For M2M communication: (1) Use the Client Credentials grant with private_key_jwt client authentication (stronger than client_secret). (2) Use short-lived access tokens (5-15 minutes) to limit damage from leaks. (3) Implement token caching on the client to avoid repeated token requests. (4) Use audience-restricted tokens — each token is scoped to a specific downstream service. (5) Implement token exchange for multi-hop requests (service A -> service B -> service C). (6) Rotate client credentials periodically. (7) Use mTLS for additional transport security.

## Q27: What is the OAuth 2.0 Authorization Server Metadata (RFC 8414) and how is it used?
**A:** RFC 8414 defines a well-known metadata endpoint (`.well-known/oauth-authorization-server`) that returns the authorization server's configuration as JSON. It includes: `authorization_endpoint`, `token_endpoint`, `issuer`, `jwks_uri`, `scopes_supported`, `response_types_supported`, `grant_types_supported`, `code_challenge_methods_supported`, `token_endpoint_auth_methods_supported`, and more. Clients dynamically discover server capabilities, eliminating hardcoded configuration. OIDC extends this with `.well-known/openid-configuration`.

## Q28: How does the JWT Algorithm Confusion attack work and how do you prevent it?
**A:** The JWT Algorithm Confusion attack exploits the server's trust in the JWT header. If the server accepts tokens signed with public-key algorithms (RS256), an attacker can change the header to use a symmetric algorithm (HS256) and sign the token using the server's public key as the HMAC secret. Prevention: (1) Always validate that the algorithm matches what's expected — reject tokens with unexpected `alg` values. (2) Use a library that explicitly binds the expected algorithm. (3) Use `alg: "none"` rejection as a baseline. (4) Use asymmetric algorithms with strict key type validation. (5) Set algorithm-specific key usage restrictions.

## Q29: How do you implement a secure JWT key management and rotation system?
**A:** A production JWT key management system should: (1) Use asymmetric keys (RS256/ES256) with the private key held only by the authorization server. (2) Expose public keys via a JWKS (JSON Web Key Set) endpoint at a well-known URL. (3) Use key IDs (`kid`) in JWT headers to identify which key was used. (4) Rotate keys on a schedule (every 30-90 days) and immediately on compromise. (5) Support overlapping key validity — keep the previous key's public key available for token verification until all tokens signed with it expire. (6) Store private keys in a hardware security module (HSM) or secrets manager.

## Q30: What is the JWK Thumbprint and how is it used in OAuth 2.0?
**A:** A JWK Thumbprint (RFC 7638) is a hash of a JWK's required members, computed using SHA-256. It provides a unique, deterministic identifier for a public key. In OAuth 2.0, it's used in: (1) DPoP — the `cnf` claim contains the JWK Thumbprint binding the token to a specific key. (2) OAuth Proof of Possession — client authentication without shared secrets. (3) Token Binding — identifying the bound key. The thumbprint is computed from the JWK representation of the public key, making it format-independent.

## Q31: How does NIST SP 800-63 define authentication assurance levels (AAL)?
**A:** NIST SP 800-63 defines three Authenticator Assurance Levels: AAL1 — single-factor authentication (password, biometric). AAL2 — multi-factor authentication with at least two different factors, where at least one factor is verifier-impersonation resistant (e.g., password + TOTP or FIDO2). AAL3 — multi-factor with hardware-based authenticators and verifier-impersonation resistance (e.g., FIDO2 hardware security key + PIN). AAL3 requires the authenticator to be a hardware device that prevents duplication and provides tamper resistance.

## Q32: How do you implement authentication for a serverless architecture (AWS Lambda, Cloud Functions)?
**A:** Serverless authentication patterns: (1) API Gateway Lambda Authorizer — custom function validates tokens and returns an IAM policy dictating access. (2) Cognito User Pools with API Gateway — Cognito validates tokens directly, no Lambda needed. (3) Lambda@Edge for CloudFront-based auth. (4) JWT validation in the function runtime itself (using middleware). Best practices: (1) Cache token validation results to reduce latency. (2) Use Lambda authorizer caching with configurable TTL. (3) Keep JWKS cached in Lambda's global scope (outside handler). (4) Minimize function cold start by keeping auth libraries lightweight.

## Q33: How does the BFF pattern handle token refresh in a SPA?
**A:** The BFF manages the entire OAuth flow: (1) SPA calls a `/api/auth/login` endpoint on the BFF. (2) BFF redirects to the authorization server with PKCE, or uses a server-side OAuth flow. (3) Authorization server redirects back to the BFF's callback endpoint. (4) BFF exchanges the code for tokens, stores them in an HTTP-only secure session cookie. (5) SPA uses the session cookie (opaque to JavaScript). (6) When the access token expires, the SPA gets a 401, the BFF transparently uses the refresh token to get a new access token, retries the request. The SPA never knows about tokens.

## Q34: What is the OAuth 2.0 Response Mode parameter and what are its values?
**A:** The `response_mode` parameter controls how the authorization response is returned to the client. Values: `query` — parameters in URL query string (default for Authorization Code flow). `fragment` — parameters in URL fragment (default for Implicit flow, not recommended). `form_post` — parameters in an HTML form POST auto-submitted to the redirect URI (useful when URL length is a concern). `jwt` (extension) — response as a JWT. `query.jwt`, `fragment.jwt` — combinations. `form_post.jwt` is used with JAR and PAR for secure transmission.

## Q35: How do you detect and prevent credential stuffing attacks at the authentication layer?
**A:** Detection methods: (1) Monitor failed login rate per IP, per user, and across the platform. (2) Look for patterns — high velocity attempts from distributed IPs, use of common passwords, attempts on multiple accounts from same session. (3) Check leaked credential databases (Have I Been Pwned API). (4) Use device fingerprinting to detect automation. Prevention: (1) Rate limiting with exponential backoff. (2) CAPTCHA after N failed attempts. (3) Account lockout with progressive delays. (4) Require MFA for suspicious attempts. (5) Passwordless authentication eliminates the attack vector entirely.

## Q36: How do you implement a secure password change flow with session management?
**A:** A secure password change flow: (1) Require current password verification. (2) Enforce new password policy (length, complexity, not in breached password DB). (3) Invalidate all existing sessions except the current one. (4) Notify the user via email/SMS about the password change. (5) Require re-authentication for high-risk operations after password change. (6) Update password history to prevent reuse. (7) If password was changed by an admin (not the user), force user re-authentication with a temporary password. (8) Emit a security event for RISC/CAEP subscribers.

## Q37: How do you implement authentication for a GraphQL API?
**A:** GraphQL authentication strategies: (1) Token in the Authorization header (same as REST) — validate in a middleware before the GraphQL handler. (2) Per-resolver authorization — extract user context from token and check permissions in each resolver. (3) Schema directives — `@auth`, `@requires` directives for declarative authorization. (4) Use a data loader pattern that checks authorization during data fetching. (5) For GraphQL over WebSocket (subscriptions), validate the connection handshake and re-validate for sensitive data. Never pass credentials in GraphQL query variables — use headers.

## Q38: How does the OAuth 2.0 Token Revocation endpoint work and what are its limitations?
**A:** The Token Revocation endpoint (RFC 7009) allows clients to revoke access or refresh tokens. The client POSTs the token (and optionally `token_type_hint`) to the revocation endpoint, authenticating with its client credentials. The server marks the token as invalid. Limitations: (1) Only revocable if the server maintains state — stateless JWTs cannot be revoked without a blacklist. (2) The client must be authenticated, so public clients (SPAs, mobile apps) without a client secret cannot reliably use revocation. (3) Revocation is at the server level — already-issued token validation results at resource servers may be cached.

## Q39: How do you implement an OAuth 2.0 Token Introspection endpoint?
**A:** Token Introspection (RFC 7662) allows resource servers to query the authorization server for token validity and metadata. The resource server POSTs the token to the introspection endpoint (authenticated with its own client credentials). The response includes `active: true/false`, plus claims like `scope`, `client_id`, `username`, `token_type`, `exp`, `iat`, `nbf`, `sub`, `aud`, `iss`, and `cnf`. This enables resource servers to make authorization decisions without validating JWTs directly. Performance concern: every request requires a network call to the introspection endpoint, so responses should be cached appropriately.

## Q40: What is the difference between opaque tokens and structured tokens (JWTs) for authorization servers?
**A:** Opaque tokens are random strings stored in the authorization server's database. They require token introspection for validation. Benefits: (1) Can be revoked immediately (delete from DB). (2) No sensitive data exposed in token. (3) Smaller token size. Structured tokens (JWTs) are self-contained. Benefits: (0) No server-side lookup needed for validation. (1) Rich claims available to resource servers. (2) Works offline. Trade-offs: JWTs cannot be revoked until expiration without a blacklist; opaque tokens require network calls for validation. Many authorization servers use opaque tokens for the resource server and provide JWTs for ID tokens.

## Q41: How does federated authentication work with SAML 2.0 in enterprise environments?
**A:** In SAML 2.0 federation: (1) The user accesses a Service Provider (SP) resource. (2) SP generates an AuthnRequest and redirects the user to the Identity Provider (IdP). (3) The user authenticates at the IdP (password, MFA, etc.). (4) IdP generates a SAML Assertion (XML signed with the IdP's certificate) containing the user's identity, attributes (groups, roles), and authentication context. (5) IdP POSTs the assertion to the SP's ACS (Assertion Consumer Service) URL. (6) SP validates the signature, processes the assertion, and creates a local session. Federation metadata (XML) is exchanged between SP and IdP to establish trust.

## Q42: What is SAML Attribute Query and when is it used?
**A:** SAML Attribute Query allows an SP to request additional user attributes from the IdP after the initial authentication. The SP sends an `<AttributeQuery>` message referencing the user (by `NameID`) and specifying requested attributes. The IdP responds with a SAML assertion containing the requested attribute values. This is useful when: (1) The original assertion didn't include all needed attributes. (2) Attributes change post-authentication. (3) The SP needs attribute updates during a long session. Attribute Queries use the same SOAP binding as Artifact Resolution.

## Q43: What is SAML Artifact Binding and when would you use it?
**A:** SAML Artifact Binding is an alternative to the HTTP POST binding for SAML responses. Instead of sending the full SAML assertion via the browser (which can be large and exposes assertions to the browser), the IdP sends a small artifact (a reference to the assertion) via the browser redirect. The SP then resolves the artifact by making a direct back-channel SOAP call to the IdP's Artifact Resolution Service to retrieve the actual assertion. Benefits: smaller URLs, assertions never exposed to the browser, works with larger assertion payloads. Used in high-security enterprise deployments.

## Q44: How does SAML Single Logout (SLO) work and what are its failure modes?
**A:** SAML SLO propagates logout across all SPs. The user initiates logout at one SP, which sends a LogoutRequest to the IdP. The IdP identifies all sessions and sends LogoutRequests to all other SPs. Each SP responds with a LogoutResponse. After collecting all responses, the IdP responds to the initiating SP. Failure modes: (1) If any SP is offline, SLO hangs or fails. (2) Session management across SPs is inconsistent — some SPs may not support SLO. (3) Browsers may block third-party cookies needed for session identification. (4) The SLO flow is synchronous and timeout-prone. Many deployments use a session timeout approach instead of SLO.

## Q45: What is the SAML Metadata XML structure and how is it used for federation?
**A:** SAML Metadata is an XML document that describes an IdP or SP and its capabilities. For IdPs: `SingleSignOnService` URLs (bindings), `SingleLogoutService` URLs, X.509 signing and encryption certificates, `NameIDFormat` supported, attributes offered. For SPs: `AssertionConsumerService` URLs, `SingleLogoutService` URLs, certificate for signature verification. Metadata is exchanged between organizations (via URL or file) to establish trust. The metadata can be signed to prevent tampering. Dynamic metadata exchange is supported via the `md:EntitiesDescriptor` for representing a federation of entities.

## Q46: How does WS-Federation differ from SAML and OIDC?
**A:** WS-Federation (part of the WS-* stack) is a Microsoft-developed protocol for federated identity. It uses XML-based messages (like SAML) over SOAP. Key differences: (1) WS-Federation supports multiple token formats (SAML, Kerberos, X.509) while SAML and OIDC use their own format. (2) WS-Federation uses a passive requestor profile (HTTP redirect) similar to SAML WebSSO. (3) WS-Federation is tightly integrated with Windows/.NET and Active Directory Federation Services (ADFS). (4) It's less commonly used outside Microsoft ecosystems, while SAML and OIDC are more widely adopted. WS-Federation is considered legacy and is increasingly replaced by OIDC.

## Q47: How do you implement cross-domain single sign-on (SSO) without a centralized IdP?
**A:** Distributed SSO patterns: (1) Shared cookie domain — set a cookie on a shared parent domain (e.g., `.example.com`) that all subdomains read. Only works if all apps share a parent domain. (2) Iframe + postMessage — the main domain embeds an iframe from the auth domain, uses postMessage for session propagation. (3) Web message relay — the user is redirected through each domain, carrying a token. (4) OIDC federation — each domain is an independent RP that trusts the same OP. (5) SAML federation — each domain is an SP with trust relationships. For full cross-domain SSO without a central IdP, protocols like OIDC Federation or SAML are required.

## Q48: How does LinkedIn's OAuth 2.0 flow differ from Google's for social login?
**A:** While both use OAuth 2.0 + OIDC, there are implementation differences: (1) Google uses the `/auth` and `/token` endpoints with PKCE support. Google returns an ID token (JWT) alongside the access token. (2) LinkedIn's v2 API uses OAuth 2.0 with specific scopes like `openid`, `profile`, `email`. LinkedIn requires the `openid` scope to return an ID token. (3) Google uses RS256 signed JWTs with a JWKS endpoint for validation. LinkedIn similarly uses RS256 with a JWKS endpoint but the `sub` claim format differs. (4) LinkedIn's profile data requires specific API calls using the access token; Google includes claims in the ID token. Both follow OIDC but with different claim sets and API shapes.

## Q49: How do you implement authentication for a Kafka messaging system?
**A:** Kafka authentication supports: (1) SASL/PLAIN — username/password, simple but sends credentials in clear. (2) SASL/SCRAM — salted challenge-response, more secure than PLAIN. (3) SASL/GSSAPI (Kerberos) — enterprise-grade for integration with Active Directory. (4) SASL/OAUTHBEARER — OAuth 2.0 token-based authentication, allows using existing IdP tokens. (5) mTLS — mutual TLS with client certificates. Best practice: use SASL/SSL with SCRAM for password-based or mTLS for certificate-based. OAuthBearer is ideal when Kafka is part of a microservice ecosystem already using OAuth 2.0.

## Q50: How do you implement authentication for WebSocket connections?
**A:** WebSocket authentication methods: (1) Token in the connection URL query string — `ws://example.com?token=jwt` — simple but tokens may be logged in server access logs. (2) Token in the WebSocket upgrade request headers — the `Sec-WebSocket-Protocol` header can carry a subprotocol that includes the token. (3) Authenticate via REST first, then use a session cookie that's automatically sent during the WebSocket upgrade handshake. (4) Post-connection authentication — establish the WebSocket, then send a `{"type": "auth", "token": "..."}` message, and have the server validate before allowing data messages. Best practice: use cookies (with SameSite=Strict) for web apps, or the Authorization header if the client supports setting headers.

## Q51: How does the OAuth 2.0 Authorization Server handle cross-origin requests for CORS?
**A:** The authorization server must return appropriate CORS headers for its endpoints that are called from browser-based clients: (1) The `token_endpoint` and `par_endpoint` need proper CORS configuration. (2) The `authorization_endpoint` does NOT need CORS — it's a redirect, not an XHR. (3) The `jwks_uri` needs `Access-Control-Allow-Origin: *` for SPA JWKS fetching. (4) The `userinfo_endpoint` needs CORS for front-channel access. (5) The `revocation_endpoint` and `introspection_endpoint` typically don't need CORS (back-channel calls from servers). CORS should be scoped to the minimum set of allowed origins needed by client applications.

## Q52: How do you implement authentication for mobile apps using biometric integration?
**A:** Mobile biometric authentication flows: (1) Initial authentication uses standard username/password or social login. (2) On success, the server returns a long-lived refresh token and a short-lived access token. (3) The refresh token is stored in the device's secure enclave/keystore (iOS Keychain, Android EncryptedSharedPreferences). (4) For subsequent launches, the app authenticates using biometrics (Face ID, fingerprint) to unlock the keychain. (5) The unlocked refresh token is used to obtain new access tokens. (6) Biometric authentication alone was never done — it unlocks a stored credential. (7) The refresh token on the server should be bound to a device ID for additional security.

## Q53: What is the difference between authentication request throttling and rate limiting?
**A:** Throttling gradually increases delays based on failed attempts — first failure: 1 second delay, second: 5 seconds, third: 30 seconds, etc. Rate limiting enforces hard counts — maximum N attempts per time window. In authentication: throttle on the same account (prevent brute force) and rate-limit on the same IP (prevent distributed attacks). Combine both: rate limiting at the perimeter (CDN/WAF level) and throttling at the application level. Throttling is preferred over lockout because it doesn't enable denial-of-service attacks against legitimate users.

## Q54: How do you implement authentication for a headless CMS or API-first architecture?
**A:** Headless CMS authentication strategies: (1) API keys for server-to-server communication (backend SDK usage). (2) JWT-based authentication for client requests (SPA, mobile app). (3) OAuth 2.0 with Client Credentials for service accounts. (4) Preview/draft access uses a separate authentication context (authors authenticate to view unpublished content). (5) Webhook authentication uses HMAC signatures (verify that webhook requests come from the CMS). (6) Fine-grained PAT (Personal Access Token) scoping for developer tools. (7) Content delivery CDN endpoints typically don't require authentication — only the management API and preview endpoints do.

## Q55: How do you implement authentication for real-time collaborative applications (like Google Docs)?
**A:** Collaborative app authentication: (1) Primary authentication via OIDC — user gets a token on the page. (2) WebSocket connection upgrade includes the token (via subprotocol header or cookie). (3) Each collaboration operation sent through the WebSocket includes the token or session identifier. (4) Token validation happens at connection time and periodically during the session. (5) When the token expires, the server closes the WebSocket, and the client re-authenticates and reconnects. (6) For offline-tolerant collaboration, operations are queued locally and submitted when reconnected. (7) The collaboration server evaluates authorization for each document the user accesses.

## Q56: How does the NIST AAL3 requirement for verifier-impersonation resistance work?
**A:** Verifier impersonation (phishing) resistance means the authentication protocol prevents a fake verifier from capturing authentication secrets. At AAL3, this requires: (1) The authenticator cryptographically proves its identity to a specific verifier (e.g., FIDO2's `rpId` origin binding). (2) The authentication protocol operates over a protected channel (TLS) with the verifier's identity validated. (3) The authenticator is a hardware device that prevents cloning. FIDO2/WebAuthn is the primary protocol meeting AAL3 requirements because the private key never leaves the authenticator and the signature is bound to the website origin.

## Q57: How do you implement a secure session regeneration strategy?
**A:** Session regeneration creates a new session ID (or token) at specific security boundaries: (1) After login — prevent session fixation. (2) After privilege escalation (e.g., user becomes admin). (3) After password change — invalidate all sessions except current. (4) On session timeout extension. (5) After sensitive operations (high-value transactions). Implementation: (1) Generate a cryptographically random session ID. (2) Copy session data from old session to new session. (3) Set the new session cookie. (4) Invalidate the old session. (5) Store old->new session mapping for a transition period to handle concurrent requests.

## Q58: How do you implement authentication and session management in a microservices architecture with an API gateway?
**A:** API gateway authentication pattern: (1) The gateway validates the user's access token on every request. (2) The gateway extracts user identity and claims from the token and forwards them as HTTP headers (`X-User-Id`, `X-User-Roles`, `X-User-Org`) to downstream services. (3) Downstream services trust the gateway's headers (enforced by network policy — services should only accept requests from the gateway). (4) The gateway handles token refresh transparently (or the client does). (5) For WebSocket, the gateway validates the session on upgrade. (6) For internal service-to-service calls, use mTLS or short-lived service tokens.

## Q59: How does the OAuth 2.0 Authorization Server use a JWKS endpoint and how should it be secured?
**A:** The JWKS (JSON Web Key Set) endpoint publishes the authorization server's public keys for token signature verification. It's typically at `/.well-known/jwks.json`. Security considerations: (1) Only include public keys — never expose private keys. (2) Rotate keys periodically and include both old and new keys during transition. (3) Use `kid` (key ID) headers in JWTs to specify which key was used. (4) Cache the JWKS response on the client side — avoid fetching on every request. (5) Serve JWKS over HTTPS with proper caching headers (`Cache-Control: max-age=3600`). (6) Protect against JWKS injection — the client should validate that the issuer matches the expected authorization server.

## Q60: How do you implement an authentication state machine with proper state management?
**A:** An authentication state machine defines transitions between states: `UNAUTHENTICATED -> AUTHENTICATING -> AUTHENTICATED` or `AUTHENTICATED -> STEP_UP_REQUIRED -> STEP_UP_AUTHENTICATING -> AUTHENTICATED`. Key states: (1) `ANONYMOUS` — no identity. (2) `IDENTIFIED` — user identified but not authenticated. (3) `PARTIALLY_AUTHENTICATED` — some factors verified (e.g., password OK, MFA pending). (4) `AUTHENTICATED` — all required factors verified. (5) `ELEVATED` — step-up completed for sensitive operations. (6) `LOCKED` — account temporarily locked. (7) `EXPIRED` — session expired. State transitions trigger specific actions (session regeneration, event emission, notification).

## Q61: How do you implement authentication for a command-line interface (CLI) tool?
**A:** CLI authentication patterns: (1) Device Authorization Grant — the CLI displays a code and URL, the user authenticates in a browser, the CLI polls for the token. This is the best UX for CLI tools. (2) API key authentication — the user generates a personal access token from the web UI and configures it in the CLI. (3) OAuth 2.0 with a local web server — the CLI starts a local HTTP server on a random port and opens the browser for authorization; the authorization server redirects to `http://localhost:port/callback`. (4) Password prompts — `read -s` for interactive input. Best practice: Device Code flow for most CLIs, PAT for CI/CD automation.

## Q62: How does the OAuth 2.0 Authorization Code + PKCE flow prevent the authorization code interception attack?
**A:** In the code interception attack, a malicious app registered with a custom scheme captures the authorization code from a legitimate app's redirect. PKCE prevents this because: (1) The legitimate app generates a `code_verifier` (random high-entropy string) and sends a hashed `code_challenge` via `S256` during the authorization request. (2) The attacker's intercepted code is useless without the original `code_verifier`. (3) When exchanging the code, the app sends the original `code_verifier`. The authorization server hashes it and compares with the stored `code_challenge`. Only the app that originated the request knows the verifier. PKCE is now mandatory for all OAuth public clients per OAuth 2.1.

## Q63: How do you implement a secure one-time password (OTP) delivery system?
**A:** OTP delivery best practices: (1) Generate OTP using cryptographically secure random (not `rand()`). Use at least 6 digits for security vs. usability balance. (2) Set a short TTL (60-120 seconds). (3) Implement rate limiting — max 3-5 OTP requests per phone/email per hour. (4) Invalidate OTP after successful verification or timeout. (5) Never log OTP values. (6) For SMS, use a reputable provider with proper SS7 protection and anti-SIM-swap detection. (7) For email, the OTP should be in a primary delivery channel (not promotions folder). (8) TOTP is preferred over delivery OTP for higher security needs.

## Q64: How do you implement authentication audit logging for compliance?
**A:** Authentication audit logs should capture: (1) Event type: LOGIN_SUCCESS, LOGIN_FAILURE, LOGOUT, PASSWORD_CHANGE, MFA_ENROLL, TOKEN_REFRESH, SESSION_EXPIRY, ACCOUNT_LOCKOUT, PRIVILEGE_ESCALATION. (2) Subject: user ID, IP address, user agent, device fingerprint. (3) Context: timestamp (UTC), location (geoIP), session ID. (4) Result: success/failure with reason code. (5) Actor: who performed the action (user or admin impersonating). Logs should be: (1) Tamper-evident — use append-only storage or cryptographic chaining. (2) Forwarded to a SIEM (Splunk, ELK, Datadog). (3) Retained per compliance requirements (SOC2: 1 year, HIPAA: 6 years, PCI DSS: 1 year). (4) Never include secrets, tokens, or passwords in logs.

## Q65: How do you implement a secure "remember me" (persistent login) feature?
**A:** Persistent login implementation: (1) On "remember me" check, generate a high-entropy random token. (2) Store a hash of the token in the database along with user ID, device identifier, and expiration (30 days typical). (3) Set a long-lived HTTP-only secure SameSite cookie with the token. (4) On return visit, the application matches the cookie token against the stored hash. (5) After successful auto-login, issue a new session and delete the used persistent token (token rotation). (6) Never use a static cookie — it must be a one-time-use rotating token. (7) Allow users to view and revoke remembered devices from their security settings.

## Q66: How does the OAuth 2.0 framework handle token expiration for long-running processes?
**A:** For long-running processes (batch jobs, data syncs): (1) Use the Client Credentials grant for service accounts — tokens are scoped to the service, not a user. (2) Implement token caching — the process obtains a token before starting and refreshes it proactively before expiry. (3) For user-delegated long-running tasks, use a refresh token with rotation. (4) Consider offline access scope — request `offline_access` to get a refresh token. (5) For very long processes (hours), implement checkpoint/restart — if the token expires and refresh fails, the process can resume from the last checkpoint after re-authentication. (6) Use service accounts instead of user tokens for unattended processes.

## Q67: How do you implement authentication for multi-tenant SaaS applications?
**A:** Multi-tenant authentication considerations: (1) Identify the tenant during the authentication request (e.g., via subdomain `tenant.example.com` or a `tenant` parameter). (2) Route the request to the tenant's identity provider or tenant-specific IdP configuration. (3) The access token should include the tenant ID as a claim (e.g., `tid` or `org_id`). (4) Each tenant can have its own authentication policies (password policy, MFA requirements, session timeout). (5) Support tenant-specific SSO (each tenant can bring their own IdP via SAML/OIDC). (6) Tenant isolation — ensure tenant A's users cannot authenticate against tenant B's context. (7) Consider user roaming — a user may exist in multiple tenants with separate identities/roles.

## Q68: How do you implement authentication for IoT devices with limited connectivity?
**A:** IoT authentication for constrained devices: (1) Pre-provisioned device certificates (X.509) during manufacturing — the device identity is embedded at the factory. (2) Raw public key (RPK) for devices that can't handle full X.509 certificates. (3) Pre-shared keys (PSK) for the simplest devices. (4) OAuth 2.0 Device Authorization Grant for devices with some UI capability. (5) Token-based authentication where the device receives a token from a bootstrap server. (6) Offline authentication — the device can authenticate cached tokens (with limited validity) when the network is unavailable. (7) Certificate rotation — devices should support receiving new certificates before the current one expires. (8) The protocol should be lightweight — CoAP with DTLS or MQTT with TLS.

## Q69: How do you implement Zero Trust Authentication (never trust, always verify)?
**A:** Zero Trust authentication principles: (1) Authenticate every request — not just at the perimeter. (2) Use device posture checks — the device must be managed, patched, and not jailbroken. (3) Implement location-aware authentication — access from non-corporate locations triggers step-up. (4) Use real-time risk scoring — combine user behavior analytics (UBA), device health, location, and request pattern to compute a risk score. (5) Continuous evaluation — re-evaluate authentication on every request, not just at session start. (6) Session token scope is per-request, not per-session. (7) Use micro-segmentation — each resource has its own authentication and authorization policy. BeyondCorp and ZTNA architectures implement these principles.

## Q70: How does the OAuth 2.0 Authorization Server handle audience restriction for tokens?
**A:** Audience restriction (via the `aud` claim in JWTs) limits which resource servers can accept a token. Implementation options: (1) The client specifies the `audience` parameter in the authorization request. (2) The authorization server validates that the client is allowed to request that audience. (3) The issued access token includes the `aud` claim set to the requested resource server's identifier (typically its URL). (4) The resource server validates that the token's `aud` matches its own identifier. (5) For multi-audience tokens (token exchange), the `aud` can be an array. Audience restriction prevents token misuse — a token issued for the payments API cannot be used against the users API.

## Q71: How do you implement secure authentication for a mobile app that must work offline?
**A:** Offline mobile authentication: (1) During the initial online authentication, the server issues a long-lived refresh token stored in the device's secure keystore. (2) The refresh token has an extended validity (7-30 days) specifically for offline use. (3) The device caches an access token with a short expiry (1 hour). (4) Offline, the app uses the cached access token. (5) When the cached token expires offline, the device uses biometric authentication to unlock the refresh token, generates a new access token locally (if the device can verify the refresh token's validity without server), or waits until connectivity is restored. (6) On reconnecting, the app authenticates with the server, syncs changes, and obtains fresh tokens.

## Q72: How does the OAuth 2.0 Resource Server validate tokens without contacting the Authorization Server?
**A:** A resource server can validate JWTs locally by: (1) Fetching the JWKS from the authorization server's `jwks_uri` endpoint. (2) Caching the JWKS with appropriate TTL. (3) Validating the JWT signature using the key identified by the `kid` header. (4) Checking `exp`, `nbf`, and `iat` claims. (5) Validating that `iss` matches the expected authorization server. (6) Validating that `aud` matches the resource server's identifier. (7) For additional security, check the `azp` (authorized party) claim to ensure the token was issued to the expected client. Local validation is faster and enables offline operation, but cannot detect immediate revocation.

## Q73: How do you implement an OTP-based passwordless authentication flow?
**A:** Passwordless OTP flow: (1) User enters their email or phone number. (2) Server generates a cryptographically random OTP (6-8 digits) with a short TTL (2-5 minutes). (3) OTP is sent via email/SMS. (4) Server rate-limits OTP generation per user/phone (max 3 attempts per minute, 10 per hour). (5) User enters OTP on the web page. (6) Server validates OTP and checks expiration. (7) On success, server creates a session and optionally issues tokens. (8) OTP is invalidated after successful verification (one-time use). (9) Consider magic links as an alternative — when clicked, the link validates a token and logs the user in directly. Passwordless OTP is vulnerable to SIM swap and phishing — it's a convenience upgrade, not a security upgrade.

## Q74: How do you implement authentication in a Hexagonal (ports and adapters) architecture?
**A:** In hexagonal architecture, authentication is an external concern handled by adapters: (1) Define an `AuthenticationPort` interface that provides identity extraction, token validation, and user resolution. (2) Implement adapters for different authentication mechanisms: JWT adapter, session adapter, API key adapter, SAML adapter. (3) The core domain never handles tokens directly — it receives domain objects like `AuthenticatedUser` with identity and roles. (4) The driving adapter (HTTP controller, gRPC interceptor, message queue handler) validates authentication and injects the authenticated user into the domain context. (5) The core domain calls `AuthenticationPort` only for step-up verification — not for basic identity. This keeps domain logic authentication-mechanism-agnostic.

## Q75: How does the OAuth 2.0 framework handle consent for delegated authorization?
**A:** OAuth 2.0 consent is the user's approval for a client to access scoped resources. Implementation: (1) The authorization server presents a consent screen listing the requesting client (name, icon, publisher) and requested scopes. (2) Scopes are described in user-friendly terms ("View your email address", "Post on your behalf"). (3) The user can approve, reject, or (ideally) customize which scopes to grant. (4) Consent can be "persistent" — once approved, the user isn't prompted again for the same client+scopes combination. (5) OAuth 2.1 recommends periodic re-consent. (6) The authorization server records the consent grant and issues tokens accordingly. (7) Users should be able to review and revoke consents from a settings page.

## Q76: How do you implement authentication with behavioral biometrics?
**A:** Behavioral biometrics analyzes patterns in user behavior for continuous authentication: (1) Keystroke dynamics — timing between key presses, hold time, typing speed, error patterns. (2) Mouse dynamics — movement trajectories, click patterns, scroll behavior. (3) Touch dynamics — swipe patterns, pressure, gesture paths on mobile. (4) Gait analysis — walking patterns (wearables). (5) Behavioral data is collected passively in the background. (6) A machine learning model creates a behavioral profile for each user. (7) Real-time scoring compares current behavior against the profile. (8) Significant deviations trigger step-up authentication or session termination. Behavioral biometrics detects session hijacking and impersonation that traditional authentication cannot.

## Q77: How do you implement authentication for a B2B SaaS with tenant-specific IdP configuration?
**B2B SaaS authentication:** (1) Each tenant (customer organization) can configure their own identity provider — SAML 2.0 (Okta, Azure AD, OneLogin) or OIDC (Google Workspace). (2) The SaaS application functions as the SP/RP, accepting authentication assertions from the tenant's IdP. (3) Tenant administrators configure IdP metadata (SAML metadata URL, certificate, ACS URL) through a settings portal. (4) During login, the application identifies the tenant (via email domain or subdomain) and initiates the appropriate SAML/OIDC flow. (5) Just-in-time (JIT) provisioning creates user accounts on first login. (6) SCIM integration synchronizes user lifecycle from the customer's IdP to the SaaS app. The application must support multiple simultaneous IdP configurations.

## Q78: How does the OAuth 2.0 Authorization Server prevent CSRF attacks on the authorization endpoint?
**A:** The state parameter is OAuth 2.0's primary CSRF protection. Implementation: (1) The client generates a unique, unguessable, session-linked value for the `state` parameter. (2) The `state` is included in the authorization request to the authorization server. (3) When the user is redirected back to the client, the `state` value is returned unchanged. (4) The client must validate that the returned `state` matches the one it generated and is tied to the current session. (5) Without state validation, an attacker could initiate an authorization flow and trick the user into completing it, with the attacker intercepting the resulting authorization code. The state parameter also serves as CSRF protection for the redirect itself.

## Q79: How do you implement authentication for a distributed system using JWT with public key cryptography?
**A:** JWT with asymmetric keys (RS256/ES256) in a distributed system: (1) One central authorization service holds the private key and issues tokens. (2) The public key is distributed to all resource services as a JWKS. (3) Each service validates tokens independently using the cached JWKS — no network call to the auth service per request. (4) The JWKS is periodically refreshed (every hour) or pushed when keys rotate. (5) The `kid` in the JWT header tells each service which key to use. (6) Services validate: signature, `exp`, `iss`, `aud`, `nbf`, and optionally `azp`. (7) For immediate revocation, combine with a short token TTL (5-15 minutes) and maintain a local blacklist synchronized via pub/sub. This architecture handles thousands of requests per second with low latency.

## Q80: What is the difference between a stateless and stateful authorization server?
**A:** A stateless authorization server produces self-contained JWTs with all claims embedded. Validation requires no database lookup — any service with the public key can validate. Benefits: infinitely scalable, no database dependency for token operations. Drawbacks: cannot revoke tokens until expiration. A stateful authorization server stores token metadata in a database (session store, Redis). Tokens may be opaque references. Benefits: immediate revocation, session management (view active sessions, terminate). Drawbacks: each request requires a database lookup (latency), shared state complexity. Many production systems use a hybrid approach: stateless JWTs for access tokens (short TTL) with stateful refresh tokens (long TTL, revocable).

## Q81: How do you implement an API key authentication system that's secure and auditable?
**A:** API key system design: (1) Generate API keys using cryptographically secure random (e.g., `sk_live_` + 32 bytes base64-encoded = 48 chars). (2) Never store the full key in the database — store a SHA-256 hash. (3) The full key is shown once during creation (like GitHub, Stripe). (4) Allow multiple keys per user/service for rotation without downtime. (5) Support key scopes — restrict what operations each key can perform. (6) Support key expiration or allow explicit revocation. (7) Audit log every API call with key ID, IP, action, timestamp. (8) Implement key rotation policies — send warning emails for keys older than 90 days. (9) For high security, require client IP allowlisting per key.

## Q82: How does the OAuth 2.0 framework support token have rotation vs. rotation with reuse detection?
**A:** Token rotation with reuse detection (the most secure approach): (1) Each time a refresh token is used, a new refresh token is returned and the old one is invalidated. (2) If a stolen, previously-used refresh token is presented, the authorization server detects the reuse and invalidates ALL refresh tokens for that client+user combination. (3) The user must re-authenticate completely. This limits the window of vulnerability for stolen refresh tokens to a single refresh cycle. Without reuse detection, stolen refresh tokens could be used until they expire. Rotation with reuse detection is recommended by OAuth 2.1 and is standard practice for high-security applications.

## Q83: How do you implement authentication for a WebSocket-based real-time API to handle token expiration?
**A:** WebSocket token expiration handling: (1) Initial token validation during the WebSocket upgrade handshake. (2) Set a shorter token lifetime for WebSocket connections. (3) The server monitors token expiration and sends a close frame with a specific reason code before the token expires. (4) The client reconnects with a fresh token. (5) Alternatively, the client sends a token refresh message over the WebSocket channel (e.g., `{"type": "refresh", "token": "new_jwt"}`). (6) The server validates the new token and updates the session association. (7) For long-lived WebSocket connections, use server-sent heartbeats that double as token validity checks. (8) If token validation fails mid-session, the server closes the connection; the client re-authenticates and reconnects.

## Q84: How do you implement authentication for edge computing (CDN-level)?
**A:** Edge authentication patterns: (1) Token validation at the CDN edge (CloudFront Lambda@Edge, Cloudflare Workers, Fastly Compute@Edge) — the edge validates JWTs before requests reach the origin. (2) Cookie-based authentication at the edge — edge validates encrypted session cookies and sets headers for the origin. (3) Token exchange at the edge — the edge exchanges a short-lived session token for an origin token. (4) Edge-side authentication uses cached JWKS for performance. (5) For private content, use signed URLs or signed cookies (CloudFront, Akamai). (6) Edge authentication reduces origin load by rejecting invalid requests at the network edge. (7) Edge workers have execution limits (CPU time, memory) — authentication logic must be lightweight and fast.

## Q85: How does the OAuth 2.0 for Browser-Based Applications BCP (Best Current Practice) improve SPA security?
**A:** OAuth 2.0 for Browser-Based Apps (BCP, RFC 9700) provides updated security guidance for SPAs: (1) Do NOT use the Implicit flow — it's deprecated. Use Authorization Code with PKCE. (2) Do NOT store tokens in browser storage (localStorage/sessionStorage) — they're accessible to JavaScript and vulnerable to XSS. (3) The BFF pattern is strongly recommended. (4) If tokens must be stored in the browser, use an encrypted session cookie with Secure, HttpOnly, SameSite attributes. (5) Do not include the client secret in SPA code (it's not secret in a public client). (6) Use the PAR endpoint to push authorization parameters server-side. (7) Implement refresh token rotation with reuse detection.

## Q86: How do you implement authentication with HMAC-based request signing for API security?
**A:** HMAC request signing: (1) The client and server share a secret key. (2) The client constructs a canonical request string using: HTTP method, URI path, query string, request body hash, timestamp, and unique nonce. (3) The client signs the canonical string with the shared secret using HMAC-SHA256. (4) The signature is sent in the `Authorization: HMAC <key-id>:<signature>` header. (5) The server recomputes the signature using the stored secret for the given `key-id`. (6) The server validates the timestamp (within 5-minute skew) and checks nonce uniqueness (to prevent replay). (7) This is the foundation for AWS Signature V4, Stripe's webhook signing, and many custom API security schemes. The request body is included in the signing, preventing tampering in transit.

## Q87: How do you implement authentication for gRPC services?
**A:** gRPC authentication methods: (1) SSL/TLS — mandatory in production, supports client certificate authentication (mTLS). (2) Token-based — interceptors on both client and server sides. (3) Client sends the token via metadata (gRPC's equivalent of HTTP headers): `metadata.append("authorization", "Bearer " + token)`. (4) Server interceptor extracts and validates the token, populates the context with user identity. (5) gRPC Interceptors work at both unary and streaming call levels. (6) For multi-service architectures, use token exchange for downstream calls — the client propagates the token (after validation) to downstream services. (7) gRPC + OAuth + mTLS provides defense in depth: transport security + credential authentication + token delegation.

## Q88: How do you implement authentication for server-side rendering (SSR) applications like Next.js or Nuxt?
**A:** SSR authentication patterns: (1) GetInitialProps/getServerSideProps — the server checks the session cookie/token on each page request before rendering. (2) The session/token is typically stored in an HTTP-only cookie set by the SSR server. (3) The SSR server validates the token, fetches authenticated data, and renders the page. (4) API calls from the client side use the same cookie (automatically sent with requests). (5) For Next.js API routes, middleware can validate the token before route handlers run. (6) The BFF pattern integrates naturally — the SSR server acts as the BFF. (7) For ISR (Incremental Static Regeneration) pages, authentication happens client-side (since pages are static). (8) Middleware-based authentication runs at the edge for Next.js, enabling geographic-aware authentication logic.

## Q89: How do you implement authentication for a cross-origin iframe widget?
**A:** Cross-origin iframe authentication: (1) The parent page communicates with the iframe via `postMessage`. (2) The iframe loads a session token from its own domain (where it has access to cookies). (3) The iframe authenticates with its backend using its own cookies/tokens. (4) On the parent side, the user may or may not be authenticated with the iframe provider. (5) If not, the iframe shows a login UI or redirects to the provider's IdP within the iframe (if third-party cookies are allowed) or in a popup window. (6) OAuth 2.0 implicit + postMessage pattern: the iframe provider uses OIDC to authenticate the user in a popup, then sends the ID token via postMessage to both the parent and iframe. (7) Third-party cookie deprecation makes iframe authentication challenging — redirect-based flows (top-level navigation) or popup flows are more reliable.

## Q90: How does the OAuth 2.0 Token Introspection Response caching work for resource servers?
**A:** Resource servers can cache token introspection results to avoid network calls on every request: (1) Cache keyed by token hash (or token value). (2) Cache `active: true` results only for the token's remaining lifetime (or shorter). (3) Cache `active: false` results for a shorter period (seconds) to avoid rapid rechecking of invalid tokens. (4) Use `Cache-Control` headers or TTL configuration. (5) Invalidate cache on token revocation events (if the resource server receives push notifications). (6) For high-security deployments, do not cache introspection results — always check with the authorization server. (7) For performance-critical deployments, prefer local JWT validation over introspection + caching.

## Q91: How do you implement authentication for a mobile app using the AppAuth pattern with PKCE?
**A:** AppAuth pattern: (1) The app opens the system browser (not embedded WebView) to the authorization server's endpoint. (2) The app generates PKCE `code_verifier` (128 chars of cryptographically random) and stores it locally. (3) The app sends the `code_challenge` (S256 hash of verifier) in the authorization request. (4) The user authenticates in the browser. (5) The authorization server redirects to the app's registered scheme (e.g., `myapp://callback?code=...`). (6) The app extracts the authorization code and exchanges it with the code_verifier for tokens. (7) Tokens are stored in the device's secure storage (iOS Keychain, Android EncryptedSharedPreferences). (8) The refresh token rotation ensures ongoing access. AppAuth libraries handle most of this automatically.

## Q92: How do you implement authentication for a headless browser automation (Cypress, Playwright)?
**A:** Authentication strategies for E2E testing: (1) API-based authentication — execute the login API call directly rather than through the UI, obtaining a token. (2) Set the token in the browser's storage before page load (Playwright's `page.addInitScript` or Cypress' `cy.setCookie`). (3) Use the app's internal authentication hooks — many frameworks allow bypassing the login UI for test environments. (4) For OAuth flows, mock the authorization server responses using route interception. (5) For SAML, mock the IdP response. (6) Use dedicated test users with known credentials. (7) Rotate test credentials regularly. (8) Ensure test authentication does not trigger security policies intended for real users (MFA, CAPTCHA, rate limiting).

## Q93: How do you implement authentication for a Progressive Web App (PWA) that works offline?
**A:** PWA offline authentication: (1) Service worker intercepts fetch requests. (2) If the request requires authentication and the user is offline, the service worker checks the Cache API for a cached token. (3) The token must support offline validation — a pre-verified JWT works (the signature was already validated when online). (4) The service worker can use `IndexedDB` to store the user's session token securely (with proper access controls). (5) Login occurs when online — the service worker stores the result. (6) Offline cache includes previously accessed authenticated resources. (7) When coming back online, the service worker revalidates the token and syncs any queued mutations. (8) Important: PWA authentication state must survive service worker updates without losing the session.

## Q94: How does the OAuth 2.0 Authorization Server handle dynamic client registration?
**A:** Dynamic Client Registration (RFC 7591) allows clients to register with the authorization server at runtime: (1) The client sends a registration request with its metadata (redirect URIs, token endpoint auth method, grant types, JWKS, software statement). (2) The server creates a client ID and optionally a client secret. (3) The server returns a `client_id`, optionally `client_secret`, `registration_access_token` (for future management), and `registration_client_uri`. (4) The client can update or delete its registration using the management endpoint. (5) Software statements (JWT signed by a trusted authority) provide proof of the client's legitimacy. (6) Dynamic registration enables OIDC federation and ecosystem-level interoperability without manual per-client configuration.

## Q95: How do you implement authentication for a headless CMS preview (draft content access)?
**A:** Preview authentication for headless CMS: (1) Preview tokens — short-lived tokens (15-60 minutes) that authorize access to unpublished content. (2) The content editor triggers "Preview" from the CMS admin. (3) The CMS invokes a webhook or callback with a signed preview token. (4) The frontend application validates the preview token and fetches content with a special header or query parameter indicating preview mode. (5) The content API checks the preview authorization and returns draft/pending content. (6) Preview tokens are single-use or strict time-limited. (7) For SSO-enabled preview, the CMS uses OIDC to authenticate the editor and passes the authenticated user's identity to the preview endpoint. (8) Preview URLs should be unpredictable and expire.

## Q96: How does the FIDO2 CTAP (Client to Authenticator Protocol) work for roaming authenticators?
**A:** CTAP defines the communication between a client platform (browser, OS) and a roaming authenticator (USB/NFC/BLE security key): (1) CTAP1 (U2F) — legacy protocol for simple registration and authentication. (2) CTAP2 — current protocol supporting WebAuthn, PIN, biometrics, and large blob storage. (3) CTAP2.1 — enhanced with credential management (list/delete credentials on the key), alwaysUv (user verification required), and enterprise attestation. (4) The authenticator has its own processing unit, secure storage for private keys, and optionally a biometric sensor. (5) Communication uses CBOR-encoded messages over HID, NFC, or BLE. (6) CTAP protects against: physical tampering (secure element), malware (cryptographic confirmation of user presence), and phishing (origin binding).

## Q97: How do you implement authentication with a custom Identity Provider built on OIDC?
**A:** Building a custom OIDC IdP: (1) Implement mandatory OIDC endpoints: `/.well-known/openid-configuration`, authorization endpoint, token endpoint, JWKS endpoint, userinfo endpoint. (2) Support response types `code` and `code id_token`. (3) Support authorization code flow with PKCE. (4) Implement authentication methods: at minimum a login form, optionally MFA, WebAuthn, social login. (5) Issue ID tokens as signed JWTs with required claims (`iss`, `sub`, `aud`, `exp`, `iat`) and optional claims (`email`, `name`, `email_verified`). (6) Issue access tokens (opaque or JWT) with appropriate scopes and expiration. (7) Support refresh token grant with rotation. (8) Implement client management — register and manage client applications with their redirect URIs and allowed scopes. (9) Consider using a proven library (Ory Hydra, Keycloak, Dex) rather than building from scratch.

## Q98: How do you implement authentication using Passkeys with conditional UI?
**A:** Passkeys with conditional UI: (1) The browser displays autofill suggestions for passkeys when the user focuses on the username field. (2) The website calls `navigator.credentials.get({ mediation: 'conditional', publicKey: ... })` on page load (not in response to user action). (3) The browser shows passkey suggestions inline in the autofill dropdown. (4) When the user selects a passkey, the browser performs the WebAuthn assertion. (5) The server validates the assertion and logs the user in without requiring them to enter a password. (6) Conditional UI improves the user experience — logging in is as simple as selecting a username from the autofill and authenticating with biometrics. (7) This requires the `mediation: 'conditional'` option and must be triggered on page load, not by a button click.

## Q99: How does the OAuth 2.0 Authorization Server handle refresh token granularity and scope reduction?
**A:** Refresh token scope management: (1) When a client obtains a refresh token with certain scopes, subsequent refresh token exchanges typically return an access token with the same scopes (scope preservation). (2) The authorization server may support scope reduction — the refresh grant request can include a reduced `scope` parameter, producing an access token with narrower permissions than the original. (3) Scope reduction is useful for: delegating to a child process with limited permissions, switching from admin to user context. (4) Scope expansion (getting more permissions than originally granted) must not be allowed via refresh — the user must re-consent. (5) The authorization server should record the granted scopes with the refresh token and validate that any reduced-scope request is a subset.

## Q100: How do you implement authentication for a multi-protocol API gateway supporting REST, gRPC, and GraphQL?
**A:** Multi-protocol authentication at the API gateway: (1) The gateway validates the authentication token regardless of protocol. (2) For REST — standard `Authorization: Bearer` header validation. (3) For gRPC — metadata-based token extraction and validation via interceptor. (4) For GraphQL — the token is extracted from the Authorization header (same as REST) before the GraphQL resolver chain. (5) The gateway extracts user identity and authorization context from the validated token. (6) For HTTP, these are set as request headers passed to upstream services. (7) For gRPC, the context carries the authenticated identity via `grpc-meta` headers or the gRPC context object. (8) For GraphQL, the authenticated user is set in the GraphQL context for resolver-level authorization. (9) The gateway handles token validation (signature, expiration, issuer, audience) centrally, and upstream services can skip re-validation if they trust the gateway's headers (via network policy).
