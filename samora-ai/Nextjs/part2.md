# Next.js Interview Questions and Answers - Part 2

## Q1: How does Next.js handle routing when both Pages Router and App Router are present?
**A:** Next.js supports incremental adoption — the `app/` directory and `pages/` directory can coexist. Routes in `app/` take precedence for the same path. Middleware applies to both directories. API routes can exist in both `pages/api/` and route handlers in `app/api/`, but path conflicts should be avoided. This coexistence is intended only as a transitional pattern during migration, not for long-term production use.

## Q2: How do you implement custom error formatting in Next.js API routes (Pages Router)?
**A:** Wrap API handlers with a try-catch boundary: `export default async function handler(req, res) { try { await reqHandler(req, res) } catch (err) { console.error(err); res.status(err.statusCode || 500).json({ error: err.message || 'Internal Server Error', code: err.code || 'INTERNAL_ERROR', ...(process.env.NODE_ENV === 'development' && { stack: err.stack }) }) } }`. For reusable error handling, create a higher-order function: `function withErrorHandler(handler) { return async (req, res) => { try { await handler(req, res) } catch (err) { res.status(500).json({ error: 'Internal server error' }) } } }`.

## Q3: How do you implement dynamic OG image generation with parameters in Next.js?
**A:** Use the `@vercel/og` library (ImageResponse) with Edge Runtime: `export const runtime = 'edge'; export async function GET(req) { const { searchParams } = new URL(req.url); const title = searchParams.get('title') || 'Default'; return new ImageResponse(<div style={{ display: 'flex', fontSize: 60, background: 'linear-gradient(to right, #ff7e5f, #feb47b)', width: '100%', height: '100%', padding: 50 }}>{title}</div>, { width: 1200, height: 630 }) }`. Reference in metadata: `openGraph: { images: [{ url: `/api/og?title=${encodeURIComponent(post.title)}` }] }`. Edge Runtime generates images quickly at the network edge.

## Q4: How do you implement authentication with middleware-based route protection in Next.js App Router?
**A:** Create `middleware.ts` at the project root: `export function middleware(request) { const token = request.cookies.get('session-token')?.value; const isAuthPage = request.nextUrl.pathname.startsWith('/login'); if (!token && !isAuthPage) return NextResponse.redirect(new URL('/login', request.url)); if (token && isAuthPage) return NextResponse.redirect(new URL('/dashboard', request.url)); return NextResponse.next() }`. Configure matcher: `export const config = { matcher: ['/dashboard/:path*', '/profile/:path*', '/login'] }`. This checks authentication before the request reaches the page, preventing flash of unauthenticated content.

## Q5: How do you implement role-based access control (RBAC) in Next.js App Router with Server Components?
**A:** Create an authorization function: `async function authorize(requiredRole) { const session = await getServerSession(); if (!session) redirect('/login'); if (session.user.role !== requiredRole && session.user.role !== 'admin') throw new Error('Forbidden'); return session }`. Use in Server Components: `export default async function AdminPage() { const session = await authorize('admin'); const data = await db.query('SELECT * FROM sensitive_table'); return <AdminTable data={data} /> }`. For client-side UI enforcement, check role in Client Components but never rely on client-only checks for security.

## Q6: How do you implement incremental migration from Pages Router to App Router?
**A:** Steps: 1) Create `app/` directory alongside `pages/`. 2) Keep shared components in a common location. 3) Migrate leaf pages first. 4) Convert `getStaticProps`/`getServerSideProps` into async Server Component data fetching. 5) Move layouts from `_app.js` to `app/layout.js`. 6) Move `_document.js` content into the root layout's `<head>` and `<body>`. 7) Migrate `pages/api/` routes to `app/api/` route handlers. Test each migration step in isolation. Both routers can coexist during this transition.

## Q7: How do you implement A/B testing in Next.js with middleware?
**A:** Assign variants via middleware: `export function middleware(req) { const variant = req.cookies.get('ab-test')?.value || (Math.random() < 0.5 ? 'A' : 'B'); const response = NextResponse.next(); if (!req.cookies.get('ab-test')) response.cookies.set('ab-test', variant, { maxAge: 86400 }); if (req.nextUrl.pathname === '/' && variant === 'B') { const url = req.nextUrl.clone(); url.pathname = '/variant-b'; return NextResponse.rewrite(url) } return response }`. Variant B rewrites `/` to `/variant-b` without changing the URL. This ensures consistent user experience and avoids SEO duplication.

## Q9: How do you implement Next.js with Redis for session storage?
**A:** For custom sessions, store session data in Redis and validate via middleware: `export async function middleware(req) { const sessionId = req.cookies.get('session-id')?.value; if (sessionId) { const session = await redis.get(`session:${sessionId}`); if (session) { const parsed = JSON.parse(session); const headers = new Headers(req.headers); headers.set('x-user-id', parsed.userId); return NextResponse.next({ request: { headers } }) } } return NextResponse.next() }`. In Server Components, read the headers: `const userId = headers().get('x-user-id')`. Use `ioredis` or `@upstash/redis` for Redis connections.

## Q10: How do you implement WebSocket connections in Next.js?
**A:** Next.js route handlers don't natively support WebSocket upgrade. Use a custom server: `const { createServer } = require('http'); const next = require('next'); const { WebSocketServer } = require('ws'); const dev = process.env.NODE_ENV !== 'production'; const app = next({ dev }); const handle = app.getRequestHandler(); app.prepare().then(() => { const server = createServer(handle); const wss = new WebSocketServer({ server }); wss.on('connection', (ws) => { ws.on('message', (msg) => { wss.clients.forEach(client => { if (client.readyState === 1) client.send(msg) }) }) }); server.listen(3000) })`. Deploy to platforms that support custom servers (not Vercel serverless).

## Q11: How do you implement optimistic updates in Next.js with Server Actions?
**A:** Use `useOptimistic` hook from React: `'use client'; import { useOptimistic } from 'react'; export function Messages({ messages, sendMessage }) { const [optimisticMessages, addOptimisticMessage] = useOptimistic(messages, (state, msg) => [...state, { text: msg, sending: true }]); async function formAction(formData) { addOptimisticMessage(formData.get('text')); await sendMessage(formData); } return (<form action={formAction}> <input name="text" /> <ul>{optimisticMessages.map((m, i) => <li key={i}>{m.text}{m.sending && '...'}</li>)}</ul> </form>) }`. The UI updates immediately while the Server Action runs. On completion, revalidate to get the authoritative server state.

## Q13: How do you implement Next.js middleware that runs only for specific HTTP methods?
**A:** Check `request.method` in middleware: `export function middleware(request) { if (request.method === 'POST') { const ct = request.headers.get('content-type'); if (!ct?.includes('application/json')) return new NextResponse('Content-Type must be application/json', { status: 415 }) } if (request.method === 'GET' && request.nextUrl.pathname.startsWith('/api')) { if (!request.nextUrl.searchParams.has('api_key')) return NextResponse.json({ error: 'API key required' }, { status: 401 }) } return NextResponse.next() }`. The `method` property is available in Edge Middleware. Note middleware cannot read the request body.

## Q15: How do you implement Next.js middleware for cookie-based theming?
**A:** Set and propagate theme preference: `export function middleware(request) { const theme = request.cookies.get('theme')?.value || 'light'; const response = NextResponse.next(); if (!request.cookies.get('theme')) response.cookies.set('theme', 'light', { maxAge: 31536000 }); const headers = new Headers(request.headers); headers.set('x-theme', theme); return NextResponse.next({ request: { headers } }) }`. In the root layout: `const theme = headers().get('x-theme') || 'light'`. Apply the theme class to `<html>`: `<html className={theme}>`. This eliminates CLS from theme switching.

## Q16: How do you implement next/script with different loading strategies in Next.js?
**A:** The Script component supports: `beforeInteractive` (critical — loaded before page hydration, use sparingly), `afterInteractive` (default — loaded after hydration for most third-party scripts), `lazyOnload` (loaded when browser is idle). Example: `import Script from 'next/script'; export default function Layout({ children }) { return (<>{children}<Script src="https://www.googletagmanager.com/gtag/js?id=GA_ID" strategy="afterInteractive" /><Script id="ga-init" strategy="lazyOnload">{`window.dataLayer=window.dataLayer||[];`}</Script></>) }`. The `worker` strategy (experimental) runs scripts in a Web Worker.

## Q20: How do you implement Partial Prerendering (PPR) in Next.js?
**A:** PPR combines static and dynamic rendering on the same page. Enable: `experimental: { ppr: true }` in `next.config.js`. Wrap dynamic content in `<Suspense>`: `export default function Page() { return (<div><StaticHeader /><Suspense fallback={<Skeleton />}><DynamicProducts /></Suspense></div>) }`. The static shell is served immediately from the CDN while dynamic content streams in. PPR provides the performance of static pages with the freshness of dynamic data.

## Q22: How do you implement file uploads with FormData in Next.js route handlers?

## Q23: How do you implement Next.js with React 19's use() hook in Server Components?

## Q100: What are the key differences between Next.js Pages Router and App Router rendering models?
**A:** The App Router uses React Server Components by default (components run on server, reducing client JS), while Pages Router components are Client Components by default. App Router data fetching uses async Server Components with `fetch` extensions; Pages Router uses `getStaticProps`/`getServerSideProps`. App Router has nested layouts, loading states, error boundaries, and parallel/intercepting routes built-in. App Router supports streaming SSR with Suspense. App Router uses route handlers (`route.ts`) instead of API routes (`pages/api/`). App Router metadata uses the Metadata API; Pages Router uses `next/head`. The App Router is the recommended approach for new projects.
