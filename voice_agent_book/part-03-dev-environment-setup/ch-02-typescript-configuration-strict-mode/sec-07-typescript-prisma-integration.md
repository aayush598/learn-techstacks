# Section 07: TypeScript & Prisma Integration

## Overview

Prisma's type generation is one of its most powerful features. The Prisma client generates TypeScript types directly from the schema, providing end-to-end type safety from the database to the API response. This section covers how to leverage Prisma-generated types, create type overrides, use input types, and maximize type inference with select/include patterns.

## Type Generation Architecture

```text
┌────────────────┐     ┌──────────────────┐     ┌────────────────┐
│  schema.prisma  │────►│  prisma generate  │────►│  Generated      │
│  (source of     │     │  (build step)     │     │  Types (node_   │
│   truth)        │     │                   │     │  modules/)      │
└────────────────┘     └──────────────────┘     └────────────────┘
                                                         │
                                                         ▼
                                               ┌────────────────┐
                                               │  @voice-agent/  │
                                               │  db re-exports  │
                                               │                 │
                                               │  - client.ts    │
                                               │  - types.ts     │
                                               │  - repos/*.ts   │
                                               └────────────────┘
```

## Generated Type Categories

Prisma generates several type categories for each model:

```typescript
// Generated from: model Agent { ... }
// These types are auto-generated by `prisma generate`

// 1. Model Type (the entity itself)
type Agent = {
  id: string;
  organizationId: string;
  name: string;
  description: string | null;
  voiceProvider: string;
  status: string;
  createdAt: Date;
  updatedAt: Date;
};

// 2. Create Input Type (for creating records)
type AgentCreateInput = {
  id?: string;
  organization: OrganizationCreateNestedOneWithoutAgentsInput;
  name: string;
  description?: string | null;
  voiceProvider?: string;
  status?: string;
  // ...
};

// 3. Update Input Type (for updating records)
type AgentUpdateInput = {
  name?: string;
  description?: string | null;
  voiceProvider?: string;
  status?: string;
  // ...
};

// 4. Where Input Type (for filtering)
type AgentWhereInput = {
  id?: StringFilter;
  organizationId?: StringFilter;
  name?: StringFilter;
  status?: StringFilter;
  // ...
};

// 5. Select Input Type (for selecting specific fields)
type AgentSelect = {
  id?: boolean;
  name?: boolean;
  status?: boolean;
  // ...
};

// 6. Include Input Type (for including relations)
type AgentInclude = {
  organization?: boolean | OrganizationArgs;
  calls?: boolean | CallArgs;
  campaigns?: boolean | CampaignArgs;
};
```

## Re-exporting Generated Types

```typescript
// packages/db/src/types.ts

// Re-export Prisma-generated types for domain use
import type {
  Agent as PrismaAgent,
  Call as PrismaCall,
  Campaign as PrismaCampaign,
  Contact as PrismaContact,
  Organization as PrismaOrganization,
} from "@prisma/client";

// Domain types (Prisma types with renames for clarity)
export type AgentRecord = PrismaAgent;
export type CallRecord = PrismaCall;
export type CampaignRecord = PrismaCampaign;
export type ContactRecord = PrismaContact;
export type OrganizationRecord = PrismaOrganization;

// Types with relations
import type {
  Prisma,
} from "@prisma/client";

export type AgentWithOrg = Prisma.AgentGetPayload<{
  include: { organization: true };
}>;

export type CallWithRelations = Prisma.CallGetPayload<{
  include: {
    agent: true;
    contact: true;
    campaign: true;
  };
}>;

// Create input types (Omit internal fields)
export type CreateAgentInput = Omit<
  Prisma.AgentCreateInput,
  "organization" | "calls" | "campaigns"
> & {
  organizationId: string;
};
```

## Type-Safe Select/Include Patterns

### The Select Pattern

Instead of fetching entire entities, use `select` to fetch only needed fields:

```typescript
// packages/db/src/repositories/agent.repo.ts
import { prisma } from "../client";
import type { Prisma } from "@prisma/client";

// Define reusable select objects
const agentSelect = {
  id: true,
  name: true,
  status: true,
  voiceProvider: true,
  createdAt: true,
} satisfies Prisma.AgentSelect;

// The return type is inferred: { id: string; name: string; status: string; ... }
type AgentSummary = Prisma.AgentGetPayload<{
  select: typeof agentSelect;
}>;

export async function getAgentSummaries(
  organizationId: string,
): Promise<AgentSummary[]> {
  return prisma.agent.findMany({
    where: { organizationId },
    select: agentSelect,
    orderBy: { createdAt: "desc" },
  });
}
```

### The Include Pattern

Use `include` for eager-loading related data:

```typescript
// Define reusable include objects
const agentWithOrgInclude = {
  organization: {
    select: {
      id: true,
      name: true,
      plan: true,
    },
  },
} satisfies Prisma.AgentInclude;

type AgentWithOrg = Prisma.AgentGetPayload<{
  include: typeof agentWithOrgInclude;
}>;

export async function getAgentWithOrg(
  id: string,
): Promise<AgentWithOrg | null> {
  return prisma.agent.findFirst({
    where: { id },
    include: agentWithOrgInclude,
  });
}
```

## Type Override Strategy

Sometimes Prisma-generated types need overrides for domain-specific concerns:

```typescript
// packages/db/src/overrides.ts

// Prisma generates voiceProvider as string, but we want a union type
import type { VoiceProvider } from "@voice-agent/types";

// Override the Agent type for domain use
declare module "@prisma/client" {
  interface Agent {
    // Override the string type with our union type
    voiceProvider: VoiceProvider;
    llmProvider: LLMProvider;
    status: AgentStatus;
  }

  // Override input types to accept our domain types
  interface AgentCreateInput {
    voiceProvider: VoiceProvider;
    llmProvider: LLMProvider;
    status: AgentStatus;
  }
}
```

## Input Type Validation with Zod

Even with generated types, runtime validation is essential:

```typescript
// packages/db/src/validation.ts
import { z } from "zod";
import type { Prisma } from "@prisma/client";

// Zod schema that mirrors the Prisma input type
export const CreateAgentSchema = z.object({
  name: z.string().min(1).max(100),
  description: z.string().max(1000).nullable().optional(),
  voiceProvider: z.enum(["elevenlabs", "cartesia", "deepgram"]),
  voiceId: z.string(),
  greetingMessage: z.string().max(500),
  maxCallDuration: z.number().int().positive().max(14400),
  temperature: z.number().min(0).max(1).default(0.7),
  llmProvider: z.enum(["openai", "anthropic"]),
  llmModel: z.string(),
  status: z.enum(["draft", "active", "paused", "archived"]).default("draft"),
});

export type CreateAgentSchema = z.infer<typeof CreateAgentSchema>;

// Transform validated data to Prisma input type
export function toPrismaCreateInput(
  data: CreateAgentSchema,
  organizationId: string,
): Prisma.AgentCreateInput {
  return {
    name: data.name,
    description: data.description ?? null,
    voiceProvider: data.voiceProvider,
    voiceId: data.voiceId,
    greetingMessage: data.greetingMessage,
    maxCallDuration: data.maxCallDuration,
    temperature: data.temperature,
    llmProvider: data.llmProvider,
    llmModel: data.llmModel,
    status: data.status,
    organization: { connect: { id: organizationId } },
  };
}
```

## Nullable Field Handling

Prisma uses `T | null` for nullable database columns. TypeScript strict mode requires explicit null handling:

```typescript
// packages/db/src/repositories/call.repo.ts

export async function getCallWithTranscript(
  callId: string,
): Promise<CallWithTranscript | null> {
  const call = await prisma.call.findUnique({
    where: { id: callId },
    include: {
      agent: { select: { name: true } },
    },
  });

  if (!call) return null;

  // Handle nullable fields explicitly
  return {
    ...call,
    duration: call.duration ?? 0, // Default for null
    recordingUrl: call.recordingUrl ?? null,
    // Transcript might not be ready yet
    transcript: null,
  };
}
```

## Transaction Types

Prisma supports interactive transactions with type-safe callbacks:

```typescript
// packages/db/src/repositories/call.repo.ts

export async function createCallWithRecording(
  data: CreateCallInput,
  recordingData: CreateRecordingInput,
): Promise<CallWithRecording> {
  return prisma.$transaction(async (tx) => {
    // tx has the same type as prisma but is scoped to the transaction
    const call = await tx.call.create({
      data: {
        ...data,
        status: "initiating",
      },
    });

    const recording = await tx.recording.create({
      data: {
        ...recordingData,
        callId: call.id,
      },
    });

    return { ...call, recording };
  });
}
```

## Batch Operations with Types

```typescript
// packages/db/src/repositories/contact.repo.ts

export async function bulkCreateContacts(
  contacts: CreateContactInput[],
): Promise<ContactRecord[]> {
  // TypeScript ensures each item matches Prisma.ContactCreateInput
  return prisma.contact.createMany({
    data: contacts,
    skipDuplicates: true,
  });
  // Note: createMany returns a count, not the records
  // Use createManyAndReturn in Prisma 5+ for returned records
}

// Prisma 5.12+ supports createManyAndReturn
export async function bulkCreateContactsWithReturn(
  contacts: CreateContactInput[],
): Promise<ContactRecord[]> {
  return prisma.contact.createManyAndReturn({
    data: contacts,
    skipDuplicates: true,
  });
}
```

## Type-Specific Queries

Enums in Prisma generate TypeScript enums or union types:

```typescript
// Generated by Prisma
enum CallStatus {
  initiating
  ringing
  in_progress
  transferring
  completed
  failed
  busy
  no_answer
}

// Usage with type safety
export async function getCallsByStatus(
  organizationId: string,
  status: CallStatus,
): Promise<CallRecord[]> {
  return prisma.call.findMany({
    where: { organizationId, status },
    orderBy: { createdAt: "desc" },
    take: 100,
  });
}
```

## Design Decisions

### Why not use Prisma types directly everywhere?

Prisma types are comprehensive but include database-specific fields (relations, internal counters) that shouldn't leak into the API layer. By re-exporting and wrapping in the `@voice-agent/db` package, we control what's exposed.

### Select/Include vs. Raw Queries

| Approach | Type Safety | Performance | Flexibility |
|----------|-------------|-------------|-------------|
| Prisma select/include | Full | Good (only fetches needed fields) | Limited to Prisma API |
| Raw SQL with typed results | Manual validation | Best | Full SQL power |

**Decision**: Use Prisma's typed API for 95% of queries. Reserve raw SQL for complex aggregation queries where Prisma's API is insufficient. When using raw SQL, validate results with Zod.

## Integration Points

- **Package exports**: `@voice-agent/db` exports Prisma types, input types, and repository classes
- **API layer**: Validates Zod schemas, transforms to Prisma input types, returns Prisma output types
- **Service layer**: Composes repository calls, maps Prisma types to domain types
- **Event system**: Prisma's `$use` middleware can emit domain events on database changes

## Production Considerations

1. **Client generation in production**: Run `prisma generate` during the build step, not at runtime. The generated client should be part of the build artifact
2. **Type synchronization**: Prisma types are only as current as the last `prisma generate`. Add a `postinstall` script or CI step to regenerate after schema changes
3. **Select/include reuse**: Define reusable select/include objects to avoid repetition. The `satisfies` operator ensures they match Prisma's expected types
4. **Migration safety**: Never change a column type without a corresponding schema change. Prisma's type generation will immediately surface type mismatches at compile time
5. **Performance monitoring**: Use Prisma middleware to log slow queries. The logged query duration can be typed for monitoring dashboards
