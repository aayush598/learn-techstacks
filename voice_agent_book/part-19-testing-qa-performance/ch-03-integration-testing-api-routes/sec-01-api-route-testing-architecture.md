# Section 01: API Route Testing Architecture

## Overview

API route testing validates the request-response lifecycle of the platform's HTTP endpoints, including authentication, authorization, input validation, business logic execution, and response formatting. For the voice AI platform, API routes handle agent management, call control, transcript retrieval, analytics queries, and billing operations. Tests use Supertest for HTTP assertions against a fully-initialized application with real database connections.

The test architecture creates a lightweight application instance with test-specific configuration, bypassing middleware that's irrelevant to the test (like rate limiting) while exercising security middleware (authentication, authorization). Each test is isolated with its own database transaction that rolls back after completion, ensuring clean state between tests.

## Design Decisions

- **In-Browser Simulator**: Full conversation simulation with WASM runtime. No backend needed.
- **Utterance Testing**: Pre-defined test utterances with assertions on path and response.
- **Flow Validation**: Graph analysis for unreachable nodes, infinite loops, missing required fields.
## Implementation Approach

```typescript
// API route test setup
import supertest from 'supertest';
import { createApp } from '@/app';
import { createTestDatabase } from '@test/helpers/database';
import { createAuthToken } from '@test/helpers/auth';

describe('Agent API Routes', () => {
  let app: Express.Application;
  let db: PrismaClient;
  let authToken: string;

  beforeAll(async () => {
    db = await createTestDatabase();
    app = await createApp({ database: db, testMode: true });
  });

  afterAll(async () => {
    await db.$disconnect();
  });

  beforeEach(() => {
    authToken = createAuthToken({ role: 'admin', orgId: 'test-org' });
  });

  describe('POST /api/agents', () => {
    it('creates agent with valid data', async () => {
      const response = await supertest(app)
        .post('/api/agents')
        .set('Authorization', `Bearer ${authToken}`)
        .send({
          name: 'Test Agent',
          language: 'en-US',
          voice: 'natural-female',
          config: { greeting: 'Hello!', temperature: 0.7 },
        })
        .expect(201);

      expect(response.body).toMatchObject({
        name: 'Test Agent',
        status: 'draft',
      });
      expect(response.body.id).toBeDefined();
    });

    it('returns 400 for missing required fields', async () => {
      await supertest(app)
        .post('/api/agents')
        .set('Authorization', `Bearer ${authToken}`)
        .send({})
        .expect(400);
    });

    it('returns 401 without auth token', async () => {
      await supertest(app)
        .post('/api/agents')
        .send({ name: 'Test Agent' })
        .expect(401);
    });
  });
});
```

## Integration Points

- **Database**: Tests use real database with migrations applied before each suite
- **Auth Service**: Authentication tokens generated by test helper matching production JWT format
- **External APIs**: External API calls (AI providers, Twilio) mocked by MSW
- **File Storage**: File upload endpoints tested with test storage provider
- **Rate Limiting**: Rate limiting disabled in test mode for deterministic behavior

## Production Considerations

- **Test Database Size**: Keep test database small; large datasets slow tests
- **Connection Pool**: Test database needs sufficient connection pool for parallel tests
- **Migration Speed**: Use migration squashing to speed up test database initialization
- **Auth Token Expiry**: Test auth tokens should not expire during test suite
- **Middleware Performance**: Disable non-essential middleware in test mode for speed
